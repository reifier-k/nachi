# RFC 003: バージョニング、互換性、非推奨化

> Language: [English](./003-versioning.md) / 日本語 (このページ)

- **ステータス:** 1.0リリースゲート向けに提案中
- **適用範囲:** すべての公開 `@nachi/*` パッケージおよび `nachi-effect` アセットフォーマット
- **規範的参照:** [RFC 001](./001-api.md)、特に §12、§13、§16

## 1. バージョンラインと1.0の意味

Nachiのパッケージはセマンティックバージョニングを使用し、独立してバージョン管理される。パッケージの
`0.x` ラインは実装およびコントラクト開発のラインである。マイナーリリースには、チェンジセットがそれを
識別した場合、破壊的変更が含まれてもよい。パッケージを `0.x` から `1.0.0` に移行するということは、
明示的に実験的とマークされていない限り、そのパッケージの `exports` マップを通じて到達可能なすべての
エクスポートが、本RFCの互換性契約の対象となることを意味する。

`1.0.0` は、FAパリティレポートに残された差分を除いた機能パリティを意味するものではない。これは、
文書化されたコードファーストのサーフェス、構造化された診断、およびバージョン1のアセットドキュメントが、
以下のルールに従って信頼できることを意味する。RFC 001 §16でサポートされていない機能は、バージョン
番号によって暗示されることはない。

初期のチェンジセットは、すべてのパッケージがまだ `0.0.0` である間に `major` を要求し、協調された
`1.0.0` リリースプランを生成する。リリースオーナーはFAの後にそのプランを適用する。本RFCはバージョン
バンプを実行しない。

## 2. パッケージのSemVerポリシー

- **パッチ:** 文書化された型と挙動を保持する修正、意味的変更を伴わないパフォーマンス改善、ドキュメント、
  およびすでに無効であった入力に対する追加のバリデーション。
- **マイナー:** 追加的なエクスポート、オプションの設定フィールド、新しいモジュール／アクション／診断、
  および既存の挙動を保持するデフォルトを持つバックエンドサポートの拡張。
- **メジャー:** §3で定義された任意の破壊的変更。独立してバージョン管理された各パッケージは、必要な
  バンプを受ける。依存パッケージも、その公開型またはランタイムの互換範囲を変更する必要がある場合は
  バンプされる。

1.0未満のパッケージも、結果として得られる数値バージョンが `0.x` の慣習を使用する場合であっても、
チェンジセットにおいて同じ分類に従う。チェンジセットは、実装の差分のみではなく、ユーザーが観測可能な
影響を記述しなければならない。

## 3. 破壊的変更

以下は、1.0以降において破壊的変更となる:

1. 公開エクスポート、オプション、モジュールステージ、ライフサイクル遷移、クロック境界、またはRFC 001で
   文書化されたバックエンドフォールバックの型／意味／デフォルト／順序を削除、リネーム、または変更する
   こと。
2. サポートされている入力ドメイン、リソース制限、peer範囲、またはブラウザ／バックエンド要件を厳格化し、
   以前は有効だったプログラムが失敗するようにすること。ただし、旧挙動が文書化されたセキュリティ上または
   正確性上の欠陥であり、リリースノートに例外の根拠が示されている場合を除く。
3. 構造化された `NACHI_*` 診断コードを削除またはリネームすること、あるいはその文書化されたフェーズ、
   重大度、パスのセマンティクス、またはトリガーを変更することで、利用者が同じ失敗を処理できなくなる
   こと。新しいコードの追加は追加的変更である。既存のコードを分割する場合は、非推奨化ウィンドウまたは
   メジャーリリースを通じて旧コードを保持しなければならない。
4. シリアライズされたモジュール／設定の `version` または `nachi-effect` エンベロープを非互換に変更する
   こと、サポートされているアセットバージョンの読み込みを停止すること、またはマイグレーションパスを
   削除すること。アセットフォーマットの変更はさらに§4に従う。パッケージのメジャー変更のみで、旧
   ドキュメントを暗黙に再解釈することは決してない。
5. RFC 001が結果を明示的に非決定的またはアダプター依存としている場合を除き、同一の文書化されたシード、
   定義、タイムステップ分割、バックエンド、および物理スロット順序に対する決定的な結果を変更すること。

パッケージの `exports` マップを通じて到達不可能な内部ファイル、playgroundのDOMデータセット、生成された
レポート、およびテストヘルパーは公開APIではない。ほとんどのユーザーがより高レベルのヘルパーを利用する
場合であっても、公開された低レベルのコンパイラ／ランタイムのエクスポートは対象に含まれる。

## 4. アセットフォーマットのバージョニング

パッケージバージョンとアセットバージョンは別個の軸である。現在のエンベロープは以下の通りである:

```ts
{ format: 'nachi-effect', version: 1, effect: /* closed declarative data */ }
```

旧リーダーが安全に拒否できるオプションフィールドの追加は、パッケージのマイナーリリースで出荷しても
よいが、それによって同一バージョンがリーダー間で曖昧になる場合、書き手はそれをバージョン1として出力
してはならない。非互換な形状または意味の変更は、エンベロープ／モジュールフォーマットバージョンを
インクリメントし、`EffectAssetMigrationRegistry` に明示的なワンステップのマイグレーションを提供する。
リーダーは決して推測せず、未知のフィールドを黙って破棄せず、バージョン1を再解釈しない。サポートされて
いるリーダーは、パッケージのメジャーバージョンが存続する限りバージョン1を受け入れ続ける。削除には
パッケージのメジャーリリースとリリースノートでのマイグレーションプランが必要である。

RFC 001に記録されたバージョン1の制約は規範的なままである。インライン関数は登録／参照境界を必要とし、
シミュレーションキャッシュの埋め込みはエンベロープの一部ではない。

## 5. 実験的API

1.0候補には現在、**実験的とマークされた公開エクスポートは存在しない**。コードのみのTSLスクラッチ
サーフェスである `tslModule()`、`gridTslModule()`、`grid3DTslModule()`、および
`neighborGridTslModule()` はエスケープハッチであるが、それらの作成／シリアライズ境界はすでに
RFC 001において規範的であるため、SemVerの対象外ではない。

RFC 001 §16の項目は繰延された機能であり、実験的な約束ではない。将来的に実験的サーフェスが必要になった
場合、それは `experimental` サブパスを通じて明示的にエクスポートされるか、`unstable_` プレフィックスを
使用しなければならず、`@experimental` ドキュメントタグを付与し、本節に記載しなければならない。実験的
APIはマイナーリリースで変更されてもよいが、その削除には、現実的な場合、リリースノートおよび少なくとも
1回のマイナーリリース分の事前通知が依然として必要である。

## 6. 非推奨化の手順

1. 承認されたRFC、またはRFC 001への規範的な修正において、代替手段と動機を記録する。
2. TypeScriptの宣言に `@deprecated` を付与し、パッケージ／ルートのドキュメントを更新し、チェンジセット
   を追加する。ランタイムでの使用が観測可能な場合、所有するインスタンス／システムごとに1つの構造化
   された警告を発行する。パーティクルごとやフレームごとに警告してはならない。
3. 少なくとも1回のマイナーリリースの間、また広く使用されているサーフェスについては1回の通常リリース
   サイクルの間、挙動を保持する。名前またはデータ形状が変更される場合は、機械的なマイグレーション例を
   提供する。
4. 同じウィンドウの間、診断のエイリアスおよびアセットマイグレーションを維持する。非推奨化は、アセット
   バージョンの決定を伴わずにシリアライズされた出力を変更してはならない。
5. メジャーリリースにおいてのみ削除し、`CHANGELOG.md` にその削除を記載し、明確な失敗経路または
   マイグレーション経路を保持する。セキュリティ修正はウィンドウを短縮してもよいが、その理由と影響を
   文書化する。

## 7. RFC 001 §16との整合性

本ポリシーは、RFC 001 §16の繰延された決定を確定または公開するものではない。特に、決定的なクロス
アダプター割り当て、完全なWebGL2ライフサイクルパリティ、reverse-z／MSAAの深度所有権、マルチレンダラー
エミッター、永続ID、Gridボリュームレイマーチング、またはEffekseerインポートを約束するものではない。
これらは、公開APIを追加できるようになる前に、記録された設計／スパイク作業を必要とする。それまでの間、
明示的な未サポート診断とパリティレポートが互換性の境界であり続ける。
