## 1. Goals and non-goals

This RFC fixes the ideal authoring vocabulary before the compiler and runtime are implemented. Every later public API is expected to conform to it or update this RFC explicitly. The primary goal is to make a Niagara user's mental model transferable while keeping authoring code-first, framework-independent, and serializable.

The RFC defines:

- the data model accepted by the module compiler;
- stage ordering and declared data access;
- dynamic particle attributes and parameter namespaces;
- composition of emitters, non-particle elements, effects, and timelines;
- the boundary between JSON assets and executable TSL or JavaScript code;
- validation and diagnostic behavior;
- the minimum runtime surface of `VFXSystem`.

It does not choose GPU buffer layout, workgroup sizes, indirect drawing mechanics, the exact WebGL2 support subset, renderer algorithms, or editor UI. Those decisions belong to the M0 spikes and later milestones.

## 2. Design principles

### 2.1 Niagara stage semantics are the public execution model

An emitter is evaluated in `Spawn -> Init -> Update -> Event -> Render` order. A module belongs to exactly one stage in its serialized form. A registered implementation MAY support more than one stage, but each use resolves to one stage before compilation.

Stage names are stable asset-format vocabulary. Internal compiler passes MAY fuse or reorder work only when the declared read/write dependencies prove the result equivalent.

### 2.2 Configuration is data

Definitions returned by `defineEmitter()` and `defineEffect()`, module `config`, value generators, timeline entries, resource references, and access manifests MUST serialize to JSON without preserving JavaScript object identity.

Functions, Three.js resources, DOM objects, GPU objects, class instances, symbols, and cyclic graphs MUST NOT appear in the serialized document. Authoring helpers may be fluent functions, but their output must be data.

### 2.3 TSL is an explicit escape boundary

Custom shader behavior uses `tslModule()`. TSL factories execute only while a compiler builds a node graph; they do not execute once per particle in JavaScript. A serializable asset references a registered TSL factory by stable ID and version.

The inline `tslModule(callback)` form from PLAN.md remains available for local authoring, prototyping, and tests. Serialization MUST reject an unresolved inline factory rather than embed source code or silently drop it.

### 2.4 All shared data has a namespace

Compiler-visible values use `System.*`, `Emitter.*`, `Particles.*`, or `User.*` paths. Bare names are permitted only in local convenience positions such as `emitTo(..., { inherit: ['position'] })`; they are normalized to fully qualified paths during compilation.

### 2.5 Particle attributes are emitter-local and compiled

Built-in and custom attributes form an emitter's schema. The compiler derives a physical GPU layout from the attributes actually read or written by its modules and renderers. Two emitters are not required to share a layout.

Attribute type, default value, read/write declarations, alignment, and backend support are compile-time concerns. Runtime code cannot add an attribute to an already compiled emitter.

### 2.6 Determinism and explicit ownership

Random generators MUST derive values from stable inputs such as system seed, emitter seed, physical particle index, the particle's own `Particles.spawnGeneration`, and module slot. Public modules MUST NOT call `Math.random()`. A physical slot increments its generation every time the free list reallocates it, before Init runs, so recycling a slot never repeats its previous random stream. `Emitter.spawnGeneration` identifies an emission batch/loop for scheduling and diagnostics; it is not a substitute for the per-particle generation after allocation. A Spawn-policy range is the sole pre-allocation exception and uses this batch generation because no physical particle exists yet.

This determinism guarantee assumes the same physical-slot allocation order. When multiple particles die in one dispatch, atomic append order is a parallel hardware race and is not deterministic across adapters or executions; SwiftShader is stable in current tests because its execution is effectively serialized. Fully deterministic allocation using a persistent-particle-ID equivalent remains a future opt-in mode (§16).

`VFXSystem` owns compiled resources, schedules effects, and supplies world/render integration. Effect instances own local time and `User.*` overrides. Emitters own particle state. Modules own no mutable runtime state unless the compiler allocates an explicit `Emitter.*` value for it.

Deterministic random streams identify a module slot from the module stage plus its non-empty `label` when present, otherwise from the stage plus its normalized index within that stage. The stage identity is always mixed into the slot, so equal indexes or labels in different stages do not alias the same stream. Labels MUST be unique within one emitter stage. Adding or reordering an unlabeled random module may change later unlabeled streams; a stable label preserves its stream across inheritance overrides and reordering within the same stage.

Labels beginning with `$` are reserved for compiler-synthesized modules and MUST be rejected on user-authored modules. This keeps compiler-owned identities stable without risking a collision with an author label.

## 3. Core concepts

### 3.1 Module

A **Module** is a versioned configuration object plus a registry-resolved compiler implementation. It declares its stage and data access:

```ts
interface ModuleDefinition<Stage, Config> {
  kind: 'module';
  type: string;
  version: number;
  label?: string;
  stage: Stage;
  config: Readonly<Config>;
  access?: {
    reads: readonly DataReference[];
    optionalReads?: readonly DataReference[];
    writes: readonly DataReference[];
  };
}
```

`type` is a stable registry ID such as `core/gravity`; it is not a JavaScript constructor name. `version` versions the module config independently from the asset schema.

`label` is serialized author intent and provides stable identity for inheritance patches, diagnostics, profiling, and deterministic RNG slots. An absent label uses the normalized stage index as its identity. `access` may be absent only on an authoring definition whose manifest is derived during normalization, notably a traced `tslModule`; every compiled and serialized module has a resolved access manifest.

Built-in helper calls such as `gravity(-9.8)` return module definitions with a fixed manifest. Third-party modules register their config validator, access manifest, and TSL compiler implementation under a namespaced type ID.

### 3.2 Emitter

An **Emitter** is a capacity-bounded particle simulation and its renderers. It contains a spawn policy, initialization modules, update modules, event handlers, render modules, optional custom attributes, and lifecycle configuration. `defineEmitter()` normalizes structural data but allocates no GPU resource. Definitions deliberately use structural typing; there is no runtime-only brand that authoring helpers fail to materialize.

### 3.3 Effect

An **Effect** is a named graph of elements plus a timeline. An element may be an emitter or a non-particle visual element supplied by packages such as `mesh-fx`, `trails`, or `post`. Element keys are the targets accepted by `play()` and `stop()`.

An effect definition is reusable immutable authoring data. A spawned effect instance contains mutable playback state and parameter overrides.

### 3.4 VFXSystem

`VFXSystem` is the runtime coordinator bound to a renderer and scene/world integration:

```ts
const fx = new VFXSystem(renderer, scene);
const instance = fx.spawn(effect, {
  position,
  rotation,
  seed: 42,
  parameters: { 'User.intensity': 1.5 },
});

instance.setParameter('User.intensity', 0.8);
instance.stop();
instance.release();
```

Compilation MAY be lazy on first spawn or explicit in a future API. Compilation failure MUST be observable before an instance reports itself active.

### 3.5 Namespaced parameters

- `System.*`: frame and world values shared across the system, for example `System.deltaTime` and `System.time`. Normally read-only to particle modules.
- `Emitter.*`: emitter-local counters, transform, age, loop state, and module scratch values.
- `Particles.*`: per-particle attributes stored in dynamic GPU buffers.
- `User.*`: typed inputs supplied by application code per definition or effect instance.

## 4. Stage and module contract

### 4.1 Stage order

| Stage  | Responsibility                                                             | Typical writes                                                  | Restrictions                                                       |
| ------ | -------------------------------------------------------------------------- | --------------------------------------------------------------- | ------------------------------------------------------------------ |
| Spawn  | Decide how many particles are allocated and when                           | `Emitter.spawnCount`, allocation metadata                       | Does not initialize arbitrary particle attributes                  |
| Init   | Initialize newly allocated particles                                       | `Particles.position`, `velocity`, `lifetime`, custom attributes | Runs once per spawn generation                                     |
| Update | Advance alive particle state                                               | Any declared mutable `Particles.*`/`Emitter.*` value            | Must use system/emitter local time, not wall-clock time            |
| Event  | Consume generated events and enqueue effects/sub-emitters/callback records | Event queues and explicitly declared inherited attributes       | Cross-emitter work becomes visible at a defined scheduler boundary |
| Render | Produce draw inputs and renderer state                                     | Render outputs only                                             | Simulation attributes are read-only                                |

The Update row describes the intended contract. M2 does not yet implement declarations or materialization for author-defined mutable `Emitter.*` values; author modules must not rely on such writes until that support lands. Compiler/runtime-owned paths are stricter regardless of stage: author modules cannot write `Particles.spawnGeneration`, `Particles.spawnOrder`, `Emitter.spawnCount`, or `Emitter.allocation.*`, and only Event modules can write `Emitter.events.*`.

The current core implementation is stricter than the general Spawn-stage contract above: only the three built-in spawn modules `core/burst`, `core/rate`, and `core/per-distance` may write `Emitter.spawnCount`. Other author modules cannot declare that write even when their stage is Spawn.

Events raised during a dispatch are buffered. They do not recursively execute an unbounded emitter chain in the same GPU dispatch. Event handlers write `Emitter.events.<eventName>` queue paths. M5 fixes consumption at the **next scheduler frame**. Each producer queue has two banks: frame N appends only to bank `N & 1`, while consumers read only bank `(N - 1) & 1`. Core clears the new write bank before any emitter advances, so effect element order cannot create same-frame recursion. A fixed-time substep is a scheduler frame; prewarm steps internal to one activation retain one bank and therefore preserve bounded overflow behavior.

`Emitter.eventPayload.<field>` is the typed, read-only view of a consuming event record. `emitTo(target, { inherit: ['position'] })` maps the producer's `Particles.position` snapshot to `Emitter.eventPayload.position`; after ordinary target Init runs, core overwrites target `Particles.position` from that field. Non-inherited attributes keep their normal Init results. The snapshot is taken in the compiler-owned death epilogue before the producer slot is recycled.

M2 compiles Spawn modules through the same versioned module registry and access validation used by Init/Update. Spawn modules execute in author order and add requests to `Emitter.spawnCount`; only the allocator may translate that count into particle indices. On WebGPU, a spawn-request uniform drives a GPU-written indirect dispatch, successful invocations pop scattered indices from the free list, increment `Particles.spawnGeneration`, and then run Init exactly once. Requests beyond the available free count are clamped, counted, and reported as `NACHI_SPAWN_CAPACITY_EXCEEDED`; they never wrap an atomic counter or overwrite an alive particle. CPU-side request clamping reports immediately. GPU-side overflow reporting requires `aliveCountReadbackInterval`, because without that explicit interval M2 performs no counter readback or GPU/CPU synchronization; “reported” in this paragraph is conditional on that option.

### 4.2 Read/write declaration

Every compiled module has a `ModuleAccess` manifest of fully qualified paths:

```ts
const gravityRegistration: ModuleRegistration<'update', GravityConfig> = {
  type: 'core/gravity',
  version: 1,
  stage: 'update',
  access: {
    reads: ['Emitter.deltaTime', 'Particles.velocity'],
    writes: ['Particles.velocity'],
  },
  compile(context, config) {
    // Build TSL nodes; no per-particle JavaScript callback.
  },
};
```

The compiler MUST:

1. resolve each path against built-ins, custom attributes, or parameter declarations;
2. reject a missing required read;
3. allocate values that are written and subsequently read;
4. verify stage write permissions;
5. detect read/write declarations that disagree with a traced TSL graph where tracing is available;
6. preserve author order for observable read-after-write and write-after-write dependencies;
7. report an unused declared custom attribute as a warning, not an error.

Item 4 is enforced for Spawn, Init, Update, Event, and Render manifests. Spawn may write only `Emitter.spawnCount` or compiler-owned `Emitter.allocation.*`; Init may write `Particles.*`; Update may write declared mutable `Particles.*`/`Emitter.*`; Event may write particle inheritance and its normalized event queue; Render may not write simulation paths.

During normalization the compiler recursively visits module config. Every nested `parameter('Namespace.name')` generator implicitly adds that path to `reads`; this is merged with and deduplicated against the module's declared manifest. Likewise, placement under `events.onDeath` replaces the authoring-only `Emitter.events.pending` placeholder with `Emitter.events.onDeath` in each handler's writes. Helper-derived access is canonical serialized data, not an exception to dependency analysis.

`optionalReads` allows capability-dependent inputs such as depth. A module implementation must define its fallback behavior when an optional value is absent. Resolution is intentionally asymmetric with required reads: a known built-in optional attribute is allocated, while an absent optional custom attribute is not diagnosed or allocated and the module fallback is used.

### 4.3 Reserved integration module

Velocity does not update position implicitly inside force modules. Unless `integration: 'none'` is selected, M1 compile-time normalization appends a compiler-owned module with stable type `core/integrate`, label `$integrate`, and this manifest. `defineEmitter()` only normalizes authoring data and does not synthesize or compose `$integrate`:

```ts
{
  reads: ['Emitter.deltaTime', 'Particles.position', 'Particles.velocity'],
  writes: ['Particles.position'],
}
```

The default integrator is semi-implicit Euler (`velocity` is finalized before position integration). Forces and the integrator read `Emitter.deltaTime`, not `System.deltaTime`: the emitter-local value includes effect `timeScale` and `hitStop`, so velocity and position cannot advance on different clocks. `System.deltaTime` remains the unscaled host-frame value for modules that explicitly need it. Supplying a custom position writer requires `integration: 'none'` or an explicit future integrator selection; the compiler must never apply both silently.

Core particle positions and velocities are stored in world space, matching `positionSphere()` which
applies `Emitter.transform` during Init. `killVolume()` is the deliberate authoring exception: its
box, sphere, and plane dimensions are expressed in emitter-local space. The Update implementation
transforms each world-space `Particles.position` by `inverse(Emitter.transform)` before testing the
volume. Consequently a spawned instance's translation and rotation move the kill volume with the
emitter; no scale is currently exposed by the public emitter transform API. The plane normal is
normalized before evaluation, so `offset` remains a local-space distance; its inside half-space is
`dot(normalize(normal), localPosition) <= offset`. Death writes `Particles.alive = false` and is
recovered by the normal compiler-owned free-list epilogue.

`turbulence()` samples deterministic 3D simplex noise in particle world space. Its octave count,
frequency doubling, amplitude halving, and per-channel offsets are compile-time/fixed constants;
neither wall time nor a random stream participates. Given identical initial particle storage and
dispatch order, two runs therefore build and execute the same node graph bit-for-bit.

`curlNoise()` uses the same centered `snoise` scalar and fixed per-channel offsets to construct a
3D potential field. Central finite differences evaluate `curl(psi)`; the resulting field is
divergence-free up to finite-difference and floating-point error. A measured, fixed
derivative-amplitude correction makes `strength` the approximate maximum acceleration, matching
the public normalization used by `turbulence()` without applying a position-dependent scale that
would reintroduce divergence. Frequency changes spatial scale without changing that normalization,
and no clock or random stream participates.

`vortex()` and `pointAttractor()` accept `space: 'world' | 'emitter'`, defaulting to `world`.
Emitter-space evaluation transforms the world-space particle position by
`inverse(Emitter.transform)`, evaluates local centers and axes, then transforms acceleration back
to world space with direction weight zero. Translation therefore follows the emitter without
contaminating acceleration, while emitter rotation affects both the field and returned force.

M6 analytic colliders use the same explicit `space: 'world' | 'emitter'` convention as those force
modules and default to `world`; this differs intentionally from the older emitter-local-only
`killVolume()`. `collidePlane()` treats `dot(normalize(normal), position) < offset` as penetration.
`collideSphere()` and `collideBox()` treat their volume as solid and push an interior particle to
the nearest surface. `mode: 'bounce'` preserves an outward normal component, reflects an inward
component with `bounce` restitution, and multiplies tangent velocity by `1 - friction`; `stick`
zeros velocity and `kill` recycles the particle. Both coefficients are clamped to `[0, 1]` at
materialization and statically invalid constants are diagnosed. Collision position correction is a
sanctioned author Update write and may precede compiler-owned `$integrate` without triggering the
custom-integrator double-write diagnostic.

`collideSceneDepth()` uses the same response modes in world space. Compute does not read
`viewportDepthTexture` directly: that node is render-pass scoped and sampling an active attachment
would create a feedback hazard. Renderer integration copies completed scene-pass depth into a
normalized, sampleable color texture and binds that previous-frame copy to the kernel adapter. The
particle is projected with `System.viewMatrix` and `System.projectionMatrix`. The latter is
explicitly a WebGPU-coordinate-system projection matrix: NDC depth is `z ∈ [0, 1]`, so projected
depth is used directly and inverse projection receives sampled depth directly (no OpenGL
`z * 0.5 + 0.5` / `z * 2 - 1` conversion). The depth copy must preserve normalized depth in a
linear, non-sRGB float color texture; output color transforms and normalized 8-bit render targets
are invalid because they encode and quantize depth. `System.viewportSize` supplies depth-gradient
texel offsets. Neighboring depth samples are inverse-projected to reconstruct a view-space normal,
which is transformed back to world space.

On three.js r185, a `pass(scene, camera)` node is not a deterministic way to drive this copy outside
the presentation loop: `PassNode` uses `FRAME` update-before semantics, and `NodeFrame` suppresses a
second update for the same frame id, so repeated `RenderPipeline.render()` calls can silently sample
the prior scene configuration. The recommended integration instead attaches a non-MSAA
`DepthTexture` to a `RenderTarget`, selects that target and calls `renderer.render(scene, camera)`
explicitly for every capture, then runs a separate TSL copy pipeline that samples the depth texture
into the linear, non-sRGB float color target described above.

M6 v1 therefore requires WebGPU, a non-MSAA resolved depth copy, matching capture/simulation camera
and viewport, and accepts one-frame latency for moving geometry. A missing binding is diagnosed as
`NACHI_SCENE_DEPTH_UNAVAILABLE`, WebGL2 and MSAA sources are rejected explicitly, and an unset
camera or detected reverse-z projection produces a runtime warning; none are treated as silent
fallbacks.

`collideSceneDepth({ thickness })` rejects penetration whose linear view-depth difference exceeds
`thickness` (default `0.1`). This prevents a particle already hidden far behind foreground geometry
from teleporting to that foreground surface. `surfaceOffset` remains the small world-space
separation applied only after an eligible collision.

`collideSdf()` samples a serializable `SdfRef` in world space. Negative distance is penetration;
the normal is the normalized central-difference gradient, and correction moves the particle by the
signed distance to the zero surface. Optional `thickness` rejects deeply embedded samples in the
same way as scene depth. The public `bakeSdf()` utility creates corner-aligned float32 fields from a
union of analytic spheres and boxes for tests and procedural assets. Mesh-to-SDF baking is deferred
to M12 because its voxelization and watertightness policy is an asset-pipeline decision.

`positionMeshSurface({ mesh, mode: 'surface' })` samples an area-weighted triangle CDF with three
deterministic particle random samples, writes world-space position, and retains the sampled normal
in compiler-known `Particles.surfaceNormal`. `velocityMeshNormal({ speed })` consumes that normal.
The Three v1 integration rebuilds triangle positions and area CDF on the CPU from
`SkinnedMesh.applyBoneTransform()` and uploads both float textures each frame. This deliberately
keeps bone textures out of core and gives static and skinned meshes one `MeshRef` contract, at the
cost of CPU skinning/upload proportional to source triangle count; large crowds require a future
GPU-skinning data interface. A resource keeps fixed source topology and uses flat triangle normals;
topology changes require rebuilding the resource, and smooth deformed vertex-normal interpolation
is deferred with the GPU-skinning path. The v1 triangle-position texture is a single row containing
three texels per triangle, so `3 * triangleCount` MUST NOT exceed the active device's
`maxTextureDimension2D`. The Three helper accepts that limit and defaults to the WebGPU guaranteed
minimum of 8192 texels (at most 2730 complete triangles); it reports
`NACHI_MESH_SURFACE_TEXTURE_TOO_WIDE` before allocating an oversized texture. Multi-row/2D tiled
triangle and CDF textures are deferred to M12.

The public `turbulence.strength` approximates a maximum acceleration magnitude, not a raw simplex
multiplier. Octaves are amplitude-sum normalized, centered Three r185 `snoise` is multiplied by
`1 / 0.286` (its measured effective scalar amplitude is approximately `±0.286`), and the resulting
three-component vector is length-clamped to one before applying `strength`.

The `$integrate` label uses the compiler-reserved `$` prefix. User-authored module labels MUST NOT begin with `$`; normalization reports a compile diagnostic rather than renaming them.

The same M1 compile-time normalization materializes two additional compiler-owned modules in the kernel description. `$defaults` (`core/defaults`) is the first init module and writes every resolved `Particles.*` attribute from its declared default. When `age` and `lifetime` are present in the resolved lifecycle schema, `$age` (`core/age`) is the first update module and has this manifest:

```ts
{
  reads: ['Emitter.deltaTime', 'Particles.age', 'Particles.lifetime'],
  writes: ['Particles.age', 'Particles.normalizedAge'],
}
```

`$age` advances age and derives clamped normalized age before author update modules. These operations are described modules rather than hidden statements injected while materializing TSL kernels.

### 4.4 Module config and implementation separation

The serialized module contains `type`, `version`, optional `label`, `stage`, `config`, and resolved `access`. Executable validation and compilation functions live in a `VfxRegistry`. This makes the asset JSON inspectable and prevents bundler-specific function names from becoming format identifiers.

## 5. Dynamic attributes

### 5.1 Built-in attributes

The baseline particle schema includes:

| Name                                 | Logical type     | Purpose                                                    |
| ------------------------------------ | ---------------- | ---------------------------------------------------------- |
| `position`                           | `vec3`           | emitter-local or world position according to emitter space |
| `velocity`                           | `vec3`           | linear velocity                                            |
| `age` / `lifetime` / `normalizedAge` | `f32`            | lifetime evaluation                                        |
| `alive`                              | `bool`           | lifecycle state; defaults false until allocation           |
| `spawnGeneration`                    | `u32`            | per-slot allocation generation mixed into particle RNG     |
| `spawnOrder`                         | `u32`            | deterministic birth order for order-sensitive renderers    |
| `color`                              | `color` (`vec4`) | linear RGBA                                                |
| `size` / `scale`                     | `f32` / `vec3`   | renderer size                                              |
| `rotation`                           | `quat`           | mesh/general orientation                                   |
| `spriteRotation`                     | `f32`            | billboard rotation                                         |
| `surfaceNormal`                      | `vec3`           | normal sampled by mesh-surface Init modules                |
| `mass`                               | `f32`            | force integration                                          |

Built-ins are logical API names, not a fixed physical struct. The compiler MAY omit unused built-ins. `normalizedAge` MAY be derived instead of stored.

### 5.2 Custom declarations

Custom attributes are declared on the emitter:

```ts
const heat = attribute('heat', {
  type: 'f32',
  default: 0,
});

const embers = defineEmitter({
  capacity: 4096,
  attributes: { heat },
  spawn: burst({ count: 128 }),
  init: [
    /* modules that initialize heat */
  ],
  update: [
    /* modules that read/write heat */
  ],
  render: billboard({ blending: 'additive' }),
});
```

Supported logical types initially are `f32`, `i32`, `u32`, `bool`, `vec2`, `vec3`, `vec4`, `color`, `quat`, `mat3`, and `mat4`. Backend layout rules are compiler-owned. Attribute defaults MUST be deterministic and type-compatible.

Redeclaring a built-in name with a different type is an error. A custom attribute that may remain unwritten needs a default. Attributes marked `transient` are excluded from sim caching and serialization of particle snapshots; the compiler must reject a transient attribute required by a cache replay path.

The `attributes` record key is canonical and MUST exactly match `AttributeDefinition.name` (`attributes.heat.name === 'heat'`). Authoring normalization rejects a mismatch; it never renames one side silently. The duplicated name is retained so a standalone attribute definition remains self-describing when moved, inspected, or referenced by tooling.

## 6. Namespaced parameters

Parameter declarations use a full path and logical type:

```ts
const intensity = defineParameter('User.intensity', {
  type: 'f32',
  default: 1,
  mutable: true,
});

const gravityScale = parameter('User.gravityScale', 1);
```

| Namespace     | Declared by                  |      Mutable from application code | Lifetime                               |
| ------------- | ---------------------------- | ---------------------------------: | -------------------------------------- |
| `System.*`    | runtime/compiler             | no, except system integration APIs | VFXSystem/frame                        |
| `Emitter.*`   | compiler/module registration |            no direct public writes | emitter instance                       |
| `Particles.*` | built-ins/emitter attributes |            no direct public writes | particle                               |
| `User.*`      | effect/emitter author        |          yes when declared mutable | definition default + instance override |

Unknown paths are compile errors. Runtime writes to unknown, immutable, or type-incompatible `User.*` parameters are runtime validation errors. Parameter namespaces are case-sensitive and reserved exactly as written.

Effect and emitter definitions may both declare `parameters`. Effect declarations define the public instance surface; emitter declarations define reusable emitter-local defaults and are mapped into the containing effect during M9 composition. `EffectInstance.setParameter()` and spawn-time `parameters` derive their path and value types from the effect's declared `User.*` schema. The exact override/forwarding rule between effect and child-emitter declarations remains an M9 decision.

### 6.1 Reserved `Emitter.*` paths

| Path                           | Meaning                                                                          |
| ------------------------------ | -------------------------------------------------------------------------------- |
| `Emitter.transform`            | emitter-to-world transform                                                       |
| `Emitter.localTime`            | emitter-local playback time                                                      |
| `Emitter.deltaTime`            | emitter-local scaled delta                                                       |
| `Emitter.age`                  | emitter lifecycle age                                                            |
| `Emitter.loopIndex`            | current loop number                                                              |
| `Emitter.seed`                 | deterministic emitter seed                                                       |
| `Emitter.spawnGeneration`      | current emission batch/loop generation (not the particle RNG generation)         |
| `Emitter.spawnCount`           | count requested by the spawn stage                                               |
| `Emitter.allocation.*`         | compiler-owned free-list/allocation metadata                                     |
| `Emitter.eventWriteBank`       | compiler-owned current event append bank (`0` or `1`)                            |
| `Emitter.eventReadBank`        | compiler-owned previous-frame event consumption bank                             |
| `Emitter.events.pending`       | authoring-only event queue placeholder; removed by normalization                 |
| `Emitter.events.<eventName>`   | normalized queue written by handlers for that event                              |
| `Emitter.eventPayload.<field>` | typed read-only consuming field mapped from an inherited `Particles.*` attribute |

Packages may reserve additional paths only through an RFC update. Application-defined emitter parameters must use a package-qualified suffix to avoid collisions.

`Emitter.events.pending` is never present in a compiled or serialized normalized manifest. Event placement supplies the concrete name, replaces the placeholder with `Emitter.events.<eventName>`, and then dependency analysis sees only the concrete queue.

M5 reserves `Emitter.eventPayload.<field>` only for fields listed by `emitTo(..., { inherit })`. Other suffixes are unknown paths, and all event-payload fields are compiler-owned read-only values. The producer and target attributes must exist with identical logical types and component counts or compilation fails. Float-domain fields (`f32`, `vec2`, `vec3`, `vec4`, `color`, and `quat`) of up to four components are packed in first-use order into vec4 groups; a field never straddles groups. Integer, boolean, and matrix payloads are rejected in M5 rather than reinterpreted silently. `CompiledEmitterMeta.eventQueues` exposes each field's group/lane, logical type, queue capacity, handlers, and payload group count.

## 7. Value generators

Generators are immutable JSON values evaluated by a stage compiler. Their conceptual signatures are:

```ts
function range<T extends number | Vec2 | Vec3 | Vec4>(min: T, max: T): RangeGenerator<T>;

function curve<T extends number | Vec2 | Vec3 | Vec4>(
  first: readonly [time: number, value: T],
  second: readonly [time: number, value: T],
  ...rest: readonly [time: number, value: T][]
): CurveGenerator<T>;

function gradient(first: ColorInput, second: ColorInput, ...rest: ColorInput[]): GradientGenerator;

function parameter<T extends JsonValue>(path: ParameterPath, fallback?: T): ParameterGenerator<T>;
```

`range()` uses deterministic uniform random selection by default. Its random stream slot combines the containing stage with the module's `label` when present and otherwise with its normalized stage index. Vector ranges sample each component from a distinct deterministic sample offset rather than broadcasting one scalar across every component. Future distributions must be explicit config fields rather than behavior changes.

`curve()` key time is domain-independent. Modules such as `sizeOverLife()` define a normalized `[0, 1]` domain, while timeline/world-space consumers may define another domain. At least two keys are required. M1 supports linear interpolation only; omitted interpolation defaults to linear, while `constant`, `cubic`, and unknown interpolation values produce an explicit compile diagnostic.

`gradient(colorA, colorB, ...)` creates at least two evenly spaced stops. A future explicit-stop overload may accept `{ position, color }`; it must serialize to the same canonical stop array. String colors such as hex values are interpreted as sRGB and converted to linear values with the Three.js transfer function before LUT interpolation. Numeric color arrays are already-linear values and are not converted.

Curves and gradients are baked to LUT textures or buffers. Baking precision, resolution, and cache keys are compiler options, not hidden module behavior.

## 8. TSL escape hatch

### 8.1 Inline authoring form

The PLAN.md form is valid:

```ts
tslModule(({ position, velocity, age }) => ({
  velocity: velocity.add(myCustomField(position)),
}));
```

The factory receives typed TSL expressions, not CPU particle values. It returns a partial mapping of attributes to new expressions. Returned keys are writes. Accessed input bindings are reads. The compiler traces both and produces the final manifest. `tslModule()` defaults to the `update` stage; `stage: 'init'` opts into initialization.

Attribute expressions supplied to inline factories and by `KernelModuleBuildContext.attribute()` are read-only logical values. Modules MUST return an output mapping or call `KernelModuleBuildContext.write()` to update particle storage; calling `assign`, `addAssign`, or another in-place mutation on an attribute expression is invalid and materialization reports an error. This rule is independent of whether the compiler places the attribute in a dedicated or packed physical buffer.

The factory MUST be pure with respect to compilation: no frame-dependent JavaScript state, I/O, random calls, GPU submission, scene mutation, or resource allocation outside registered compiler facilities. It MAY compose TSL functions from `@nachi/tsl-kit` and the pinned Three.js adapter layer.

An optional `access` manifest may be supplied, with two deliberately distinct states:

- property absent: the manifest is unknown at authoring time and is derived completely from the traced TSL graph during normalization;
- property present, including `{ reads: [], writes: [] }`: the author supplied a declaration and the traced reads/writes must be subsets of it.

Normalization materializes the traced result before serialization. It never replaces an absent declaration with an empty declaration. Explicit access is required for dynamic property access or registry integrations that cannot be traced.

Built-in binding names have no open string index signature, so `velocty` is a type error. Custom bindings use declared keys of the form `custom.<name>`:

```ts
type HeatBindings = TslParticleBindings<{ heat: number }>;

tslModule<HeatBindings>((bindings) => ({
  'custom.heat': bindings['custom.heat'],
}));
```

Deriving `HeatBindings` automatically from an emitter's `attributes` schema is postponed to M2 or later and listed in §16. Until then, custom bindings are supplied explicitly.

Returned write keys are additionally resolved against the emitter schema during trace validation. TypeScript catches misspelled built-in input bindings and explicitly typed `custom.*` bindings, but it cannot prove every structurally inferred object key; a write-key typo that passes authoring type inference is therefore a compile-time trace-validation error, not an accepted custom attribute.

### 8.2 Serializable registered form

```ts
const followField = defineTslFunction(
  'game/follow-field',
  ({ position, velocity }) => ({
    velocity: velocity.add(field(position)),
  }),
  1,
);

registry.registerTsl(followField);

const module = tslModule(followField.ref, {
  stage: 'update',
  access: {
    reads: ['Particles.position', 'Particles.velocity'],
    writes: ['Particles.velocity'],
  },
});
```

The JSON stores only `{ kind: 'function-ref', id, version }`. Deserialization resolves it through the registry. Registry IDs MUST be stable, namespaced, and explicitly versioned. Missing or version-incompatible entries are compile errors.

### 8.3 TSL version boundary

Public callbacks receive the library's `TslExpression<T>` facade. Direct Three.js TSL nodes may be admitted through a narrowly typed adapter, but raw Three.js node types are not serialized and are not promised semver stability by this library. The adapter is the only package that should change when pinned Three.js TSL APIs move.

## 9. Emitter API

The canonical shape is:

```ts
interface EmitterConfig<Attributes, Parameters> {
  capacity: number;
  attributes?: Attributes;
  parameters?: Parameters;
  spawn: SpawnModule | readonly SpawnModule[];
  init?: readonly InitModule[];
  update?: readonly UpdateModule[];
  events?: Partial<Record<ParticleEventName, EventModule | readonly EventModule[]>>;
  integration?: 'euler' | 'none';
  render: RenderModule | readonly RenderModule[];
  lifecycle?: {
    duration?: number;
    loopCount?: number | 'infinite';
    prewarm?: number;
    startDelay?: number;
  };
}
```

`capacity` is a hard logical maximum and must be a positive integer. Spawn and event queues clamp overflow explicitly; silently corrupting or resizing GPU buffers is forbidden.

Multiple modules in a stage run in author order unless dependency analysis proves a safe optimization. Multiple handlers for one event run in author order and each writes the normalized `Emitter.events.<eventName>` queue dependency. Renderer modules cannot mutate simulation attributes. M7 batch 1 supports exactly one render module per emitter and diagnoses additional render modules with `NACHI_RENDER_MODULE_LIMIT`; multiple render modules require per-draw indirect-argument slots and remain a later M7 batch. `integration` defaults to `euler`; M1 compile-time normalization synthesizes and composes the reserved `$integrate` module from §4.3. Authoring-time normalization in `defineEmitter()` does not perform that composition.

Emitter inheritance planned for M9 will use an asset reference plus declarative overrides. It will not use JavaScript prototype inheritance. The exact patch/merge representation remains open until the asset format milestone, but overrides must preserve module identity so a child can replace, remove, or reorder one module without copying the whole parent.

An override addresses a labeled module by label. An unlabeled module is addressed by normalized stage index and is therefore less stable under edits. Duplicate labels in one stage are compile errors.

M2 lifecycle normalization uses `startDelay` before the first activation, runs an emitter for `duration` seconds per loop, and interprets `loopCount` as the total number of activations including the first (`'infinite'` never completes emission). An omitted duration is a zero-duration one-shot burst. `prewarm` advances the first activation by the declared number of effect-local seconds before it becomes externally visible. A looping emitter requires a positive duration.

M2 provides `burst({ count, cycles?, interval? })`, `rate({ rate })` (particles per emitter-local second), and `perDistance({ rate })` (particles per world-space unit travelled by `Emitter.transform`). Rate and distance modes retain fractional remainders; fixed timesteps therefore produce the same integer totals for equivalent delta partitions. Per-distance compares successive transforms supplied by the runtime and consumes each travelled segment once.

Prewarm equivalence covers deterministic emitter-local Spawn/Init/Update state when compared with the same sequence of simulation steps. It deliberately excludes unscaled `System.*` clock values, rendering/presentation, readback timing, callbacks, and external integrations. An emitter with no module that declares a `Particles.lifetime` write has an unbounded conservative lifetime (`Infinity`) unless exact alive-count completion is enabled. If any module other than the statically understood `core/lifetime` writes lifetime, the same conservative `Infinity` rule applies.

An undeclared lifetime is also the supported form for particles that live until an explicit Update
module, such as `killVolume()`, writes `Particles.alive = false`. Reaching `capacity` never
overwrites a living slot: Spawn compares requests with the atomic free count, drops unavailable
invocations, and exposes them through the existing overflow diagnostic. A slot becomes available
only through the compiler-owned death/recycling epilogue. Unbounded rate emitters SHOULD provide
an explicit death path; otherwise capacity stays full and later spawns remain suppressed.

### 9.1 M5/M6 events and sub-emitters

`onDeath: emitTo(target, { inherit })` is the M5 executable event contract. Effect compilation resolves `target` against emitter element keys and reports `NACHI_EVENT_TARGET_UNKNOWN` before materialization. Each producer event kind owns one append queue shared by all of its handlers. The inherited field union determines one payload layout, so two handlers that inherit `position` do not duplicate that lane.

An event queue uses three WebGPU resources: a four-word atomic state buffer, a double-banked vec4 payload buffer, and one three-word indirect-dispatch buffer. State words are the two bank counts, cumulative overflow count, and cumulative emitted-event count. Payload allocation costs `2 * producerCapacity * max(1, payloadGroupCount) * 16` bytes. State costs 16 bytes and indirect arguments cost 12 bytes per producer event kind. These resources are listed in `CompiledEmitterMeta.storageBuffers` and count toward device storage-buffer budgets. A consumer uses the M2 free-list and `dispatchIndirect` path; event count is a separate spawn batch, Init runs normally, then inherited attributes overwrite the declared outputs.

Event-consumed Init uses the ordinary uniform values retained from the previous scheduler frame; current-frame timing uniforms are installed when that emitter advances after event consumption. Event bank uniforms are the scheduler-owned exception. Effect completion conservatively drains one scheduler frame per edge in the deepest event chain, so a write-bank append remains consumable without an event-count readback. When exact alive readback is enabled, emission completion and every event-consumption frame force the final compact/readback regardless of `aliveCountReadbackInterval`.

Atomic append may increment a bank count beyond producer capacity, but no payload address is formed for those slots. Such records are dropped, the cumulative overflow word is incremented, and opt-in readback reports `NACHI_EVENT_QUEUE_OVERFLOW`. Consumer capacity pressure uses the existing `NACHI_SPAWN_CAPACITY_EXCEEDED` counter and never overwrites a living target slot. Queue counts are not recursively consumed in their write frame.

`onCollision: emitTo(target, { inherit })` is executable in M6. Built-in analytic and scene-depth
colliders append whenever their penetration test produces a response. Position correction is
written before append, so `inherit: ['position']` carries the resolved surface position through the
same vec4 payload path used by `onDeath`; consumption remains next-scheduler-frame and bounded by
the same double-bank and overflow rules. The payload `position` and `velocity` are sampled after
position correction and velocity response. A `mode: 'kill'` collision appends `onCollision`, then
writes `alive = false`, so the compiler death epilogue also appends `onDeath` when both handlers
exist. Non-kill responses do not write `Particles.alive` and therefore cannot resurrect a particle
killed by an earlier author module. Resting contact uses strict penetration; gravity/integration
may move a corrected particle back inside on a later frame, and every new penetration response
emits another `onCollision`. Continuous contact is intentionally an event stream rather than a
begin-contact-only event in M6 v1. `onSpawn` is reserved for a future compiler-owned
lifecycle producer and produces `NACHI_EVENT_KIND_UNIMPLEMENTED`. The planned custom producer
surface is a `tslModule` context method `context.emitEvent(condition)`, valid only from Update and
lowering to a conditional append into `onCustom`. The condition must be a scalar boolean TSL
expression and inherited fields still come from the handler's `emitTo` declaration. Until that
context is materialized, authoring `onCustom` produces `NACHI_EVENT_ON_CUSTOM_UNIMPLEMENTED`.

### 9.2 M3 sprite and mesh renderers

`billboard()` consumes particle color, position, size, and optional orientation inputs without
mutating simulation attributes. `flipbook(texture, { cols, rows })` assigns frame 0 to the
top-left atlas cell and advances left-to-right, then top-to-bottom. Three.js textures using
`flipY` therefore resolve the atlas row as `rows - 1 - floor(frame / cols)`; textures with
`flipY: false` use `floor(frame / cols)` directly. Frame interpolation is enabled by default. If
the emitter has no lifetime writer, `normalizedAge` remains at its default zero and the flipbook
remains on frame 0.

Motion-vector interpolation requires both interpolation and a motion-vector `TextureRef`.
Requesting motion vectors without a resource falls back to plain interpolation with a warning;
requesting them while `interpolate: false` ignores them with a warning. UV warps are clamped to
their source atlas cell so filtering cannot leak into adjacent frames.

`cutout.vertices` accepts 4 through 8 vertices. Four vertices retain the full quad. Five through
eight vertices form a convex polygon contained by the quad, reducing overdraw by cutting away its
texture corners; triangulation uses `(vertices - 2) * 3` indices and preserves UVs in the original
quad's `[0, 1]` range.

The four blending modes have fixed M3 semantics. `alpha` uses source-alpha blending;
`premultiplied` uses normal blending with premultiplied-alpha factors; `additive` adds source color;
and `multiply` multiplies the destination by source color. Billboards always use the transparent
render path with depth testing enabled and depth writes disabled. Meshes keep depth writes enabled;
`alpha` and `premultiplied` select the transparent render path, while `additive` and `multiply`
remain on the opaque render-list path.

Billboard alignment is one of `camera-facing` (default), `custom-axis`, `velocity-aligned`, or
`velocity-stretch`; the stretch form optionally scales its velocity contribution with `factor`.
Mesh alignment is one of `none` (default), `custom-axis`, `velocity`, or `quaternion`. Velocity and
custom-axis mesh modes rotate the geometry's local +Y axis toward the selected direction, while
quaternion mode consumes `Particles.rotation` directly.

`orientToVelocity()` is an Update module that writes both billboard-compatible
`Particles.spriteRotation` and a quaternion rotating local +Y toward `Particles.velocity`. Its
quaternion path uses direct quaternion-vector math rather than Three r185's transposed TSL
`rotate()` Euler convention. `spriteRotation` is the velocity angle projected onto the world XY
plane, whereas renderer-side velocity-aligned billboards derive their angle in view space. Zero
velocity preserves the previous orientation. Use it with camera-facing billboards or
`meshRenderer({ alignment: { mode: 'quaternion' } })`; combining it with renderer-side velocity
alignment applies the same intent twice and SHOULD be avoided.
`faceCamera(options)` is the explicit render-stage spelling of
`billboard({ ...options, alignment: { mode: 'camera-facing' } })`; it does not introduce camera
state into deterministic Update simulation.

`rotationOverLife()` accepts an `f32` curve and writes the isotropic scalar
`Particles.spriteRotation` in radians. Per-axis Euler and quaternion curves require a future typed
orientation module. `velocityOverLife()` likewise accepts only an `f32` curve and applies one
absolute isotropic scale to all velocity components. Each Update applies
`lut(t) / lut(tPrevious)`, where `tPrevious` is reconstructed from `Emitter.deltaTime` and particle
lifetime; equivalent elapsed-time partitions therefore reach the same scale instead of repeatedly
compounding the absolute LUT value.

`billboard({ soft: true })` uses a calibrated default fade distance of `0.035`.
`billboard({ soft: { fadeDistance } })` overrides it. The value is measured in Three.js
linearized normalized camera-depth units and must be positive and finite. The default comes from
the accepted `spike-depth` partial-intersection calibration; it is not a world-space distance.

`meshRenderer({ geometry, alignment })` resolves a serializable `GeometryRef` to an arbitrary
Three.js `BufferGeometry` at materialization time. It consumes per-particle position, scale, and
color, with optional velocity, custom-axis, or quaternion orientation. Like billboards, mesh
instances use the compact alive-index indirection and alive-count indirect draw arguments.

### 9.3 M7 ribbon/trail renderer

`@nachi/trails` owns `ribbon()`, `ribbonId()`, its render compiler registration, and the Three.js
WebGPU materializer. Applications add those implementations to the existing core registry with
`registerTrails(createCoreKernelModuleRegistry())` and supply that registry through
`VfxSystemOptions.registry` or `compileEmitter({ registry })`. Core provides only the generic
render-registration context and lifecycle ordering resources; it does not re-export the trails
authoring helpers. This is the first external package proof of the module/render registry boundary
from §13.

`ribbon({ width, maxRibbons, uv, taper, blending, map })` reads `Particles.position`, `color`,
`ribbonId`, and the compiler-owned `spawnOrder`. `ribbonId()` initializes the custom `u32`
`ribbonId` attribute to a constant or an alternating modulo sequence. `maxRibbons` is an explicit
scan bound (v1: 1 through 64), not a dynamically resized GPU allocation. `uv.mode: 'stretched'`
maps each strip's ordered point ordinal to `[0, 1]`; `uv.mode: 'tiled'` accumulates world-space
segment length and divides by positive `tileLength`. Taper fractions are evaluated against the
same per-ribbon parametric coordinate. The endpoint width is exactly zero when its corresponding
taper fraction is nonzero, while the untapered body uses the authored `width`. Fragment color is
still read from `Particles.color`, so `colorOverLife()` remains orthogonal, and additive, alpha,
multiply, and premultiplied blending retain the §9.2 meanings.

Ribbon order MUST NOT consume compacted `aliveIndices`. Before each spawn dispatch, a one-invocation
GPU preparation kernel atomically reserves a consecutive `spawnOrder` range of
`min(requested, freeCount)`. The scheduler submits prepare, spawn, and finalize strictly in order,
so `freeCount` cannot change between reservation and allocation; failed saturated spawn attempts
therefore consume neither an order value nor a birth-ring slot. Successful invocation `i` receives
`base + i`, independently of free-list physical-slot selection. Spawn writes the physical index
into a capacity-sized birth ring at `spawnOrder % capacity`. Before rendering,
`@nachi/trails` runs one GPU invocation that scans the ring chronologically for each `ribbonId`,
rejects stale entries by comparing the particle's stored `spawnOrder`, skips dead particles, and
emits adjacent surviving pairs into compact segment/UV/width buffers plus a renderer-owned indirect
draw record. Scattered death and non-deterministic alive compaction therefore cannot reorder or
cross-connect a strip, and killing an interior control point reconnects its surviving predecessor
and successor. Normal operation performs no CPU readback; the exposed readback helper is diagnostic
and smoke-test only.

M7 batch 1 ribbons require WebGPU storage buffers and indirect draw. Materialization on the WebGL2
path fails explicitly with `NACHI_RIBBON_WEBGL2_UNSUPPORTED`; there is no silent sprite/line
fallback. The v1 preparation cost is `O(maxRibbons * capacity)` in a deliberately serial GPU pass,
and ordering uses wrapping `u32` values. `NACHI_SPAWN_ORDER_WRAP_RISK` warns once at the `2^31`
safety threshold. Without counter readback, normal-spawn request accumulation is a conservative
upper bound; enabled lifecycle readback also checks the actual GPU counter, including event births.
An emitter must be restarted before the 32-bit order wraps, and sim-cache work in M11 must preserve
or rebase this counter.

The birth ring retains only the latest `capacity` successful births. If one unusually long-lived
control point survives while at least `capacity` later successful births churn through shorter-lived
slots, its ring entry is overwritten and that point drops out of the strip even though it remains
alive. M7 batch 1 has no zero-readback runtime diagnostic for this condition; authors must size
capacity/lifetime/rate so every required control point remains inside the retention horizon.

Ribbon smoothing, curve tessellation, width-over-life/parametric width curves (fixed body width plus
endpoint taper only), per-control-point twist, facing modes beyond camera-facing width, and
Niagara's link-order policy variants are not yet implemented. Each segment also computes its side
vector independently; at sharp curves with wide ribbons this can expose cracks, whereas Niagara
averages point tangents across adjacent segments. M7 batch 1 retains the current one-render-module-
per-emitter limit; per-draw indirect slots remain a later M7 batch.

## 10. Effect, timeline, and runtime API

### 10.1 Effect composition

```ts
const effect = defineEffect({
  elements: {
    arc,
    sparks,
    flash,
    shockwave,
  },
  timeline: [
    at(0.0, play('flash')),
    at(0.05, play('arc'), cameraShake({ strength: 0.3 }), hitStop(40)),
    at(0.08, play('sparks')),
    at(0.1, play('shockwave')),
  ],
});
```

Typed public parameters are declared in the same definition:

```ts
const chargedEffect = defineEffect({
  elements: { flash },
  parameters: {
    'User.intensity': defineParameter('User.intensity', {
      type: 'f32',
      default: 1,
      mutable: true,
    }),
  },
});
```

`defineEffect()` infers element keys. `play()` and `stop()` preserve literal target names so TypeScript and the compiler can reject unknown targets. Timeline entries are sorted by time during validation; author order breaks ties.

### 10.2 Timeline actions

```ts
function at(timeSeconds: number, ...actions: TimelineAction[]): TimelineEntry;
function play<Target extends string>(target: Target): PlayAction<Target>;
function stop<Target extends string>(target: Target): StopAction<Target>;
function cameraShake(options: {
  strength: number;
  duration?: number;
  frequency?: number;
}): CameraShakeAction;
function hitStop(durationMs: number, timeScale?: number): HitStopAction;
```

Timeline time is effect-local time. `hitStop()` affects the effect-local clock and integrations explicitly subscribed to that effect; it must not freeze unrelated browser or game clocks. `cameraShake()` produces an integration command consumed by a registered camera adapter rather than mutating an assumed global camera.

Callbacks from timeline/event modules use registered callback references. They are queued to a documented synchronization point; arbitrary JavaScript is never invoked from inside a GPU dispatch. `EffectInstance.on('death', callback)` is deliberately aggregate-only: callback receives `{ event: 'death', count }` accumulated since the previous event-counter readback, never a particle payload or one callback per particle. Event totals and overflow totals normally use the same `aliveCountReadbackInterval` cadence as authoritative alive count; emission completion and event-consumption frames force a final readback so the last aggregate is flushed before completion. When that option or renderer readback support is absent, GPU events still drive sub-emitters but no JavaScript event notification or overflow diagnostic is promised. This avoids an unconditional GPU/CPU synchronization point.

### 10.3 Runtime instances

`VFXSystem.spawn()` accepts immutable effect data and returns an `EffectInstance`. Position, rotation, seed, local time scale, and `User.*` overrides belong to the instance. Three-component rotations are Euler angles in radians and XYZ order; four-component rotations are quaternions. Spawn-time `parameters` and `instance.setParameter()` are mapped types derived from the definition, so an unknown path or wrong value type is a TypeScript error and remains a runtime validation error for untyped input. `setTransform()` updates `Emitter.transform` and supplies the delta used by per-distance spawning. `attachTo(source)` reads world position/quaternion from an external transform provider before initialization and every scheduled system step; `detach()` retains the last transform and stops polling. Backend sugar such as `createThreeTransformSource(Object3D | Bone)` adapts engine objects without importing them into core. `release()` returns pooled resources when safe. In M2, `stop()` means immediate scheduling stop: it does not advance or drain particles and retains resources until `release()`. A future explicit policy may add graceful drain without changing this default silently.

`VFXSystem.update(deltaSeconds?)` advances a system clock and schedules each live instance through the renderer-independent kernel adapter. Core accepts a structural `VfxRuntimeRenderer` bridge containing the M1 `KernelTslAdapter`, compute/indirect submission, optional readback/uniform/resource hooks, and device-loss notification; backend packages wrap their native renderer without importing it into core. `VFXSystem.setCamera({ viewMatrix, projectionMatrix, viewportSize })` copies column-major camera matrices and positive depth-texture dimensions into every current emitter and initializes later spawns with the same state. A fixed step partitions unscaled world delta first. Each instance then multiplies that substep by its `timeScale` (and active hit-stop scale) once to produce `Emitter.deltaTime`; `System.deltaTime` and `System.time` remain unscaled. Excess fixed-step backlog is discarded explicitly rather than producing an unbounded frame.

M2 uses a scattered WebGPU free list. Update atomically appends dead physical indices; Spawn allocates from the current tail; a separate atomic compaction pass writes alive particle indices and the authoritative alive count. Four counters, the free-list region, and the compact alive-index region share one writable lifecycle state buffer. Spawn-dispatch and draw-indirect arguments share a second indirect-argument buffer so an indirect source is never also writable storage in the same compute-pass synchronization scope. The draw arguments begin at the byte offset exposed by `BuiltEmitterKernels.drawIndirectOffsetBytes`; core intentionally leaves that region's word 0 (`indexCount`) untouched, so the M3 renderer must prime it once on the CPU before the first draw and bind the same resource with that offset. Renderer modules consume the compact index indirection using `aliveIndicesOffset` when binding particle attributes. WebGL2, which lacks the required atomics/indirect calls, has only an explicitly declared reduced `burst` path, and its Spawn/Init transform-feedback varying budget is validated before node materialization. Schemas over the backend limit and all `rate`/`perDistance` emitters are rejected with `NACHI_BACKEND_SPAWN_UNSUPPORTED` rather than silently degrading. `CompiledEmitterMeta.capabilities` and `backendBudgets` make these distinctions inspectable.

Spawn executes before Update in a scheduler step, so particles born in that step receive one full `Emitter.deltaTime` of aging immediately. If `$age` reaches lifetime in a step, compiler-owned death handling skips the remaining author Update modules for that particle and recycles it before the step's alive compaction.

Exact completion is opt-in with `aliveCountReadbackInterval`: after emission completes, a zero authoritative alive count completes the instance. Without it, core conservatively drains for the maximum statically provable lifetime; an unprovable or undeclared lifetime is `Infinity`.

## 11. Serialization and registries

### 11.1 Canonical document

The asset envelope is versioned independently from modules:

```ts
interface VfxAssetDocument {
  kind: 'vfx-effect';
  schema: 'com.nachi.effect';
  schemaVersion: number;
  effect: JsonValue;
}
```

Serialization canonicalizes helper output, expands shorthand where required, records module versions, and rejects non-JSON values. Object key order is not semantic. Timeline array order and module array order are semantic.

### 11.2 Resources

Textures, meshes, SDFs, vector fields, VAT data, and other external assets use serializable references such as:

```ts
interface AssetRef<AssetType extends string> {
  kind: 'asset-ref';
  assetType: AssetType;
  uri: string;
}

type TextureRef = AssetRef<'texture'>;
type FieldRef = AssetRef<'vector-field'>;
type MeshRef = AssetRef<'mesh'>;
type SdfRef = AssetRef<'sdf'>;
```

The runtime resolves refs through an asset resolver. Raw `THREE.Texture` or `Object3D` values may be wrapped for ephemeral code-only effects, but serialization must reject them until replaced by refs.

ASCII FGA vector fields are parsed by the renderer-independent `parseFga()` helper. The header is
three positive integer resolution values followed by minimum and maximum vec3 bounds; exactly
`resolution.x * resolution.y * resolution.z` vec3 samples follow in X-major order.
`vectorField({ field, strength, tiling })` stores only its serializable `FieldRef`. During kernel
materialization the backend resolver maps that ref to a bounds-aware 3D texture. Particle world
position is normalized by the bounds. FGA samples are corner-aligned: normalized coordinate zero
addresses the first texel center and one addresses the last, using the per-axis transform
`u * (resolution - 1) / resolution + 0.5 / resolution`. The default clamps normalized positions
before this transform. `tiling: true` leaves coordinates unbounded and uses repeat sampler wrapping,
including interpolation across the last/first texel seam; it does not implement repetition as
`fract()` followed by a clamped sample. Float32-filterable backends use trilinear sampling; other
backends fall back to nearest sampling. The sample is treated as acceleration and integrated with
`Emitter.deltaTime`.

`MeshRef` surface resources contain a deformed triangle-position texture and normalized area CDF
texture. `SdfRef` resources contain float distance samples plus bounds/resolution metadata. Both
are resolved by backend integration and remain serializable references in effect definitions.

### 11.3 Executable registrations

There are three executable registry categories:

1. module compiler implementations (`type` + `version`);
2. TSL factories (`id` + `version`);
3. JavaScript callbacks/integration commands (`id` + `version`).

The three categories use distinct reference types: module lookup uses `(type, version)`, TSL uses `TslFunctionRef`, and JavaScript/integration callbacks use `CallbackRef`. `VfxRegistry` exposes `resolveModule`, `resolveTsl`, and `resolveCallback` accordingly; one category's reference cannot resolve through another category.

The JSON contains references only. Registries are supplied by application/package setup and are not embedded in assets. Deserialization may produce a parsed document before all registrations exist, but compilation cannot succeed with unresolved required registrations.

Registration conflicts for the same ID/version are errors unless the implementation identity is exactly the same registration object. Version lookup is exact, not semver-compatible: a request for version 1 is incompatible when only version 2 is registered. Migration from an old version is explicit and belongs to `@nachi/format`; after migration the document references the exact registered version.

The integrated `VfxRegistry` implementation is introduced in M12 together with serialization and executable-reference loading. M1 uses `KernelModuleRegistry`; it already enforces the same module-category conflict rule (an identical registration object is idempotent, while replacement at the same type/version is an error) and exact version lookup.

### 11.4 Inline function policy

`tslModule(callback)` and future callback-taking authoring helpers are marked non-serializable until registered. The callback may be held as non-enumerable authoring metadata, but it is not JSON. `serializeEffect()` reports its exact path and suggests the registration API.

## 12. Error model

### 12.1 Structured diagnostics

Compiler and loader operations return or throw diagnostics with:

```ts
interface VfxDiagnostic {
  code: string;
  severity: 'error' | 'warning';
  phase: 'compile' | 'deserialize' | 'runtime' | 'serialize';
  message: string;
  path?: string;
  hint?: string;
}
```

Public diagnostic codes are stable enough for tests and tooling. Human messages may improve without a breaking release.

### 12.2 Compile-time validation

The following are compile errors before GPU work is submitted:

- unknown module, function, callback, asset, attribute, parameter, event target, or timeline target;
- version mismatch without a registered migration;
- invalid capacity, lifecycle, curve, gradient, or renderer config;
- attribute type/default mismatch or incompatible built-in redeclaration;
- undeclared required read, illegal stage write, read/write trace mismatch, or dependency cycle;
- backend capability required with no fallback;
- non-deterministic or unsupported TSL graph;
- unresolved inline function during serialization;
- resource layout exceeding known device limits.

Validation should accumulate independent diagnostics rather than fail at the first module. No partially compiled effect enters the active scheduler.

### 12.3 Runtime validation

Runtime errors are reserved for state that cannot be known at compile time:

- device loss or renderer/backend removal;
- asset load failure after a ref was valid;
- type-invalid or immutable `User.*` parameter writes from application code;
- spawn after system disposal or use of a released instance;
- runtime device limit changes and external integration failures.

Device loss and other runtime failures transition every affected `EffectInstance.state` to `'error'` and emit a `VfxDiagnostic` with phase `'runtime'`. They must not continue reporting plausible FPS or silently switch semantic behavior. Development builds SHOULD include diagnostic paths; production builds MAY use compact codes.

Released-instance methods (`setParameter`, `setTimeScale`, `setTransform`, `applyHitStop`, `stop`, and emitter inspection) throw `NACHI_INSTANCE_RELEASED`; `release()` itself remains idempotent. Device loss is latched by `VFXSystem`: existing and subsequently spawned instances receive `NACHI_DEVICE_LOST` directly instead of a secondary materialization/submission error.

## 15. Differences from PLAN.md

There is no intended surface-syntax divergence from the north-star sketch. The compile test uses its emitter, mesh element, effect, timeline, runtime spawn, and inline TSL forms.

This RFC makes two boundaries that were implicit in PLAN.md explicit:

1. `tslModule(callback)` cannot itself be JSON. It is accepted for authoring but must be converted to a stable registered function reference before serialization. This is required to satisfy both the inline escape hatch and the rule that serialized module configuration is data.
2. Names such as `explosionTex` and `noiseTex` in the sketch are typed as `TextureRef` for serializable effects. Passing a raw Three.js texture is allowed only for an explicitly ephemeral, non-serializable definition.

These are serialization constraints, not changes to the composition or stage model.

## 16. Open decisions

M4 visual regression screenshots use a changed-pixel ratio tolerance below `0.5%` rather than PNG
byte equality. The firefly additive pass can compact equal simulation state into a different draw
order, and floating-point additive blending then changes a small set of output pixels without a
behavioral regression. Exact simulation/readback checks remain deterministic and byte-based.

The following decisions are deliberately deferred. Implementations must not bake an undocumented answer into public API before the named spike is reviewed.

### Spike 1: 100k-particle TSL compute

SoA is constrained by the device's per-stage storage-buffer limit (the WebGPU default is 8, while an adapter may expose a higher opt-in limit). When a resolved schema exceeds the active device limit, compilation reports `NACHI_STORAGE_BUFFER_LIMIT`.

決定(2026-07-11): SoA-per-attribute+予算診断を基本構成として維持。vec4パッキング(スカラー/小型属性の共有vec4配列への束ね)はM3バッチ1でレンダラ頂点ステージのストレージバッファ予算と統合設計して実装する

追記(2026-07-11、M3バッチ1): vec4パッキングは既定で有効。`f32`/`vec2`/`vec3`、`i32`、`u32`/`bool` の3つの成分型ドメインを別々の共有vec4配列(`vec4`/`ivec4`/`uvec4`)へ格納し、型のビット再解釈は行わない。解決済み論理属性の決定的順序で、最初に収まるvec4グループの最小連続レーンへ割り当てる(first-fit、vec4境界を跨がない)。1粒子あたり複数グループが必要な場合も同じ物理バッファ内のストライドとして保持する。`color`/`quat`/`vec4`/`mat3`/`mat4` は専用バッファを維持する。論理属性から物理buffer/group/laneへの対応は `ResolvedAttribute.physical` と `CompiledEmitterProgram.meta.storageBuffers[].attributes` に公開するが、カーネルおよびレンダラは論理属性名だけを読み書きする。

- structure-of-arrays vs interleaved/hybrid storage;
- alignment and packing rules exposed in diagnostics;
- workgroup size and dispatch partitioning;
- whether all custom attribute types work in storage buffers on the pinned Three.js version;
- compiler tracing mechanics for TSL reads/writes;
- automatic derivation of `TslParticleBindings` custom keys and value types from an emitter's attribute schema;
- practical buffer/device limits and capacity diagnostics.

### Spike 2: `drawIndirect` / `dispatchIndirect`

M2 resolves the core decisions: physical slots use an atomic free list because deaths are scattered and stable physical indices keep SoA storage and deterministic identity cheap; a compact alive-index region supplies dense render/dispatch order. The index regions and allocator/authoritative-alive/success/overflow counters share one writable lifecycle state buffer. Spawn-dispatch and draw-indirect arguments share a second, separate indirect-argument buffer so an indirect source is never also writable storage in the same compute-pass synchronization scope. Core owns the argument regions and updates them after Spawn/Update compaction. CPU readback is permitted for diagnostics, opt-in low-frequency completion, and the reduced WebGL2 count path. Births and deaths are visible to renderers after the current scheduler step's compaction.

Deterministic allocation order across parallel adapters is intentionally unresolved. A future opt-in may assign persistent particle IDs or use a deterministic allocation pass; its cost, serialized representation, and interaction with events/sim cache remain open.

追記(2026-07-11、M7バッチ1): リボンはallocation/compaction順を決定化せず、別の
`Particles.spawnOrder` とbirth-index ringを用いる。各spawn/event prepareの単一invocationが
連続範囲を予約し、spawn invocation indexを加えたキーを保存するため、physical slotのatomic
競合を順序意味論から分離できる。この決定はリボン順序だけを保証し、一般のfree-list割当順や
加算描画順を決定化するものではない。詳細・制約は§9.3。

### Spike 3: WebGL2 backend parity

M2's practical lifecycle runtime is not currently supported on WebGL2. The backend has no atomics or indirect operations, `rate` and `perDistance` are rejected, and a normal `burst` Init schema with lifetime state exceeds the usual transform-feedback separate-varying limit of 4. `buildKernels()` statically compares the required Spawn/Init varyings with the backend limit and rejects over-budget burst emitters with `NACHI_BACKEND_SPAWN_UNSUPPORTED`; it never submits a kernel whose writes would otherwise disappear silently. A small prefix-only burst schema may remain usable when it fits the measured limit, but it is the reduced path rather than lifecycle parity.

- which non-lifecycle module capabilities compile through the WebGL2 backend;
- whether a packed transform-feedback layout can expand the reduced burst tier safely;
- storage-texture fallbacks and limits introduced by later milestones;
- whether WebGL2 should add a CPU allocation cursor for `rate`/`perDistance` instead of the current explicit rejection;
- the serialized capability/quality declaration used to prevent silent semantic fallback.

### Spike 4: depth access and TSL post pipeline

M6 batch 1 resolves compute depth collision through an explicit previous-frame normalized-depth
color copy. Fragment soft particles continue using `linearDepth(viewportDepthTexture(screenUV))` in
their active render pass. Compute uses a separately owned sampleable texture plus System camera
uniforms, avoiding attachment feedback. Reverse-depth, MSAA depth resolve, WebGL2 parity, and shared
copy ownership across multiple effects remain future capability work and must be diagnosed rather
than inferred.

### Later milestone decisions

- custom-event TSL context materialization beyond the reserved `context.emitEvent(condition)` contract (post-M5);
- alpha sorting/OIT selection and its serialized quality policy (M10);
- effect/emitter inheritance patch format (M9/M12);
- stop/release semantics, pooling, and callback synchronization (M9/M11);
- sim-cache snapshot representation and custom/transient attribute rules (M11);
- grid/neighbor stage extension of `ModuleStage` (M12).
- vector-field `space` selection and an explicit field transform; `vectorField()` is currently
  fixed in world space with neither option.

Until resolved, these are compiler capability questions. They must not change the meaning of existing authoring data silently.
