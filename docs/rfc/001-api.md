## 1. Goals and non-goals

This RFC fixes the ideal authoring vocabulary before the compiler and runtime are implemented. Every later public API is expected to conform to it or update this RFC explicitly. The primary goal is to make a Niagara user's mental model transferable while keeping authoring code-first, framework-independent, and serializable.

The RFC defines:

- the data model accepted by the module compiler;
- stage ordering and declared data access;
- dynamic particle attributes and parameter namespaces;
- composition of emitters, non-particle elements, effects, and timelines;
- the boundary between JSON assets and executable TSL or JavaScript code;
- validation and diagnostic behavior;
- the minimum runtime surface of `VFXSystem`.

It does not choose GPU buffer layout, workgroup sizes, indirect drawing mechanics, the exact WebGL2 support subset, renderer algorithms, or editor UI. Those decisions belong to the M0 spikes and later milestones.

## 2. Design principles

### 2.1 Niagara stage semantics are the public execution model

An emitter is evaluated in `Spawn -> Init -> Update -> Event -> Render` order. A module belongs to exactly one stage in its serialized form. A registered implementation MAY support more than one stage, but each use resolves to one stage before compilation.

Stage names are stable asset-format vocabulary. Internal compiler passes MAY fuse or reorder work only when the declared read/write dependencies prove the result equivalent.

### 2.2 Configuration is data

Definitions returned by `defineEmitter()` and `defineEffect()`, module `config`, value generators, timeline entries, resource references, and access manifests MUST serialize to JSON without preserving JavaScript object identity.

Functions, Three.js resources, DOM objects, GPU objects, class instances, symbols, and cyclic graphs MUST NOT appear in the serialized document. Authoring helpers may be fluent functions, but their output must be data.

### 2.3 TSL is an explicit escape boundary

Custom shader behavior uses `tslModule()`. TSL factories execute only while a compiler builds a node graph; they do not execute once per particle in JavaScript. A serializable asset references a registered TSL factory by stable ID and version.

The inline `tslModule(callback)` form from PLAN.md remains available for local authoring, prototyping, and tests. Serialization MUST reject an unresolved inline factory rather than embed source code or silently drop it.

### 2.4 All shared data has a namespace

Compiler-visible values use `System.*`, `Emitter.*`, `Particles.*`, or `User.*` paths. Bare names are permitted only in local convenience positions such as `emitTo(..., { inherit: ['position'] })`; they are normalized to fully qualified paths during compilation.

### 2.5 Particle attributes are emitter-local and compiled

Built-in and custom attributes form an emitter's schema. The compiler derives a physical GPU layout from the attributes actually read or written by its modules and renderers. Two emitters are not required to share a layout.

Attribute type, default value, read/write declarations, alignment, and backend support are compile-time concerns. Runtime code cannot add an attribute to an already compiled emitter.

### 2.6 Determinism and explicit ownership

Init random generators MUST derive values from the emitter seed, the deterministic
`Particles.spawnOrder` birth key, and the module/sample slot. They MUST NOT include the physical
particle index or `Particles.spawnGeneration`. The allocator assigns `spawnOrder` before Init runs,
independently of the free-list slot selected for that birth. Reusing a physical slot therefore
receives a fresh Init sample, while equal seeds and spawn schedules reproduce the same samples even
when atomic free-list append order changes.

Public modules MUST NOT call `Math.random()`. Update-stage random generators derive values from the
emitter seed, `Particles.spawnOrder`, the module/sample slot, and `Emitter.updateRandomStep`. They
MUST NOT include the physical particle index or `Particles.spawnGeneration`. The update step is a
u32 ordinal modulo 2^32 for one runtime emitter. A new runtime, including a pooled-kernel checkout,
begins with current value zero. Immediately before each simulation Update submission, including
fixed/prewarm substeps and event-input fallback simulation, the runtime uploads and the dispatch
consumes the current value. Only after the renderer submission's await succeeds does the private
counter advance to the next value modulo 2^32. A rejected submission leaves the next value
unchanged. Compile-purpose Update submission during pipeline preparation does not consume or
advance this ordinal. A paused, fully hit-stopped, culled, zero-delta, or otherwise undispatched
interval also does not advance it. Lifecycle loops do not reset it: the sequence continues through
`0xffffffff` and naturally wraps to zero rather than restarting with `Emitter.localTime`.
`kernels.uniforms['Emitter.updateRandomStep'].value` is initialized to zero and, after a simulation
Update has run, remains the current value consumed by the last such dispatch; the private counter
already holds the next value. Therefore time-varying Update ranges change on every executed step,
while equal seeds, spawn schedules, and step/dispatch sequences reproduce equal values. A Spawn-policy
range is the pre-allocation exception and uses `Emitter.spawnGeneration`, because no particle birth
has been allocated at that stage yet. Event-spawn Init continues to use the Init birth-key contract.
The general physical allocation order is still not deterministic: when multiple particles die in
one dispatch, atomic append order is a parallel hardware race across adapters or executions. Init
and Update randomness are explicitly isolated from that race, but fully deterministic physical
allocation and order-dependent simulation remain a future opt-in mode (§16).

`VFXSystem` owns compiled resources, schedules effects, and supplies world/render integration. Effect instances own local time and `User.*` overrides. Emitters own particle state. Modules own no mutable runtime state unless the compiler allocates an explicit `Emitter.*` value for it.

Deterministic random streams identify a module slot from the module stage plus its non-empty `label` when present, otherwise from the stage plus its normalized index within that stage. The stage identity is always mixed into the slot, so equal indexes or labels in different stages do not alias the same stream. Labels MUST be unique within one emitter stage. Adding or reordering an unlabeled random module may change later unlabeled streams; a stable label preserves its stream across inheritance overrides and reordering within the same stage.

The PCG RXS-M-XS input preserves separate odd-multiplied axes for the birth key, emitter seed,
module/sample slot, and progression key before XOR mixing. Init fixes the progression key to zero;
Update uses `Emitter.updateRandomStep`; the pre-allocation Spawn CPU mirror retains its emission
generation. It is not an `order + stream` lattice, so adjacent orders, update steps, modules, or
sample offsets cannot receive shifted copies of neighboring streams by construction. As with every
32-bit hash, unrelated tuples may still collide with ordinary 32-bit probability; the contract is
stream separation, not global injectivity over the Cartesian product.

H1-10 compatibility note (2026-07-14): an external registered Init implementation that calls
`context.random()` MUST declare `Particles.spawnOrder` in both the module definition and registered
implementation `access.reads`; omitting it from either declaration fails materialization with
`NACHI_INIT_RANDOM_SPAWN_ORDER_ACCESS_REQUIRED` and a diagnostic that identifies the required
access update. `@nachi-vfx/format` serializes module access manifests, so a JSON asset exported before
this contract change may still contain the former
`Particles.spawnGeneration` Init read. Loading that asset against the updated implementation fails
closed with `NACHI_MODULE_ACCESS_MISMATCH` instead of silently rewriting its executable contract.
No asset migration is provided because all affected packages remain unpublished at version
`0.0.0`; such development assets must be re-exported or have their access manifest re-authored.

H2-3 compatibility note (2026-07-15): an external registered Update implementation that calls
`context.random()` MUST declare `Emitter.seed`, `Particles.spawnOrder`, and
`Emitter.updateRandomStep` in both the module definition and registered implementation
`access.reads`. Omitting any input fails materialization with
`NACHI_UPDATE_RANDOM_STABLE_KEY_ACCESS_REQUIRED`. When a legacy definition also fails to include
reads declared by the registered implementation, the generic `NACHI_MODULE_ACCESS_MISMATCH`
validation fails earlier, before implementation random tracing; a definition-side stable superset
paired with a legacy implementation reaches the Update-specific diagnostic instead. Both
one-sided mismatches therefore fail closed. Serialized development assets that still declare the
former `Particles.spawnGeneration` Update read fail under these registry access rules and must be
re-exported or re-authored. Inline `tslModule()` does not expose
`KernelModuleBuildContext.random()`; registered kernel implementations are the affected escape
hatch. The M1 all-slots compatibility Init path and event-spawn Init path do not consume
`Emitter.updateRandomStep` and retain their prior bits.

Labels beginning with `$` are reserved for compiler-synthesized modules and MUST be rejected on user-authored modules. This keeps compiler-owned identities stable without risking a collision with an author label.

## 3. Core concepts

### 3.1 Module

A **Module** is a versioned configuration object plus a registry-resolved compiler implementation. It declares its stage and data access:

```ts
interface ModuleDefinition<Stage, Config> {
  kind: 'module';
  type: string;
  version: number;
  label?: string;
  stage: Stage;
  config: Readonly<Config>;
  access?: {
    reads: readonly DataReference[];
    optionalReads?: readonly DataReference[];
    writes: readonly DataReference[];
  };
}
```

`type` is a stable registry ID such as `core/gravity`; it is not a JavaScript constructor name. `version` versions the module config independently from the asset schema.

`label` is serialized author intent and provides stable identity for inheritance patches, diagnostics, profiling, and deterministic RNG slots. An absent label uses the normalized stage index as its identity. `access` may be absent only on an authoring definition whose manifest is derived during normalization, notably a traced `tslModule`; every compiled and serialized module has a resolved access manifest.

Built-in helper calls such as `gravity(-9.8)` return module definitions with a fixed manifest. Third-party modules register their config validator, access manifest, and TSL compiler implementation under a namespaced type ID.

### 3.2 Emitter

An **Emitter** is a capacity-bounded particle simulation and its renderers. It contains a spawn policy, initialization modules, update modules, event handlers, render modules, optional custom attributes, and lifecycle configuration. `defineEmitter()` normalizes structural data but allocates no GPU resource. Definitions deliberately use structural typing; there is no runtime-only brand that authoring helpers fail to materialize.

### 3.3 Effect

An **Effect** is a named graph of elements plus a timeline. An element may be an emitter or a non-particle visual element supplied by packages such as `mesh-fx`, `trails`, or `post`. Element keys are the targets accepted by `play()` and `stop()`.

An effect definition is reusable immutable authoring data. A spawned effect instance contains mutable playback state and parameter overrides.

### 3.4 VFXSystem

`VFXSystem` is the runtime coordinator bound to a renderer and scene/world integration:

```ts
const fx = new VFXSystem(renderer, scene);
const instance = fx.spawn(effect, {
  position,
  rotation,
  seed: 42,
  parameters: { 'User.intensity': 1.5 },
});

instance.setParameter('User.intensity', 0.8);
instance.stop();
instance.release();
```

Compilation MAY be lazy on first spawn or explicit in a future API. Compilation failure MUST be observable before an instance reports itself active.

### 3.5 Namespaced parameters

- `System.*`: frame and world values shared across the system, for example `System.deltaTime` and `System.time`. Normally read-only to particle modules.
- `Emitter.*`: emitter-local counters, transform, age, loop state, and module scratch values.
- `Particles.*`: per-particle attributes stored in dynamic GPU buffers.
- `User.*`: typed inputs supplied by application code per definition or effect instance.

## 4. Stage and module contract

### 4.1 Stage order

| Stage  | Responsibility                                                             | Typical writes                                                  | Restrictions                                                       |
| ------ | -------------------------------------------------------------------------- | --------------------------------------------------------------- | ------------------------------------------------------------------ |
| Spawn  | Decide how many particles are allocated and when                           | `Emitter.spawnCount`, allocation metadata                       | Does not initialize arbitrary particle attributes                  |
| Init   | Initialize newly allocated particles                                       | `Particles.position`, `velocity`, `lifetime`, custom attributes | Runs once per spawn generation                                     |
| Update | Advance alive particle state                                               | Any declared mutable `Particles.*`/`Emitter.*` value            | Must use system/emitter local time, not wall-clock time            |
| Event  | Consume generated events and enqueue effects/sub-emitters/callback records | Event queues and explicitly declared inherited attributes       | Cross-emitter work becomes visible at a defined scheduler boundary |
| Render | Produce draw inputs and renderer state                                     | Render outputs only                                             | Simulation attributes are read-only                                |

The Update row describes the intended contract. M2 does not yet implement declarations or materialization for author-defined mutable `Emitter.*` values; author modules must not rely on such writes until that support lands. Compiler/runtime-owned paths are stricter regardless of stage: author modules cannot write `Particles.spawnGeneration`, `Particles.spawnOrder`, `Emitter.spawnCount`, or `Emitter.allocation.*`, and only Event modules can write `Emitter.events.*`.

The current core implementation is stricter than the general Spawn-stage contract above: only the three built-in spawn modules `core/burst`, `core/rate`, and `core/per-distance` may write `Emitter.spawnCount`. Other author modules cannot declare that write even when their stage is Spawn.

Events raised during a dispatch are buffered. They do not recursively execute an unbounded emitter chain in the same GPU dispatch. Event handlers write `Emitter.events.<eventName>` queue paths. M5 fixes consumption at the **next scheduler frame**. Each producer queue has two banks: frame N appends only to bank `N & 1`, while consumers read only bank `(N - 1) & 1`. Core clears the new write bank before any emitter advances, so effect element order cannot create same-frame recursion. A fixed-time substep is a scheduler frame; prewarm steps internal to one activation retain one bank and therefore preserve bounded overflow behavior.

`Emitter.eventPayload.<field>` is the typed, read-only view of a consuming event record. `emitTo(target, { inherit: ['position'] })` maps the producer's `Particles.position` snapshot to `Emitter.eventPayload.position`; after ordinary target Init runs, core overwrites target `Particles.position` from that field. Non-inherited attributes keep their normal Init results. The snapshot is taken in the compiler-owned death epilogue before the producer slot is recycled.

M2 compiles Spawn modules through the same versioned module registry and access validation used by Init/Update. Spawn modules execute in author order and add requests to `Emitter.spawnCount`; only the allocator may translate that count into particle indices. On WebGPU, a spawn-request uniform drives a GPU-written indirect dispatch, successful invocations pop scattered indices from the free list, increment `Particles.spawnGeneration`, and then run Init exactly once. Requests beyond the available free count are clamped, counted, and reported as `NACHI_SPAWN_CAPACITY_EXCEEDED`; they never wrap an atomic counter or overwrite an alive particle. CPU-side request clamping reports immediately. GPU-side overflow reporting requires `aliveCountReadbackInterval`, because without that explicit interval M2 performs no counter readback or GPU/CPU synchronization; “reported” in this paragraph is conditional on that option.

### 4.2 Read/write declaration

Every compiled module has a `ModuleAccess` manifest of fully qualified paths:

```ts
const gravityRegistration: ModuleRegistration<'update', GravityConfig> = {
  type: 'core/gravity',
  version: 1,
  stage: 'update',
  access: {
    reads: ['Emitter.deltaTime', 'Particles.velocity'],
    writes: ['Particles.velocity'],
  },
  compile(context, config) {
    // Build TSL nodes; no per-particle JavaScript callback.
  },
};
```

The compiler MUST:

1. resolve each path against built-ins, custom attributes, or parameter declarations;
2. reject a missing required read;
3. allocate values that are written and subsequently read;
4. verify stage write permissions;
5. detect read/write declarations that disagree with a traced TSL graph where tracing is available;
6. preserve author order for observable read-after-write and write-after-write dependencies;
7. report an unused declared custom attribute as a warning, not an error.

Item 4 is enforced for Spawn, Init, Update, Event, and Render manifests. Spawn may write only `Emitter.spawnCount` or compiler-owned `Emitter.allocation.*`; Init may write `Particles.*`; Update may write declared mutable `Particles.*`/`Emitter.*`; Event may write particle inheritance and its normalized event queue; Render may not write simulation paths.

During normalization the compiler recursively visits module config. Every nested `parameter('Namespace.name')` generator implicitly adds that path to `reads`; this is merged with and deduplicated against the module's declared manifest. Likewise, placement under `events.onDeath` replaces the authoring-only `Emitter.events.pending` placeholder with `Emitter.events.onDeath` in each handler's writes. Helper-derived access is canonical serialized data, not an exception to dependency analysis.

`optionalReads` allows capability-dependent inputs such as depth. A module implementation must define its fallback behavior when an optional value is absent. Resolution is intentionally asymmetric with required reads: a known built-in optional attribute is allocated, while an absent optional custom attribute is not diagnosed or allocated and the module fallback is used.

### 4.3 Reserved integration module

Velocity does not update position implicitly inside force modules. Unless `integration: 'none'` is selected, M1 compile-time normalization appends a compiler-owned module with stable type `core/integrate`, label `$integrate`, and this manifest. `defineEmitter()` only normalizes authoring data and does not synthesize or compose `$integrate`:

```ts
{
  reads: ['Emitter.deltaTime', 'Particles.position', 'Particles.velocity'],
  writes: ['Particles.position'],
}
```

The default integrator is semi-implicit Euler (`velocity` is finalized before position integration). Forces and the integrator read `Emitter.deltaTime`, not `System.deltaTime`: the emitter-local value includes effect `timeScale` and `hitStop`, so velocity and position cannot advance on different clocks. `System.deltaTime` remains the unscaled host-frame value for modules that explicitly need it. Supplying a custom position writer requires `integration: 'none'` or an explicit future integrator selection; the compiler must never apply both silently.

Core particle positions and velocities are stored in world space, matching `positionSphere()` which
applies the per-birth emitter transform during Init. `killVolume()` is the deliberate authoring
exception: its box, sphere, and plane dimensions are expressed in emitter-local space. The Update
implementation transforms each world-space `Particles.position` by the inverse of the shared H2-6
Update-midpoint transform before testing the volume. Consequently a spawned instance's translation
and rotation move the kill volume with the emitter; no scale is currently exposed by the public
emitter transform API. The plane normal is
normalized before evaluation, so `offset` remains a local-space distance; its inside half-space is
`dot(normalize(normal), localPosition) <= offset`. Death writes `Particles.alive = false` and is
recovered by the normal compiler-owned free-list epilogue.

`positionSphere({ radius, center?, arc?, surfaceOnly? })` authors its shape in emitter-local space.
`center` is a `ValueInput<Vec3>` and defaults to `[0, 0, 0]`. `arc` restricts sampling to a spherical
cap with `thetaMax: ValueInput<number>` in degrees (the same unit convention as
`velocityCone.angle`) around `axis`, which defaults to `[0, 1, 0]` in emitter-local space. Cap area
is sampled uniformly by drawing `cos(theta)` uniformly from `[cos(thetaMax), 1]`; when
`surfaceOnly` is false, the existing cube-root radial sample continues to provide volume-uniform
sampling in the spherical sector subtended by that surface cap, with the sphere center as its
vertex. Omitting both `center` and `arc` preserves the original full-sphere kernel graph and
deterministic output. Static center components must be finite, `thetaMax` must remain in `(0, 180]`,
and an authored axis must be finite and non-zero.

The exhaustive current module-space and unit inventory is [RFC 004](./004-module-spaces.md). Omitted
defaults are module-specific: the five H1-5 force/collider selectors default to `emitter`, while the
H2-6 `velocityCone` and `linearForce` selectors default to `world` for v1 compatibility. `gravity`
remains world-only. Fixed consumers such as scene depth, SDFs, vector fields, and decals are not
silently reinterpreted.

`turbulence()` samples deterministic 3D simplex noise in particle world space. Its octave count,
frequency doubling, amplitude halving, and per-channel offsets are compile-time/fixed constants;
neither wall time nor a random stream participates. Given identical initial particle storage and
dispatch order, two runs therefore build and execute the same node graph bit-for-bit.

`curlNoise()` uses the same centered `snoise` scalar and fixed per-channel offsets to construct a
3D potential field. Central finite differences evaluate `curl(psi)`; the resulting field is
divergence-free up to finite-difference and floating-point error. A measured, fixed
derivative-amplitude correction makes `strength` the approximate maximum acceleration, matching
the public normalization used by `turbulence()` without applying a position-dependent scale that
would reintroduce divergence. Frequency changes spatial scale without changing that normalization,
and no clock or random stream participates.

`vortex()` and `pointAttractor()` accept `space: 'world' | 'emitter'`, defaulting to `emitter`.
Emitter-space evaluation transforms the world-space particle position by
the inverse of the shared Update-midpoint transform, evaluates local centers and axes, then
transforms acceleration back to world space with direction weight zero using that exact same
transform expression. Translation therefore follows the emitter without contaminating
acceleration, while emitter rotation affects both the field and returned force.

`velocityCone()` and `linearForce()` accept the same selector but default to `world`, preserving
their v1 direction and acceleration. With explicit `space: 'emitter'`, cone direction uses that
birth's `Emitter.spawnInterpolatedTransform`, while linear force uses the shared Update midpoint;
both use direction form (`w = 0`) so translation cannot enter a vector. `gravity()` remains fixed
world-space acceleration and has no selector. An authored selector outside the two literals is
rejected as `NACHI_MODULE_SPACE_INVALID`.

The H2-6 implementations of `velocityCone`, `linearForce`, `vortex`, `pointAttractor`, the three
analytic colliders, and `killVolume` are module version 2. Their version-1 registry entries preserve
pre-H2-6 execution: cone/linear are world-space even if a generic old asset contains an ignored
emitter field, and emitter-space Update consumers sample the current endpoint. Current helpers emit
version 2. Format load/serialize preserves a v1 record as v1 rather than implicitly upgrading its
meaning, while a pre-H2-6 reader safely rejects `type@2` as `NACHI_MODULE_UNKNOWN`. The asset
envelope could consequently remain version 1 when H2-6 was implemented without violating RFC 003
§4. H2-7 later raised the canonical envelope to version 2 for the renderer boundary; its default
envelope-only migration preserves these H2-6 module records and versions unchanged.

M6 analytic colliders use the same explicit `space: 'world' | 'emitter'` convention as those force
modules and default to `emitter`, matching the emitter-local-only `killVolume()` authoring rule.
Emitter-space analytic colliders evaluate and write their response through the same shared
Update-midpoint transform; inverse and forward conversion MUST NOT independently resample it.
`collidePlane()` treats `dot(normalize(normal), position) < offset` as penetration.
`collideSphere()` and `collideBox()` treat their volume as solid and push an interior particle to
the nearest surface. `mode: 'bounce'` preserves an outward normal component, reflects an inward
component with `bounce` restitution, and multiplies tangent velocity by `1 - friction`; `stick`
zeros velocity and `kill` recycles the particle. Both coefficients are clamped to `[0, 1]` at
materialization and statically invalid constants are diagnosed. Collision position correction is a
sanctioned author Update write and may precede compiler-owned `$integrate` without triggering the
custom-integrator double-write diagnostic.

`collideSceneDepth()` uses the same response modes in world space. Compute does not read
`viewportDepthTexture` directly: that node is render-pass scoped and sampling an active attachment
would create a feedback hazard. Renderer integration copies completed scene-pass depth into a
normalized, sampleable color texture and binds that previous-frame copy to the kernel adapter. The
particle is projected with `System.viewMatrix` and `System.projectionMatrix`. The latter is
explicitly a WebGPU-coordinate-system projection matrix: NDC depth is `z ∈ [0, 1]`, so projected
depth is used directly and inverse projection receives sampled depth directly (no OpenGL
`z * 0.5 + 0.5` / `z * 2 - 1` conversion). The depth copy must preserve normalized depth in a
linear, non-sRGB float color texture; output color transforms and normalized 8-bit render targets
are invalid because they encode and quantize depth. `System.viewportSize` supplies depth-gradient
texel offsets. Neighboring depth samples are inverse-projected to reconstruct a view-space normal,
which is transformed back to world space.

The x/y conversion is deliberately asymmetric. Three r185 `screenUV` and copied render-target
textures use a top-left origin (`v` grows down), while WebGPU NDC `y` grows up. Projecting a particle
therefore uses `uv = (ndc.x * 0.5 + 0.5, 0.5 - ndc.y * 0.5)`, and reconstruction uses
`ndc = (uv.x * 2 - 1, 1 - uv.y * 2)`. The y `oneMinus` operation is mandatory in both directions.
The `/m6-collision/` regression scene rotates an occluder around X and tests a particle at nonzero
y; symmetric frontal or Y-rotated planes cannot detect this mirror error.

On three.js r185, a `pass(scene, camera)` node is not a deterministic way to drive this copy outside
the presentation loop: `PassNode` uses `FRAME` update-before semantics, and `NodeFrame` suppresses a
second update for the same frame id, so repeated `RenderPipeline.render()` calls can silently sample
the prior scene configuration. The recommended integration instead attaches a non-MSAA
`DepthTexture` to a `RenderTarget`, selects that target and calls `renderer.render(scene, camera)`
explicitly for every capture, then runs a separate TSL copy pipeline that samples the depth texture
into the linear, non-sRGB float color target described above.

M6 v1 therefore requires WebGPU, a non-MSAA resolved depth copy, matching capture/simulation camera
and viewport, and accepts one-frame latency for moving geometry. A missing binding is diagnosed as
`NACHI_SCENE_DEPTH_UNAVAILABLE`, WebGL2 and MSAA sources are rejected explicitly, and an unset
camera or detected reverse-z projection produces a runtime warning; none are treated as silent
fallbacks.

`collideSceneDepth({ thickness })` rejects penetration whose linear view-depth difference exceeds
`thickness` (default `0.1`). This prevents a particle already hidden far behind foreground geometry
from teleporting to that foreground surface. `surfaceOffset` remains the small world-space
separation applied only after an eligible collision.

`collideSdf()` samples a serializable `SdfRef` in world space. Negative distance is penetration;
the normal is the normalized central-difference gradient, and correction moves the particle by the
signed distance to the zero surface. Optional `thickness` rejects deeply embedded samples in the
same way as scene depth. The public `bakeSdf()` utility creates corner-aligned float32 fields from a
union of analytic spheres and boxes for tests and procedural assets. Mesh-to-SDF baking is deferred
to M12 because its voxelization and watertightness policy is an asset-pipeline decision.

`positionMeshSurface({ mesh, mode: 'surface' })` samples an area-weighted triangle CDF with three
deterministic particle random samples, writes world-space position, and retains the sampled normal
in compiler-known `Particles.surfaceNormal`. `velocityMeshNormal({ speed })` consumes that normal.
The backend resource stores triangle vertices in **mesh-local space**, including CPU-skinned
deformed positions before the object transform. The Init kernel applies `Emitter.transform` exactly
once to produce world-space `Particles.position` and applies its direction form (`w = 0`) to the
sampled normal. Mesh resources therefore MUST NOT pre-apply `matrixWorld`; doing so would double
transform static and skinned samples. The CDF is built from those same local-space deformed
triangles, so its probabilities remain proportional to the geometry actually sampled.
The Three v1 integration rebuilds triangle positions and area CDF on the CPU from
`SkinnedMesh.applyBoneTransform()` and uploads both float textures each frame. This deliberately
keeps bone textures out of core and gives static and skinned meshes one `MeshRef` contract, at the
cost of CPU skinning/upload proportional to source triangle count; large crowds require a future
GPU-skinning data interface. A resource keeps fixed source topology and uses flat triangle normals;
topology changes require rebuilding the resource, and smooth deformed vertex-normal interpolation
is deferred with the GPU-skinning path. The v1 triangle-position texture is a single row containing
three texels per triangle, so `3 * triangleCount` MUST NOT exceed the active device's
`maxTextureDimension2D`. The Three helper accepts that limit and defaults to the WebGPU guaranteed
minimum of 8192 texels (at most 2730 complete triangles); it reports
`NACHI_MESH_SURFACE_TEXTURE_TOO_WIDE` before allocating an oversized texture. Multi-row/2D tiled
triangle and CDF textures are deferred to M12.

The public `turbulence.strength` approximates a maximum acceleration magnitude, not a raw simplex
multiplier. Octaves are amplitude-sum normalized, centered Three r185 `snoise` is multiplied by
`1 / 0.286` (its measured effective scalar amplitude is approximately `±0.286`), and the resulting
three-component vector is length-clamped to one before applying `strength`.

The `$integrate` label uses the compiler-reserved `$` prefix. User-authored module labels MUST NOT begin with `$`; normalization reports a compile diagnostic rather than renaming them.

The same M1 compile-time normalization materializes two additional compiler-owned modules in the kernel description. `$defaults` (`core/defaults`) is the first init module and writes every resolved `Particles.*` attribute from its declared default. When `age` and `lifetime` are present in the resolved lifecycle schema, `$age` (`core/age`) is the first update module and has this manifest:

```ts
{
  reads: ['Emitter.deltaTime', 'Particles.age', 'Particles.lifetime'],
  writes: ['Particles.age', 'Particles.normalizedAge'],
}
```

`$age` advances age and derives clamped normalized age before author update modules. These operations are described modules rather than hidden statements injected while materializing TSL kernels.

### 4.4 Module config and implementation separation

The serialized module contains `type`, `version`, optional `label`, `stage`, `config`, and resolved `access`. Executable validation and compilation functions live in a `VfxRegistry`. This makes the asset JSON inspectable and prevents bundler-specific function names from becoming format identifiers.

### 5.1 Built-in attributes

The baseline particle schema includes:

| ------------------------------------ | ---------------- | ---------------------------------------------------------- |
| `position`                           | `vec3`           | emitter-local or world position according to emitter space |
| `velocity`                           | `vec3`           | linear velocity                                            |
| `age` / `lifetime` / `normalizedAge` | `f32`            | lifetime evaluation                                        |
| `alive`                              | `bool`           | lifecycle state; defaults false until allocation           |

| `color`                              | `color` (`vec4`) | linear RGBA                                                |
| `intensity`                          | `f32`            | light renderer intensity and priority                      |

Built-ins are logical API names, not a fixed physical struct. The compiler MAY omit unused built-ins. `normalizedAge` MAY be derived instead of stored.

### 5.2 Custom declarations

Custom attributes are declared on the emitter:

```ts
const heat = attribute('heat', {
  type: 'f32',
  default: 0,
});

const embers = defineEmitter({
  capacity: 4096,
  attributes: { heat },
  spawn: burst({ count: 128 }),
  init: [
    /* modules that initialize heat */
  ],
  update: [
    /* modules that read/write heat */
  ],
  render: billboard({ blending: 'additive' }),
});
```

Supported logical types initially are `f32`, `i32`, `u32`, `bool`, `vec2`, `vec3`, `vec4`, `color`, `quat`, `mat3`, and `mat4`. Backend layout rules are compiler-owned. Attribute defaults MUST be deterministic and type-compatible.

Redeclaring a built-in name with a different type is an error. A custom attribute that may remain unwritten needs a default. Attributes marked `transient` are excluded from sim caching and serialization of particle snapshots; the compiler must reject a transient attribute required by a cache replay path.

The `attributes` record key is canonical and MUST exactly match `AttributeDefinition.name` (`attributes.heat.name === 'heat'`). Authoring normalization rejects a mismatch; it never renames one side silently. The duplicated name is retained so a standalone attribute definition remains self-describing when moved, inspected, or referenced by tooling.

## 6. Namespaced parameters

Parameter declarations use a full path and logical type:

```ts
const intensity = defineParameter('User.intensity', {
  type: 'f32',
  default: 1,
  mutable: true,
});

const gravityScale = parameter('User.gravityScale', 1);
```

| Namespace     | Declared by                  |      Mutable from application code | Lifetime                               |
| ------------- | ---------------------------- | ---------------------------------: | -------------------------------------- |
| `System.*`    | runtime/compiler             | no, except system integration APIs | VFXSystem/frame                        |
| `Emitter.*`   | compiler/module registration |            no direct public writes | emitter instance                       |
| `Particles.*` | built-ins/emitter attributes |            no direct public writes | particle                               |
| `User.*`      | effect/emitter author        |          yes when declared mutable | definition default + instance override |

Unknown paths are compile errors. Runtime writes to unknown, immutable, or type-incompatible `User.*` parameters are runtime validation errors. Parameter namespaces are case-sensitive and reserved exactly as written.

Effect and emitter definitions may both declare `parameters`. Effect declarations define the public instance surface; emitter declarations define reusable emitter-local defaults and are mapped into the containing effect during M9 composition. `EffectInstance.setParameter()` and spawn-time `parameters` derive their path and value types from the effect's declared `User.*` schema. The exact override/forwarding rule between effect and child-emitter declarations remains an M9 decision.

M9 batch 1 resolves that rule. `defineEffect()` lifts every child-emitter `User.*` declaration into
the effect's public parameter schema. Declarations of the same lifted path must have identical
logical type, default, and mutability. An explicit effect-level declaration is authoritative for
default and mutability, but every child declaration of that path must still have the same logical
type. The composed schema is forwarded into each child before emitter compilation, so an explicit
effect-level declaration materializes as a child uniform and one runtime write can update every
materialized child uniform. A generator authored inside a standalone emitter still declares its
path on that reusable emitter so `defineEmitter()` can validate it before composition. Conflicts
report `NACHI_EFFECT_PARAMETER_CONFLICT` rather than selecting an element by object/property order.

Spawn-time overrides may set a declared immutable parameter for that instance; subsequent
`setParameter()` calls require `mutable: true`. Both paths validate the logical scalar/vector/matrix
shape at runtime in addition to their mapped TypeScript types. Every effect instance owns its own
value record and uniform nodes, even when definitions, compiled programs, or pooled GPU resources
are shared. Composition runtimes such as `@nachi-vfx/timeline` MUST perform the same validation against
the composed effect schema before retaining a write, even when no child emitter is currently
materialized; forwarding to active children is not the validation boundary.

### 6.1 Reserved `Emitter.*` paths

| Path                                 | Meaning                                                                          |
| ------------------------------------ | -------------------------------------------------------------------------------- |
| `Emitter.transform`                  | emitter-to-world transform                                                       |
| `Emitter.previousTransform`          | composed emitter-to-world transform used by the preceding simulation step        |
| `Emitter.spawnInterpolatedTransform` | compiler-provided per-birth transform between previous and current transforms    |
| `Emitter.updateInterpolatedTransform` | compiler-provided phase-0.5 transform shared by Update consumers                  |
| `Emitter.interpolationActive`        | compiler-owned `0`/`1` exact-transform-change flag                               |
| `Emitter.localTime`                  | emitter-local playback time                                                      |
| `Emitter.deltaTime`                  | emitter-local scaled delta                                                       |
| `Emitter.updateRandomStep`           | current u32 ordinal modulo 2^32 consumed by a simulation Update random stream    |
| `Emitter.age`                        | emitter lifecycle age                                                            |
| `Emitter.loopIndex`                  | current loop number                                                              |
| `Emitter.seed`                       | deterministic emitter seed                                                       |
| `Emitter.spawnGeneration`            | current emission batch/loop generation (not the particle RNG generation)         |

Packages may reserve additional paths only through an RFC update. Application-defined emitter parameters must use a package-qualified suffix to avoid collisions.

`Emitter.events.pending` is never present in a compiled or serialized normalized manifest. Event placement supplies the concrete name, replaces the placeholder with `Emitter.events.<eventName>`, and then dependency analysis sees only the concrete queue.

M5 reserves `Emitter.eventPayload.<field>` only for fields listed by `emitTo(..., { inherit })`. Other suffixes are unknown paths, and all event-payload fields are compiler-owned read-only values. The producer and target attributes must exist with identical logical types and component counts or compilation fails. Float-domain fields (`f32`, `vec2`, `vec3`, `vec4`, `color`, and `quat`) of up to four components are packed in first-use order into vec4 groups; a field never straddles groups. Integer, boolean, and matrix payloads are rejected in M5 rather than reinterpreted silently. `CompiledEmitterMeta.eventQueues` exposes each field's group/lane, logical type, queue capacity, handlers, and payload group count.

## 7. Value generators

Generators are immutable JSON values evaluated by a stage compiler. Their conceptual signatures are:

```ts
function range<T extends number | Vec2 | Vec3 | Vec4>(min: T, max: T): RangeGenerator<T>;

function curve<T extends number | Vec2 | Vec3 | Vec4>(
  first: readonly [time: number, value: T],
  second: readonly [time: number, value: T],
  ...rest: readonly [time: number, value: T][]
): CurveGenerator<T>;

function gradient(first: ColorInput, second: ColorInput, ...rest: ColorInput[]): GradientGenerator;

function parameter<T extends JsonValue>(path: ParameterPath, fallback?: T): ParameterGenerator<T>;
```

`range()` uses deterministic uniform random selection by default. Its random stream slot combines the containing stage with the module's `label` when present and otherwise with its normalized stage index. In Init, the value is keyed by `Particles.spawnOrder` and the emitter seed. In Update, it is keyed by `Particles.spawnOrder`, emitter seed, and `Emitter.updateRandomStep`; physical slot and `Particles.spawnGeneration` never participate. Vector ranges sample each component from a distinct deterministic sample offset rather than broadcasting one scalar across every component. Future distributions must be explicit config fields rather than behavior changes.

The GPU mapping multiplies a 32-bit hash by `2^-32`. Its mathematical range is `[0, 1)`, but f32
rounding can produce exactly `1.0` with probability of roughly `3e-8`; shader-side consumers MUST
treat the contract as the closed interval `[0, 1]`. The JavaScript f64 mirror remains `[0, 1)`.

`curve()` key time is domain-independent. Modules such as `sizeOverLife()` define a normalized `[0, 1]` domain, while timeline/world-space consumers may define another domain. At least two keys are required. M1 supports linear interpolation only; omitted interpolation defaults to linear, while `constant`, `cubic`, and unknown interpolation values produce an explicit compile diagnostic.

`gradient(colorA, colorB, ...)` creates at least two evenly spaced stops. A future explicit-stop overload may accept `{ position, color }`; it must serialize to the same canonical stop array. String colors accept `#RGB`, `#RRGGBB`, `#RGBA`, and `#RRGGBBAA`; RGB channels are interpreted as sRGB and converted to linear values with the Three.js transfer function before LUT interpolation, while alpha is decoded directly to the linear `[0, 1]` range. Numeric color arrays are already-linear values and are not converted.

Curves and gradients are baked to LUT textures or buffers. Baking precision, resolution, and cache keys are compiler options, not hidden module behavior.

## 8. TSL escape hatch

### 8.1 Inline authoring form

The PLAN.md form is valid:

```ts
tslModule(({ position, velocity, age }) => ({
  velocity: velocity.add(myCustomField(position)),
}));
```

The factory receives typed TSL expressions, not CPU particle values. It returns a partial mapping of attributes to new expressions. Returned keys are writes. Accessed input bindings are reads. The compiler traces both and produces the final manifest. `tslModule()` defaults to the `update` stage; `stage: 'init'` opts into initialization.

Attribute expressions supplied to inline factories and by `KernelModuleBuildContext.attribute()` are read-only logical values. Modules MUST return an output mapping or call `KernelModuleBuildContext.write()` to update particle storage; calling `assign`, `addAssign`, or another in-place mutation on an attribute expression is invalid and materialization reports an error. This rule is independent of whether the compiler places the attribute in a dedicated or packed physical buffer.

The factory MUST be pure with respect to compilation: no frame-dependent JavaScript state, I/O, random calls, GPU submission, scene mutation, or resource allocation outside registered compiler facilities. It MAY compose TSL functions from `@nachi-vfx/tsl-kit` and the pinned Three.js adapter layer.

An optional `access` manifest may be supplied, with two deliberately distinct states:

- property absent: the manifest is unknown at authoring time and is derived completely from the traced TSL graph during normalization;
- property present, including `{ reads: [], writes: [] }`: the author supplied a declaration and the traced reads/writes must be subsets of it.

Normalization materializes the traced result before serialization. It never replaces an absent declaration with an empty declaration. Explicit access is required for dynamic property access or registry integrations that cannot be traced.

Built-in binding names have no open string index signature, so `velocty` is a type error. Custom bindings use declared keys of the form `custom.<name>`:

```ts
type HeatBindings = TslParticleBindings<{ heat: number }>;

tslModule<HeatBindings>((bindings) => ({
  'custom.heat': bindings['custom.heat'],
}));
```

Deriving `HeatBindings` automatically from an emitter's `attributes` schema is postponed to M2 or later and listed in §16. Until then, custom bindings are supplied explicitly.

Returned write keys are additionally resolved against the emitter schema during trace validation. TypeScript catches misspelled built-in input bindings and explicitly typed `custom.*` bindings, but it cannot prove every structurally inferred object key; a write-key typo that passes authoring type inference is therefore a compile-time trace-validation error, not an accepted custom attribute.

Binding-operation arguments accept existing TSL nodes unchanged and lower the following plain
authoring literals before the operation is materialized. This mapping is derived from the same
logical attribute-type table used for GPU layout:

| Input shape                       | TSL logical type |
| --------------------------------- | ---------------- |
| `boolean`                         | `bool`           |
| finite `number`                   | `f32`            |
| finite numeric array of length 2  | `vec2`           |
| finite numeric array of length 3  | `vec3`           |
| finite numeric array of length 4  | `vec4`           |
| finite numeric array of length 9  | `mat3`           |
| finite numeric array of length 16 | `mat4`           |

For example, `binding.velocity.add([0, 1, 0])` is equivalent to passing an explicitly constructed
`vec3` node. Strings, `undefined`, `null`, functions, and nodes are passed through unchanged for TSL
operations that use non-literal metadata. A finite numeric array of unsupported length, or a plain
object that is neither a node nor a supported literal, is rejected during trace compilation with
`NACHI_TSL_BINDING_INPUT_INVALID` and the module path. A numeric array containing a non-finite
component is also rejected. It never reaches GPU submission.

Known limitation: a mixed array containing a non-numeric element (for example, `[1, 'a']`) is
passed through so it can be distinguished from node-array metadata, and may therefore fail later
at GPU submission. Shape and finiteness validation is complete for all-numeric arrays.

### 8.2 Serializable registered form

```ts
const followField = defineTslFunction(
  'game/follow-field',
  ({ position, velocity }) => ({
    velocity: velocity.add(field(position)),
  }),
  1,
);

registry.registerTsl(followField);

const module = tslModule(followField.ref, {
  stage: 'update',
  access: {
    reads: ['Particles.position', 'Particles.velocity'],
    writes: ['Particles.velocity'],
  },
});
```

The JSON stores only `{ kind: 'function-ref', id, version }`. Deserialization resolves it through the registry. Registry IDs MUST be stable, namespaced, and explicitly versioned. Missing or version-incompatible entries are compile errors.

### 8.3 TSL version boundary

Public callbacks receive the library's `TslExpression<T>` facade. Direct Three.js TSL nodes may be admitted through a narrowly typed adapter, but raw Three.js node types are not serialized and are not promised semver stability by this library. The adapter is the only package that should change when pinned Three.js TSL APIs move.

## 9. Emitter API

The canonical shape is:

```ts
interface EmitterConfig<Attributes, Parameters> {
  capacity: number;
  attributes?: Attributes;
  offset?: Vec3;
  parameters?: Parameters;
  spawn: SpawnModule | readonly SpawnModule[];
  init?: readonly InitModule[];
  update?: readonly UpdateModule[];
  events?: Partial<Record<ParticleEventName, EventModule | readonly EventModule[]>>;
  integration?: 'euler' | 'none';
  render: RenderModule | readonly RenderModule[];
  lifecycle?: {
    duration?: number;
    loopCount?: number | 'infinite';
    prewarm?: number;
    startDelay?: number;
  };
}
```

`capacity` is a hard logical maximum and must be a positive integer. Spawn and event queues clamp overflow explicitly; silently corrupting or resizing GPU buffers is forbidden.

`offset` is a finite emitter-local translation of that emitter element's origin inside the effect
instance frame. Runtime materialization composes the single `Emitter.transform` uniform as
`instanceTransform * translate(offset)`. Init placement, every module using `space: 'emitter'`,
emitter-local kill volumes and bounds, sorting, and other transform consumers therefore observe the
same frame. Timeline `play()` passes the effect position and rotation to core unchanged, so the same
composition applies to immediately played and later-played emitter elements. Omission and an exact
zero offset preserve the unmodified instance transform.

Multiple modules in a stage run in author order unless dependency analysis proves a safe optimization. Multiple handlers for one event run in author order and each writes the normalized `Emitter.events.<eventName>` queue dependency. Renderer modules cannot mutate simulation attributes. M7 supports exactly one render module per emitter and diagnoses additional render modules with `NACHI_RENDER_MODULE_LIMIT`; multiple render modules require per-draw indirect-argument slots and are deferred to M8 or later as recorded in §16. `integration` defaults to `euler`; M1 compile-time normalization synthesizes and composes the reserved `$integrate` module from §4.3. Authoring-time normalization in `defineEmitter()` does not perform that composition.

M9 batch 1 emitter inheritance uses `defineEmitter(base, overrides)`. It does not use JavaScript
prototype inheritance and does not retain `base` object identity in the returned definition.
Authoring normalization eagerly resolves the chain into ordinary emitter data, so JSON output has
the same shape as a standalone `EmitterDefinition`; M12 may encode an asset-reference form that
normalizes to this same result.

`capacity`, `integration`, `offset`, and `spawn` are child-wins values (`spawn` is replaced as one policy,
never concatenated). Lifecycle is field-wise child-wins. Attributes and parameters merge by key,
but an inherited key cannot change logical type. Event maps merge by event name and the child's
handler value replaces that event as a unit.

`init`, `update`, and `render` accept an `EmitterModuleListOverride`:

```ts
const child = defineEmitter(parent, {
  update: {
    mode: 'merge', // default; alternatives are 'append' and 'replace'
    modules: [{ ...gravity(-4), label: 'gravity' }],
    remove: ['drag'],
    order: ['gravity'],
  },
});
```

`merge` matches a labeled child module to the same parent label and replaces it in place. An
unlabeled child module matches its normalized parent index; an unmatched child module appends.
`remove` accepts a label or normalized index. `order` moves the listed identities to the front in
the given order and leaves every unlisted module in stable relative order. `append` applies
removals and then appends all child modules without implicit replacement. `replace` replaces the
whole stack and therefore ignores `remove` and `order`. An explicitly `undefined` override is
treated as absent and does not erase the inherited `capacity`, `spawn`, lifecycle field, schema
entry, or event handler. Unknown selectors, duplicate/invalid order selectors, stage mismatches,
attribute or parameter type mismatches, and cyclic raw authoring graphs are compile diagnostics.
The inheritance diagnostics use the six stable names
`NACHI_EMITTER_INHERITANCE_BASE_TYPE_MISMATCH`,
`NACHI_EMITTER_INHERITANCE_CYCLE`,
`NACHI_EMITTER_INHERITANCE_STAGE_MISMATCH`,
`NACHI_EMITTER_INHERITANCE_TARGET_UNKNOWN`,
`NACHI_EMITTER_INHERITANCE_ATTRIBUTE_TYPE_MISMATCH`, and
`NACHI_EMITTER_INHERITANCE_PARAMETER_TYPE_MISMATCH`.

This is the code-first equivalent of Niagara emitter inheritance/versioning: inherited stack
entries remain visible and unchanged until a child explicitly overrides/disables/reorders an
identified entry. It deliberately avoids both blind array concatenation (which would double-apply
forces/renderers) and implicit whole-array replacement (which would make a small child edit copy
the parent stack).

An override addresses a labeled module by label. An unlabeled module is addressed by normalized stage index and is therefore less stable under edits. Duplicate labels in one stage are compile errors.

M2 lifecycle normalization uses `startDelay` before the first activation, runs an emitter for
`duration` seconds per loop, and interprets `loopCount` as the total number of activations including
the first (`'infinite'` never completes emission). When `lifecycle.duration` is undefined—whether
the lifecycle object itself is omitted or only partially authored—an emitter containing `rate` or
`perDistance` derives an unbounded active duration. A burst-only emitter instead derives a finite
duration as the largest authored `cycles * interval` envelope plus the maximum statically provable
particle lifetime as grace; without a multi-cycle burst, that derived duration remains zero. An
explicit finite numeric duration always wins in every authoring and asset-format path, and authored
`Infinity` remains invalid. `prewarm` advances the first activation by the declared number of
effect-local seconds before it becomes externally visible, and `startDelay` composes ahead of the
derived window. Each finite or infinite `loopCount` activation repeats a finite derived envelope.
For an omitted-duration continuous emitter, the first activation is itself unbounded, so even a
finite `loopCount` never reaches a later activation; `stop()`, timeline stop/end, or `release()` owns
termination. A zero-duration `at(0)` timeline shorthand has no later boundary and continues to let
the played child determine completion; a positive-duration final track boundary truncates it. A
looping emitter requires a positive explicit duration, a positive derived
multi-cycle burst envelope, or a duration-omitted continuous spawn policy.

Untrusted assets are rejected before allocation or synchronous kernel expansion when emitter
`capacity` exceeds `2^22`, PBD distance-constraint `iterations` exceeds `64`, or lifecycle `prewarm`
exceeds `300` seconds. Core compilation and format validation enforce the same limits.

M2 provides `burst({ count, cycles?, interval? })`, `rate({ rate })` (particles per emitter-local second), and `perDistance({ rate })` (particles per world-space unit travelled by `Emitter.transform`). Rate and distance modes retain fractional remainders; fixed timesteps therefore produce the same integer totals for equivalent delta partitions. Per-distance compares successive simulation-step transform snapshots and consumes the straight chord between them once. Multiple `setTransform()` calls before one simulation step do not accumulate a polyline; only the net displacement from the previously simulated transform to the final current transform contributes distance.

H1-7 interpolates transform-consuming Init placement within each ordinary spawn batch. Let `i` be
the birth's zero-based local spawn order, independent of the physical free-list slot and alive
compaction order. A `rate`/`burst` batch of `N` births uses `phase = (i + 0.5) / N`. For a
per-distance module with previous fractional remainder `r`, scaled rate `R`, and current segment
length `D`, birth `i` uses the accumulator's actual firing point,
`phase = ((1 - r) + i) / (R * D)`. Phases are clamped to `[0, 1]` only at GPU materialization to
cover the existing spawn epsilon. Multiple per-distance modules retain their own remainders and
batches; timed modules share one batch for the step.

For timed batches, the phase denominator `N` is fixed to the batch size after CPU-side clamping
against the request, physical capacity, and logical availability. GPU free-list saturation may
reduce the successful birth count further, but phases are not redistributed: successful births
retain the leading firing positions of that clamped batch. This matches the approved per-distance
saturation semantics of retaining the first available firing positions and dropping the rest
without redistribution.

`Emitter.spawnInterpolatedTransform` is a compiler-provided Init expression, not an author-writable
uniform. Its translation is `lerp(previous, current, phase)` and its rotation is shortest-path
quaternion slerp. Interpolation covers rotation and translation only because the public transform
API exposes no scale. `Emitter.previousTransform` is the complete composed transform after applying
the emitter-local `offset`. When the CPU observes exact equality between previous and current matrices,
`Emitter.interpolationActive` is zero and the kernel takes the existing direct
`Emitter.transform` path; no lerp/slerp expression executes. Thus stationary births preserve the
existing output bits and random stream. Event-consumed Init and the all-slots compatibility Init
have no timed/distance phase and continue to read the current transform directly.

H2-6 module-v2 implementations add `Emitter.updateInterpolatedTransform`, another compiler-provided
expression rather than a host uniform. Every Update-stage emitter-space force, analytic collider,
and kill volume shares one
sample at exact phase `0.5`: translation is a previous/current lerp and rotation is shortest-path
quaternion slerp. Exact stationary transforms retain the direct-current branch and prior output
bits. NeighborGrid bucket insertion and every NeighborGrid visitor lookup deliberately continue to
use the current endpoint so one rebuild has a coherent grid frame. The midpoint is a one-sample
temporal integration approximation, not CCD, swept-volume intersection, multi-point integration,
or synthesized substeps; a thin volume may still be missed between sample points.

H1-7 deliberately does not backdate `Particles.age` within a step. Optional spawn-time age
correction may be proposed later, but it must be an explicit option so stationary-emitter golden
behavior does not change implicitly.

Prewarm equivalence covers deterministic emitter-local Spawn/Init/Update state when compared with the same sequence of simulation steps. It deliberately excludes unscaled `System.*` clock values, rendering/presentation, readback timing, callbacks, and external integrations. An emitter with no module that declares a `Particles.lifetime` write has an unbounded conservative lifetime (`Infinity`) unless exact alive-count completion is enabled. If any module other than the statically understood `core/lifetime` writes lifetime, the same conservative `Infinity` rule applies.

An undeclared lifetime is also the supported form for particles that live until an explicit Update
module, such as `killVolume()`, writes `Particles.alive = false`. Reaching `capacity` never
overwrites a living slot: Spawn compares requests with the atomic free count, drops unavailable
invocations, and exposes them through the existing overflow diagnostic. A slot becomes available
only through the compiler-owned death/recycling epilogue. Unbounded rate emitters SHOULD provide
an explicit death path; otherwise capacity stays full and later spawns remain suppressed.

### 9.1 M5/M6 events and sub-emitters

`onDeath: emitTo(target, { inherit })` is the M5 executable event contract. Effect compilation resolves `target` against emitter element keys and reports `NACHI_EVENT_TARGET_UNKNOWN` before materialization. Each producer event kind owns one append queue shared by all of its handlers. The inherited field union determines one payload layout, so two handlers that inherit `position` do not duplicate that lane.

An event queue uses three WebGPU resources: a four-word atomic state buffer, a double-banked vec4 payload buffer, and one three-word indirect-dispatch buffer. State words are the two bank counts, cumulative overflow count, and cumulative emitted-event count. Payload allocation costs `2 * producerCapacity * max(1, payloadGroupCount) * 16` bytes. State costs 16 bytes and indirect arguments cost 12 bytes per producer event kind. These resources are listed in `CompiledEmitterMeta.storageBuffers` and count toward device storage-buffer budgets. A consumer uses the M2 free-list and `dispatchIndirect` path; event count is a separate spawn batch, Init runs normally, then inherited attributes overwrite the declared outputs.

Event-consumed Init uses the ordinary uniform values retained from the previous scheduler frame; current-frame timing uniforms are installed when that emitter advances after event consumption. Event bank uniforms are the scheduler-owned exception. Effect completion conservatively drains one scheduler frame per edge in the deepest event chain, so a write-bank append remains consumable without an event-count readback. When exact alive readback is enabled, emission completion and every event-consumption frame force the final compact/readback regardless of `aliveCountReadbackInterval`.

Atomic append may increment a bank count beyond producer capacity, but no payload address is formed for those slots. Such records are dropped, the cumulative overflow word is incremented, and opt-in readback reports `NACHI_EVENT_QUEUE_OVERFLOW`. Consumer capacity pressure uses the existing `NACHI_SPAWN_CAPACITY_EXCEEDED` counter and never overwrites a living target slot. Queue counts are not recursively consumed in their write frame.

When more than one producer queue routes into one target, core consumes links in the canonical
logical tuple `(target emitter key, source emitter key, event name, handler target, sorted inherited
field names)`. Effect-element insertion order, object spread order, and handler-array order do not
select the first consumer under target capacity pressure. Two handlers whose tuple is identical
are semantically indistinguishable: they read the same queue record, inherit the same fields, and
target the same emitter, so retaining their relative authored order cannot change the result.

This cross-queue routing guarantee does **not** order records inside one producer/event queue. GPU
producer invocations reserve append positions with an atomic operation; their winning append order
is a parallel hardware race and may differ across adapters or runs. Consumer Init and child
`spawnOrder` follow that queue record order, and queue or target saturation may therefore choose a
different record from the same queue. Queue separation can isolate independent producer classes,
and sufficient event-target/queue capacity avoids an overflow winner, but neither makes one queue's
parallel record order deterministic. Separate emitter seeds isolate random streams only; changing a
seed is not an ordering fix.

`onCollision: emitTo(target, { inherit })` is executable in M6. Built-in analytic and scene-depth
colliders append whenever their penetration test produces a response. Position correction is
written before append, so `inherit: ['position']` carries the resolved surface position through the
same vec4 payload path used by `onDeath`; consumption remains next-scheduler-frame and bounded by
the same double-bank and overflow rules. The payload `position` and `velocity` are sampled after
position correction and velocity response. A `mode: 'kill'` collision appends `onCollision`, then
writes `alive = false`, so the compiler death epilogue also appends `onDeath` when both handlers
exist. Non-kill responses do not write `Particles.alive` and therefore cannot resurrect a particle
killed by an earlier author module. Resting contact uses strict penetration; gravity/integration
may move a corrected particle back inside on a later frame, and every new penetration response
emits another `onCollision`. Continuous contact is intentionally an event stream rather than a
begin-contact-only event in M6 v1. `onSpawn` is reserved for a future compiler-owned
lifecycle producer and produces `NACHI_EVENT_KIND_UNIMPLEMENTED`. The planned custom producer
surface is a `tslModule` context method `context.emitEvent(condition)`, valid only from Update and
lowering to a conditional append into `onCustom`. The condition must be a scalar boolean TSL
expression and inherited fields still come from the handler's `emitTo` declaration. Until that
context is materialized, authoring `onCustom` produces `NACHI_EVENT_ON_CUSTOM_UNIMPLEMENTED`.

### 9.2 M3 sprite and mesh renderers

`billboard()` consumes particle color, position, size, and optional orientation inputs without
mutating simulation attributes. `flipbook(texture, { cols, rows })` assigns frame 0 to the
top-left atlas cell and advances left-to-right, then top-to-bottom. Three.js textures using
`flipY` therefore resolve the atlas row as `rows - 1 - floor(frame / cols)`; textures with
`flipY: false` use `floor(frame / cols)` directly. Frame interpolation is enabled by default. If
the emitter has no lifetime writer, `normalizedAge` remains at its default zero and the flipbook
remains on frame 0.

Motion-vector interpolation requires both interpolation and a motion-vector `TextureRef`.
Requesting motion vectors without a resource falls back to plain interpolation with a warning;
requesting them while `interpolate: false` ignores them with a warning. UV warps are clamped to
their source atlas cell so filtering cannot leak into adjacent frames.

`cutout.vertices` accepts 4 through 8 vertices. Four vertices retain the full quad. Five through
eight vertices form a convex polygon contained by the quad, reducing overdraw by cutting away its
texture corners; triangulation uses `(vertices - 2) * 3` indices and preserves UVs in the original
quad's `[0, 1]` range.

The four blending modes have fixed M3 semantics. `alpha` uses source-alpha blending;
`premultiplied` uses normal blending with premultiplied-alpha factors; `additive` adds source color;
and `multiply` multiplies the destination by source color. Billboards always use the transparent
render path with depth testing enabled and depth writes disabled. Meshes keep depth writes enabled;
`alpha` and `premultiplied` select the transparent render path, while `additive` and `multiply`
remain on the opaque render-list path.

Billboard alignment is one of `camera-facing` (default), `custom-axis`, `velocity-aligned`, or
`velocity-stretch`; the stretch form optionally scales its velocity contribution with `factor`.
Mesh alignment is one of `none` (default), `custom-axis`, `velocity`, or `quaternion`. Velocity and
custom-axis mesh modes rotate the geometry's local +Y axis toward the selected direction, while
quaternion mode consumes `Particles.rotation` directly.

`orientToVelocity()` is an Update module that writes both billboard-compatible
`Particles.spriteRotation` and a quaternion rotating local +Y toward `Particles.velocity`. Its
quaternion path uses direct quaternion-vector math rather than Three r185's transposed TSL
`rotate()` Euler convention. `spriteRotation` is the velocity angle projected onto the world XY
plane, whereas renderer-side velocity-aligned billboards derive their angle in view space. Zero
velocity preserves the previous orientation. Use it with camera-facing billboards or
`meshRenderer({ alignment: { mode: 'quaternion' } })`; combining it with renderer-side velocity
alignment applies the same intent twice and SHOULD be avoided.
`faceCamera(options)` is the explicit render-stage spelling of
`billboard({ ...options, alignment: { mode: 'camera-facing' } })`; it does not introduce camera
state into deterministic Update simulation.

`rotationOverLife()` accepts an `f32` curve and writes the isotropic scalar
`Particles.spriteRotation` in radians. Per-axis Euler and quaternion curves require a future typed
orientation module. `velocityOverLife()` likewise accepts only an `f32` curve and applies one
absolute isotropic scale to all velocity components. Each Update applies
`lut(t) / lut(tPrevious)`, where `tPrevious` is reconstructed from `Emitter.deltaTime` and particle
lifetime; equivalent elapsed-time partitions therefore reach the same scale instead of repeatedly
compounding the absolute LUT value.

`billboard({ soft: true })` uses a calibrated default fade distance of `0.035`.
`billboard({ soft: { fadeDistance } })` overrides it. The value is measured in Three.js
linearized normalized camera-depth units and must be positive and finite. The default comes from
the accepted `spike-depth` partial-intersection calibration; it is not a world-space distance.

`meshRenderer({ geometry, alignment })` resolves a serializable `GeometryRef` to an arbitrary
Three.js `BufferGeometry` at materialization time. It consumes per-particle position, scale, and
color, with optional velocity, custom-axis, or quaternion orientation. Like billboards, mesh
instances use the compact alive-index indirection and alive-count indirect draw arguments.

### 9.3 M7 ribbon/trail renderer

`@nachi-vfx/trails` owns `ribbon()`, `ribbonId()`, its render compiler registration, and the Three.js
WebGPU materializer. Applications add those implementations to the existing core registry with
`registerTrails(createCoreKernelModuleRegistry())` and supply that registry through
`VfxSystemOptions.registry` or `compileEmitter({ registry })`. Core provides only the generic
render-registration context and lifecycle ordering resources; it does not re-export the trails
authoring helpers. This is the first external package proof of the module/render registry boundary
from §13.

`ribbon({ width, maxRibbons, uv, taper, blending, map })` reads `Particles.position`, `color`,
`ribbonId`, and the compiler-owned `spawnOrder`. `ribbonId()` initializes the custom `u32`
`ribbonId` attribute to a constant or an alternating modulo sequence. `maxRibbons` is an explicit
scan bound (v1: 1 through 64), not a dynamically resized GPU allocation. `uv.mode: 'stretched'`
maps each strip's ordered point ordinal to `[0, 1]`; `uv.mode: 'tiled'` accumulates world-space
segment length and divides by positive `tileLength`. Taper fractions are evaluated against the
same per-ribbon parametric coordinate. The endpoint width is exactly zero when its corresponding
taper fraction is nonzero, while the untapered body uses the authored `width`. Fragment color is
still read from `Particles.color`, so `colorOverLife()` remains orthogonal, and additive, alpha,
multiply, and premultiplied blending retain the §9.2 meanings.

Constant IDs and alternating offsets are non-negative u32-compatible integers; alternating counts
are positive u32 integers whose resulting range remains within u32, and the mode tag is exact.
Count `2^32` is invalid because its shader representation would wrap to zero. UV modes
outside `stretched | tiled` are rejected rather than treated as stretched. Compile-time cross-module
validation requires every statically authored ID to be less than `maxRibbons`;
`NACHI_RIBBON_ID_VALUE_INVALID`, `NACHI_RIBBON_ID_COUNT_INVALID`,
`NACHI_RIBBON_ID_OFFSET_INVALID`, `NACHI_RIBBON_ID_MODE_INVALID`,
`NACHI_RIBBON_UV_MODE_INVALID`, and `NACHI_RIBBON_ID_OUT_OF_RANGE` identify these errors.

Ribbon order MUST NOT consume compacted `aliveIndices`. Before each spawn dispatch, a one-invocation
GPU preparation kernel atomically reserves a consecutive `spawnOrder` range of
`min(requested, freeCount)`. The scheduler submits prepare, spawn, and finalize strictly in order,
so `freeCount` cannot change between reservation and allocation; failed saturated spawn attempts
therefore consume neither an order value nor a birth-ring slot. Successful invocation `i` receives
`base + i`, independently of free-list physical-slot selection. Spawn writes the physical index
into a capacity-sized birth ring at `spawnOrder % capacity`. Before rendering,
`@nachi-vfx/trails` runs one GPU invocation that scans the ring chronologically for each `ribbonId`,
rejects stale entries by comparing the particle's stored `spawnOrder`, skips dead particles, and
emits adjacent surviving pairs into compact segment/UV/width buffers plus a renderer-owned indirect
draw record. Scattered death and non-deterministic alive compaction therefore cannot reorder or
cross-connect a strip, and killing an interior control point reconnects its surviving predecessor
and successor. Normal operation performs no CPU readback; the exposed readback helper is diagnostic
and smoke-test only.

M7 batch 1 ribbons require WebGPU storage buffers and indirect draw. Materialization on the WebGL2
path fails explicitly with `NACHI_RIBBON_WEBGL2_UNSUPPORTED`; there is no silent sprite/line
fallback. The v1 preparation cost is `O(maxRibbons * capacity)` in a deliberately serial GPU pass,
and ordering uses wrapping `u32` values. `NACHI_SPAWN_ORDER_WRAP_RISK` warns once at the `2^31`
safety threshold. Without counter readback, normal-spawn request accumulation is a conservative
upper bound; enabled lifecycle readback also checks the actual GPU counter, including event births.
An emitter must be restarted before the 32-bit order wraps, and sim-cache work in M11 must preserve
or rebase this counter.

The birth ring retains only the latest `capacity` successful births. If one unusually long-lived
control point survives while at least `capacity` later successful births churn through shorter-lived
slots, its ring entry is overwritten and that point drops out of the strip even though it remains
alive. M7 batch 1 has no zero-readback runtime diagnostic for this condition; authors must size
capacity/lifetime/rate so every required control point remains inside the retention horizon.

Ribbon smoothing, curve tessellation, width-over-life/parametric width curves (fixed body width plus
endpoint taper only), per-control-point twist, facing modes beyond camera-facing width, and
Niagara's link-order policy variants are not yet implemented. Each segment also computes its side
vector independently; at sharp curves with wide ribbons this can expose cracks, whereas Niagara
averages point tangents across adjacent segments. M7 retains the current one-render-module-per-
emitter limit; per-draw indirect slots and multiple render modules are deferred to M8 or later as
recorded in §16.

The reduced WebGL2 burst path assigns `spawnOrder` only as the invocation's number inside that
burst; it is not a persistent global birth order. Order-sensitive renderers reject WebGL2, so this
restricted meaning cannot silently drive a ribbon.

### 9.4 M7 light and decal renderers

`lightRenderer({ maxLights = 8, priority = 'intensity', radiusScale = 1 })` uses a bounded
`THREE.PointLight` pool. A single-invocation WebGPU kernel scans physical particle storage, ignores
dead or non-positive candidates, and keeps only the top `maxLights` entries by `Particles.intensity`
(or intensity multiplied by radius). It writes one fixed small buffer containing
`N × { position, color, intensity, radius, physicalIndex, spawnOrder }` plus counts. Runtime double-buffers this
selection output and performs exactly one asynchronous readback of the previous frame's buffer;
it never reads every particle attribute. Pool updates therefore have one-frame latency. Candidate
count greater than `maxLights` emits `NACHI_LIGHT_LIMIT_EXCEEDED` once and discarded lights remain
inactive. Equal-priority candidates are resolved by ascending unsigned `Particles.spawnOrder`: the
earlier logical birth wins. GPU replacement and CPU readback ordering use the same
`(priority descending, spawnOrder ascending)` tuple; physical index remains diagnostic metadata and
never selects a winner before `spawnOrder`. The light module adds `spawnOrder` to the resolved
particle schema only when used, preserving non-light layouts and budgets. After the u32 birth key
wraps, two simultaneously alive particles can have the same key and exact-tie identity is no longer
guaranteed; applications that could approach the existing half-range warning MUST restart the
emitter before wrap. `colorOverLife()` writes `Particles.color`, while `lightIntensity()` and
`intensityOverLife()` write `Particles.intensity`; the renderer only reads both, preserving module
orthogonality.

Every pooled PointLight stays `visible = true`; inactive slots use `intensity = 0`. Three r185 hashes
the visible analytic-light id list into `LightsNode.customCacheKey`, so visibility toggles during a
flash would create shader variants and pipeline-compilation hitches. `dispose()` is idempotent,
zeros every intensity, and removes the pool group from its scene. Renderer integration MUST call it
when the effect completes, errors, stops, or is released; `update(renderer, effectState)` performs
that transition when a non-active state is supplied.

Three r185 WebGPU constructs one analytic light node per scene light rather than exposing a fixed
portable `maxLights` constant. `/golden-slash/` therefore probes 8, 16, and 32 PointLights by
offscreen render/readback and publishes a warmed GPU render duration for each count as
`pointLightProbe.gpuRenderMs`. The runtime default remains 8 to bound shader growth and lighting
cost; adapter-specific timing values are baseline data rather than a portable API constant. The
selected design is the PointLight pool because it affects ordinary lit scene materials
in M7 without requiring post processing. Emissive sprites plus bloom are complementary appearance
work deferred to M10, not a semantic fallback for particle lights.

`decalRenderer({ sizeScale = 1, fadeOverLife = true, map, blending })` draws an instanced projection
box for every compacted alive particle. Position, quaternion orientation, scalar size, linear color,
and normalized age come from particle attributes. The fragment stage samples the M6 previous-frame
linear float depth copy, converts `screenUV` to clip x/y, uses sampled WebGPU NDC z directly in
`[0, 1]`, and converts top-left-origin `screenUV.v` with `ndc.y = 1 - 2 * screenUV.y` before inverse
projection. It then uses the camera world matrix to reconstruct world position. The point is
transformed into the particle's oriented box; fragments outside the box are transparent, while
inside x/y provide decal UV. Default lifetime fade multiplies alpha by
`1 - normalizedAge`, and `colorOverLife()` remains an independent input.

M7 decals are WebGPU-only and require the same non-MSAA, non-sRGB, linear float depth copy and
matching camera/viewport as scene-depth collision. `NACHI_DECAL_WEBGL2_UNSUPPORTED`,
`NACHI_DECAL_DEPTH_MSAA_UNSUPPORTED`, and `NACHI_DECAL_SCENE_DEPTH_UNAVAILABLE` are explicit
diagnostics. The capture is single-camera and one frame late, so moving receivers can lag by one
frame. M7 does not reject projection by receiver normal or back-facing surfaces, and reversed-z is
not yet guarded in the decal materializer; integrations must use the documented forward-z capture.
CPU `DecalGeometry` is not used because rebuilding clipped CPU geometry for particle-driven
instances would defeat GPU simulation and bounded scheduling.

### 9.5 M10 transparency ordering

Alpha and premultiplied rendering has two independent ordering levels. At the particle level,
module-v2 `billboard()`, `meshRenderer()`, and `decalRenderer()` default `sorted` to `true`; an
explicit `false` selects M2's compact alive-index array and deliberately has no semantic order
across death/reuse. Additive and multiply billboard/mesh draws default to `false`. Module-v1 data
retains the old meanings: billboard/mesh omission is false, and decals never particle-sort.
`sorted: true` is accepted only for alpha or premultiplied blending.

A sorted draw leaves simulation/event storage untouched and allocates separate float-depth and uint
index buffers. Every compaction expands alive particles into a power-of-two extent. Padding uses the
farthest camera-z sentinel in the sorted prefix; vertex fetch starts at
`paddedCapacity - aliveCount`, so no sentinel is drawn. Preparation writes camera-space z from
`System.viewMatrix`, then every bitonic `(k,j)` stage is a separate ordered compute submission.
Ascending camera-z is back-to-front under Three's forward-camera convention. Equal depth compares
physical index, which is byte-deterministic for fixed input but is not persistent particle identity.
Sorting is rebuilt by `VFXSystem.update()` after compaction; `setCamera()` only changes camera state.

The sorted indirection is renderer-owned draw data, not a new simulation ordering. Events,
compaction, and future sim caches continue to observe physical/alive storage. Capacity is limited
to 65,536 and padded to the next `2^n`; the dispatch count is
`log2(P) * (log2(P) + 1) / 2`, each dispatch compares `P / 2` pairs, for
`O(P log^2 P)` comparisons and one ordered backend submission per bitonic stage after emitter
compaction. WebGL2 reports
`NACHI_PARTICLE_SORT_WEBGL2_UNSUPPORTED`; no CPU particle-sort fallback is hidden.

At the draw level, `VFXSystem` ranks each participating alpha/premultiplied compiled draw by its
transformed emitter-local `sortCenter`, defaulting to the emitter origin. Ranks increase far to near;
equal depth uses numeric per-system instance creation sequence, emitter key, then draw path/index.
Version-2 billboard, mesh, and decal draws participate, as do legacy billboard/mesh draws; v1 decals,
ribbons, additive, and multiply draws do not. The runtime sends `(drawIndex, rank)` assignments, so
late materialization and multiple registered draws cannot collapse to one kernel-wide scalar.

Three composes host order exactly as
`bucket = base + renderOrderOffset` and
`renderOrder = bucket + (rank + 1) / 2^20` for an automatic draw. Sprite/mesh materializers default
`base` to 1000 and decals retain 10. `base`, the v2 module's `renderOrderOffset`, and their sum are
signed 32-bit integers; at most `2^20 - 1` automatic entries exist per system. Therefore an external
object at `bucket` precedes every automatic draw in that bucket and one at `bucket + 1` follows them.
Materialized sprite/mesh/decal objects expose `setRenderOrderBase()` for persistent host changes.
Direct `Object3D.renderOrder` assignment may be overwritten on the next rank, activation, or pool
transition. Pooling clears the previous generation's assignment. Version-2 transparent mesh
materials keep depth testing but set `depthWrite = false`; v1 mesh materials retain `true`.

Any active particle-sorted draw, or multiple automatic transparent draws, without a configured
camera records `NACHI_ALPHA_SORT_CAMERA_UNSET` once per affected instance. See
[RFC 006](./006-transparent-draw-order.md) for the renderer-version, exact-number, asset-migration,
decal spawn-orientation, and pooling contracts.

Weighted OIT is the third, approximate path and normally replaces both particle and emitter
sorting for the objects routed through it. `sorted: true` is allowed to remain on an authoring
definition but supplies no quality benefit inside WBOIT; integrations SHOULD disable it to avoid
the compute cost. Ordinary sorted alpha and WBOIT objects may coexist only as separate render
groups with an explicit composition boundary.

### 9.6 M10 lit billboard particles

`billboard({ lit: true })` opts a sprite into Three r185's standard physical-lighting path.
`lit` may instead be `{ normalMap, roughness, metalness }`; roughness defaults to `0.8`, metalness
to `0`, and both are finite unit values. The ordinary `map` or flipbook remains base color and
alpha. The tangent-space `normalMap` is an independent `TextureRef` and MUST resolve to a
`NoColorSpace` texture. Omitting it uses the flat camera-facing normal.

The Three materializer constructs `MeshStandardNodeMaterial` and replaces only its
`setupPositionView()` method with the pinned r185 `SpriteNodeMaterial` implementation. This keeps
`MeshStandardNodeMaterial.setupVariants()`, `setupLightingModel()`, environment handling, and the
renderer-provided `builder.lightsNode` intact, so ordinary Three analytic lights—including the M7
PointLight pool—affect particles without a private light list or readback. A custom
`lightingContext` was rejected because it would duplicate Three's physical material setup and
silently diverge as the pinned implementation changes. Unlit billboards continue to instantiate
`SpriteNodeMaterial`; enabling M10 does not change their shader graph or response to scene lights.

Sprite geometry is authored directly in the camera-facing view XY plane. Let `r` be the exact
rotation node used by the sprite vertex path after alignment and per-particle rotation. The normal
map basis in view space is:

```text
T = ( cos(r), sin(r), 0 )
B = (-sin(r), cos(r), 0 )
N = (      0,      0, 1 )
n_view = normalize(T*n_tangent.x + B*n_tangent.y + N*n_tangent.z)
```

The fragment result is assigned directly to `normalNode`. This is intentional: Three r185's
`normalNode` contract is already view space, so applying a model normal transform again would be a
double transform. Combining `lit` with `velocity-stretch` does not apply inverse-scale correction
to the normal basis in M10, so non-uniform stretch can leave a small lighting approximation residual.
`/m10-lit/` swaps an M7 PointLight between asymmetric left/right positions and
requires the measured bright side of a flat-lit pair to reverse, compares an asymmetric
normal-mapped sprite with flat-lit output, and renders the same unlit definition under both light
positions as an invariant control.

## 10. Effect, timeline, and runtime API

### 10.1 Effect composition

```ts
const effect = defineEffect({
  elements: {
    arc,
    sparks,
    flash,
    shockwave,
  },
  timeline: [
    at(0.0, play('flash')),
    at(0.05, play('arc'), cameraShake({ strength: 0.3 }), hitStop(40)),
    at(0.08, play('sparks')),
    at(0.1, play('shockwave')),
  ],
});
```

Typed public parameters are declared in the same definition:

```ts
const chargedEffect = defineEffect({
  elements: { flash },
  parameters: {
    'User.intensity': defineParameter('User.intensity', {
      type: 'f32',
      default: 1,
      mutable: true,
    }),
  },
});
```

`defineEffect()` infers element keys. `play()` and `stop()` preserve literal target names so
TypeScript can reject unknown targets. Core `defineEffect()` is a composition and compatibility-data
factory; runtime validation of timeline targets and values belongs to `@nachi-vfx/timeline`, including
when a definition was constructed with core's timeline factories. Timeline entries are sorted by
time during timeline-package validation; author order breaks ties.

M9 batch 2 moves the integration surface to `@nachi-vfx/timeline`. That package re-exports the timeline
data factories and supplies its own `defineEffect()` adapter and `VFXSystem` composition runtime.
The north-star array form remains valid. Whole-track controls use an equally serializable wrapper:

```ts
timeline: timeline([at(0, play('flash')), at(0.05, play('arc'))], {
  duration: 0.5,
  loop: 3,
  speed: 1.25,
});
```

`loop: true` is infinite; a positive integer is an exact cycle count. A loop requires positive
`duration`, and duration cannot precede the last entry. The array shorthand is one cycle at speed
one. At each loop boundary every playing element stops immediately, then time-zero actions of the
next cycle run in author order. A child emitter whose lifecycle outlives the track is deliberately
truncated, while a shorter child stays stopped until another `play()`. Each `play()` creates a fresh
child lifecycle and may reuse core's pooled materialization. Child `loopCount` never leaks across
sequencer cycles.

The timeline runtime defensively revalidates core-authored definitions before scheduling. Entry
times must be non-negative and finite, and speed must be positive and finite. Violations produce a
runtime diagnostic and return the spawned timeline instance in `error`; they are not clamped. This
preserves one deterministic time meaning across authoring paths and exposes corrupt definitions
instead of silently changing their choreography.

Timeline-system spawn validates `timeScale` with the same non-negative finite rule as
`setTimeScale()` before retaining an instance. Timeline and core also share the transform value
contract: spawn position is optional, while `setTransform()` and attachment samples require a finite
vec3 position; rotation, when present, is a finite Euler vec3 or quaternion vec4. Timeline validates
before allocating an ID or mutating its stored pose, child GPU uniforms, attachment source, or mesh
clone. On the first update, attachment synchronization runs before time-zero actions even when world
delta is zero; `at(0, play(...))` therefore receives the latest attachment pose.
Core system spawn reads `timeScale` and `priority` exactly once, validates the resulting primitives
before allocating an ID, and uses those same snapshots for the instance clock and significance.
Timeline system spawn likewise reads `timeScale`, position, rotation, seed, camera-shake target, and
parameters exactly once before ID allocation and creates a plain frozen own-data options record for
construction. The public `TimelineEffectInstance` constructor revalidates that owned record and raw
direct-constructor options; no internal snapshot argument is exposed. Thus a getter returning a valid
value and then `NaN` cannot create a post-validation TOCTOU failure or consume an ID.
An attachment sample is bound to an operation revision captured before invoking its getter, both for
direct `attachTo()` and scheduled synchronization. Any nested attach, detach, or release advances the
revision and discards the outer sample. The same rule applies to same-source reentry and to an invalid
nested attach whose exception the getter catches: the failed attach leaves the prior source and pose
unchanged, while its newer revision still prevents the older outer operation from becoming
authoritative. Replacement poses and first-update time-zero actions therefore remain authoritative
in the same update.
Core and timeline normalize each public transform operation to an owned frozen snapshot. The
position and rotation properties, array length and indexed components, or object `x/y/z` components
are each read once; validation, matrix construction, stored pose, child spawn/update, mesh mutation,
and GPU writes use only the snapshot. Attachment sampling checks its revision immediately after the
source getter, then snapshots all returned transform accessors and checks the revision again directly
before commit. Source-getter release/replacement therefore remains a quiet discard, while reentry
from a transform property or component accessor also invalidates the stale sample.

### 10.2 Timeline actions

```ts
function at(timeSeconds: number, ...actions: TimelineAction[]): TimelineEntry;
function play<Target extends string>(target: Target): PlayAction<Target>;
function stop<Target extends string>(target: Target): StopAction<Target>;
function cameraShake(options: {
  strength: number;
  duration?: number;
  frequency?: number;
}): CameraShakeAction;
function hitStop(durationMs: number, timeScale?: number): HitStopAction;
function marker(name: string, payload?: JsonValue): MarkerAction;
```

Timeline time is effect-local time. `hitStop()` affects the effect-local clock and integrations explicitly subscribed to that effect; it must not freeze unrelated browser or game clocks. `cameraShake()` produces an integration command consumed by a registered camera adapter rather than mutating an assumed global camera.

`applyHitStop()` uses replacement semantics: applying it again replaces both the remaining duration
and time scale with the new call rather than stacking, extending, or taking the maximum. The current
replacement is forwarded to every active child; children played afterward receive the then-remaining
replacement.

`TimelineEffectInstance` also exposes the additive companion contract:

```ts
bindCompanion(instance: VfxEffectInstance): void;
unbindCompanion(instance: VfxEffectInstance): void;
```

Binding immediately copies the timeline's effective time scale (instance scale multiplied by
timeline speed) and any remaining hit-stop replacement. Later `setTimeScale()` calls and hit-stop
applications are forwarded synchronously in the same frame. A separately updated companion system
MUST consume the same world-step sequence. Exact local-time equality through stop and resume is
guaranteed only when the hit-stop action boundary is also a companion update boundary. Advancing the
companion system first avoids a full-frame-late stop, but it does not sub-segment a non-aligned
companion update: in that case the timeline alone splits at the action and the clocks may differ by
less than one companion update interval. Socket-following integrations MUST use the consumed-pose
latching pattern in [RFC 005 §3.1](./005-effect-local-clock.md#31-bindcompanion-phase-limit-and-socket-driving).

Binding is imperative rather than exclusive clock ownership. Initial binding overwrites the
companion's current time scale and remaining hit stop. Subsequent direct companion writes and
timeline forwards are last-writer-wins for their respective controls; unbinding leaves the last
values in place. Bindings are weak, so released companions are removed automatically and cannot be
retained by a timeline. Every initial, residual, hit-stop, and time-scale forwarding path MUST skip
an `error` or `released` companion. An unavailable bind, or a bound companion entering `error`, adds
the warning diagnostic `NACHI_TIMELINE_COMPANION_UNAVAILABLE` instead of forwarding an operation.

M9 defines the camera adapter as a callback receiving an absolute additive translation/rotation
sample with linear decay, effect id, cycle, and effect-local sample time. It has no Three camera
type. Integrations may add it to any camera rig and combine samples by effect id; termination always
emits a zero sample. Noise uses core's PCG hash with effect seed, shake sequence, axis, and frequency
lattice index plus smooth interpolation. Equal seeds and equal world-step sequences therefore
produce byte-equal samples. Shake duration advances in effect-local time, so hit stop holds the
current shake without consuming its decay.

`marker()` is pure data. `TimelineEffectInstance.onAction()` observes every action at the
post-clock/pre-next-submission synchronization boundary; `onMarker(name, callback)` is its filtered
gameplay form. Entries are stable-sorted by time, ties retain author order, and actions within one
entry retain argument order. For an emitter `play()` action, the event's optional `emitter` field is
the `VfxEmitterRuntimeView` created by that action. Integrations that materialize renderer objects
MUST capture that field instead of assuming the timeline definition is one monolithic core effect:
each played emitter is an independently scheduled single-element sub-definition and can later be
replaced or released by `stop()`, replay, or completion. Mesh-fx and non-play actions leave the field
undefined. `on('death', ...)` remains the aggregate GPU-event path below and is forwarded to each
currently playing child emitter. An `onAction`/`onMarker` callback exception is converted to
`NACHI_TIMELINE_ACTION_CALLBACK_FAILED`, transitions only that timeline instance to `error`, and is
absorbed so the system consumes the full caller-supplied world delta and continues updating
unrelated instances.

Callbacks from timeline/event modules use registered callback references. They are queued to a documented synchronization point; arbitrary JavaScript is never invoked from inside a GPU dispatch. `EffectInstance.on('death', callback)` is deliberately aggregate-only: callback receives `{ event: 'death', count }` accumulated since the previous event-counter readback, never a particle payload or one callback per particle. Event totals and overflow totals normally use the same `aliveCountReadbackInterval` cadence as authoritative alive count; emission completion and event-consumption frames force a final readback so the last aggregate is flushed before completion. When that option or renderer readback support is absent, GPU events still drive sub-emitters but no JavaScript event notification or overflow diagnostic is promised. This avoids an unconditional GPU/CPU synchronization point.

### 10.3 Runtime instances

`VFXSystem.spawn()` accepts immutable effect data and returns an `EffectInstance`. Position, rotation, seed, local time scale, and `User.*` overrides belong to the instance. Three-component rotations are Euler angles in radians and XYZ order; four-component rotations are quaternions. Spawn-time `parameters` and `instance.setParameter()` are mapped types derived from the definition, so an unknown path or wrong value type is a TypeScript error and remains a runtime validation error for untyped input. `setTransform()` updates `Emitter.transform` and supplies the delta used by per-distance spawning. `attachTo(source)` reads world position/quaternion from an external transform provider before initialization and every scheduled system step; `detach()` retains the last transform and stops polling. Backend sugar such as `createThreeTransformSource(Object3D | Bone)` adapts engine objects without importing them into core. `release()` returns pooled resources when safe. In M2, `stop()` means immediate scheduling stop: it does not advance or drain particles and retains resources until `release()`. A future explicit policy may add graceful drain without changing this default silently.

Before each positive simulation step, runtime snapshots the last transform actually consumed by
simulation as `Emitter.previousTransform`, then resolves attachment/socket/manual transform changes
for the current step. A newly spawned runtime and every M9 pool reacquisition initialize previous
to current, including the composed emitter offset, so neither the origin nor a previous pooled
generation becomes an interpolation endpoint. `setTransform()` is interpreted as continuous motion
from the preceding simulated transform and is therefore interpolated. A future explicit
previous-transform reset/teleport API is required for instantaneous relocation; H1-7 does not infer
teleports from distance or velocity thresholds.

Transform history follows every positive system step even while scalability culls an instance or
its effective local time is zero (including `timeScale: 0` and zero-scale hit stop). Those frames do
not emit per-distance births, and their movement distance is intentionally discarded instead of
being accumulated for a burst on resume. The first resumed step therefore interpolates only from
the latest stopped/culled transform, preventing stale interpolation beads.

M9 batch 1 adds a per-definition bounded effect-resource pool. `VfxSystemOptions.maxPoolSize`
defaults to 16 retained effect bundles per live definition; zero disables retention, and
`getPooledInstanceCount(definition)` exposes that definition's current count. Both the retained
resources and their count live in the value of a definition-keyed `WeakMap`, so collecting a
definition cannot leave a detached system-wide count that consumes capacity. `release()` removes
the handle from live scheduling immediately. If a system update is in flight, retention/disposal
is deferred until that queued update settles, so a new spawn cannot race pending GPU submissions.
Poolability (including the error-state exclusion) is captured synchronously when `release()` is
called. At the per-definition bound, resources are disposed and the released handle records
`NACHI_EFFECT_POOL_LIMIT_EXCEEDED`.

The existing effect-definition `WeakMap` remains the cache of renderer-independent compiled
programs. The pool is a separate per-definition cache of materialized kernels and their GPU
storage/indirect/event buffers. Before a bundle enters the pool, core calls the renderer's
`prepareKernelsForPooling()` hook for every emitter. The adapter must synchronously make the bundle
non-drawable; an indirect renderer writes draw `instanceCount` to zero and schedules that word for
upload before a stale external draw object can render it. Reuse constructs a fresh public instance and fresh CPU lifecycle,
clock, listener, diagnostic, transform, seed, and parameter state around that bundle. Before any
activation, the initialization kernel clears every resolved attribute lane, the free list, alive/
success/overflow counters, event counters, `spawnGeneration`,
`nextSpawnOrder`, `currentSpawnBase`, and the capacity-sized birth ring. The birth ring uses the
invalid `0xffffffff` sentinel while its counter is zero. Thus the first successful allocation after
reuse again has particle generation 1 and spawn order 0, and equal definition/seed/parameters/
step sequence matches a cold spawn bit-for-bit subject to the §2.6 physical-allocation boundary.
The draw-indirect `instanceCount` is zeroed by the pooling hook, then rewritten from the compacted
alive count by `finalizeIndirect`; initialize does not overwrite draw word 0 (`indexCount`).

`VFXSystem.update(deltaSeconds?)` advances a system clock and schedules each live instance through the renderer-independent kernel adapter. Core accepts a structural `VfxRuntimeRenderer` bridge containing the M1 `KernelTslAdapter`, compute/indirect submission, optional readback/uniform/resource hooks, and device-loss notification; backend packages wrap their native renderer without importing it into core. `VFXSystem.setCamera({ viewMatrix, projectionMatrix, viewportSize })` copies column-major camera matrices and positive depth-texture dimensions into every current emitter and initializes later spawns with the same state.

When `deltaSeconds` is omitted, the first call samples `VfxSystemOptions.now` and advances by zero.
Each later omitted call uses `max(0, (now - previousNow) / 1000)`. Equal or reversing finite
timestamps therefore produce zero while still becoming the next baseline. A non-finite sample is a
`RangeError`. The measured delta is capped by `maxMeasuredDeltaSeconds`, which defaults to `0.25`,
must be a positive finite number, and accepts `Infinity` to disable the cap. Only `undefined`
selects the default; `null`, numeric strings, zero, negative values, and `NaN` are synchronous
constructor errors. An explicit finite,
non-negative `update(deltaSeconds)` bypasses this cap completely and neither samples nor resets the
last measured timestamp. Consequently, the next omitted update measures from the preceding omitted
call; applications that intentionally catch up should keep supplying explicit deltas.

Every sampled `now()` value, including a non-finite value, is stored as `previousNow` before delta
validation. An initial non-finite sample therefore rejects rather than establishing a zero update.
After `NaN`, the next finite sample also rejects because its difference is `NaN`; that finite sample
becomes the new baseline, so a following finite sample can recover normally. After `+Infinity`, the
next finite difference is a clock reversal and succeeds immediately with zero. After `-Infinity`,
the next finite difference is `+Infinity` and rejects once, while installing the finite recovery
baseline. Thus `[1000, NaN, 2000, 2100]` produces `[ok(0), RangeError, RangeError, ok(0.1)]`.
Rejected non-finite or bridge deltas add no dropped seconds because validation precedes clamping.

Measurement, timestamp reservation, and measured-drop accounting happen synchronously when
`update()` is invoked, before its work enters the FIFO update queue. Concurrent omitted calls thus
reserve clock samples in invocation order, and a sample remains consumed even if that queued update
later fails. A newly constructed system starts without a timestamp and with zero counters; effect
stop/release, pooling, and explicit updates do not reset either state. Core has no system-clock reset
operation.

`measuredDeltaDroppedSeconds`, `fixedStepDroppedSeconds`, and `droppedSeconds` are cumulative,
non-frame-local public counters. The last is the sum of the first two. The measured cap first
discards `raw - used`; only `used` enters `FixedStepAccumulator`, whose existing `maxSubSteps`
backlog discard is reported separately. The same seconds are never counted in both components.
Explicit deltas never add measured drops, but can still add fixed-step drops. These counters are not
part of the frame-local profile snapshot and remain monotonic (normal JavaScript numeric overflow
may eventually yield `Infinity`, never a wrapped value).

A fixed step partitions the already capped unscaled world delta. Each instance then multiplies that
substep by its `timeScale` (and active hit-stop scale) once to produce `Emitter.deltaTime`;
`System.deltaTime` and `System.time` remain unscaled. Hit stop, continuous and per-distance spawning,
lifecycle, and transform interpolation retain their existing meanings. Excess fixed-step backlog is
discarded explicitly rather than producing an unbounded frame.

M2 uses a scattered WebGPU free list. Update atomically appends dead physical indices; Spawn allocates from the current tail; a separate atomic compaction pass writes alive particle indices and the authoritative alive count. Four counters, the free-list region, and the compact alive-index region share one writable lifecycle state buffer. Spawn-dispatch and draw-indirect arguments share a second indirect-argument buffer so an indirect source is never also writable storage in the same compute-pass synchronization scope. The draw arguments begin at the byte offset exposed by `BuiltEmitterKernels.drawIndirectOffsetBytes`; core intentionally leaves that region's word 0 (`indexCount`) untouched, so the M3 renderer must prime it once on the CPU before the first draw and bind the same resource with that offset. Renderer modules consume the compact index indirection using `aliveIndicesOffset` when binding particle attributes. WebGL2, which lacks the required atomics/indirect calls, has only an explicitly declared reduced `burst` path. Its initialize kernel writes every physical attribute storage, so `initializeVaryingCount` and `initializeVaryings` validate that complete transform-feedback write set before node materialization. Schemas over the backend limit and all `rate`/`perDistance` emitters are rejected with `NACHI_BACKEND_SPAWN_UNSUPPORTED` rather than silently degrading. `CompiledEmitterMeta.capabilities` and `backendBudgets` make these distinctions inspectable.

The two spawn-order counters and capacity-word birth ring are appended to lifecycle state only when
the resolved schema contains `Particles.spawnOrder`. Ordinary sprite, mesh, and decal emitters pay
neither those words nor birth-ring writes; ribbons and lights that select top-N by logical birth
order opt in explicitly. Render compiler budget descriptions are stage-accurate: ribbon vertex
shading binds position/color plus its three segment buffers and does not count lifecycle state,
which is consumed only by the separate preparation compute pass.

Spawn executes before Update in a scheduler step, so particles born in that step receive one full `Emitter.deltaTime` of aging immediately. If `$age` reaches lifetime in a step, compiler-owned death handling skips the remaining author Update modules for that particle and recycles it before the step's alive compaction.

Exact completion is opt-in with `aliveCountReadbackInterval`: after emission completes, a zero authoritative alive count completes the instance. Without it, core conservatively drains for the maximum statically provable lifetime; an unprovable or undeclared lifetime is `Infinity`.

The M9 `@nachi-vfx/timeline` system partitions an update at action, loop, mesh-life, shake-end, and
hit-stop boundaries before delegating every world-time segment to core. A hit stop authored at 0.05
seconds can therefore affect the remainder of a large caller update without moving `System.time`
backward or freezing unrelated game time. With fixed stepping, timeline owns M2's
`FixedStepAccumulator` and delegates its fixed world steps to a core system without a second
accumulator. Timeline and child-emitter clocks consume the same exact step sequence.
Timeline also owns its optional-delta wall clock and applies the same `maxMeasuredDeltaSeconds`
contract before fixed partitioning. Its internal core system receives only explicit boundary
segments, so core never measures or reports the same wall gap a second time. Timeline's three drop
getters therefore report only the timeline-owned measured cap and fixed accumulator. Mesh life, VAT
clocks, actions, shakes, and child emitters all consume the same capped/partitioned segments.
React `useFrame` integrations and the showcase pass explicit deltas, so this default cap does not
change them.

### 10.4 M11 quality, culling, and significance

Nachi has four ordered `QualityTier` values: `low`, `medium`, `high`, and `epic`. An emitter may
declare serializable overrides under `quality[tier]` for `spawnRateScale`, `capacityScale`, and the
`soft`/`lit`/`sorted` feature gates. The system presets are respectively 0.25, 0.5, 0.75, and 1.0
for spawn and logical capacity. Low disables all three gated features; medium enables soft; high
enables soft, lit, and sorted; epic also enables all three. An override replaces only its named values. Every
scale is finite and in `[0,1]`. The authored capacity remains the physical allocation ceiling, so a
live tier change never reallocates or truncates existing GPU storage.

`VFXSystemOptions.qualityTier` accepts a tier or `auto`. Omission selects epic for backward
compatibility. `detectDeviceQualityTier()` obtains `navigator.gpu` adapter limits/features when
available and returns a `QualityTierSelection` containing the selected tier, normalized device
profile, and human-readable reasons. WebGL2 always selects low. WebGPU selects medium by default,
high at the 128 MiB storage-binding/256 invocation/8 storage-buffer plus timestamp-query threshold,
and epic at 256 MiB/512/10 plus timestamp-query and shader-f16. An explicit override wins and is
reported as such; these heuristics are policy, not a claim about a specific marketed GPU model.
Synchronous `VFXSystem` auto-selection cannot discover optional feature names from the renderer
adapter boundary. If `deviceProfile` is omitted it therefore tops out at medium; callers that want
high/epic auto-selection MUST pass the feature-bearing profile returned by device detection.
The root package exports `detectDeviceQualityTier()`, `selectDeviceQualityTier()`, and their device
profile/selection types. Tier application, structural-key generation, bounding/frustum transforms,
and significance-score mathematics are scheduler/compiler implementation details, not public API.

`setQualityTier()` immediately changes CPU spawn-policy scaling and the logical capacity ceiling.
Those are uniform/scheduler values and reuse the same compiled program and materialized pool.
Turning authored `soft`, `lit`, or `sorted` behavior on/off changes the renderer graph, storage
indirection, or shader interface. That is the recompile boundary. Compiled programs and GPU pools
are keyed by the resulting structural renderer variant, not merely by tier name. A live instance
keeps its existing structural variant to preserve simulation state and records
`NACHI_QUALITY_RESTART_REQUIRED`; the next spawn uses the requested compiled/pool variant. This is
an explicit restart boundary and never silently hot-swaps stateful kernels.
Restart diagnostics are deduplicated by `(code, destinationTier)`, so a tier round-trip reports each
structural destination once. The sorted gate applies symmetrically to billboard, mesh, and decal
renderers; low/medium degradation removes sorted variants, while high/epic retain the correctness
default. Automatic WebGL2 selection chooses low and therefore removes unsupported particle-sort
paths before materialization. An explicit high/epic WebGL2 configuration must override sorted to
false or receives the normal unsupported diagnostic.

Each emitter may declare a conservative local sphere `bounds: { center?, radius }`; omission uses a
radius-1000 sphere so frustum culling errs toward rendering. Effect spheres are deterministic unions
of transformed emitter spheres. `effect.scalability.culling.distance` linearly fades from
`fadeStart` to `fadeEnd`, while `frustum` uses the camera matrices supplied to `setCamera()`.
`VfxCameraState.coordinateSystem` declares `webgpu` (`0 <= clipZ <= clipW`, the default) or `webgl`
(`-clipW <= clipZ <= clipW`); near-plane extraction MUST use that explicit convention because a
perspective matrix alone cannot identify it. Fade is the `System.visibility` uniform multiplied
into billboard, mesh, decal, ribbon, and other material opacity. External light pools are registered
with the same kernel visibility channel and are disabled at fade zero. At fade zero, core skips event,
spawn, update, compaction, sort, and rendering work. Its time contract is Niagara-style pause:
effect-local time, emitter lifecycle, and particle state all freeze. System/world time continues;
re-entry resumes from the exact frozen state without catch-up simulation or deferred burst. A
future advance-while-hidden mode would need an explicit reset/catch-up contract and is not inferred.

Significance is evaluated once per scheduling boundary from the same camera/state snapshot. For a
world sphere and user/authored priority `p`, the score is:

```text
distanceScore = 1 / (1 + viewDistance)
screenOccupancy = clamp((abs(projection[5]) * radius / max(abs(viewZ), radius, 1e-6))^2, 0, 1)
score = 4 * p + 2 * screenOccupancy + distanceScore
```

Candidates sort by descending score and then ascending numeric per-system instance creation
sequence. The sequence is the same internal key used by equal-depth coarse alpha ordering; the
public decimal ID string is display/diagnostic/profile identity only. `maxActiveInstances` culls the
remaining tail. `maxParticles` charges each accepted instance its tier-scaled declared capacities;
an otherwise accepted instance that would exceed that budget keeps updating/drawing existing
particles but suppresses new spawn. This conservative accounting avoids mandatory per-frame GPU
readback. `EffectInstance.scalability` exposes action (`full`, `spawn-suppressed`, or `culled`), fade,
reasons, total score, and all score components. Equal camera/state/ids therefore produce identical
culling and budget decisions.

Particle-budget spawn suppression applies to ordinary scheduler births (`burst`, `rate`, and
`perDistance`), not births requested by `emitTo` event consumption. The GPU logical-capacity clamp
remains the authoritative upper bound for both paths.

Budget ordering has a fixed `0.05` score hysteresis: instances admitted at the previous scheduling
boundary receive that bonus only for the next budget-order comparison. The exposed significance
score is unchanged. A challenger therefore must exceed the current admitted instance by more than
the band, preventing near-threshold visible/culled flapping and the pause/resume time stutter it
would otherwise cause.

Tier-scaled logical capacity is enforced even when alive-count readback is disabled. Core keeps a
conservative ledger of dispatched births and expires each batch only after the statically proven
maximum lifetime. Unknown lifetime is `Infinity`, intentionally making this mode conservative until
restart. The WebGPU allocator independently clamps normal and event births against the logical-capacity
uniform using its exact free-list occupancy, so the CPU estimate cannot permit an over-cap birth.
An authoritative readback resets the estimate, after which births since that sample are added
immediately. When logical capacity equals authored physical capacity, the CPU does not clamp
against a stale alive sample; it submits up to the physical request ceiling and leaves exact free-slot
clamping to the GPU allocator, preserving pre-M11 behavior. WebGL2's full-quantity readback refreshes
the estimate every compaction.

`@nachi-vfx/timeline` treats its emitter children as implementation details of the parent choreography:
child sub-effects do not inherit effect culling and the timeline core omits the global significance
budget. Children therefore cannot freeze independently and permanently block parent completion.
Timeline/mesh clocks and completion remain parent-owned; applications that cull timelines MUST do so
as a whole rather than budgeting individual play-action children.

### 10.5 M11 simulation cache

Simulation caches are renderer-ready snapshots, not serialized simulation kernels. Baking uses the
same seed, module PCG, and constant frame step as live execution:

```ts
const cache = await bakeSimulation(system, effect, {
  frames: 121,
  frameRate: 60,
  sampleStartFrame: 120, // warm up for two seconds before recording
  compression: 'float32', // or 'quantized-u16'
  interpolation: 'linear',
  loop: true,
  spawn: { seed: 42 },
});

const player = await replaySimulation(system, effect, cache, {
  loop: true,
  timeScale: 1,
});
player.play();
await player.update(worldDelta);
player.stop();
```

`bakeSimulation()` advances the supplied system and therefore SHOULD receive a dedicated baking
system, not a system shared with live effects. A system configured with `fixedTimeStep` is rejected
as `NACHI_SIM_CACHE_FIXED_TIMESTEP_UNSUPPORTED`: its accumulator could split, defer, or discard the
exact `1 / frameRate` cache steps.
Each bake step executes the same simulation Update dispatches and therefore advances
`Emitter.updateRandomStep` from the bake runtime's zero origin. Replay uploads recorded frames and
does not re-evaluate Update random streams.
Simulation cache v2 records emitter render-read attributes and lifecycle indirection only; it does
not record Grid2D or Grid3D state. Baking an effect containing a grid executes its live stages and
returns `NACHI_SIM_CACHE_GRID2D_NOT_RECORDED` or `NACHI_SIM_CACHE_GRID3D_NOT_RECORDED` in
`SimulationCache.diagnostics`. Replay materializes the effect definition but does not restore or
advance its baked grid state.
The bake instance MUST remain at scalability action `full`; `culled` and `spawn-suppressed` bake
targets are rejected as `NACHI_SIM_CACHE_SCALABILITY_SUPPRESSED` rather than producing frozen or
empty assets. Readback of an emitter whose initialization kernel has not run is rejected as
`NACHI_SIM_CACHE_EMITTER_UNINITIALIZED`.

The first sample is the initialized frame at `sampleStartFrame / frameRate`; `sampleStartFrame`
defaults to zero and advances that many constant simulation steps as warmup before any readback.
Subsequent samples are separated by exactly `1 / frameRate`. The cache is
`{ kind, metadata, diagnostics, data: ArrayBuffer }` so the
JSON metadata remains independently serializable while binary payload transport can use ordinary
asset loaders. Metadata version 2 records emitter keys/capacities, logical attribute types and
component counts, per-frame alive counts/physical indices, a lossless lineage stream, fixed byte
offsets/strides, compression, frame timing, interpolation, source backend, endpoint continuity
(including separate lineage-set and integer-attribute matches), bake-time quality tier, and upload
bytes per frame. Every emitter records one lossless u32 `Particles.spawnOrder` lineage value for
every physical slot in every frame. A v1 cache, or metadata with a missing version, is rejected as
`NACHI_SIM_CACHE_VERSION_UNSUPPORTED` with an explicit instruction to re-bake; v2 never guesses at
or silently migrates an older payload. Replay on a backend different from that source is rejected as
`NACHI_SIM_CACHE_BACKEND_MISMATCH`; v2 does not claim cross-backend lifecycle or physical-slot
equivalence.
All multi-byte scalar fields in the binary payload use little-endian byte order. Bake and load check
host endianness once and reject non-little-endian hosts as `NACHI_SIM_CACHE_ENDIAN_UNSUPPORTED`.
Baking uses a cache-private, bake-only compile variant to materialize lineage when an ordinary
emitter schema does not contain `Particles.spawnOrder`. The original effect definition and ordinary
live/replay emitter schemas remain unchanged: an emitter whose renderer does not consume
`Particles.spawnOrder` does not acquire a permanent birth ring or next-order counter. If a renderer
does consume `Particles.spawnOrder` (notably ribbons), the cache additionally stores the existing
lifecycle birth-index ring and `nextSpawnOrder`; replay restores them before renderer preparation.
When the bake-only variant is needed, baking retires it completely with `releaseUnpooled()` before
sequentially spawning and releasing the original definition once to warm the ordinary pool. The
variant and ordinary GPU resource bundles are not retained together, avoiding a doubled peak. The
observable tradeoff is one additional compile/materialization path during baking and advancement of
the system's instance sequence; callers should continue to use the dedicated baking system described
above.

Attribute selection follows every render module's declared `Particles.*` reads, cross-checked with
the compiled draw vertex attributes. Render `optionalReads` are selected only when they exist in the
ordinary replay schema; an attribute materialized solely by the bake-only lineage variant does not
satisfy an optional read or change the renderer's fallback semantics. Simulation-only state is
omitted. Values are deinterleaved into logical capacity-order streams, so a selected lane does not
accidentally serialize unrelated or transient lanes from the same packed GPU buffer. A transient
attribute required by a render path is
`NACHI_SIM_CACHE_TRANSIENT_RENDER_ATTRIBUTE`; silently retaining it is forbidden. Alive membership,
indices, and lineage are always lossless u32. `bool`/`u32` and `i32` attributes remain lossless
u32/i32.
Floating attributes use either Float32 or per-logical-component global min/max u16 quantization.
Non-finite values reject quantization instead of receiving an implicit replacement.

Loop caches contain a duplicated endpoint: `duration = (frameCount - 1) / frameRate`. Endpoint
comparison builds a lineage-to-physical-slot map for each endpoint, requires the same alive lineage
set, and compares integer and floating attributes at the physical slots mapped by each lineage.
Alive-index order and physical-slot assignment are therefore not continuity criteria. Metadata
publishes the set result as `loop.lineageMatch`. Baking with `loop: true` requires `lineageMatch`,
equal integer attributes, and float endpoint error at or below `loopTolerance`; otherwise it raises
`NACHI_SIM_CACHE_LOOP_DISCONTINUITY`. Playback wraps the duplicated endpoint to frame zero. This makes
the seam an authored/baked contract rather than hiding a discontinuity in the player. A loop bake
SHOULD warm up for at least one complete authored period and record a post-warmup window whose
particle lifetimes, spawn cadence, and deterministic init state are periodic at the seam.
`loop: false` is the non-looping opt-out: it still records endpoint metrics for tooling but never
rejects a discontinuous endpoint, so a second validation-skip option is not needed.

Alive lineage values MUST be unique within each emitter frame. Baking rejects missing or duplicate
alive lineage as `NACHI_SIM_CACHE_LINEAGE_DUPLICATE`. To keep unsigned lineage comparisons
unambiguous, baking also rejects a `nextSpawnOrder` at or beyond the u32 half range (`0x80000000`),
including during warmup, as `NACHI_SIM_CACHE_LINEAGE_WRAP_RISK`; the emitter must be restarted and a
shorter window re-baked before u32 wrap. During warmup, a conservative CPU upper bound of requested
births only triggers an authoritative GPU lifecycle readback; it never rejects by itself because
allocator saturation and failed births can make that bound overestimate progress. Only the actual
GPU `nextSpawnOrder` value can raise the wrap-risk diagnostic.

M11 v2 chooses per-frame CPU decode plus storage upload (`writeBuffer` in a native WebGPU adapter)
instead of making every frame GPU-resident. Web effects commonly load/precompute caches before play,
and this path bounds GPU residency to the existing current-frame attribute/lifecycle buffers while
allowing binary download compression. It also reuses billboard, mesh, ribbon, light, and decal
render-resource bindings: replay reconstructs the compiled physical packing and alive indirection.
Renderer-derived preparation such as particle sorting or ribbon segment construction may still run;
spawn/init/update/event/compaction simulation kernels do not. An all-frame-resident buffer/texture
mode remains a future opt-in for workloads where upload bandwidth is more expensive than residency.
`writeStorage()` replay uploads are flushed before `seek()`, `update()`, or initialization resolves,
so immediate debug/lifecycle readback observes the selected cache frame even before a render call.

`nearest` selects one complete frame. `linear` interpolates floating attributes only when the same
physical particle slot is alive in both adjacent frames and its lineage value matches in both
frames. A reused slot with different lineage and the same lineage appearing in different physical
slots both use the nearest complete frame; interpolation never moves values across physical storage.
Fractions below `0.5` select the left frame and fractions at or above `0.5` select the right frame.
Integer attributes and alive membership use that same nearest-frame rule. New/dead/reused slots
therefore never interpolate against unrelated or stale storage. Given the same cache, playback
options, and time sequence, frame selection, decode, upload bytes, and draw input are deterministic.
Float32 live-vs-replay attribute acceptance is bit equality where the backend preserves readback
bytes; the M11 smoke uses absolute `1e-6` for decoded float comparison and a linear pixel threshold
of `2 / 255` to accommodate render-target conversion. Quantized caches use their recorded
per-component quantization error bounds instead of the Float32 criterion.

For emitter `e`, frame count `F`, capacity `C_e`, selected attribute component counts `k_ea`, bytes
per encoded component `b_ea`, selected physical GPU storage buffers `S_e`, and lifecycle state word
count `L_e`, the budgets are:

```text
binaryBytes = sum_e F * (4 * C_e + 4 * C_e + birthOrder_e * 4 * C_e + sum_a(C_e * k_ea * b_ea)) + alignmentPadding
metadataBytes = UTF8(JSON.stringify(metadata)).byteLength
totalAssetBytes = binaryBytes + metadataBytes
uploadBytesPerFrame = sum_e (sum_s_in_S_e physicalBytes(s) + 4 * L_e + 4)
```

The first `4 * C_e` binary term is the fixed alive-index stream and the second is the mandatory
lineage stream. Thus lineage adds exactly `4 * C_e * F` bytes before alignment for every emitter.
`birthOrder_e` is one when renderer preparation needs the birth ring and zero otherwise (its
next-order scalar stays in metadata). The lineage stream is a CPU-side identity check and is not an
additional ordinary per-frame replay upload; the upload formula is unchanged. The final upload `4`
is the indirect instance-count word; shared packed storage and mat3 padding are counted once in
`physicalBytes(s)`.
Runtime GPU residency remains the compiled current-frame storage plus lifecycle/renderer auxiliaries,
independent of `F`. The public `estimateSimulationCacheMemory()` reports the four asset/upload terms
above.

WebGL2 packed simulation storage is restricted to group 0. If a behavioral spawn/init/update kernel
reads or writes an attribute in group 1 or above, compilation fails with
`NACHI_BACKEND_PACKED_STORAGE_UNSUPPORTED`; Three r185 would otherwise omit the group from the
transform-feedback destination address, alias the access onto group 0, and silently corrupt state.
The compiler-authored defaults pass alone does not trigger this gate because it only establishes
allocated storage; it does not make a higher-group attribute safe for later behavioral access.
Reduced WebGL2 emitters whose behavioral accesses remain entirely in group 0 continue to be a valid
runtime subset; this rule does not revoke those M2 runtime configurations.

Simulation-cache baking has no supported renderable WebGL2 fixture under this strict gate. All float
attributes are packed into one `packed_float` storage, and lifecycle contributes `age`, `lifetime`,
and `normalizedAge` implicitly. A renderable lifecycle emitter requiring `position` therefore needs
at least six float components and reaches group 1 even with an otherwise minimal schema. WebGL2 bake
is consequently treated as unsupported in practice and MUST fail with a structured diagnostic; the
implementation and `/m11-cache/?backend=webgl` probe must not seek numerical equivalence through a
schema that aliases packed storage.

Replay is independently diagnosed as `NACHI_SIM_CACHE_REPLAY_WEBGL2_UNSUPPORTED` in v2:
the current Three renderer integration has no WebGL2 indirect-draw materialization path into which
the restored alive indirection can be bound. It must not claim replay while silently drawing a prefix.
Packed readback is backend-specific: WebGPU preserves `particle * groupCount + group` vec4
addressing, while Three r185 WebGL2 transform feedback emits one vec4 per storage/particle and omits
the packed group from the physical destination address. This physical limitation is exposed through
the strict behavioral-access diagnostic, never normalized into apparently valid WebGL2 cache bytes.

### 10.6 M11 debug snapshots and profiling

Debugging is a code-first runtime surface. An effect instance exposes asynchronous logical-attribute
capture, while the owning system exposes the latest completed update's profiler counters:

```ts
const attributes = await instance.debug.captureAttributes('sparks', {
  attributes: ['position', 'velocity', 'age', 'spawnOrder'],
  offset: 0,
  limit: 256,
});

const profile = await system.debug.captureProfile({ gpuTiming: perfBaseline.gpu });
```

`captureAttributes()` reads the same materialized storage resources and physical-to-logical packing
metadata as simulation-cache baking. It returns one row per selected alive entry with compact
`aliveIndex`, `physicalSlot`, logical typed values, and `spawnGeneration`/`spawnOrder` lineage when
those attributes are declared. Boolean values are booleans, integer and float scalars are numbers,
and vectors/matrices are logical-component arrays (mat3 padding is never exposed). Omitted
`attributes` means every declared compiled logical attribute. Omitted `limit` means every remaining
alive particle; an explicit non-negative `offset`/`limit` selects compact alive order. Every result
contains `{ totalAlive, returned, offset, limit, truncated }`, so neither selection nor backend
limits can silently discard rows. Unknown columns and unavailable readback are structured runtime
diagnostics.

Both capture APIs are asynchronous, non-mutating operations serialized on the owning system's update
queue. A capture runs after every update already submitted when it was called and completes before
any update submitted after it starts. Consequently lifecycle membership, every storage readback, and
profiler counters describe one scheduler boundary even when callers issue `update()` concurrently.
Backend command completion can make an attribute snapshot effectively one frame late relative to
presentation, reported as `latencyFrames: 1`. WebGPU membership follows the compact alive-index
buffer. WebGL2 derives membership from the reduced path's logical `Particles.alive` storage because
it has no compact indirect lifecycle path.
Calling attribute capture before the emitter initialization kernel runs is
`NACHI_DEBUG_EMITTER_UNINITIALIZED`. The root package exposes the instance/system capture surfaces
and snapshot/profile types; formatter, renderer-readback, and frame-aggregation helpers are internal.

The backend-specific packed address rule from §10.5 is non-injective on WebGL2: Three r185 transform
feedback omits the packed group, so logical `(group >= 1, offset)` aliases the corresponding
`(group 0, offset)` physical component. The strict build gate rejects behavioral kernels that access
such columns instead of permitting silent aliasing; group-0-only reduced runtime configurations
remain supported. Debug capture does not weaken that gate. It may still expose a declared
higher-group column that was only mirrored by the compiler-authored defaults pass. For that
inspection case, capture preserves the physical alias value instead of inventing or removing data,
marks the column with `aliased: true`, and adds a warning with code
`NACHI_DEBUG_WEBGL2_ATTRIBUTE_ALIASED` to `snapshot.diagnostics`. Debugger DOMs MUST display the alias
annotation rather than presenting the value as an ordinary available logical attribute. An alias
annotation is diagnostic evidence, not a promise that behavioral access to the column is supported.

Profiler counters reset at each top-level `VFXSystem.update()` and aggregate all fixed substeps and
prewarm work performed inside it. Their definitions are normative:

- `alive` is the latest authoritative GPU/readback value. Without configured readback it is
  `unavailable`, never a conservative estimate presented as fact.
- `capacity` is the active tier-scaled logical capacity charged to the emitter.
- `spawnCount` is the number of normal scheduler spawn invocations submitted after tier, budget,
  and CPU capacity clamps. GPU free-list rejection under concurrent saturation is not relabeled as
  a successful birth.
- `computeDispatches` counts every direct or indirect compute/transform-feedback kernel submission
  made for that emitter, including initialization, spawn preparation/finalization, update,
  compaction, events, and sort stages.
- `indirectDraws` is the number of materialized indirect draw records made renderable for the
  emitter in the most recently completed render frame. It is intentionally one frame late: a
  `captureProfile()` FIFO-serialized after `update()` but before the current render observes the
  preceding completed frame. Fully culled, paused, and never-materialized frames report zero once
  they become the completed frame. WebGL2 reports
  `NACHI_PROFILE_INDIRECT_DRAW_WEBGL2_UNAVAILABLE` instead of zero.
- `cpuUpdateMs` is host wall time spent awaiting and coordinating that emitter's scheduler work,
  including adapter promise overhead. It is not GPU execution time and is not used as a substitute
  for it.

GPU timing is pass-granularity in M11. `captureProfile()` never allocates or resolves timestamp
queries itself. The application supplies the cached `gpu` record owned by the existing
`nachi.perf-baseline` v1 monitor, preventing double instrumentation and timestamp-pool growth.
Available `computeMs`/`renderMs` remain pass totals and are not divided among emitters. Missing,
pending, errored, or unsupported values stay typed as unavailable with a reason and null value.
WebGL2 compute timing specifically reports
`NACHI_PROFILE_GPU_TIMESTAMP_WEBGL2_COMPUTE_UNAVAILABLE`; timestamp-query absence is never replaced
with a CPU estimate. Correctness/long-run loops SHOULD use a renderer with timestamp tracking off,
then run the short perf capture separately, preserving the established timestamp-pool defense.
The shared perf snapshot marks a zero Three current-frame `computeCalls` counter as unreliable with
an explicit reason because Three may have reset it after submitted compute work.

### 10.7 M12 Grid2D/Grid3D data interfaces and simulation stages

Simulation stages are effect elements, separate from an emitter's Spawn/Init/Update/Event/Render
module stack. The target is the key of a Grid2D or Grid3D data-interface element; using a string
instead of a live object reference keeps the complete graph declarative and makes broken references
diagnosable:

```ts
const fluid = defineGrid2D({
  resolution: [64, 64],
  channels: {
    density: { type: 'f32', default: 0 },
    temperature: { type: 'f32', default: 0 },
    velocity: { type: 'vec2', default: [0, 0] },
    pressure: { type: 'f32', default: 0 },
  },
});

const pressure = defineSimStage({
  target: 'fluid',
  phase: 'after-particles',
  iterations: 8,
  update: gridPressureJacobi(),
});

const effect = defineEffect({ elements: { fluid, pressure } });
```

`phase` is `before-particles` or `after-particles` and defaults to the latter. At one effect
scheduler boundary, all stages for a grid target retain author element order within their phase.
The boundary is:

```text
before-particles grid stages -> particle event/spawn/init/update -> after-particles grid stages
```

Independent grid targets have no cross-target visibility contract. A dependency between grids
requires a future explicit multi-target data-interface declaration. Grid stages use effect-local
scaled delta time and pause with the effect. Emitter prewarm currently advances inside the ordinary
particle boundary rather than interleaving one Grid2D step around every internal prewarm substep;
coupled grid/particle prewarm therefore requires explicit external fixed steps in v1.

Every positive `iterations` count is literal. One iteration submits the update kernel, then submits
a scratch-to-state commit kernel. These are separate backend submissions. WebGPU defines storage
visibility between consecutive dispatches in one pass, but the M10 sorting experiments observed
that dependent dispatches recorded through Three r185 did not provide the required behavior.
Grid2D therefore conservatively uses separate Three submissions for stage and commit. Grid clear
is a separate one-time submission. Initialization intentionally runs both stage phases once with
`deltaTime = 0`; together with clear, the 4x3 scheduling smoke's two ordinary updates therefore
assert `submissionCount = 13`. `Grid2DRuntimeView.submissionCount` exposes these submissions for
debug and smoke assertions.

Grid2D v1 uses storage buffers rather than storage textures. Logical `f32` and `vec2` channels are
packed deterministically into one or more vec4 records per cell, but every record remains in the
same physical buffer. Runtime state therefore binds one current buffer and one scratch buffer even
for density, temperature, velocity, and pressure. This avoids spending the device's commonly small
per-stage storage-buffer budget on SoA grid channels and works with Three r185's established
`instancedArray` storage path. Addressing is x-fast row major (`index = y * width + x`), coordinates
are cell-centered, boundaries clamp, and `sample()` is four-tap bilinear filtering performed in TSL.
`gridCellIndex()`, `sampleGrid2DBilinear()`, and `grid2DSnapshotChannel()` are CPU mirrors for tests
and tooling.

`gridTslModule(factory)` receives `index`, integer cell coordinates, effect-local `deltaTime`,
`read(channel)`, and `sample(channel, cell)`, and returns channel writes. It is the code-first
scratch-pad escape hatch. Inline factories are non-serializable. A
`defineGrid2DStageFunction(id, factory, version)` registration serializes only its exact
`grid2d-function-ref`; `Grid2DStageRegistry` supplies execution. Built-in modules serialize their
stable `core/grid2d-*` source and JSON config directly.

Particle transfer is a data-interface operation on normalized particle positions. WebGPU
`rasterizeParticles(points, scalarChannel, value)` clears a u32 accumulation buffer, dispatches one
particle per invocation with fixed-point atomic addition at scale 4096, then resolves into the
float grid state in a separate submission. Duplicate deposits are additive; values are
non-negative and quantized to `1/4096`; negative or non-finite authored values are rejected before
submission. Every point coordinate supplied to rasterization or sampling must be finite; finite
coordinates outside `[0, 1]` retain the documented edge-clamp behavior. For one deposit, callers
must satisfy `value * 4096 <= 0xffffffff`, which runtime validates. Callers must also ensure the sum
of all fixed-point deposits targeting one cell is at most `0xffffffff`. The runtime does not perform
a CPU cell histogram to validate that aggregate because doing so would duplicate the GPU
rasterization; an overflowing atomic sum has ordinary u32 modulo-`2^32` wrap semantics and the
resolved float therefore contains the wrapped, not saturated, value.
`sampleParticles(points, scalarChannel)` dispatches one particle per invocation and returns the
same cell-centered bilinear result used by grid stages. V1 transfer accepts at most one grid-cell
count of points per call and scalar target/source channels; direct binding of an emitter's packed
particle storage and vec2 transfer channels remain later extensions. These explicit methods still
avoid inventing a CPU interpolation result: both deposition and sampling execute against live GPU
Grid2D storage. Rasterization reset/resolve and sampling are each dispatched over the grid's full
cell capacity, so their dispatch work is always `O(C)` for `C = cellCount`, even when fewer points
are supplied.

All public `capture()`, `rasterizeParticles()`, and `sampleParticles()` calls returned by
`getGrid2D()`, `getGrid3D()`, and `getNeighborGrid()` enter the same FIFO as `VFXSystem.update()` and
attribute/profile captures. A call made during an in-flight update runs only after that update is
complete, and it blocks subsequent updates until its submissions/readbacks finish. In particular,
a NeighborGrid capture reads counts, slots, and stats from one completed rebuild rather than
allowing an update to interleave between those three reads.

The `/m12-grid/` reference loop uses source injection, first-order semi-Lagrangian advection,
exponential per-channel dissipation, temperature-minus-density buoyancy, repeated Jacobi pressure
solves, and pressure-gradient velocity projection. Jacobi is deliberately approximate: eight
iterations reduce local divergence but do not converge low-frequency pressure on large grids.
There is no vorticity confinement, obstacle mask, MacCormack/BFECC correction, adaptive timestep,
or mass-conserving advection. Semi-Lagrangian advection is stable but numerically diffusive; clamp
boundaries behave as closed edge replication rather than a physically derived wall condition.
Velocity channels and advection coordinates use grid-cell units (velocity is cells per second),
while `gridInject()` center and radius use normalized grid coordinates; injected channel values are
per-second additions, with injected velocity values likewise expressed in cells per second.

Simulation cache v2 does not record Grid2D buffers. As specified in §10.5, bake runs live Grid2D
stages but reports `NACHI_SIM_CACHE_GRID2D_NOT_RECORDED`, while replay does not schedule Grid2D
simulation; no baked Grid2D evolution is reconstructed.

WebGL2 cannot provide arbitrary indexed storage writes, atomics, or indirect dispatch through the
Three r185 transform-feedback fallback. Grid2D materialization reports
`NACHI_GRID2D_WEBGL2_UNSUPPORTED`; no texture ping-pong, CPU fluid, or prefix transform-feedback
simulation is selected silently. The WebGL2 `/m12-grid/` smoke asserts this diagnostic only.

Grid2D and sim-stage definitions entered the declarative subset while the `nachi-effect` envelope
was version 1. The current serializer round-trips the same closed payload fields in canonical
envelope v2; the default v1 → v2 migration does not rewrite grid or stage definitions. Inline
Grid2D TSL reports `NACHI_ASSET_INLINE_FUNCTION`; registered references and built-in configs remain
JSON.

M12 batch 3 extends the same contract to `defineGrid3D()`. Grid3D channels are `f32` or `vec3`,
addressing is x-fast (`index = (z * height + y) * width + x`), and coordinates are cell-centered
with clamp boundaries. Logical channels are packed into vec4 records in one current buffer and one
scratch buffer. `sample()` uses eight-tap trilinear interpolation. Every clear, stage, commit,
particle raster, resolve, and particle sample kernel guards `instanceIndex < cellCount`; dispatch
rounding must never rely on out-of-range storage behavior. A Grid3D target accepts only a
`grid3d-stage-module`, while a Grid2D target accepts only a `grid2d-stage-module`; mismatches report
`NACHI_SIM_STAGE_TARGET_KIND_MISMATCH`. Both retain the same before/after-particle boundary,
literal iteration count, independent submit contract, pause behavior, and ping-pong semantics.

For resolution `W × H × D`, let `C = W * H * D` and `G` be the packed vec4 records per cell.
Current plus scratch consumes `2 * C * G * 16` bytes. The v1 grid-to/from-particle path reserves a
vec3 position store with 16-byte WGSL array stride, a float sample store, and a u32 atomic store, so
its additional budget is `C * (16 + 4 + 4)` bytes. The public `estimateGrid3DMemory()` result is
therefore `totalBytes = 32 * C * G + 24 * C`, with an explicit allocation breakdown. The common
smoke schema uses `G = 2`: 32³ consumes 2.75 MiB, 48³ about 9.28 MiB, and doubling every dimension
multiplies memory by eight. Before allocating, runtime compares the largest individual binding with
the smaller available `maxStorageBufferBindingSize`/`maxBufferSize` limit and reports
`NACHI_GRID3D_STORAGE_LIMIT_EXCEEDED`; it never retries at a hidden lower resolution.
Grid2D applies the symmetric safe-product and active device-buffer checks and reports
`NACHI_GRID2D_RESOLUTION_INVALID` or `NACHI_GRID2D_STORAGE_LIMIT_EXCEEDED` before allocation.

Grid3D built-ins are the direct 3D forms of injection, first-order semi-Lagrangian
advection/exponential dissipation, y-axis buoyancy, six-neighbor Jacobi pressure solve, and
three-axis pressure-gradient projection. Velocity remains grid cells per second, while injection
centers/radii use normalized volume coordinates. The Grid2D limitations above also apply to Grid3D:
first-order backtracing is numerically diffusive, repeated Jacobi leaves low-frequency divergence
when iteration counts are small (more visibly so as a volume grows), edge clamp is not a physical
wall solve, and v1 has no obstacle mask, vorticity confinement, MacCormack/BFECC correction,
adaptive timestep, or mass-conserving transport. These limits are part of the method, not silently
corrected by the renderer.

Volume rendering is staged deliberately. V1 samples density at normalized 3D particle positions
with the same storage-buffer data-interface path used by Grid2D, then renders those samples as
depth-layered particles or simple slices/billboards. This proves live grid-to-particle coupling and
gives a bounded SwiftShader path without adding a second volume resource system. A future stage may
bind the packed density field as a filterable 3D texture and add ray marching, empty-space skipping,
shadow integration, temporal reprojection, and physically based extinction/scattering. Full ray
marching is not a v1 Grid3D requirement.

`defineGrid3DStageFunction()`/`Grid3DStageRegistry` are the serializable custom-stage boundary;
inline `grid3DTslModule()` remains code-only. Grid3D declarations and registered/built-in stages
were originally serialized in the then-current `nachi-effect` version 1 envelope. They now
round-trip with the same payload and module versions in canonical envelope v2; default migration
changes only the envelope version. WebGL2 materialization reports only
`NACHI_GRID3D_WEBGL2_UNSUPPORTED`; it does not fall back to CPU volume simulation or texture
ping-pong. Simulation cache v2 likewise warns with `NACHI_SIM_CACHE_GRID3D_NOT_RECORDED` because it
does not record volume state.

M12 batch 4 adds `defineNeighborGrid()` as the particle-oriented data interface at the same
before-particles scheduler boundary. A declaration fixes `resolution`, cubic `cellSize`,
emitter-local minimum-corner `origin`, and `cellCapacity` (default 32). `cellSize`, search distance,
and PBD distance remain world lengths because the current emitter transform composes translation
and rotation, but not scale. Exactly one
emitter may reference a v1 neighbor grid through `boids()`, `pbdDistanceConstraint()`, or
`neighborGridTslModule()`; zero consumers report `NACHI_NEIGHBOR_GRID_UNBOUND`, multiple consumers
report `NACHI_NEIGHBOR_GRID_MULTIPLE_EMITTERS`, and a missing key reports
`NACHI_NEIGHBOR_GRID_TARGET_UNKNOWN`. This one-source restriction avoids suggesting that two
independently packed emitter schemas form one coherent index space.

Immediately before every ordinary particle update, runtime submits a clear kernel and then a
bucket kernel. The bucket kernel reads the emitter's alive flag and world-space position, maps that
position through `inverse(Emitter.transform)`, computes
`floor((emitterLocalPosition - origin) / cellSize)`, and reserves a cell slot with `atomicAdd`.
The same cell-coordinate helper is used by bucket insertion and the current-cell lookup shared by
boids, custom neighbor visitors, and PBD. `Emitter.transform` already composes the instance
translation/rotation with `EmitterConfig.offset`, so spawn transforms, `setTransform()`, attachment
updates, and offsets move and rotate the grid automatically. Counts are one
u32 atomic per cell and indices are a fixed cell-major `cellCount * cellCapacity` u32 array. A
reservation below `cellCapacity` writes the physical particle index; a later reservation is
dropped, increments the `dropped` counter, and does not stop simulation. Positions outside the
declared volume are dropped separately and increment `outOfBounds`. `getNeighborGrid(key).capture()`
reads counts, fixed slots, and both counters; counts retain attempted inserts and can therefore
exceed capacity. A nonzero drop also returns `NACHI_NEIGHBOR_GRID_CELL_OVERFLOW` in the capture's
diagnostics. Clear writes the `0xffffffff` empty-slot sentinel. Every clear, bucket, and
constraint kernel guards its rounded invocation against its logical slot/cell/particle count.

`setTransform()` changes the grid frame used by the next rebuild; it does not retroactively move
existing world-space particle positions. The m12 live-transform fixture therefore updates the
transform before its first spawn. Applications that move an emitter after particles exist must
choose whether those world-space particles should remain near the newly moved local grid volume.

Capture also derives `inBounds` by summing attempted cell counts. When `total = inBounds +
outOfBounds` is nonzero and `outOfBounds / total` is strictly greater than 50%, it returns
`NACHI_NEIGHBOR_GRID_OUT_OF_BOUNDS_DOMINANT`. The diagnostic context includes both counts, total,
ratio, grid key/definition, emitter path, and the `neighbor-grid` kernel kind. It is accumulated on
the owning instance and delivered at most once during that NeighborGrid runtime lifetime; repeated
captures and later dominant rebuilds do not redeliver it. A pool checkout constructs a new runtime
and rearms delivery. Snapshot diagnostics continue to describe every captured rebuild, and cell
overflow and dominant out-of-bounds may coexist. Exactly 50% and zero total do not trigger it.

The bucket submission also snapshots each particle's world-space position and velocity. Neighbor update code
then scans the cubic cell neighborhood and each retained fixed slot, reading the snapshot by the
stored particle index. The current particle's position and velocity are instead read from the live
attributes, coherently within the invocation, so writes from earlier update modules remain the
base state; only neighbor state comes from the bucket-time snapshot. This live-self/snapshot-neighbor
rule applies to built-in boids and `neighborGridTslModule()` context state.
`neighborGridTslModule({ grid, radius, access }, factory)` is the code-only scratch-pad abstraction:
its context exposes the current index/live position/live velocity and `forEachNeighbor(visitor)`
with neighbor index/snapshot position/snapshot velocity. V1 deliberately snapshots those two
classical flocking/constraint neighbor attributes; arbitrary custom-attribute snapshots remain a
later schema extension. Accumulation inside the visitor uses a TSL local (`value.toVar()` followed
by `addAssign()`/`assign()`) because cell offsets and fixed slots are runtime GPU loops. The inline
helper is not serializable. Built-in `boids()` and
`pbdDistanceConstraint()` are ordinary declarative modules and do serialize.

`radius` is a non-negative integer measured in cells. The scan cost per particle is
`O((2 * radius + 1)^3 * cellCapacity)` branches before distance rejection, so radius 1 and the
default capacity 32 are the intended real-time baseline (at most 864 retained-slot tests per
particle). The fixed capacity makes memory and shader work predictable but biases an overfull cell
toward whichever atomic reservations win; atomic insertion order is not stable across devices or
runs. Even below capacity, retained slot order is the atomic reservation order, so floating-point
neighbor accumulation can differ at low bits. When `cellCapacity` overflows, the retained neighbor
set and overflow winner can differ materially. Generous `cellCapacity` headroom avoids lossy winner
selection but does not promise a canonical bucket order; order-sensitive simulations requiring
bitwise replay need a future deterministic neighbor mode. Overflow is therefore an explicit lossy
approximation, never a hidden resize or CPU fallback.
Before allocation, the largest slot/snapshot binding is checked against the active buffer limits;
an oversized declaration reports `NACHI_NEIGHBOR_GRID_STORAGE_LIMIT_EXCEEDED` rather than silently
reducing resolution or capacity.

`boids()` implements classical separation, alignment, and cohesion. Alignment steers toward mean
neighbor velocity, cohesion toward mean neighbor position, and separation accumulates inverse
squared displacement inside `separationRadius`; the combined acceleration is bounded by
`maxAcceleration` and integrated into velocity with emitter-local delta time. The `/m12-neighbors/`
control omits the neighbor module and gates on a materially smaller dispersion radius in the live
flock, so zeroed/no-op neighbor reads cannot pass.

`pbdDistanceConstraint({ distance, stiffness, iterations })` is the minimal pair-distance push-out.
Each iteration performs clear, bucket/snapshot, and a separate constraint submission. Every
particle independently computes half of each penetration correction from the iteration snapshot,
averages all incident corrections, and applies `stiffness` (default 1); this is Jacobi-style rather
than an in-place Gauss-Seidel solve. `iterations` defaults to 1. Rebuilding and submitting between
iterations is mandatory because Three r185 cannot supply a whole-grid barrier between dependent
dispatches in one compute pass. One to a few iterations remove soft overlaps, but finite iteration
count, averaging, overflow dropping, and `stiffness < 1` make rigidity approximate. V1 has no
inverse-mass weighting, pinned particles, compliance/XPBD, graph coloring, friction, shape matching,
or guaranteed convergence for dense contact graphs. Exactly coincident pairs also remain coincident
because v1 does not inject a deterministic jitter direction.

PBD's submit-separated iterations execute before the emitter's ordinary Update-module kernel.
Consequently an authored stack that visually reads `integrate -> pbdDistanceConstraint` still runs
the constraint solve first and the ordinary integration/update modules second; v1 does not promise
author-stack interleaving across this separate-kernel boundary.

Grid2D, Grid3D, and NeighborGrid storage is allocated anew for every non-pooled effect spawn. The
current effect pool reuses emitter kernels only; pooling these data-interface buffers is a future
optimization and must preserve the same clear/initialization and FIFO visibility contracts.

Neighbor grids require storage atomics and arbitrary indexed reads. WebGL2 materialization reports
`NACHI_NEIGHBOR_GRID_WEBGL2_UNSUPPORTED`; it never selects a CPU boids solver or transform-feedback
approximation. Neighbor-grid declarations were introduced in the then-current envelope v1 with
closed fields (`kind`, `version`, `resolution`, `cellCapacity`, `cellSize`, `origin`). Current
serialization places those unchanged declaration fields in canonical envelope v2, and the default
v1 → v2 migration adds or migrates no payload fields. H2-5 changes the runtime interpretation of
existing `origin` values from world-space to emitter-local. Code or assets that deliberately paired
an absolute origin with a transformed emitter must migrate that origin into the emitter frame.

## 11. Serialization and registries

### 11.1 Canonical document

The asset envelope is versioned independently from modules:

```ts
interface EffectAssetDocumentV2 {
  format: 'nachi-effect';
  version: 2;
  effect: JsonValue;
}
```

`@nachi-vfx/format` exports `effectAssetSchemaV2`, the `EffectAssetDocumentV2` type (also the current
`EffectAssetDocument` alias), `serializeEffect(definition)`, and `loadEffect(json)`. Serialization
operates on normalized `defineEffect()` output and emits only the canonical envelope above. Loading
accepts native v2 or a historical v1 object (or either JSON string representation), applies the
configured migration boundary, and returns an ordinary eager `EffectDefinition`; it never returns
a partially validated document as a runnable definition. `effectAssetSchemaV1` and
`EffectAssetDocumentV1` remain compatibility exports for inspection and explicit migration tooling,
not the current serializer contract.

Grid-stage source strings resolve only against the built-in `core/grid2d-*` or `core/grid3d-*`
vocabulary. A serialized custom `grid2d-function-ref` or `grid3d-function-ref` must resolve exactly
through `LoadEffectOptions.grid2DStageRegistry` or `grid3DStageRegistry`; unknown built-ins report
`NACHI_ASSET_GRID_STAGE_SOURCE_UNKNOWN`, and missing registrations report
`NACHI_ASSET_GRID_STAGE_FUNCTION_UNRESOLVED` during validation/load rather than reaching core
materialization.

Current v2 validation closes the envelope, effect/emitter/module/access/parameter/attribute/
timeline/scalability structures, known tagged value generators, and resource references. Unknown
fields on those format-owned structures are errors. Module-config validation is version aware:
owned built-ins such as renderer@2 use their closed nested shape, while an otherwise unowned
registered module config remains an extensible JSON object for its registry validator. After
structural validation, the loader runs the existing `defineEmitter()` and `defineEffect()`
normalization so attribute, lifecycle, parameter composition, and M9 merge invariants remain one
contract. Timeline target/value runtime validation remains owned by `@nachi-vfx/timeline`; the
format layer does not move that responsibility into core.

Format validation nevertheless enforces the same serializable action value domains as timeline
authoring: non-negative hit-stop duration/scale, non-negative camera-shake strength/duration,
positive shake frequency, and non-empty marker names. The stable load diagnostics are
`NACHI_ASSET_TIMELINE_HIT_STOP_INVALID`, `NACHI_ASSET_TIMELINE_CAMERA_SHAKE_INVALID`, and
`NACHI_ASSET_TIMELINE_MARKER_INVALID`.

Serialization preserves normalized helper output, module versions, `User.*` defaults, immutable
curve/gradient tuples, color strings (including `#RGBA` and `#RRGGBBAA`), semantic module/timeline
array order, and all render/quality/scalability data. Object key order is not semantic. It rejects
`undefined`, non-finite numbers, bigint, symbols, functions, symbol keys, cyclic graphs, class
instances, Three.js/DOM/GPU objects, and other non-JSON values. Diagnostics use the stable
`NACHI_ASSET_*` prefix with phase `serialize` or `deserialize` and an exact data path. Loading does
not silently discard an unknown field, version, action, reference, or type mismatch. JSON-bearing
configuration is limited to 256 nested containers, and parsed-object sparse arrays are rejected, so
host recursion errors and array holes are also reported through `NACHI_ASSET_*` diagnostics.

The asset envelope version is independent of module versions. The default registry performs one
v1 → v2 envelope-only migration: it changes only the top-level `version`, does not mutate the input,
and preserves the effect payload and every module version unchanged. A renderer module-v1 record
therefore remains module v1 after migration. `EffectAssetMigrationRegistry.register(fromVersion,
toVersion, migrate)` is the explicit custom-migration boundary; every step must return a
`nachi-effect` envelope whose `version` exactly equals the registered destination. Validation then
uses the migrated document's actual module versions, so a custom migration that creates renderer@2
must satisfy its strict config contract. A v1 envelope cannot reserve renderer@2 in an actual render
slot, and a v1-only old reader rejects canonical envelope v2 before compilation. Missing paths,
conflicting registrations, cycles, or malformed results are diagnostics. There is no semver
inference or best-effort downgrade. RFC 006 §9 owns the renderer-specific guard and safe-rejection
details.

### 11.2 Resources

Textures, meshes, SDFs, vector fields, VAT data, and other external assets use serializable references such as:

```ts
interface AssetRef<AssetType extends string> {
  kind: 'asset-ref';
  assetType: AssetType;
  uri: string;
}

type TextureRef = AssetRef<'texture'>;
type FieldRef = AssetRef<'vector-field'>;
type MeshRef = AssetRef<'mesh'>;
type SdfRef = AssetRef<'sdf'>;
```

The runtime resolves refs through an asset resolver. Raw `THREE.Texture` or `Object3D` values may be wrapped for ephemeral code-only effects, but serialization must reject them until replaced by refs.

The M9 `timeline/mesh-fx` placeholder is also a reference boundary. JSON stores only its resource
ID and duration. After `loadEffect()`, `bindMeshFxResources()` resolves those IDs to live Three.js
meshes and attaches them out of band through the timeline runtime's WeakMap. This is required before
playback but does not mutate or enrich the JSON document. Missing resources are explicit timeline
diagnostics. Golden #5 uses this path, with the JSON-loaded definition driving the ordinary timeline
and GPU renderers while a same-seed code definition supplies the equivalence control.

External asset/document IDs beginning with `$` are reserved for runtime-owned identities. In
particular, `$root` denotes the loader's root document and MUST NOT be used as a caller-supplied
asset ID.

ASCII FGA vector fields are parsed by the renderer-independent `parseFga()` helper. The header is
three positive integer resolution values followed by minimum and maximum vec3 bounds; exactly
`resolution.x * resolution.y * resolution.z` vec3 samples follow in X-major order.
`vectorField({ field, strength, tiling })` stores only its serializable `FieldRef`. During kernel
materialization the backend resolver maps that ref to a bounds-aware 3D texture. Particle world
position is normalized by the bounds. FGA samples are corner-aligned: normalized coordinate zero
addresses the first texel center and one addresses the last, using the per-axis transform
`u * (resolution - 1) / resolution + 0.5 / resolution`. The default clamps normalized positions
before this transform. `tiling: true` leaves coordinates unbounded and uses repeat sampler wrapping,
including interpolation across the last/first texel seam; it does not implement repetition as
`fract()` followed by a clamped sample. Float32-filterable backends use trilinear sampling; other
backends fall back to nearest sampling. The sample is treated as acceleration and integrated with
`Emitter.deltaTime`.

`MeshRef` surface resources contain a deformed triangle-position texture and normalized area CDF
texture. `SdfRef` resources contain float distance samples plus bounds/resolution metadata. Both
are resolved by backend integration and remain serializable references in effect definitions.

### 11.3 Asset-reference emitter inheritance

Eager code authoring continues to use `defineEmitter(base, overrides)`. The canonical v2 document
accepts this element node; a historical v1 document containing the same node reaches it unchanged
through the default envelope-only migration:

```ts
interface EmitterAssetExtension {
  kind: 'emitter-extends';
  extends: string; // "#base" or "library://effects#base"
  overrides: EmitterOverrideConfig;
}
```

The reference contains exactly one `#`: `#element` addresses an emitter in the current document,
while `asset-id#element` invokes the synchronous `LoadEffectOptions.resolveAsset(assetId)` document
resolver. Effect element keys (emitter names included) cannot contain `#`; this keeps local
references such as `#element` unambiguous and prevents them from being misrouted to the external
resolver. The loader resolves the complete chain,
then calls `defineEmitter(base, overrides)`, so capacity/spawn child-wins behavior, lifecycle/schema
merges, label/index stack replacement/removal/order, and all M9 diagnostics retain their meaning.
The loaded `EffectDefinition` contains only the resulting eager emitter—no prototype, reference
node, or parent identity enters compilation. Missing resolvers/assets/elements, non-emitter bases,
invalid reference syntax, and local or cross-document cycles are `NACHI_ASSET_*` deserialize errors.

### 11.4 Executable registrations

There are three executable registry categories:

1. module compiler implementations (`type` + `version`);
2. TSL factories (`id` + `version`);
3. JavaScript callbacks/integration commands (`id` + `version`).

The three categories use distinct reference types: module lookup uses `(type, version)`, TSL uses `TslFunctionRef`, and JavaScript/integration callbacks use `CallbackRef`. `VfxRegistry` exposes `resolveModule`, `resolveTsl`, and `resolveCallback` accordingly; one category's reference cannot resolve through another category.

The JSON contains references only. Registries are supplied by application/package setup and are not embedded in assets. Deserialization may produce a parsed document before all registrations exist, but compilation cannot succeed with unresolved required registrations.

Both `tslModule(functionRef)` and `tslModule(defineTslFunction(id, factory, version))` serialize the
module source as the same `{ kind: 'function-ref', id, version }` JSON value. The registration form
may retain its factory as non-enumerable authoring metadata on the module, but that metadata is not
walked into or emitted by the serializer.

Registration conflicts for the same ID/version are errors unless the implementation identity is exactly the same registration object. Version lookup is exact, not semver-compatible: a request for version 1 is incompatible when only version 2 is registered. An envelope-only migration does not change that reference version. An explicit custom migration that intentionally upgrades a reference belongs to `@nachi-vfx/format`; its output must reference the exact registered version.

M12 batch 1 introduces serialization and executable-reference preservation while retaining the
existing `KernelModuleRegistry` compilation boundary. That registry already enforces the same
module-category conflict rule (an identical registration object is idempotent, while replacement at
the same type/version is an error) and exact version lookup. A broader multi-category `VfxRegistry`
facade remains later M12 ecosystem work rather than being implied by JSON parsing alone.

### 11.5 Inline function policy

`tslModule(callback)` and future callback-taking authoring helpers are marked non-serializable until registered. The callback may be held as non-enumerable authoring metadata, but it is not JSON. `serializeEffect()` reports its exact path and suggests the registration API.

Simulation-cache metadata/binary embedding or references inside an effect document are a future
asset-format version. Neither historical envelope v1 nor canonical envelope v2 includes them; both
contain effect-definition JSON only.

## 12. Error model

### 12.1 Structured diagnostics

Compiler and loader operations return or throw diagnostics with:

```ts
interface VfxDiagnostic {
  code: string;
  severity: 'error' | 'warning';
  phase: 'compile' | 'deserialize' | 'runtime' | 'serialize';
  message: string;
  path?: string;
  hint?: string;
  context?: {
    emitterPath: string;
    kernel:
      | 'compaction'
      | 'event'
      | 'init'
      | 'neighbor-grid'
      | 'sim-stage'
      | 'sort'
      | 'spawn'
      | 'unknown'
      | 'update';
    neighborGrid?: {
      key: string;
      inBounds: number;
      outOfBounds: number;
      total: number;
      outOfBoundsRatio: number;
      cellCapacity: number;
      cellSize: number;
      origin: [number, number, number];
      resolution: [number, number, number];
    };
  };
}
```

Public diagnostic codes are stable enough for tests and tooling. Human messages may improve without a breaking release.

### 12.2 Compile-time validation

The following are compile errors before GPU work is submitted:

- unknown module, function, callback, asset, attribute, parameter, event target, or timeline target;
- version mismatch without a registered migration;
- invalid capacity, lifecycle, curve, gradient, or renderer config;
- attribute type/default mismatch or incompatible built-in redeclaration;
- undeclared required read, illegal stage write, read/write trace mismatch, or dependency cycle;
- backend capability required with no fallback;
- non-deterministic or unsupported TSL graph;
- unresolved inline function during serialization;
- resource layout exceeding known device limits.

Validation should accumulate independent diagnostics rather than fail at the first module. No partially compiled effect enters the active scheduler.

Authoring factories fail fast for engine-independent static configuration errors. They throw
`VfxDiagnosticError` synchronously and use the same validator as compile-time registry validation.
The compile check remains mandatory because JSON-loaded and manually constructed module data does
not pass through the factory. Constraints requiring the emitter schema, another effect element,
backend capabilities, device limits, or traced TSL execution remain compile-only.

Ordinary built-in `ValueInput` constants and every statically visible generator value MUST be finite
and match every field's scalar/vec3 shape, including nested fields. Required fields reject omission
or `undefined`; optional omission retains its documented default. Parameter generators require a
string path even when no fallback is present, and also match the declared logical type of `User.*`
and materialized built-in uniforms; the latter use the same path/type definitions as uniform
materialization. The shared factory/compiler diagnostic is
`NACHI_VALUE_INPUT_INVALID`. `turbulence.octaves` is a safe integer in `[1,4]` and reports
`NACHI_TURBULENCE_OCTAVES_INVALID`. Collision `mode` uses only `bounce | kill | stick` and reports
`NACHI_COLLISION_MODE_INVALID`; it is required for plane/sphere/box/SDF and optional only for
scene-depth collision. A `Particles.normalizedAge` reader reports
`NACHI_NORMALIZED_AGE_WITHOUT_LIFETIME` unless authored access owns writes to both `Particles.age`
and `Particles.lifetime`, or owns an explicit `Particles.normalizedAge` write. Reads that merely
allocate age/lifetime buffers do not satisfy this ownership contract.

If `Particles.lifetime` is allocated without `Particles.age` and no lifecycle is declared, the
compiler emits the `NACHI_LIFETIME_WITHOUT_AGE` warning. The warning directs authors to add a
lifecycle declaration or an explicit age write. The compiler does not inject `age`, because doing
so would silently change the declared buffer layout.

Spawn-time compile, kernel-build, and resource-materialization diagnostics are also reported through
`VfxSystemOptions.onBuildDiagnostic`. When the property is omitted, each error is sent in one call
to `console.error` and each warning in one call to `console.warn`. Passing `null` disables this
reporting; passing a function replaces it. If the handler throws, the exception is contained and
recorded exactly once as the `NACHI_BUILD_DIAGNOSTIC_HANDLER_FAILED` warning; that warning is not
recursively delivered to the failing handler. This hook does not report ordinary runtime
diagnostics. `NACHI_RUNTIME_MATERIALIZATION_FAILED` remains in this build-delivery layer despite its
historical code name and runtime phase because it is produced while constructing a spawn, before
the instance enters post-spawn scheduling.

`VfxSystemOptions.onRuntimeDiagnostic` delivers diagnostics when a runtime source observes them.
Omission uses the same one-line console formatter as build diagnostics, `null` disables delivery,
and a function replaces the formatter. Delivery configuration never suppresses storage in an
owning instance's `diagnostics`. A throwing handler never rejects spawn, prepare, update, or
capture. The owning instance records `NACHI_RUNTIME_DIAGNOSTIC_HANDLER_FAILED` once and the default
formatter reports that fallback without recursively passing it to the failed handler. An ownerless
system source has no instance storage target: its fallback is emitted to the default console at most
once per system and no synthetic failure diagnostic is stored. A later source still retries the
replacement handler. Delivery occurs after the primary diagnostic is stored and any associated
error-state transition is applied, so a synchronous handler observes the same state later exposed
to the host.

Core runtime delivery includes post-spawn GPU submission failure, attachment failure, device loss,
preparation failure/cleanup, runtime camera/quality/capacity warnings, NeighborGrid runtime warnings,
and spawn/event overflow detected by an enabled alive-count readback. Spawn-time compile, kernel,
and materialization failures remain exclusively owned by the build-delivery path described above.
Timeline delivers its own validation, action/callback, attachment/update, boundary, companion,
cleanup, and mesh preparation diagnostics through the same option. A core child diagnostic copied
into a timeline instance is not delivered again. `createThreeEffectPreparer()` routes a prepared
light draw's `NACHI_LIGHT_LIMIT_EXCEEDED` warning to the live core owner after transfer. An explicit
`ThreeLightPoolOptions.onDiagnostic` replaces that automatic route.

One source occurrence is delivered once even when its diagnostic is stored on more than one
affected instance. In particular, device loss is delivered once and stored on every live instance.
All affected instances store the primary diagnostic and enter error before the representative
owner delivers that shared occurrence.
If no instance is live when loss occurs, the system latches it and the first later affected spawn
delivers it; further spawns only store it. Independent occurrences remain independent even when
they use the same diagnostic code. Build delivery and runtime delivery are disjoint: diagnostics
already reported by `onBuildDiagnostic`, and diagnostics merely copied from another runtime owner,
do not cross the runtime seam again.

The overflow guarantee does not add a GPU readback. Exact free-list and event-queue overflow is
reported only when `aliveCountReadbackInterval` enables the existing count readback (or on a
backend that already requires it). With readback omitted, maximum-lifetime estimation and the
existing performance profile remain unchanged and no exact overflow diagnostic is promised.

### 12.3 Runtime validation

Runtime errors are reserved for state that cannot be known at compile time:

- device loss or renderer/backend removal;
- asset load failure after a ref was valid;
- type-invalid or immutable `User.*` parameter writes from application code;
- spawn after system disposal or use of a released instance;
- runtime device limit changes and external integration failures.

Device loss and other runtime failures transition every affected `EffectInstance.state` to `'error'` and emit a `VfxDiagnostic` with phase `'runtime'`. They must not continue reporting plausible FPS or silently switch semantic behavior. Development builds SHOULD include diagnostic paths; production builds MAY use compact codes.

`NACHI_GPU_SUBMISSION_FAILED` always carries `context.emitterPath` and `context.kernel`;
`emitterPath` contains the originating emitter or grid element path, and `path` contains the same
value. Its message names both context values. Kernel kinds distinguish spawn, init, update, event,
compaction, sort, neighbor-grid, sim-stage, and unknown fallback failures so a renderer
exception can be traced to the element submission that caused it.

Released-instance methods (`setParameter`, `setTimeScale`, `setTransform`, `applyHitStop`, `stop`, and emitter inspection) throw `NACHI_INSTANCE_RELEASED`; `release()` itself remains idempotent. Device loss is latched by `VFXSystem`: existing and subsequently spawned instances receive `NACHI_DEVICE_LOST` directly instead of a secondary materialization/submission error.

### 13.1 `@nachi-vfx/tsl-kit` standalone boundary

`@nachi-vfx/tsl-kit` is the framework-independent adoption entry point. It MUST NOT import
`@nachi-vfx/core` or any other nachi package. Every shader helper accepts a declarative JavaScript
configuration object and returns a Three.js TSL node that can be assigned directly to an ordinary
`NodeMaterial`. These shader-authoring objects may contain live `Texture` and `Node` instances and
are therefore not the serialized module configuration governed by §2.2; `mesh-fx` or a core
registered factory owns any later conversion from asset references to these live inputs.

M8 batch 1 chooses a direct `three` peer dependency. M12 release preparation narrows the supported
peer from the earlier `>=0.185.0 <0.186.0` range to exactly `0.185.1`. The implementation imports
`three/tsl` through one package-private adapter
module, but does not publish a second TSL abstraction or wrapper-node type. This preserves direct
interoperation for plain Three.js users while retaining one migration point for TSL API churn.
Applications MUST supply the peer; the package MUST NOT bundle a second Three.js runtime.

The standalone `polarUV(config)` in this package is the low-level shader function and immediately
returns a TSL `vec2` node. The north-star `PolarUvFactory` in core authoring types is a separate,
future `mesh-fx`/`fxMaterial` layer: it returns serializable UV authoring data and exposes fluent
composition such as `.flow()`. M8's material compiler will lower that data to `polarUV()` and
`uvFlow()` calls; the standalone function will not acquire the fluent builder return type, and the
two layers will not be made structurally interchangeable.

Every clock is explicit. `uvFlow`, `distortionUV`, and `flowMap` require a number or TSL node as
`time`; they MUST NOT read Three's global `time` node or wall-clock state implicitly. A nachi caller
binds effect-local time, while a standalone caller commonly supplies `uniform(0)`. Numeric constant
validation failures throw `TslKitDiagnosticError` with a stable code and parameter path. Dynamic
node ranges remain the caller's runtime responsibility.

### 13.2 `@nachi-vfx/tsl-kit` mathematics and conventions

All UVs use Three.js geometry UV convention. Angles and rotation inputs are radians. Colors are
working-space TSL colors. Texture wrap/filter modes remain owned by the supplied Three.js texture;
UV helpers do not apply `fract` unless the algorithm intrinsically requires a periodic phase.

For input UV `u`, time `t`, speed `s`, center `c`, and CCW rotation `r`, the UV helpers are:

- `uvFlow`: `u' = u + s t`. The result is intentionally unwrapped.
- `polarUV`: let `p = u - c` and
  `q = (p.x cos(r) - p.y sin(r), p.x sin(r) + p.y cos(r))`. The returned `vec2` is
  `(fract(atan2(q.y, q.x) / 2π + 1/2), length(q))`. Thus the positive x axis is `0.5`, the positive
  y axis is `0.75`, and positive rotation is counter-clockwise. The implementation expands this
  equation to keep that public convention visible. Three r185's vec2 `rotate()` path is also
  standard CCW; the known transpose/orientation trap is limited to the vec3/Euler path and
  hand-written matrix literals whose displayed row layout is mistaken for the constructor's column
  layout. Any substitution in those paths still requires a GPU direction test.
- `distortionUV`: sample `n = noise(u + speed t).rg`, decode `d = 2n - 1`, and return
  `u' = u + strength d`.

For unit normal `N`, unit surface-to-camera direction `V`, power `p > 0`, and color `C`, `fresnel`
returns `C (1 - clamp(dot(N,V), 0, 1))^p`. It is a standalone colored mask. `rimLight` is deliberately
the compositing preset rather than a synonym: for base color `B`, light color `L`, and non-negative
intensity `I`, it returns `B + I L (1 - clamp(dot(N,V), 0, 1))^p`.

`dissolve` samples scalar noise `n`, accepts a pure threshold input `h` (number or node), and returns
`vec4(E, A)`, where `A = step(h,n)` and
`E = edgeColor * (step(h,n) - step(h + edgeWidth,n))`. `A` is binary coverage. Assigning it to
`opacityNode` and setting a positive material `alphaTest` performs the fragment cutout; `E` is the
additive/emissive edge contribution. Lifetime evaluation is intentionally outside this package.

`flowMap` performs standard two-phase sampling. Decode `f = strength (2 flowTexture(u).rg - 1)`,
then define `p0 = fract(speed t)`, `p1 = fract(p0 + 1/2)`, `u0 = u - f p0`,
`u1 = u - f p1`, and `w0 = abs(2p0 - 1)`. The returned RGBA node is
`w0 map(u0) + (1 - w0) map(u1)`. The half-cycle offset and complementary triangular weights hide
the reset seam while requiring exactly two base-map samples and one flow-map sample.

`@nachi-vfx/tsl-kit/math` publishes CPU mirrors for these coordinate/phase rules. Unit tests use them as
independent numeric references; `/m8-tslkit/` additionally measures the compiled GPU result through
an offscreen render target.

### 13.3 `@nachi-vfx/mesh-fx` standalone boundary and geometry conventions

M8 batch 2 makes `@nachi-vfx/mesh-fx` a core-independent Three.js package. It depends on
`@nachi-vfx/tsl-kit` as its shader-part composition target and has the same exact Three `0.185.1` peer.
It does not import `@nachi-vfx/core`, allocate simulation state, submit GPU work,
or read a global clock. The public mesh factories return ordinary `THREE.Mesh` instances. Each has
stable `userData.nachiMeshFx = { kind, version: 1 }` metadata, so the M9 `defineEffect()` runtime can
adapt the object returned by the north-star `slashArc({...})` call without making this package depend
on core lifecycle types. Applications that own mesh construction may use the matching
`create*Geometry()` exports instead.

Mesh-fx diagnostics use the single `NACHI_MESH_FX_*` namespace. Standalone validation emits
`NACHI_MESH_FX_INVALID_PARAMETER` and `NACHI_MESH_FX_TEXTURE_REQUIRED`; timeline resource/lifecycle
integration emits `NACHI_MESH_FX_REFERENCE_INVALID`, `NACHI_MESH_FX_RESOURCE_UNRESOLVED`,
`NACHI_MESH_FX_MATERIAL_CLONE_UNSUPPORTED`, `NACHI_MESH_FX_TIME_BINDING_UNSUPPORTED`, and
`NACHI_MESH_FX_OPACITY_BINDING_CONFLICT`.
`NACHI_VAT_FRAME_RANGE` and the remaining VAT-specific diagnostics retain their `NACHI_VAT_*`
namespace.

All geometry is indexed, triangle-list `BufferGeometry`. Position, normal, and UV attribute counts
are equal. Seams and concentric boundaries are deliberately duplicated rather than welded, and
front-face winding agrees with the published outward normal. Segment values are author-controlled
integers and statically diagnosed before allocation.

| Factory       | Coordinates and topology                                                                          | UV and normal contract                                                                                                                     | Vertex / index count |
| ------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | -------------------- |
| `slashArc`    | XY fan strip, +X-centered degree sweep, `radius > innerRadius`, optional symmetric endpoint taper | `uv=(sweep fraction, radial fraction)`, normal +Z                                                                                          | `2(s+1)` / `6s`      |
| `ring`        | XY annulus with duplicated angular seam                                                           | `uv=(angle fraction, radial fraction)`, normal +Z                                                                                          | `2(s+1)` / `6s`      |
| `cylinder`    | Y-axis lateral surface; both ends open                                                            | `uv=(circumference fraction, height fraction)`, analytic unit radial normal                                                                | `(r+1)(h+1)` / `6rh` |
| `cone`        | Y-axis lateral surface; base open; apex duplicated once per angular column                        | same UV as cylinder, analytic unit slope normal `(cosθ, radius/height, sinθ)`                                                              | `(r+1)(h+1)` / `6rh` |
| `magicCircle` | XY disk split into duplicated concentric radial bands                                             | primary `uv=(x/(2R)+1/2,y/(2R)+1/2)` for `tsl-kit polarUV`; `uv1=(angle fraction, band-local radius)` gives one island per band; normal +Z | `2k(s+1)` / `6ks`    |

Here `s` is planar angular segments, `r`/`h` are lateral radial/height segments, and `k` is magic
circle ring count. The slash `angle` and `rotation` inputs use degrees to preserve the PLAN north-star
`angle: 140` surface; ring/cylinder/cone/magic-circle `thetaStart` follows Three.js and uses radians.
Slash taper scales half-width by `1 - taper * abs(2u - 1)`, preserving its center radius. A taper of
one reaches zero width at each tip.

The magic-circle primary UV is continuous across radial bands even though boundary vertices are
duplicated. This is the channel consumed by the low-level Cartesian-to-polar transform. `uv1`
isolates every concentric band and duplicates the angular seam, allowing band-local textures or a
future material group assignment without changing the polar primary channel. The center band has
coincident inner vertices; this intentional degenerate edge keeps the same deterministic count and
is discarded by triangle rasterization.

### 13.4 `fxMaterial` authoring, time, and diagnostics

`@nachi-vfx/mesh-fx` owns the fluent north-star authoring form. Its `polarUV(config)` returns immutable
authoring data, and `.flow({ speed })` appends immutable flow stages. `fxMaterial()` lowers that data
in author order to the standalone `@nachi-vfx/tsl-kit` `polarUV()` and `uvFlow()` node functions. This
implements the layer separation established in §13.1; the low-level tsl-kit return type remains a
TSL node and does not become fluent.

For meshes whose primary UV must remain Cartesian, notably the open cylinder beam, `mesh-fx` also
publishes immutable `uvFlow({ speed })` authoring data. `fxMaterial()` lowers it from the geometry's
unmodified `uv()` node. This is distinct from `polarUV().flow()`: the former moves `(u,v)` directly,
while the latter moves `(angle,radius)` after the polar transform.

M8 batch 2 returns a `THREE.MeshBasicNodeMaterial`. Optional base map/color, dissolve edge and
coverage, colored Fresnel, opacity, depth-write choice, and the four blending modes (`additive`,
`alpha`, `multiply`, `premultiplied`) compose into one node graph. Dissolve uses its binary coverage
as opacity with a small positive alpha-test, while its edge RGB and Fresnel RGB add to the base
working-space color. Textures retain application-owned color-space, wrapping, and filtering.

`opacity` accepts a scalar number or TSL node. A number, including the default value when omitted,
initializes a package-owned uniform exposed as `material.fx.opacity`; `setOpacity()` validates and
writes `[0,1]` values for immediate alpha/additive (and other blending-mode) updates. A TSL node is
composed directly and makes `fx.opacity` null and `setOpacity()` reject assignment. These two
ownership modes are mutually exclusive by contract.

`dissolve.uv` may be `'static'`, independent immutable `polarUV()` / `uvFlow()` authoring, or
omitted. Omission preserves the original graph by sharing the material map UV; `'static'` samples
the geometry's unmodified `uv()`. `dissolve.edgeIntensity` is a non-negative numeric emission
multiplier with default `1`. `dissolve.edgeModulate` is `'none'` by default; `'map'` requires a base
map and multiplies edge RGB by its working-space luminance. The defaults therefore preserve the
pre-existing unmodulated edge output.

`dissolve.overLife` accepts either a scalar/TSL node or a piecewise-linear array of `[time, value]`
pairs. Array curves require strictly increasing `[0,1]` times, endpoints at zero and one, and values
in `[0,1]`; they lower to nested TSL selections, not per-frame JavaScript evaluation. If `time` or
`normalizedLife` is omitted, `material.fx` exposes a package-owned uniform and checked
`setTime()`/`setNormalizedLife()` methods for standalone use. If either input is supplied as a TSL
node, it is treated as an externally owned binding and its setter is rejected. M9 timeline
integration does not replace those node bindings: it clones materials and advances omitted inputs
through their package-owned `material.fx.setTime()` and `setNormalizedLife()` controls, including
hit-stop and time-scale semantics. Supplying `time` as a number likewise lowers a fixed constant and
causes `setTime()` to reject assignment. Timeline authoring therefore diagnoses any adapted
material with an explicit `time`; omit it to provide the required writable effect-local clock.
Lowering from core `curve()` authoring objects to mesh-fx tuple arrays belongs to the M9
`@nachi-vfx/timeline` integration surface. Its `fxMaterial()` wrapper checks linear interpolation,
strictly increasing `[0,1]` keys and endpoints, then passes immutable `[time,value]` tuples to the
core-independent `@nachi-vfx/mesh-fx`. The wrapper additionally accepts `opacityOverLife` in the same
core `curve()`, tuple-array, number, or TSL-node authoring forms. Numeric and curve forms are
piecewise-linearly evaluated from normalized mesh life and written through `fx.setOpacity()` on
play and every timeline update; TSL nodes remain compile-time opacity bindings. `opacity` and
`opacityOverLife` cannot both own that channel. M8 does not infer core state and never reads Three's
wall-clock `time` node.

The same package owns the effect-element adapter. A raw self-describing mesh returned by
`slashArc()`, `ring()`, or another mesh-fx factory may be placed directly in
`defineEffect().elements`; `meshFxElement(mesh, { duration })` overrides the automatic one-second
life. The raw Three object is an explicitly ephemeral resource held outside the effect document,
which contains a serializable `timeline/mesh-fx` placeholder. Each effect instance clones the mesh
and material controls, adds it through a structural scene target, and writes effect-local time to
`fx.time` plus time since the latest `play()` to `fx.normalizedLife`. `stop()`, natural expiry, loop reset,
and release all make it non-renderable. Timeline ownership preserves standalone mesh-fx usage and
keeps Three lifecycle objects out of core.
Restoring ephemeral mesh resources from serialized placeholders belongs to `@nachi-vfx/format`.

`applyVat()` metadata participates in that adapter without making mesh-fx depend on core. After
cloning the material, timeline MUST rebuild the ordered unique union of graph-reachable VAT layers.
Position reachability begins at the latest absolute-position layer, or at the first layer when all
positions are offsets. Normal reachability is the latest layer that supplies a normal texture. A
layer selected by either channel owns one active control; a layer selected by neither MUST NOT be
cloned, returned, or driven. This means offset+normal A followed by absolute+normal B exposes only B,
while omitting B's normal exposes A for normal and B for position in original order. For timeline
`fxMaterial()` rebuilding occurs on the regenerated package graph; for an ordinary NodeMaterial it
first restores the node graph captured immediately before the VAT chain, because
Three r185's generic clone may alias the already-applied position/normal graph. An explicitly
authored pre-VAT position or normal node is an external binding and remains shared. Reapplication
preserves the standalone composition rules: offset positions compose with the existing position
node, absolute position replaces it, and a supplied VAT normal replaces the current normal node.
Detached binding metadata MAY be compacted, but position/normal textures are borrowed external
assets and MUST NOT be disposed by timeline clone, prepare, error, retained preparation, or release
paths.

Cloneable VAT metadata is active independently for position and normal only while the mesh still
references the recorded material and the exact channel root installed by the final applicable
`applyVat()`. Replacing a final position or normal root transfers that channel back to external
authoring: timeline preserves the current root as a shared clone binding, MUST NOT resurrect its
detached VAT graph, and MUST NOT return or drive a clock solely owned by that detached graph. The
other channel remains independently active; therefore replacing final position can leave the latest
VAT normal clock active, and replacing final normal does not disable VAT position clocks. Replacing
both roots or the material leaves no timeline-owned VAT clock. Between applications, replacing only
normal retires the prior VAT normal while preserving accumulated position layers; replacing position
starts a new VAT chain from the authored root. This contract observes root identity, not mutations
inside an unchanged node graph; callers transferring ownership MUST replace the root reference.

When `VatConfig.time` is omitted, each source and timeline clone owns a distinct VAT uniform. Clone
creation snapshots the source control's current value, but every element `play()` resets the clone
to zero and subsequent timeline segments write `MeshRuntime.elapsed`, in seconds since that latest
play. This is deliberately not the track-wide `fx.time`: time scaling, speed, zero-scale pause, hit
stop, natural expiry, loop reset/replay, and fresh instances follow element-local lifecycle
semantics, while explicit stop and all terminal states freeze the last value. The internal timeline
writer accepts only finite non-negative time and permits a non-looping element to outlive its VAT
clip so shader clamping holds the final frame; standalone `VatControls.setTime()` retains its
stricter authored-range diagnostic. A numeric or TSL `VatConfig.time` is non-writable and remains
the same external binding in every clone. Timeline MUST neither write it nor diagnose it.

Material cloning is a spawn-time snapshot, not a replay of the original authoring config alone.
Timeline first regenerates the package-owned node graph and writable controls, then applies Three
r185's `MeshBasicMaterial.copy()` state contract and copies the current values of compatible owned
`fx.opacity`, `fx.time`, and `fx.normalizedLife` uniforms through their checked setters. Consequently,
`setOpacity()` before `spawn()` is preserved, as are `side`, depth/stencil/blend/color-write state,
clipping planes, `name`, and the ordinary deep-JSON copy of `userData`. Package-owned uniforms,
clipping planes, and user data are independent between source and clones; textures and other
externally owned resources follow Three's normal shared-reference semantics. Timeline never copies
the source node graph, so clone controls cannot alias it. `opacityOverLife` retains exclusive
timeline ownership and its existing conflict diagnostic.

`TimelineEffectInstance.setUserVisible(meshKey, visible)` is the public per-mesh user component.
The clone publishes exactly `mesh.visible = runtimeVisible && userVisible`; runtime visibility owns
play, stop, natural expiry, loop reset/replay, completion, construction cleanup, and release, while
the default-true user component persists across those transitions and transform updates. Setting
the user component to true returns control to the current runtime value rather than forcing a draw.
Unknown/non-mesh keys and non-boolean values are rejected, released instances use the common
released guard, and a newly spawned instance starts with a fresh default-true user component.
Direct writes to the cloned `Object3D.visible` field are not a supported persistent override: the
next visibility publication by `setUserVisible()`, play, stop, expiry, or loop replay may overwrite
them. Hosts MUST use `setUserVisible()` for user-owned visibility.

Three r185 `Mesh.clone()` shares its `BufferGeometry` reference. Timeline deliberately preserves
that behavior: geometry is an immutable borrow, not a per-instance copy. Attribute/index/group,
bounding-volume, and `drawRange` mutations through the source or any clone affect all instances.
The application/resource owner must retain the geometry until every source definition, active
timeline clone, and retained preparation object is finished, and only then call
`geometry.dispose()`. Timeline release/error cleanup disposes its cloned material but never the
borrowed geometry or source mesh/material.
Preparation follows the same boundary. A non-retained success, preparer failure, or abort disposes
the temporary cloned material. A successful `{ retained: true }` transfers that detached clone and
material to the preparer, which MUST dispose the cloned material when its retained resource is
released; neither the preparer nor timeline disposes the borrowed geometry.

At clone time, timeline captures the source mesh's position, quaternion, and scale as an immutable
`authoredLocal` transform. Position and orientation are an effect-local offset, so the clone
transform is `effectTransform * authoredLocal`: the effect quaternion rotates the authored
position before the effect position is added, and the clone quaternion is the effect quaternion
multiplied by the authored quaternion. The authored scale is copied by clone initialization only.
Timeline never overwrites clone scale during `play()`, updates, attachment synchronization, or
`setTransform()`, allowing application code to animate clone scale independently.

All statically knowable authoring errors throw `MeshFxDiagnosticError` synchronously, following the
batch-1 `TslKitDiagnosticError` shape (`code`, `path`, `message`). Stable codes distinguish an invalid
parameter from a missing/non-Three texture. Dynamic TSL-node ranges remain the caller or future
compiler's runtime responsibility.

### 13.5 Blender VAT layout and runtime

M8 batch 3 targets the `VAT` add-on distributed through the official Blender Extensions catalog
and maintained in `flement/VAT-blender-addon`. This identifies the compatibility source precisely:
the catalog distribution is not a claim that Blender itself authors the extension. Its default,
non-wrapped image has width equal to vertex count and height equal to frame count. A horizontal row
contains all vertices for one frame; frames progress vertically from top to bottom. The exporter's
Y-flip option reverses that order. The generated mesh names its vertex lookup UV set `vertex_anim`,
which Three's glTF path exposes as `uv1`. Nachi uses that per-vertex U coordinate to select the VAT
column, preserving a valid exporter-defined reordering. Exact texture-width versus
`geometry.position.count` validation still rejects an imported mesh whose seam splitting changed
the vertex count. `vertexLookup: 'vertex-index'` is an explicit fallback for meshes generated in the
same order as the VAT; it is not the Blender compatibility default.

The exporter can store either offset or absolute object-space position. Offset is the runtime
default and adds the decoded sample to the material's existing `positionNode` or `positionLocal`;
absolute mode replaces it. Unnormalized position output is OpenEXR RGB half/full float. With
normalization enabled, the exporter reports one global `Min Offset` and `Max Offset`, where encoded
zero maps to the minimum and encoded one maps to the maximum. `positionEncoding: 'remapped'`
therefore decodes `p = min + sample * (max - min)` using that scalar pair before axis conversion.
The default `axisMap: 'xzy'` follows the exporter's Three.js example for Blender Z-up to Three Y-up,
but a bare Y/Z swap has determinant -1 and therefore changes chirality (it is a reflection, not a
rotation). Use `axisMap: 'xz-y'` for the right-handed `(x, z, -y)` conversion with determinant +1;
`xyz` remains available for already converted content. The mirrored `xzy` default is retained for
compatibility with exporter-authored assets rather than presented as the preferred coordinate-system
rotation.

The v1 runtime requires the position texture to be `FloatType` or `HalfFloatType` with
`NoColorSpace`, exact dimensions `(vertexCount, frameCount)`, no mip-dependent lookup, and one frame
per row. A remapped PNG position export is not sampled directly even though the Blender add-on can
write it: applications must convert it to a linear float texture, retaining the reported min/max.
This is a deliberate precision and color-management gate. The optional normal texture may be an
unorm texture and defaults to decoding `n = normalize(2 sample - 1)`; signed float normals are an
explicit alternative. Normal VAT lookup and decode occur in the TSL vertex stage and the decoded
object-space normal is transformed by the model-view normal matrix before being passed through a
varying as a normalized view-space normal. Sampling a VAT in the fragment stage through interpolated
lookup UVs is incompatible.

`applyVat(mesh, config)` accepts `interpolation: 'nearest' | 'linear'`, inclusive integer
`frameRange`, `fps`, looping, frame order, and either a package-owned or external TSL time input.
`frameOrder` defaults to `top-to-bottom` and names the GPU texel-row order after texture loading. The
compatibility profile assumes VAT textures are uploaded with `texture.flipY = false`. If a loader
flips rows during upload, callers must select the opposite `frameOrder`; an exporter Y-flip file
loaded without another flip uses `bottom-to-top`.

External numeric VAT time is finite, while `loop` and `disableFrustumCulling` are strict booleans.
Untyped strings are not coerced by truthiness, and validation precedes node/culling mutation.

For time `t`, the continuous frame is `start + fps*t`. Looping reduces it modulo
`end-start+1`; non-looping clamps it. Nearest rounds to the closest frame. Linear samples
`floor(frame)` and its next frame (wrapping end to start only in loop mode), then mixes by the
fractional frame. `controls.setTime()` and `setFrame()` diagnose non-looping or explicit frame values
outside the configured range; an external dynamic node remains the owner's responsibility, exactly
like external `fxMaterial` time/life nodes.

Stable VAT diagnostics include missing texture, non-float position texture, vertex-count mismatch,
layout mismatch, invalid frame range, and non-NodeMaterial use. Variable vertex topology,
wrap/wrap-and-crop packing, multi-row frames, texture arrays, clip atlases, tangent/orientation VAT,
and per-axis remap bounds are not compatible in v1. Animated bounds are not derivable from an
opaque GPU texture, so `applyVat` disables frustum culling by default; callers may re-enable it only
after setting conservative geometry bounds. `/m8-vat/` uses generated float position and unorm
normal textures with analytic, non-mirrored motion and compares rasterized GPU vertex positions at
known times against CPU frame interpolation.

### 13.6 `@nachi-vfx/post` pipeline, distortion sources, and bloom

M10 batch 1 makes `@nachi-vfx/post` a core-independent Three r185 package and fixes
`THREE.RenderPipeline` as its integration point, following M0 spike 4. `createPostPipeline()` owns
one render-pipeline graph rooted at `pass(scene, camera)`. The configured pass set is ordered by the
explicit `order` permutation; absent `order` uses `distortion -> radialBlur -> bloom`. Every
configured pass appears exactly once, so an unknown, duplicate, missing, or absent pass name is a
synchronous diagnostic rather than an implicit fallback. UV-domain passes retain a sampler closure
for the preceding graph. A bloom result is exposed through its pass texture, which allows later
distortion or radial blur to resample the already-bloomed composition and makes custom order
semantically real rather than metadata only.

The public `PostPipeline` constructor revalidates direct `PostPipelineConfig` data before creating
the scene pass. This is the same validation used by pass factories and covers pass tags/configs,
numeric/vector domains, source arrays, sample counts, bloom presets, the order array, and strict
`outputColorTransform`; bypassing a factory cannot create a zero-sample or non-finite graph.

Screen distortion follows Niagara's Scene Color distortion-material model: it samples the prior
scene-color composition at an offset screen UV. Shockwaves add a radial ring offset using center,
travel radius, ring width, strength, start time, speed, duration, and enable inputs. Heat haze adds
two deterministic, smoothly interpolated value-noise components inside a feathered rectangular
region. The low-frequency continuity produces Niagara-like wobble instead of pixel-scale white
noise. All spatial values are normalized screen-UV units. Sampling clamps to an inset
`[0.001, 0.999]` boundary to avoid backend-dependent edge wrap.

The M10 distortion-source connection is **`User.*`-backed uniform nodes**, not particle-storage
readback and not a dedicated distortion buffer. Gameplay already owns a hit/shockwave's screen
center, start time, radius, width, and strength; the effect binds those typed `User.*` values to TSL
uniform nodes passed into `screenDistortion()`. Numeric/tuple fields instead receive package-owned
uniforms writable through `PostPipeline.controls`. This has bounded source count and zero hidden
GPU/CPU synchronization, and it maps directly to Niagara's ordinary parameter-driven distortion
material. Reading particle positions back would add latency and a synchronization point while
duplicating values the gameplay event already knows. A dedicated low-resolution vector/distortion
buffer is the future path for hundreds or thousands of independently simulated heat particles; it
is intentionally not allocated for sparse shockwaves in batch 1.

Distortion time uses the `fxMaterial` two-layer model. Omitting `time` creates a package-owned
uniform and enables `controls.setTime()`. A supplied number is a fixed constant and a supplied TSL
node is externally owned; both reject the package setter. Effect/timeline integration supplies an
effect-local node so hit stop and time scale never fall back to wall-clock time.

Bloom wraps Three r185's existing TSL `BloomNode`. That implementation already performs a smooth
luminance-threshold extraction into a half-float target, five progressively downsampled separable
Gaussian blur levels, and weighted additive composition on both WebGPU and WebGL2 backends.
Reusing it costs one thin preset/lifecycle wrapper and tracks Three's pinned TSL boundary. A custom
implementation would duplicate at least the extraction, target-resize/disposal, ten blur passes,
mip composite, and two-backend maintenance without changing the requested appearance. The wrapper
therefore exposes `soft`, `intense`, and `cinematic` presets plus strength/radius/threshold and
resolution-scale overrides.

`/m10-post/` never presents the WebGPU canvas. It selects an offscreen render target, executes the
pipeline, compacts Three r185's padded WebGPU readback, evaluates linear-space pixels, and only then
copies bytes to a 2D canvas for the visual artifact. The readback checks also mutate externally
owned center, strength, and time uniform nodes and
require visible output differences, covering the `User.*` GPU binding path. These passes do not
sample or reconstruct depth, so forward-z versus reverse-z does not change their
mathematics. The unresolved reverse-z/MSAA/shared-depth-copy constraints in §16 still apply to M6
depth collision and M7 decal paths and to any future depth-gated post pass; post integration MUST
NOT infer that capability from color-only success.

### 13.7 `@nachi-vfx/post` weighted blended OIT

`createWboitPipeline(renderer, transparentScene, camera)` owns a two-attachment Three r185
`RenderTarget`: `nachiWboitAccum` is RGBA16F and `nachiWboitRevealage` is R8. The transparent scene
contains only `NodeMaterial`s whose `mrtNode` is produced by `createWboitOutput(color, alpha,
depth)`. Assigning this structure to `outputNode` is incorrect in r185: while a renderer MRT is
active, `NodeMaterial` merges the renderer MRT with `material.mrtNode` and the material attachment
names override the context defaults.

For linear color `C`, clamped coverage `a`, and WebGPU normalized depth `z`, the McGuire-Bavoil
weight is:

```text
w(a,z) = clamp((min(1, 10a) + 0.01)^3 * 10^8 * (1 - 0.9 clamp(z,0,1))^3,
               10^-2, 3*10^3)
accum      += vec4(C * a, a) * w(a,z)
revealage  *= (1 - a)
out.rgb     = accum.rgb / max(accum.a, 10^-5)
out.a       = 1 - revealage
```

Three r185 supports attachment-specific blending through `MRTNode.setBlendMode()` on native
WebGPU. Accum uses `ONE, ONE, ADD`. Revealage is initialized to one and outputs scalar alpha to an
R8 target with `ZERO, ONE_MINUS_SRC_COLOR, ADD`; `ONE_MINUS_SRC_ALPHA` would be incorrect because
R8 has no alpha component. A fullscreen initialization draw writes `(0,1)` before the transparent
scene, and a fullscreen composite alpha-blends the resolved color over the caller's current output
target. Compatibility-mode/WebGL2 MRT cannot preserve per-attachment blending and is rejected as
`NACHI_WBOIT_WEBGL2_UNSUPPORTED` rather than split into a silent two-pass semantic fallback.

Weighted OIT is order-independent for a fixed fragment multiset, but it is an approximation of
sorted alpha. The owned accumulation target also owns an independent depth attachment: it cannot
consume the depth written to the caller's opaque target, so opaque geometry does not currently
occlude WBOIT fragments and particles behind a wall can leak through. Callers render opaque content
first, then composite WBOIT, but that sequence supplies only the color background—not opaque depth.
An external shared-depth option is deferred because Three r185 assigns one `DepthTexture` owner
RenderTarget and disposing either owner may destroy the shared GPU texture; safe support needs an
explicit borrowed-attachment lifecycle plus size/sample/format validation. `/m10-sort/` validates two known overlapping layers against
`compositeWboitLayers()`, repeats them in reverse order, and uses linear byte tolerances that include
the intentional R8 revealage quantization.

### 13.8 `@nachi-vfx/react` React Three Fiber binding

`@nachi-vfx/react` is a lifecycle adapter, not a second runtime. `VFXSystemProvider` is mounted inside
an R3F `Canvas`, creates one core `VFXSystem` from a supplied `VfxRuntimeRenderer` and the current
R3F scene, and, by default, copies the active R3F camera's column-major `projectionMatrix` and
`matrixWorldInverse` plus the pixel `size` into `system.setCamera()` before calling
`system.update(delta)` from `useFrame`. This keeps distance/frustum culling, significance,
scene-depth uniforms, and coarse transparent sorting aligned with the rendered view. The
`syncCamera` provider prop defaults to `true`; callers that manage `setCamera()` themselves can opt
out with `syncCamera={false}`. An asynchronous update rejection is
retained as React state and thrown on the next render so an Error Boundary can observe it; the
binding never converts a failed frame into a plausible successful frame.

`useVFXSystem()` returns that system. `useEffectInstance(definition, options)` spawns on mount and
releases the exact handle on unmount or when the definition/system/spawn-only seed/priority changes.
Callers SHOULD define `definition` at module scope or otherwise preserve its reference; recreating it
while rendering intentionally triggers that respawn contract.
Position, rotation, time scale, and declared `User.*` parameter props update the live instance.
The hook forwards a changed parameter to core validation before updating its own committed-value
record. This preserves the M9 store-before-forward prohibition: an invalid/immutable value cannot
be remembered by React and replayed later merely because no emitter was active at the time.

`attachTo` accepts either core's `EffectTransformSource` or a Three `Object3D`. The latter is
adapted with `updateWorldMatrix()`, `getWorldPosition()`, and `getWorldQuaternion()` and is polled by
core before initialization and every scheduled system step. While attached, this sampled transform
overwrites the instance's complete transform, including any position or rotation passed at spawn or
through later React props. Cleanup detaches before release and
`release()` remains the authoritative resource/pool transition. `<VFXEffect>` is the null-rendering
component form of the same hook.

React, `@react-three/fiber`, and Three are peers. The package pins the supported Three peer exactly
to `0.185.1` for this release and does not bundle any of the three. Unit tests use
`react-test-renderer`, a mocked R3F frame/scene context, and the same structural fake runtime
renderer boundary used by core tests; they do not require a WebGL/WebGPU canvas.

R3F reports `state.size` in CSS pixels; the binding multiplies both dimensions by
`state.viewport.dpr` before forwarding `viewportSize`, because scene-depth texel offsets operate in
device pixels. Removing a live React parameter prop restores its declared default on the instance;
spawn-only props (`seed` and `priority`) require a respawn and are intentionally not live-patched.

### 13.9 Three r185 internal dependency audit

The Three adapter is pinned to `three@0.185.1`. Every Three upgrade MUST audit the following internal
or version-coupled surfaces: `renderer._attributes`; the locally duplicated `AttributeType`
constants; the port of `setupPositionView`; storage-node `.value` replacement; the
`texture.renderTarget.samples` probe; direct `AtomicFunctionNode` construction; and the
`isWebGLBackend` / `compatibilityMode` branches. A version bump is incomplete until each item is
confirmed against upstream sources and covered by the adapter smoke tests.

The audit also includes playground performance tooling's direct inspection of
`renderer.backend.timestampQueryPool[scope].currentQueryIndex` / `maxQueries` and its assumption
that resolving the pool resets `currentQueryIndex` to zero. If those fields are renamed or removed,
or the reset semantics change, the adaptive drain becomes a no-op, timestamp-pool exhaustion
warnings return, and pages that enforce `consoleClean` fail.

## 15. Differences from PLAN.md

There is no intended surface-syntax divergence from the north-star sketch. The compile test uses its emitter, mesh element, effect, timeline, runtime spawn, and inline TSL forms.

This RFC makes two boundaries that were implicit in PLAN.md explicit:

1. `tslModule(callback)` cannot itself be JSON. It is accepted for authoring but must be converted to a stable registered function reference before serialization. This is required to satisfy both the inline escape hatch and the rule that serialized module configuration is data.
2. Names such as `explosionTex` and `noiseTex` in the sketch are typed as `TextureRef` for serializable effects. Passing a raw Three.js texture is allowed only for an explicitly ephemeral, non-serializable definition.

These are serialization constraints, not changes to the composition or stage model.

## 16. Open decisions

M4 visual regression screenshots use a changed-pixel ratio tolerance below `0.5%` rather than PNG
byte equality. The firefly additive pass can compact equal simulation state into a different draw
order, and floating-point additive blending then changes a small set of output pixels without a
behavioral regression. Exact simulation/readback checks remain deterministic and byte-based.

The following decisions are deliberately deferred. Implementations must not bake an undocumented answer into public API before the named spike is reviewed.

- whether all custom attribute types work in storage buffers on the pinned Three.js version;
- compiler tracing mechanics for TSL reads/writes;
- automatic derivation of `TslParticleBindings` custom keys and value types from an emitter's attribute schema;
- practical buffer/device limits and capacity diagnostics.

### Spike 2: `drawIndirect` / `dispatchIndirect`

M2 resolves the core decisions: physical slots use an atomic free list because deaths are scattered and stable physical indices keep SoA storage and deterministic identity cheap; a compact alive-index region supplies dense render/dispatch order. The index regions and allocator/authoritative-alive/success/overflow counters share one writable lifecycle state buffer. Spawn-dispatch and draw-indirect arguments share a second, separate indirect-argument buffer so an indirect source is never also writable storage in the same compute-pass synchronization scope. Core owns the argument regions and updates them after Spawn/Update compaction. CPU readback is permitted for diagnostics, opt-in low-frequency completion, and the reduced WebGL2 count path. Births and deaths are visible to renderers after the current scheduler step's compaction.

Deterministic allocation order across parallel adapters is intentionally unresolved. A future opt-in may assign persistent particle IDs or use a deterministic allocation pass; its cost, serialized representation, and interaction with events/sim cache remain open.

### Spike 3: WebGL2 backend parity

M2's practical lifecycle runtime is not currently supported on WebGL2. The backend has no atomics or indirect operations, and `rate` and `perDistance` are rejected. `buildKernels()` statically compares every physical attribute storage written by initialize with the transform-feedback limit and rejects over-budget burst emitters with `NACHI_BACKEND_SPAWN_UNSUPPORTED`; it never submits a kernel whose writes would otherwise disappear silently. A small prefix-only burst schema may remain usable when initialize's complete physical write set fits the measured limit, but it is the reduced path rather than lifecycle parity.

- which non-lifecycle module capabilities compile through the WebGL2 backend;
- whether a packed transform-feedback layout can expand the reduced burst tier safely;
- storage-texture fallbacks and limits introduced by later milestones;
- whether WebGL2 should add a CPU allocation cursor for `rate`/`perDistance` instead of the current explicit rejection;
- the serialized capability/quality declaration used to prevent silent semantic fallback.

### Spike 4: depth access and TSL post pipeline

M6 batch 1 resolves compute depth collision through an explicit previous-frame normalized-depth
color copy. Fragment soft particles continue using `linearDepth(viewportDepthTexture(screenUV))` in
their active render pass. Compute uses a separately owned sampleable texture plus System camera
uniforms, avoiding attachment feedback. Reverse-depth, MSAA depth resolve, WebGL2 parity, and shared
copy ownership across multiple effects remain future capability work and must be diagnosed rather
than inferred.

M10 batch 1 resolves the color-post half of this spike with the §13.6 `@nachi-vfx/post` graph rooted at
`RenderPipeline`. Shockwave/heat-haze distortion, radial blur, and Three's TSL `BloomNode` all
compose as scene-color operations and therefore require no depth convention. This does not resolve
reverse-z, MSAA depth resolve, WebGL2 compute-depth parity, or shared depth-copy ownership; those
items remain explicitly open for depth consumers.

M10 batch 2 resolves alpha ordering with §9.5's independent sorted indirection and §13.7's native
WebGPU MRT WBOIT. This does not resolve M2 free-list/compaction allocation order or create a
persistent particle identity. Equal-depth sorting uses physical index only as a deterministic
frame-local tie key. The 65,536 capacity ceiling and `O(log^2 P)` dispatch count are part of the
opt-in quality contract. Each bitonic stage is a separate submission because one compute pass does
not provide a device-wide barrier between dispatches. Sort work is update-time work; camera mutation
without a following update deliberately leaves the previous indirection intact.

M10 batch-2 review decision (2026-07-12): alpha emitter ranks keep the fixed 1000 base while
additive draws remain at their ordinary order 0. Pulling additive into the ranked set would change
existing composition with user transparent scene objects without improving additive/additive
correctness, so the separation is an explicit v1 policy rather than an omission. Projection decals
are excluded because receiver depth, not emitter-center alpha order, controls their useful
visibility. WBOIT external-depth injection remains follow-up work: Three r185's RenderTarget setter
reassigns `DepthTexture.renderTarget`, and RenderTarget disposal destroys its cached depth texture,
so a borrowed shared attachment cannot be exposed safely as a low-cost option. A future API must
define ownership and validate dimensions, samples, format, reverse-z convention, and render-pass
load behavior before enabling opaque occlusion.

Ribbon alpha-rank integration remains follow-up work. It must bring ribbon alpha and premultiplied
draws into the emitter-center ranking policy without changing their existing trail preparation or
ordering semantics.

### Later milestone decisions

M11 batch-1 decision (2026-07-12): quality spawn/capacity scales and visibility are runtime values;
soft/lit/sorted gates are structural compiler variants isolated in the compile cache and GPU pool.
Live structural changes require restart instead of discarding particle state. Fully culled effects
pause local/lifecycle/particle time and do no catch-up work. Significance uses the normative
`4*priority + 2*screenOccupancy + 1/(1+distance)` formula from §10.4, with stable instance-id ties.
This resolves the M11 tier-selection item below; sim-cache remains a separate M11 batch.

M11 batch-2 decision (2026-07-12, lineage amendment 2026-07-15): §10.5 simulation caches store
render-read logical attributes, lossless alive indirection, and mandatory lossless per-slot
`spawnOrder` lineage in metadata+ArrayBuffer form. Float32 and per-component u16 quantization are
supported. V2 reuses current-frame GPU buffers and performs one decoded upload per frame; its lineage
stream does not add an ordinary replay upload, and all-frame GPU residency is deferred. V1 caches
must be re-baked. Nearest is whole-frame selection, while linear interpolates float attributes only
across the same physical slot when it is alive with matching lineage at both endpoints. Loop caches
duplicate the endpoint and compare the alive lineage set and attributes through lineage-to-physical
maps. WebGL2 packed simulation storage rejects behavioral access to group 1 or above rather than
allowing silent transform-feedback aliasing. Because every renderable lifecycle fixture reaches
group 1, WebGL2 cache baking is effectively unsupported and verified through structured diagnostics;
replay remains explicitly diagnosed as unsupported. This resolves the snapshot/custom/transient
attribute item below.

M11 batch-3 decision (2026-07-12): §10.6 attribute snapshots reuse simulation-cache storage
readback and expose logical typed rows plus explicit compact/physical lineage and truncation.
Profiler counters are frame-local scheduler facts. GPU timing remains the pass-level cached sample
owned by `nachi.perf-baseline` v1; core does not issue a second timestamp query or attribute a pass
duration to emitters heuristically. This resolves the M11 code-debugger surface while leaving a GUI
editor outside scope.

M12 batch-2 decision (2026-07-12): §10.7 makes Grid2D and simulation stages effect elements. Grid
channels use packed vec4 cell records in one state and one scratch storage buffer. Dependent stage
iterations are separate update/commit submissions. The reference fluid uses semi-Lagrangian
advection and an intentionally bounded Jacobi projection. Particle deposition uses u32 fixed-point
atomics and particle sampling uses GPU bilinear reads. WebGL2 is explicitly unsupported. Neighbor
grids, Grid3D, obstacle fields, and emitter-storage-direct transfer remain later M12 batches.

M7 decision: bounded PointLight pooling is the semantic light renderer. GPU top-N selection plus a
fixed one-frame-late readback bounds both CPU objects and synchronization. Emissive+bloom is deferred
to M10 because bloom is post-processing appearance and cannot replace illumination of ordinary lit
scene geometry. M7 decal projection uses depth reconstruction rather than the oriented-quad
fallback; no §16 downgrade was necessary. CPU `DecalGeometry` remains unsuitable for per-particle
GPU-driven updates.

M7 decision (2026-07-11): multiple render modules per emitter are deferred to M8 or later rather
than a later M7 batch. Supporting them requires per-draw indirect-argument slots and coordinated
renderer resource ownership; M7 keeps the explicit one-render-module limit and
`NACHI_RENDER_MODULE_LIMIT` diagnostic until that work is scheduled.

M8 batch-2 decision (2026-07-11): mesh-fx remains core-independent. Procedural factories return
self-describing Three meshes and M9 owns their effect-element lifecycle adapter. This avoids making
standalone geometry/material creation depend on compiler or scheduler types. `fxMaterial` exposes
standalone uniforms and externally supplied TSL clock/life nodes now; automatic effect-local binding
is deferred to M9 timeline integration. VAT remains a separate M8 batch because asset texture layout,
frame interpolation, bounds, and Blender exporter compatibility do not alter the procedural geometry
or material contracts above. The existing one-render-module emitter limit is not changed by mesh
effect elements, which are effect-level non-particle elements rather than additional emitter render
modules.

M8 batch-3 decision (2026-07-11): VAT follows the Blender Extensions `VAT` add-on's strict
one-frame-per-row layout and global scalar min/max remap. Position textures are float/half-float and
linear; wrapped/cropped and direct PNG position layouts are diagnosed rather than guessed. The
standalone uniform/external-node clock split matches `fxMaterial`, while M9 remains responsible for
automatic element-lifecycle binding. Golden #3 uses M8-local fixed composition; timeline
choreography remains M9 and bloom/post remains M10.

H2-10 decision (2026-07-16): M9's automatic element-lifecycle binding now includes package-owned
VAT clocks and clone graph reconstruction. The clock is latest-play mesh-element elapsed time, not
track time or `fx.time`; explicit VAT time inputs remain external and untouched. Cloneable metadata
stays internal to mesh-fx, while timeline owns reset/advance/stop/loop scheduling. This is a minor
behavioral change for VAT meshes placed on timelines and leaves standalone `applyVat()` unchanged.

M9 batch-2 decision (2026-07-11): sequencer orchestration and the mesh-fx effect adapter live in
`@nachi-vfx/timeline`. Camera shake targets a callback/rig abstraction instead of a Three camera. Track
loop reset truncates active child lifecycles and replays time-zero actions; child emitter loops stay
element-local. Timeline owns fixed-step partitioning and core consumes those exact steps, avoiding
double scaling across world time, track speed, and hit stop. Raw mesh resources remain ephemeral
while timeline actions and mesh placeholders remain serializable. This resolves §13.4 core-curve
lowering and automatic fxMaterial local-time/normalized-life binding.

M9 batch-2 review decision (2026-07-12): each timeline `play()` currently materializes a
single-element core sub-definition so independently scheduled element lifecycles remain isolated.
Consequently, M5 `emitTo()` links cannot cross timeline element keys in this batch: the producer's
sub-definition has no consumer key. A failed child compilation now copies the explicit
`NACHI_EVENT_TARGET_UNKNOWN` diagnostic to the timeline instance and transitions it to `error`;
silently remaining `active` is forbidden. Grouping event-linked elements into one scheduled child
definition is deferred because it requires coordinated play/stop ownership for every member.

M10 batch-3 decision (2026-07-12): lit billboards reuse the MeshStandard physical-lighting path and
the Sprite position path rather than introducing a second lighting model. Tangent-space normals are
rotated by the billboard's view-plane basis and supplied as view-space `normalNode` values. Golden
#2 now composes sorted smoke with `@nachi-vfx/post` shockwave distortion. Golden #5 is code-authored in
M10 and exercises the full timeline/post stack; loading the same definition from JSON remains an
M12 asset-format acceptance item rather than an implicit runtime capability.

- custom-event TSL context materialization beyond the reserved `context.emitEvent(condition)` contract (post-M5);
- WBOIT automatic tier selection remains external to core because WBOIT is an `@nachi-vfx/post`

- serialized asset-reference syntax for emitter inheritance (the M9 authoring patch normalizes
