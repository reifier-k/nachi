## 1. 目標と非目標

本RFCは、コンパイラおよびランタイムが実装される前に理想的な作成語彙を確定する。それ以降のすべてのパブリックAPIは、これに準拠するか、本RFCを明示的に更新することが期待される。主要な目標は、作成方法をコードファーストでフレームワーク非依存かつシリアライズ可能に保ちながら、Niagaraユーザーのメンタルモデルを転用可能にすることである。

本RFCが定義するのは次の事項である。

- モジュールコンパイラが受け付けるデータモデル
- ステージの順序と宣言されたデータアクセス
- 動的パーティクル属性とパラメータ名前空間
- エミッター、非パーティクル要素、エフェクト、タイムラインの合成
- JSONアセットと実行可能なTSLまたはJavaScriptコードとの境界
- 検証と診断の振る舞い
- `VFXSystem` の最小限のランタイム表面

本RFCは、GPUバッファレイアウト、ワークグループサイズ、間接描画の仕組み、WebGL2サポートの正確なサブセット、レンダラーのアルゴリズム、エディタUIについては定めない。これらの決定はM0スパイクおよび後続のマイルストーンに属する。

## 2. 設計原則

### 2.1 Niagaraのステージセマンティクスがパブリックな実行モデルである

エミッターは `Spawn -> Init -> Update -> Event -> Render` の順で評価される。モジュールはそのシリアライズされた形式において、ちょうど1つのステージに属する。登録された実装は複数のステージをサポートしてもよいが、各使用箇所はコンパイル前に1つのステージへと解決される。

ステージ名は安定したアセットフォーマットの語彙である。内部のコンパイラパスは、宣言された読み書きの依存関係によって結果が等価であると証明される場合に限り、作業を融合または並べ替えてもよい。

### 2.2 構成はデータである

`defineEmitter()` および `defineEffect()` が返す定義、モジュールの `config`、値ジェネレータ、タイムラインエントリ、リソース参照、アクセスマニフェストは、JavaScriptのオブジェクトIDを保持することなくJSONへシリアライズできなければならない。

関数、Three.jsのリソース、DOMオブジェクト、GPUオブジェクト、クラスインスタンス、シンボル、循環グラフは、シリアライズされたドキュメントに現れてはならない。作成用のヘルパーはフルーエントな関数であってもよいが、その出力はデータでなければならない。

### 2.3 TSLは明示的なエスケープ境界である

カスタムシェーダーの振る舞いには `tslModule()` を使用する。TSLファクトリはコンパイラがノードグラフを構築している間にのみ実行され、パーティクルごとにJavaScriptで1回ずつ実行されるわけではない。シリアライズ可能なアセットは、登録済みのTSLファクトリを安定したIDとバージョンによって参照する。

PLAN.md由来のインライン形式 `tslModule(callback)` は、ローカルでの作成、プロトタイピング、テストのために引き続き利用可能である。シリアライゼーションは、未解決のインラインファクトリをソースコードとして埋め込んだり黙って破棄したりするのではなく、拒否しなければならない。

### 2.4 共有されるすべてのデータは名前空間を持つ

コンパイラから可視な値は `System.*`、`Emitter.*`、`Particles.*`、`User.*` のいずれかのパスを使用する。裸の名前は `emitTo(..., { inherit: ['position'] })` のようなローカルな便宜上の位置でのみ許可され、コンパイル時に完全修飾パスへ正規化される。

### 2.5 パーティクル属性はエミッターローカルであり、コンパイルされる

組み込み属性とカスタム属性はエミッターのスキーマを構成する。コンパイラは、そのモジュールおよびレンダラーによって実際に読み書きされる属性から物理的なGPUレイアウトを導出する。2つのエミッターがレイアウトを共有する必要はない。

属性の型、デフォルト値、読み書きの宣言、アライメント、バックエンドサポートはコンパイル時の関心事である。ランタイムのコードは、すでにコンパイル済みのエミッターに属性を追加することはできない。

### 2.6 決定性と明示的な所有権

Initの乱数ジェネレータは、エミッターシード、決定的な誕生キーである
`Particles.spawnOrder`、モジュール/サンプルスロットから値を導出しなければならない。物理パーティクル
インデックスまたは `Particles.spawnGeneration` を含めてはならない。アロケータは、その誕生に選択された
フリーリストの物理スロットとは独立に、Initの実行前に `spawnOrder` を割り当てる。したがって、物理スロット
を再利用するたびに新しいInitサンプルを受け取り、アトミックなフリーリストの追加順序が変化しても、同一の
シードとスポーンスケジュールは同じサンプルを再現する。

パブリックなモジュールは `Math.random()` を呼び出してはならない。Updateステージの乱数ジェネレータは、
エミッターシード、`Particles.spawnOrder`、モジュール/サンプルスロット、`Emitter.updateRandomStep` から値を
導出する。物理パーティクルインデックスまたは `Particles.spawnGeneration` を含めてはならない。update stepは
1つのRuntimeEmitterに対する2^32を法とするu32序数である。プール済みkernelのcheckoutを含む新規runtimeは
current値0から開始する。fixed/prewarm substepとevent-input fallback simulationを含む各simulation Update submitの
直前にruntimeがcurrent値をuploadし、そのdispatchが値を消費する。renderer submissionのawaitが成功した後に限り、
private counterを2^32を法とするnext値へ進める。submitがrejectされた場合はnext値を進めない。pipeline準備中の
compile-purpose Update submitはこの序数を消費せず、進めない。pause、完全なhit stop、culling、zero-delta、その他
dispatchされない区間でも進まない。lifecycle loopではresetせず、`Emitter.localTime` とともに再開始する代わりに
`0xffffffff`から0へ自然wrapする。`kernels.uniforms['Emitter.updateRandomStep'].value` は0で初期化され、1回以上の
simulation Update後は最後のdispatchが消費したcurrent値を保持する一方、private counterはすでにnext値を保持する。
したがって、時間変化するUpdate rangeは実行stepごとに変化する一方、同じseed、spawn schedule、step/dispatch列は
同じ値を再現する。Spawnポリシーのrangeは
割り当て前の例外であり、パーティクルの誕生がまだ割り当てられていないため `Emitter.spawnGeneration` を使用
する。event-spawn InitはInitの誕生キー契約を引き続き使用する。一般の物理割り当て順序は依然として決定的では
ない。1回のディスパッチで複数のパーティクルが消滅する場合、アトミックな追加順序は、アダプターや実行を
またぐ並列ハードウェアのレースとなる。InitとUpdateの乱数はこのレースから明示的に分離されるが、完全に
決定的な物理割り当てと順序依存シミュレーションは、将来のオプトインモードとして残る(§16)。

`VFXSystem` はコンパイル済みリソースを所有し、エフェクトをスケジューリングし、ワールド/レンダリング統合を提供する。エフェクトインスタンスはローカル時間と `User.*` の上書きを所有する。エミッターはパーティクル状態を所有する。モジュールは、コンパイラがそのために明示的な `Emitter.*` の値を割り当てない限り、可変のランタイム状態を所有しない。

決定的な乱数ストリームは、モジュールのステージと、存在する場合は空でない `label` とからモジュールスロットを識別する。存在しない場合は、ステージとそのステージ内での正規化されたインデックスとから識別する。ステージのアイデンティティは常にスロットに混合されるため、異なるステージにおける等しいインデックスやラベルが同一のストリームにエイリアスすることはない。ラベルは1つのエミッターステージ内で一意でなければならない。ラベルのない乱数モジュールの追加や並べ替えは、それ以降のラベルなしストリームを変化させる可能性がある。安定したラベルは、継承のオーバーライドや同一ステージ内での並べ替えをまたいでそのストリームを保持する。

PCG RXS-M-XSの入力は、XOR混合の前に、誕生キー、エミッターシード、モジュール/サンプルスロット、進行キー
について、それぞれ独立した奇数乗算の軸を保持する。Initは進行キーを0に固定し、Updateは
`Emitter.updateRandomStep` を使用し、割り当て前のSpawn CPU mirrorは従来の発生世代を維持する。これは
`order + stream` の格子ではないため、隣接するorder、update step、module、sample offsetが隣接streamの
シフトコピーを受け取る構造にはならない。あらゆる32ビットハッシュと同様に、無関係なタプルは通常の
32ビット確率で衝突し得る。この契約はストリーム分離であり、直積全体にわたる大域的な単射性ではない。

H1-10互換性注記(2026-07-14): `context.random()` を呼び出す外部登録のInit実装は、モジュール定義と登録実装
双方の `access.reads` に `Particles.spawnOrder` を宣言しなければならない。いずれかの宣言で省略すると、必要な
access更新を示す `NACHI_INIT_RANDOM_SPAWN_ORDER_ACCESS_REQUIRED` により具現化が失敗する。`@nachi-vfx/format`
はモジュールのaccess manifestを直列化するため、この契約変更より前にエクスポートされたJSONアセットには、
以前のInit readである `Particles.spawnGeneration` が残り得る。そのアセットを更新後の実装に対してロード
すると、実行可能な契約を暗黙に書き換える代わりに `NACHI_MODULE_ACCESS_MISMATCH` でfail closedする。影響を
受けるすべてのパッケージは未公開のversion `0.0.0` のままであるため、アセットmigrationは提供しない。該当
する開発中アセットは再エクスポートするか、access manifestを再作成しなければならない。

H2-3互換性注記(2026-07-15): `context.random()` を呼び出す外部登録のUpdate実装は、モジュール定義と登録実装の
双方の `access.reads` に `Emitter.seed`、`Particles.spawnOrder`、`Emitter.updateRandomStep` を宣言しなければ
ならない。いずれかを省略すると `NACHI_UPDATE_RANDOM_STABLE_KEY_ACCESS_REQUIRED` により具現化が失敗する。
legacy definitionが登録実装の宣言readも含まない場合は、implementation random tracingより前に汎用の
`NACHI_MODULE_ACCESS_MISMATCH` validationが先に失敗する。definition側がstable入力とlegacy実装のreadを
supersetとして含む一方、実装側だけがlegacyの場合はUpdate固有診断へ到達する。したがって片側だけの不一致も
双方ともfail closedする。従来の `Particles.spawnGeneration` Update readを保持する開発中の直列化assetは、
同じregistry access規則でfailし、再エクスポートまたはaccess manifestの再作成が必要となる。inline `tslModule()` は
`KernelModuleBuildContext.random()` を公開しないため、影響を受けるescape hatchは登録済みkernel実装である。
M1 all-slots互換Init経路とevent-spawn Init経路は `Emitter.updateRandomStep` を消費せず、従来のbitを維持する。

`$` で始まるラベルはコンパイラが合成するモジュールのために予約されており、ユーザーが作成したモジュールでは拒否されなければならない。これにより、コンパイラが所有するアイデンティティが、作成者のラベルと衝突するリスクなしに安定に保たれる。

## 3. 中核概念

### 3.1 モジュール

**モジュール(Module)** とは、バージョン管理された設定オブジェクトと、レジストリによって解決されるコンパイラ実装との組み合わせである。モジュールは自身のステージとデータアクセスを宣言する。

```ts
interface ModuleDefinition<Stage, Config> {
  kind: 'module';
  type: string;
  version: number;
  label?: string;
  stage: Stage;
  config: Readonly<Config>;
  access?: {
    reads: readonly DataReference[];
    optionalReads?: readonly DataReference[];
    writes: readonly DataReference[];
  };
}
```

`type` は `core/gravity` のような安定したレジストリIDであり、JavaScriptのコンストラクタ名ではない。`version` はアセットスキーマとは独立にモジュール設定をバージョン管理する。

`label` はシリアライズされた作成者の意図であり、継承パッチ、診断、プロファイリング、決定的なRNGスロットのための安定したアイデンティティを提供する。ラベルが存在しない場合は、正規化されたステージインデックスをそのアイデンティティとして使用する。`access` が存在しなくてもよいのは、正規化の過程でマニフェストが導出される作成用定義──とりわけトレースされた `tslModule` ──に限られる。コンパイルおよびシリアライズされたすべてのモジュールは、解決済みのアクセスマニフェストを持つ。

`gravity(-9.8)` のような組み込みヘルパー呼び出しは、固定されたマニフェストを持つモジュール定義を返す。サードパーティのモジュールは、名前空間化されたtype IDのもとに、設定バリデータ、アクセスマニフェスト、TSLコンパイラ実装を登録する。

### 3.2 エミッター

**エミッター(Emitter)** とは、容量に上限のあるパーティクルシミュレーションと、そのレンダラー群である。エミッターは、スポーンポリシー、初期化モジュール、更新モジュール、イベントハンドラ、レンダーモジュール、任意のカスタム属性、およびライフサイクル設定を含む。`defineEmitter()` は構造的なデータを正規化するが、GPUリソースは一切割り当てない。定義は意図的に構造的型付けを用いており、作成用ヘルパーが具現化できないランタイム専用のブランド(brand)は存在しない。

### 3.3 エフェクト

**エフェクト(Effect)** とは、名前付きの要素グラフとタイムラインとの組み合わせである。要素はエミッターであってもよいし、`mesh-fx`、`trails`、`post` のようなパッケージが提供する非パーティクルの視覚要素であってもよい。要素のキーは `play()` および `stop()` が受け付けるターゲットである。

エフェクト定義は再利用可能な不変の作成用データである。スポーンされたエフェクトインスタンスは、可変の再生状態とパラメータのオーバーライドを含む。

### 3.4 VFXSystem

`VFXSystem` は、レンダラーおよびシーン/ワールド統合に紐づけられたランタイムコーディネータである。

```ts
const fx = new VFXSystem(renderer, scene);
const instance = fx.spawn(effect, {
  position,
  rotation,
  seed: 42,
  parameters: { 'User.intensity': 1.5 },
});

instance.setParameter('User.intensity', 0.8);
instance.stop();
instance.release();
```

コンパイルは、最初のスポーン時に遅延して行われてもよいし、将来のAPIにおいて明示的に行われてもよい。コンパイルの失敗は、インスタンスが自身をアクティブとして報告する前に観測可能でなければならない。

### 3.5 名前空間化されたパラメータ

- `System.*`: `System.deltaTime` や `System.time` など、システム全体で共有されるフレームおよびワールドの値。通常はパーティクルモジュールに対して読み取り専用である。
- `Emitter.*`: エミッターローカルなカウンタ、トランスフォーム、経過時間(age)、ループ状態、モジュールのスクラッチ値。
- `Particles.*`: 動的なGPUバッファに格納されるパーティクルごとの属性。
- `User.*`: 定義またはエフェクトインスタンスごとにアプリケーションコードから供給される型付き入力。

## 4. ステージとモジュールの契約

### 4.1 ステージの順序

| ステージ | 責務                                                                                       | 典型的な書き込み                                           | 制約                                                                                    |
| -------- | ------------------------------------------------------------------------------------------ | ---------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| Spawn    | 割り当てるパーティクル数とタイミングを決定する                                             | `Emitter.spawnCount`、割り当てメタデータ                   | 任意のパーティクル属性を初期化しない                                                    |
| Init     | 新しく割り当てられたパーティクルを初期化する                                               | `Particles.position`、`velocity`、`lifetime`、カスタム属性 | スポーン世代ごとに1回実行される                                                         |
| Update   | 生存しているパーティクルの状態を進める                                                     | 宣言された可変の `Particles.*`/`Emitter.*` の値            | ウォールクロック時間ではなく、システム/エミッターのローカル時間を使用しなければならない |
| Event    | 生成されたイベントを消費し、エフェクト/サブエミッター/コールバックレコードをエンキューする | イベントキューと明示的に宣言された継承属性                 | エミッターをまたぐ作業は、定義されたスケジューラ境界において可視になる                  |
| Render   | 描画入力とレンダラー状態を生成する                                                         | レンダー出力のみ                                           | シミュレーション属性は読み取り専用である                                                |

Updateの行は意図された契約を記述したものである。M2では、作成者が定義する可変の `Emitter.*` 値に対する宣言や具現化はまだ実装されていない。そのサポートが実現するまで、作成者のモジュールはそのような書き込みに依存してはならない。コンパイラ/ランタイムが所有するパスは、ステージにかかわらずより厳格である。作成者のモジュールは `Particles.spawnGeneration`、`Particles.spawnOrder`、`Emitter.spawnCount`、`Emitter.allocation.*` に書き込むことができず、`Emitter.events.*` に書き込めるのはEventモジュールのみである。

現在のコア実装は、上記の一般的なSpawnステージの契約よりも厳格である。`Emitter.spawnCount` への書き込みが許されるのは、組み込みの3つのスポーンモジュール `core/burst`、`core/rate`、`core/per-distance` のみである。他の作成者モジュールは、そのステージがSpawnであっても、この書き込みを宣言することはできない。

ディスパッチ中に発生したイベントはバッファリングされる。それらは同一のGPUディスパッチ内で無制限のエミッターチェーンを再帰的に実行することはない。イベントハンドラは `Emitter.events.<eventName>` というキューパスに書き込む。M5では消費を **次のスケジューラフレーム** に固定する。各プロデューサキューは2つのバンクを持ち、フレームNはバンク `N & 1` にのみ追加を行い、コンシューマはバンク `(N - 1) & 1` のみを読み取る。コアはいずれかのエミッターが進行する前に新しい書き込みバンクをクリアするため、エフェクト要素の順序が同一フレーム内の再帰を生み出すことはない。固定時間のサブステップは1つのスケジューラフレームである。1回のアクティベーション内部のプリウォームステップは1つのバンクを保持するため、有界なオーバーフロー挙動が維持される。

`Emitter.eventPayload.<field>` は、消費されるイベントレコードの型付き読み取り専用ビューである。`emitTo(target, { inherit: ['position'] })` は、プロデューサの `Particles.position` のスナップショットを `Emitter.eventPayload.position` にマップする。通常のターゲットのInitが実行された後、コアはそのフィールドからターゲットの `Particles.position` を上書きする。継承されない属性は、通常のInit結果を保持する。スナップショットは、プロデューサのスロットが再利用される前の、コンパイラが所有する消滅エピローグ内で取得される。

M2は、Init/Updateで使用されるものと同じバージョン管理されたモジュールレジストリおよびアクセス検証を通じてSpawnモジュールをコンパイルする。Spawnモジュールは作成者の順序で実行され、要求を `Emitter.spawnCount` に加算する。その数をパーティクルインデックスへ変換できるのはアロケータのみである。WebGPUでは、スポーン要求のuniformがGPU書き込みの間接ディスパッチを駆動し、成功した呼び出しはフリーリストから散在したインデックスを取り出し、`Particles.spawnGeneration` をインクリメントし、その後Initをちょうど1回実行する。利用可能な空き数を超える要求はクランプされ、カウントされ、`NACHI_SPAWN_CAPACITY_EXCEEDED` として報告される。それらがアトミックカウンタをラップしたり、生存しているパーティクルを上書きしたりすることはない。CPU側の要求クランプは即座に報告される。GPU側のオーバーフロー報告には `aliveCountReadbackInterval` が必要である。なぜなら、その明示的なインターバルがなければ、M2はカウンタの読み戻し(readback)もGPU/CPU同期も一切行わないためである。本段落における「報告される」は、このオプションに依存する。

### 4.2 読み書きの宣言

コンパイルされたすべてのモジュールは、完全修飾パスからなる `ModuleAccess` マニフェストを持つ。

```ts
const gravityRegistration: ModuleRegistration<'update', GravityConfig> = {
  type: 'core/gravity',
  version: 1,
  stage: 'update',
  access: {
    reads: ['Emitter.deltaTime', 'Particles.velocity'],
    writes: ['Particles.velocity'],
  },
  compile(context, config) {
    // Build TSL nodes; no per-particle JavaScript callback.
  },
};
```

コンパイラは次を行わなければならない。

1. 各パスを、組み込み、カスタム属性、またはパラメータ宣言と照合して解決する
2. 不足している必須の読み取りを拒否する
3. 書き込まれた後に読み取られる値を割り当てる
4. ステージの書き込み権限を検証する
5. トレースが可能な場合、トレースされたTSLグラフと一致しない読み書き宣言を検出する
6. 観測可能なread-after-writeおよびwrite-after-writeの依存関係について、作成者の順序を保持する
7. 未使用の宣言済みカスタム属性を、エラーではなく警告として報告する

項目4は、Spawn、Init、Update、Event、Renderの各マニフェストに対して強制される。Spawnは `Emitter.spawnCount` またはコンパイラが所有する `Emitter.allocation.*` のみに書き込むことができる。Initは `Particles.*` に書き込むことができる。Updateは宣言された可変の `Particles.*`/`Emitter.*` に書き込むことができる。Eventはパーティクルの継承とその正規化されたイベントキューに書き込むことができる。Renderはシミュレーションのパスに書き込むことはできない。

正規化の過程で、コンパイラはモジュールの設定を再帰的に訪問する。ネストされたすべての `parameter('Namespace.name')` ジェネレータは、そのパスを暗黙的に `reads` に追加し、これはモジュールの宣言済みマニフェストとマージおよび重複排除される。同様に、`events.onDeath` の下への配置は、各ハンドラの `writes` において、作成専用のプレースホルダである `Emitter.events.pending` を `Emitter.events.onDeath` に置き換える。ヘルパーから導出されるアクセスは正規のシリアライズ済みデータであり、依存関係解析の例外ではない。

`optionalReads` は、深度(depth)のようなケイパビリティに依存する入力を許容する。モジュールの実装は、任意の値が存在しない場合のフォールバック挙動を定義しなければならない。この解決は必須の読み取りとは意図的に非対称である。既知の組み込みの任意属性は割り当てられるが、存在しない任意のカスタム属性は診断も割り当ても行われず、モジュールのフォールバックが使用される。

### 4.3 予約された統合モジュール

速度(velocity)は、力(force)モジュール内部で暗黙に位置(position)を更新することはない。`integration: 'none'` が選択されない限り、M1のコンパイル時正規化は、安定したtype `core/integrate`、ラベル `$integrate`、および以下のマニフェストを持つコンパイラ所有のモジュールを追加する。`defineEmitter()` は作成用データを正規化するのみであり、`$integrate` を合成することも組み込むこともない。

```ts
{
  reads: ['Emitter.deltaTime', 'Particles.position', 'Particles.velocity'],
  writes: ['Particles.position'],
}
```

デフォルトのインテグレータはsemi-implicit Euler(半陰的オイラー法)である(`velocity` は位置の積分の前に確定される)。力とインテグレータは `System.deltaTime` ではなく `Emitter.deltaTime` を読み取る。エミッターローカルな値はエフェクトの `timeScale` と `hitStop` を含むため、速度と位置が異なるクロックで進行することはない。`System.deltaTime` は、それを明示的に必要とするモジュールのために、スケールされていないホストフレームの値のままである。カスタムの位置書き込みを供給するには、`integration: 'none'` または将来の明示的なインテグレータ選択が必要である。コンパイラは両者を黙って同時に適用してはならない。

コアのパーティクルの位置と速度はワールド空間に格納され、Init中に粒ごとのemitter transformを適用する
`positionSphere()` と一致する。`killVolume()` は意図的な作成上の例外であり、そのボックス、球、平面の
寸法はエミッターローカル空間で表現される。Updateの実装は、ボリュームを判定する前に、ワールド空間の
各 `Particles.position` を共有H2-6 Update-midpoint transformのinverseによって変換する。その結果、
スポーンされたインスタンスの平行移動と回転はエミッターとともにキルボリュームを移動させるが、パブリックなエミッター
トランスフォームAPIによって現在公開されているスケールはない。平面の法線は評価前に正規化されるため、
`offset` はローカル空間の距離のままである。その内側半空間は
`dot(normalize(normal), localPosition) <= offset` である。消滅は `Particles.alive = false` を書き込み、
通常のコンパイラが所有するフリーリストのエピローグによって回収される。

`positionSphere({ radius, center?, arc?, surfaceOnly? })` は、その形状をエミッターローカル空間で
作成する。`center` は `ValueInput<Vec3>` で、既定値は `[0, 0, 0]` である。`arc` は球冠へ
サンプリングを制限する。`thetaMax: ValueInput<number>` の単位は度であり
(`velocityCone.angle` と同じ単位規約)、エミッターローカルな `axis` の既定値は `[0, 1, 0]`
である。球冠の面積一様性は `cos(theta)` を `[cos(thetaMax), 1]` から一様に選ぶことで保つ。
`surfaceOnly` がfalseのときも、既存の立方根による半径サンプルを維持し、球の中心を頂点として
その球冠が張る球分(spherical sector)内で体積一様にサンプリングする。`center` と `arc` の両方を
省略した場合は、従来の全球カーネルグラフと決定的な出力を維持する。静的なcenter成分は有限、
`thetaMax` は `(0, 180]`、指定されたaxisは有限かつ非ゼロでなければならない。

モジュール座標空間と単位の完全な現行一覧は [RFC 004](./004-module-spaces.ja.md) に記録する。省略時既定は
module別である。H1-5の5つのforce/collider selectorは`emitter`、H2-6の`velocityCone`と`linearForce`は
v1互換のため`world`を既定とする。`gravity`はworld固定のままである。scene depth、SDF、vector field、
decalなどの固定consumerを黙って再解釈しない。

`turbulence()` はパーティクルのワールド空間において決定的な3Dシンプレックスノイズをサンプリングする。
そのオクターブ数、周波数の倍加、振幅の半減、チャンネルごとのオフセットはコンパイル時/固定の定数であり、
ウォールタイムも乱数ストリームも関与しない。したがって、初期パーティクルストレージとディスパッチ順序が
同一であれば、2回の実行はビット単位で同一のノードグラフを構築・実行する。

`curlNoise()` は、3Dポテンシャル場を構築するために、同じ中心化された `snoise` スカラーと固定の
チャンネルごとのオフセットを使用する。中心差分によって `curl(psi)` を評価する。結果として得られる場は、
有限差分および浮動小数点誤差の範囲内で発散フリー(divergence-free)である。測定された固定の微分振幅
補正により、`strength` はおおよその最大加速度となり、発散を再導入してしまうような位置依存のスケールを
適用することなく、`turbulence()` で使用されるパブリックな正規化と一致する。周波数はその正規化を変える
ことなく空間スケールを変化させ、クロックも乱数ストリームも関与しない。

`vortex()` と `pointAttractor()` は `space: 'world' | 'emitter'` を受け付け、デフォルトは `emitter` で
ある。エミッター空間での評価は、ワールド空間のパーティクル位置を共有Update-midpoint transformのinverseで
変換し、ローカルな中心と軸を評価した後、完全に同じtransform式を方向の重みゼロで使って加速度をworldへ
戻す。したがって、平行移動は加速度を汚染することなくエミッターに追従し、一方でエミッターの回転は場と
返却される力の両方に影響する。

`velocityCone()` と `linearForce()` も同じselectorを受け付けるが、v1の方向/加速度を維持するため既定は
`world`である。明示`space: 'emitter'`ではcone方向はそのbirthの`Emitter.spawnInterpolatedTransform`、
linear forceは共有Update midpointを使う。どちらもdirection形式(`w = 0`)なので平行移動はvectorへ入らない。
`gravity()`はworld-space加速度固定でselectorを持たない。2つ以外のselector literalは
`NACHI_MODULE_SPACE_INVALID`として拒否する。

`velocityCone`、`linearForce`、`vortex`、`pointAttractor`、解析collider 3種、`killVolume`のH2-6実装は
module version 2である。version-1 registry entryはH2-6前の実行を維持する。cone/linearは旧generic assetに
無視されるemitter fieldが含まれてもworld-space、emitter-space Update consumerはcurrent終点をsampleする。
現行helperはversion 2を出力する。format load/serializeはv1 recordを暗黙にv2へupgradeせず、H2-6前readerは
`type@2`を`NACHI_MODULE_UNKNOWN`で安全に拒否する。このためH2-6実装時点ではRFC 003 §4に違反せず、当時の
asset envelopeをversion 1のまま維持できた。H2-7はその後renderer境界のためcanonical envelopeをversion 2へ
上げたが、defaultのenvelope-only migrationはこれらH2-6 module recordとversionを変更しない。

M6の解析的コライダーは、これらの力モジュールと同じ明示的な `space: 'world' | 'emitter'` の規約を用い、
デフォルトは `emitter` である。これはエミッターローカル専用の `killVolume()` と同じ作成規則である。
emitter-space解析colliderは同じ共有Update-midpoint transformを通して評価とresponse書き戻しを行い、
inverse/forward変換で独立に再sampleしてはならない。
`collidePlane()` は `dot(normalize(normal), position) < offset` を貫通(penetration)として扱う。
`collideSphere()` と `collideBox()` はそのボリュームを中身の詰まったもの(solid)として扱い、内部の
パーティクルを最も近い表面へ押し出す。`mode: 'bounce'` は外向きの法線成分を保存し、内向きの成分を
`bounce` の反発係数で反射し、接線速度に `1 - friction` を乗算する。`stick` は速度をゼロにし、`kill` は
パーティクルを回収する。両係数は具現化時に `[0, 1]` にクランプされ、静的に無効な定数は診断される。
衝突の位置補正は許可された作成者のUpdate書き込みであり、カスタムインテグレータの二重書き込み診断を
発生させることなく、コンパイラが所有する `$integrate` に先行してもよい。

`collideSceneDepth()` は、ワールド空間において同じ応答モードを使用する。コンピュートは
`viewportDepthTexture` を直接読み取ることはない。そのノードはレンダーパスにスコープされており、
アクティブなアタッチメントをサンプリングするとフィードバックハザードが生じるためである。レンダラー
統合は、完了したシーンパスの深度を、正規化されサンプリング可能なカラーテクスチャへコピーし、その
前フレームのコピーをカーネルアダプターにバインドする。パーティクルは `System.viewMatrix` と
`System.projectionMatrix` によって投影される。後者は明示的にWebGPU座標系の投影行列であり、NDC深度は
`z ∈ [0, 1]` であるため、投影された深度はそのまま使用され、逆投影はサンプリングされた深度をそのまま
受け取る(OpenGLの `z * 0.5 + 0.5` / `z * 2 - 1` 変換は行わない)。深度コピーは、正規化された深度を
リニアかつ非sRGBの浮動小数点カラーテクスチャに保持しなければならない。出力カラー変換や正規化された
8ビットレンダーターゲットは、深度をエンコード・量子化してしまうため無効である。`System.viewportSize`
は深度勾配のためのテクセルオフセットを提供する。近傍の深度サンプルは逆投影されてビュー空間の法線を
再構築し、それがワールド空間へ変換し直される。

x/yの変換は意図的に非対称である。Three r185の `screenUV` およびコピーされたレンダーターゲット
テクスチャは左上原点を使用する(`v` は下方向に増加する)が、WebGPUのNDCの `y` は上方向に増加する。
したがって、パーティクルの投影には `uv = (ndc.x * 0.5 + 0.5, 0.5 - ndc.y * 0.5)` を使用し、再構築には
`ndc = (uv.x * 2 - 1, 1 - uv.y * 2)` を使用する。y方向の `oneMinus` 操作は両方向で必須である。
`/m6-collision/` の回帰シーンは遮蔽物をX軸周りに回転させ、非ゼロのyを持つパーティクルをテストする。
対称的な正面向きやY軸回転した平面では、このミラー誤りを検出できない。

three.js r185において、`pass(scene, camera)` ノードは、プレゼンテーションループの外でこのコピーを
駆動する決定的な方法ではない。`PassNode` は `FRAME` のupdate-beforeセマンティクスを使用し、
`NodeFrame` は同一フレームIDに対する2回目の更新を抑制するため、`RenderPipeline.render()` の
繰り返し呼び出しは、黙って以前のシーン構成をサンプリングしてしまう可能性がある。推奨される統合方法は、
代わりに非MSAAの `DepthTexture` を `RenderTarget` にアタッチし、そのターゲットを選択して、キャプチャの
たびに明示的に `renderer.render(scene, camera)` を呼び出し、その後、上述のリニアかつ非sRGBの浮動
小数点カラーターゲットへ深度テクスチャをサンプリングする別個のTSLコピーパイプラインを実行することである。

したがってM6 v1は、WebGPU、非MSAAの解決済み深度コピー、一致するキャプチャ/シミュレーションのカメラと
ビューポートを要求し、動くジオメトリに対する1フレームのレイテンシを許容する。バインディングの欠落は
`NACHI_SCENE_DEPTH_UNAVAILABLE` として診断され、WebGL2とMSAAソースは明示的に拒否され、未設定の
カメラや検出されたリバースzの投影はランタイム警告を生成する。いずれも黙ったフォールバックとして
扱われることはない。

`collideSceneDepth({ thickness })` は、リニアなビュー深度の差が `thickness`(デフォルト `0.1`)を
超える貫通を拒否する。これにより、前景のジオメトリのはるか奥に既に隠れているパーティクルが、その
前景表面へテレポートしてしまうことを防ぐ。`surfaceOffset` は、有効な衝突が発生した後にのみ適用される、
小さなワールド空間の分離量のままである。

`collideSdf()` は、シリアライズ可能な `SdfRef` をワールド空間でサンプリングする。負の距離は貫通を
意味する。法線は正規化された中心差分勾配であり、補正はゼロ面までの符号付き距離だけパーティクルを
移動させる。任意の `thickness` は、シーン深度と同様の方法で深く埋没したサンプルを拒否する。パブリック
な `bakeSdf()` ユーティリティは、テストおよび手続き的アセットのために、解析的な球とボックスの和集合
からコーナー整列のfloat32フィールドを生成する。メッシュからSDFへのベイクは、ボクセル化と水密性
(watertightness)のポリシーがアセットパイプラインの決定事項であるため、M12へ先送りされている。

`positionMeshSurface({ mesh, mode: 'surface' })` は、3つの決定的なパーティクル乱数サンプルを用いて
面積重み付き三角形CDFをサンプリングし、ワールド空間の位置を書き込み、サンプリングされた法線を
コンパイラが認識する `Particles.surfaceNormal` に保持する。`velocityMeshNormal({ speed })` はその
法線を消費する。バックエンドのリソースは、三角形の頂点を **メッシュローカル空間** に格納する。これには、
オブジェクトトランスフォーム適用前のCPUスキニングによる変形後の位置も含まれる。Initカーネルは
`Emitter.transform` をちょうど1回だけ適用して、ワールド空間の `Particles.position` を生成し、その
ディレクション形式(`w = 0`)をサンプリングされた法線に適用する。したがって、メッシュリソースは
`matrixWorld` を事前に適用してはならない。そうすると、静的サンプルとスキニングされたサンプルの両方に
対して二重の変換が行われてしまう。CDFはそれと同じローカル空間の変形後三角形から構築されるため、その
確率は実際にサンプリングされるジオメトリに対して比例したままとなる。

Three v1の統合は、`SkinnedMesh.applyBoneTransform()` からCPU上で三角形の位置と面積CDFを再構築し、
毎フレーム両方のfloatテクスチャをアップロードする。これは、ボーンテクスチャを意図的にコアの外に保ちつつ、
静的メッシュとスキニングされたメッシュに対して単一の `MeshRef` 契約を提供するものであり、その代償として
ソース三角形数に比例したCPUスキニング/アップロードのコストを伴う。大規模な群衆には、将来のGPU
スキニングデータインターフェースが必要となる。リソースは固定されたソーストポロジーを保持し、フラット
な三角形法線を使用する。トポロジーの変更にはリソースの再構築が必要であり、滑らかな変形後の頂点法線
補間はGPUスキニングパスとともに先送りされている。v1の三角形位置テクスチャは、三角形ごとに3テクセルを
含む単一の行であるため、`3 * triangleCount` はアクティブなデバイスの `maxTextureDimension2D` を超えて
はならない。Threeヘルパーはこの制限を受け入れ、WebGPUが保証する最小値である8192テクセル(最大2730個の
完全な三角形)をデフォルトとする。過大なテクスチャを割り当てる前に `NACHI_MESH_SURFACE_TEXTURE_TOO_WIDE`
を報告する。複数行/2Dタイル化された三角形およびCDFテクスチャはM12へ先送りされている。

パブリックな `turbulence.strength` は、生のシンプレックス乗数ではなく、最大加速度の大きさを近似する。
オクターブは振幅の和で正規化され、中心化されたThree r185の `snoise` は `1 / 0.286` を乗算される
(その測定された実効スカラー振幅はおよそ `±0.286` である)。結果として得られる3成分ベクトルは、
`strength` を適用する前に長さが1にクランプされる。

`$integrate` のラベルは、コンパイラが予約する `$` プレフィックスを使用する。ユーザーが作成するモジュール
のラベルは `$` で始まってはならない。正規化は、それらを改名するのではなく、コンパイル診断を報告する。

同じM1のコンパイル時正規化は、カーネル記述において追加で2つのコンパイラ所有モジュールを具現化する。
`$defaults`(`core/defaults`)は最初のinitモジュールであり、解決済みのすべての `Particles.*` 属性を、
その宣言済みデフォルト値から書き込む。解決済みのライフサイクルスキーマに `age` と `lifetime` が存在
する場合、`$age`(`core/age`)は最初のupdateモジュールであり、以下のマニフェストを持つ。

```ts
{
  reads: ['Emitter.deltaTime', 'Particles.age', 'Particles.lifetime'],
  writes: ['Particles.age', 'Particles.normalizedAge'],
}
```

`$age` は、作成者のupdateモジュールより前に、経過時間(age)を進め、クランプされた正規化age
(normalized age)を導出する。これらの操作は、TSLカーネルの具現化時に注入される隠れたステートメントで
はなく、記述されたモジュールである。

### 4.4 モジュール設定と実装の分離

シリアライズされたモジュールは、`type`、`version`、任意の `label`、`stage`、`config`、および解決済みの `access` を含む。実行可能な検証関数とコンパイル関数は `VfxRegistry` に存在する。これにより、アセットのJSONは検査可能なものとなり、バンドラ固有の関数名がフォーマットの識別子になってしまうことを防ぐ。

## 5. 動的属性

### 5.1 組み込み属性

ベースラインのパーティクルスキーマには次が含まれる。

| 名前                                 | 論理型           | 用途                                                             |
| ------------------------------------ | ---------------- | ---------------------------------------------------------------- |
| `position`                           | `vec3`           | エミッター空間に応じたエミッターローカルまたはワールド位置       |
| `velocity`                           | `vec3`           | 線形速度                                                         |
| `age` / `lifetime` / `normalizedAge` | `f32`            | ライフタイムの評価                                               |
| `alive`                              | `bool`           | ライフサイクル状態。割り当てられるまではデフォルトでfalse        |
| `spawnGeneration`                    | `u32`            | スロットごとの割り当て世代とデバッグ系譜                         |
| `spawnOrder`                         | `u32`            | 決定的な誕生順序とInit乱数キー                                   |
| `color`                              | `color` (`vec4`) | リニアRGBA                                                       |
| `intensity`                          | `f32`            | ライトレンダラーの強度と優先度                                   |
| `size` / `scale`                     | `f32` / `vec3`   | レンダラーのサイズ                                               |
| `rotation`                           | `quat`           | メッシュ/一般的な向き                                            |
| `spriteRotation`                     | `f32`            | ビルボードの回転                                                 |
| `surfaceNormal`                      | `vec3`           | メッシュサーフェスのInitモジュールによってサンプリングされる法線 |
| `mass`                               | `f32`            | 力の積分                                                         |

組み込みは論理的なAPI名であり、固定された物理構造体ではない。コンパイラは未使用の組み込みを省略してもよい。`normalizedAge` は格納される代わりに導出されてもよい。

### 5.2 カスタム宣言

カスタム属性はエミッター上で宣言される。

```ts
const heat = attribute('heat', {
  type: 'f32',
  default: 0,
});

const embers = defineEmitter({
  capacity: 4096,
  attributes: { heat },
  spawn: burst({ count: 128 }),
  init: [
    /* modules that initialize heat */
  ],
  update: [
    /* modules that read/write heat */
  ],
  render: billboard({ blending: 'additive' }),
});
```

当初サポートされる論理型は `f32`、`i32`、`u32`、`bool`、`vec2`、`vec3`、`vec4`、`color`、`quat`、`mat3`、`mat4` である。バックエンドのレイアウトルールはコンパイラが所有する。属性のデフォルト値は決定的であり、型に対して互換でなければならない。

組み込み名を異なる型で再宣言することはエラーである。書き込まれないままになる可能性のあるカスタム属性にはデフォルト値が必要である。`transient` とマークされた属性は、シミュレーションキャッシュおよびパーティクルスナップショットのシリアライゼーションから除外される。コンパイラは、キャッシュのリプレイパスで必要とされるtransient属性を拒否しなければならない。

`attributes` レコードのキーは正規のものであり、`AttributeDefinition.name` と厳密に一致しなければならない(`attributes.heat.name === 'heat'`)。作成時の正規化は不一致を拒否し、一方を黙って改名することは決してない。この名前の重複は、単体の属性定義が移動・検査・ツールによる参照が行われた際にも自己記述的であり続けるように保持されている。

## 6. 名前空間化されたパラメータ

パラメータの宣言は、完全パスと論理型を用いる。

```ts
const intensity = defineParameter('User.intensity', {
  type: 'f32',
  default: 1,
  mutable: true,
});

const gravityScale = parameter('User.gravityScale', 1);
```

| 名前空間      | 宣言者                      | アプリケーションコードから可変か | 生存期間                                |
| ------------- | --------------------------- | -------------------------------: | --------------------------------------- |
| `System.*`    | ランタイム/コンパイラ       |    いいえ(システム統合APIを除く) | VFXSystem/フレーム                      |
| `Emitter.*`   | コンパイラ/モジュール登録   |     直接のパブリック書き込みなし | エミッターインスタンス                  |
| `Particles.*` | 組み込み/エミッター属性     |     直接のパブリック書き込みなし | パーティクル                            |
| `User.*`      | エフェクト/エミッター作成者 |    宣言でmutableとされていれば可 | 定義のデフォルト + インスタンスの上書き |

未知のパスはコンパイルエラーである。未知・不変・型非互換な `User.*` パラメータへのランタイム書き込みは、ランタイム検証エラーである。パラメータの名前空間は大文字小文字を区別し、記載されたとおりに厳密に予約される。

エフェクト定義とエミッター定義はどちらも `parameters` を宣言してよい。エフェクトの宣言はパブリックなインスタンス表面を定義し、エミッターの宣言は再利用可能なエミッターローカルのデフォルトを定義し、M9の合成の過程で、それを含むエフェクトへマップされる。`EffectInstance.setParameter()` およびスポーン時の `parameters` は、そのパスと値の型を、エフェクトの宣言された `User.*` スキーマから導出する。エフェクトと子エミッターの宣言の間の正確な上書き/転送ルールは、M9の決定事項のままである。

M9バッチ1はこのルールを解決する。`defineEffect()` は、すべての子エミッターの `User.*` 宣言を、エフェクトの
パブリックなパラメータスキーマへ引き上げる。同一の引き上げられたパスの宣言は、同一の論理型、デフォルト値、
可変性を持たなければならない。明示的なエフェクトレベルの宣言は、デフォルト値と可変性について権威を持つが、
そのパスのすべての子宣言は、それでも同一の論理型を持たなければならない。合成されたスキーマは、エミッター
のコンパイル前に各子へ転送されるため、明示的なエフェクトレベルの宣言は子のuniformとして具現化され、1回の
ランタイム書き込みですべての具現化された子のuniformを更新できる。単体のエミッター内で作成されたジェネレータ
は、それでも、その再利用可能なエミッター上にそのパスを宣言するため、`defineEmitter()` は合成の前にそれを
検証できる。競合は、オブジェクト/プロパティの順序によって要素を選択するのではなく、
`NACHI_EFFECT_PARAMETER_CONFLICT` を報告する。

スポーン時のオーバーライドは、そのインスタンスについて宣言された不変パラメータを設定してよい。それ以降の
`setParameter()` 呼び出しには `mutable: true` が必要である。どちらの経路も、マップされたTypeScriptの型に
加えて、論理的なスカラー/ベクトル/行列の形状をランタイムで検証する。すべてのエフェクトインスタンスは、
定義、コンパイル済みプログラム、プールされたGPUリソースが共有されている場合であっても、独自の値レコードと
uniformノードを所有する。`@nachi-vfx/timeline` のような合成ランタイムは、現在具現化されている子エミッターが
存在しない場合であっても、書き込みを保持する前に、合成されたエフェクトスキーマに対して同じ検証を行わなければ
ならない。アクティブな子への転送は、検証の境界ではない。

### 6.1 予約された `Emitter.*` パス

| パス                                 | 意味                                                                                |

M5は `Emitter.eventPayload.<field>` を、`emitTo(..., { inherit })` によって列挙されたフィールドのためだけに予約する。それ以外のサフィックスは未知のパスであり、すべてのイベントペイロードフィールドはコンパイラが所有する読み取り専用の値である。プロデューサとターゲットの属性は、同一の論理型とコンポーネント数を持って存在しなければならず、そうでなければコンパイルは失敗する。最大4コンポーネントの浮動小数点系フィールド(`f32`、`vec2`、`vec3`、`vec4`、`color`、`quat`)は、初出順にvec4グループへパックされる。フィールドがグループをまたぐことは決してない。整数、真偽値、行列のペイロードは、黙って再解釈されるのではなく、M5では拒否される。`CompiledEmitterMeta.eventQueues` は、各フィールドのグループ/レーン、論理型、キュー容量、ハンドラ、ペイロードグループ数を公開する。

## 7. 値ジェネレータ

ジェネレータは、ステージコンパイラによって評価される不変のJSON値である。それらの概念的なシグネチャは次の通りである。

```ts
function range<T extends number | Vec2 | Vec3 | Vec4>(min: T, max: T): RangeGenerator<T>;

function curve<T extends number | Vec2 | Vec3 | Vec4>(
  first: readonly [time: number, value: T],
  second: readonly [time: number, value: T],
  ...rest: readonly [time: number, value: T][]
): CurveGenerator<T>;

function gradient(first: ColorInput, second: ColorInput, ...rest: ColorInput[]): GradientGenerator;

function parameter<T extends JsonValue>(path: ParameterPath, fallback?: T): ParameterGenerator<T>;
```

`range()` はデフォルトで決定的な一様乱数選択を使用する。その乱数ストリームのスロットは、それを含むステージと、存在する場合はモジュールの `label` とを組み合わせ、存在しない場合はその正規化されたステージインデックスと組み合わせる。Initでは値を `Particles.spawnOrder` とエミッターシードでキー化する。Updateでは `Particles.spawnOrder`、エミッターシード、`Emitter.updateRandomStep` でキー化し、物理スロットと `Particles.spawnGeneration` は一切関与しない。ベクトルのrangeは、1つのスカラー値をすべてのコンポーネントへブロードキャストするのではなく、各コンポーネントを、それぞれ異なる決定的なサンプルオフセットからサンプリングする。将来の分布は、挙動の変更ではなく、明示的な設定フィールドでなければならない。

GPUのマッピングは、32ビットのハッシュに `2^-32` を乗算する。その数学的な範囲は `[0, 1)` であるが、f32
の丸めにより、およそ `3e-8` の確率でちょうど `1.0` が生成され得る。シェーダー側の利用者は、この契約を
閉区間 `[0, 1]` として扱わなければならない。JavaScriptのf64ミラーは `[0, 1)` のままである。

`curve()` のキー時間はドメイン非依存である。`sizeOverLife()` のようなモジュールは正規化された `[0, 1]` ドメインを定義するが、タイムライン/ワールド空間の利用者は別のドメインを定義してよい。少なくとも2つのキーが必要である。M1は線形補間のみをサポートする。省略された補間はデフォルトで線形となり、`constant`、`cubic`、および未知の補間値は明示的なコンパイル診断を生成する。

`gradient(colorA, colorB, ...)` は、少なくとも2つの等間隔なストップを生成する。将来の明示的ストップのオーバーロードは `{ position, color }` を受け付けてもよいが、同じ正規のストップ配列へシリアライズされなければならない。文字列カラーは `#RGB`、`#RRGGBB`、`#RGBA`、`#RRGGBBAA` を受け付ける。RGBチャンネルはsRGBとして解釈され、LUT補間の前にThree.jsの伝達関数によってリニア値へ変換されるが、アルファはリニアの `[0, 1]` 範囲へ直接デコードされる。数値のカラー配列はすでにリニアな値であり、変換されない。

カーブとグラデーションはLUTテクスチャまたはバッファへベイクされる。ベイクの精度、解像度、キャッシュキーは、隠れたモジュールの挙動ではなく、コンパイラのオプションである。

## 8. TSLエスケープハッチ

### 8.1 インライン作成形式

PLAN.md由来の形式は有効である。

```ts
tslModule(({ position, velocity, age }) => ({
  velocity: velocity.add(myCustomField(position)),
}));
```

ファクトリはCPUのパーティクル値ではなく、型付きのTSL式を受け取る。それは属性から新しい式への部分的なマッピングを返す。返却されたキーは書き込みである。アクセスされた入力バインディングは読み取りである。コンパイラは両方をトレースし、最終的なマニフェストを生成する。`tslModule()` はデフォルトで `update` ステージとなる。`stage: 'init'` は初期化を選択する。

インラインファクトリに供給される属性式、および `KernelModuleBuildContext.attribute()` によって供給される属性式は、読み取り専用の論理値である。モジュールは、出力マッピングを返すか、パーティクルストレージを更新するために `KernelModuleBuildContext.write()` を呼び出さなければならない。属性式に対して `assign`、`addAssign`、その他のインプレース変更を呼び出すことは無効であり、具現化はエラーを報告する。このルールは、コンパイラがその属性を専用バッファに配置するかパックされた物理バッファに配置するかとは独立である。

ファクトリはコンパイルに関して純粋でなければならない。フレームに依存するJavaScriptの状態、I/O、乱数呼び出し、GPU送信、シーンの変更、登録されたコンパイラ施設の外でのリソース割り当てを行ってはならない。ファクトリは `@nachi-vfx/tsl-kit` および固定されたThree.jsアダプター層のTSL関数を合成してもよい。

任意の `access` マニフェストが供給されてもよく、意図的に区別された2つの状態を持つ。

- プロパティが存在しない場合: マニフェストは作成時点では未知であり、正規化の過程でトレースされたTSLグラフから完全に導出される。
- プロパティが存在する場合(`{ reads: [], writes: [] }` を含む): 作成者が宣言を供給しており、トレースされた読み書きはその部分集合でなければならない。

正規化はシリアライゼーションの前にトレース結果を具現化する。存在しない宣言を空の宣言で置き換えることは決してない。動的なプロパティアクセスや、トレースできないレジストリ統合には、明示的なアクセスが必要である。

組み込みのバインディング名にはオープンな文字列インデックスシグネチャがないため、`velocty` は型エラーとなる。カスタムバインディングは `custom.<name>` という形式の宣言済みキーを使用する。

```ts
type HeatBindings = TslParticleBindings<{ heat: number }>;

tslModule<HeatBindings>((bindings) => ({
  'custom.heat': bindings['custom.heat'],
}));
```

エミッターの `attributes` スキーマから `HeatBindings` を自動的に導出することは、M2以降へ延期されており、§16に記載されている。それまでは、カスタムバインディングは明示的に供給される。

返却された書き込みキーは、トレース検証の過程で、エミッタースキーマに対してさらに解決される。TypeScriptは、組み込み入力バインディングのスペルミスや、明示的に型付けされた `custom.*` バインディングを検出するが、構造的に推論されるすべてのオブジェクトキーを証明することはできない。したがって、作成時の型推論を通過する書き込みキーのタイプミスは、受け入れられるカスタム属性ではなく、コンパイル時のトレース検証エラーとなる。

バインディング演算の引数は、既存のTSLノードをそのまま受け取り、それ以外の作成用リテラルを演算の具現化前に次のようにloweringする。この対応は、GPUレイアウトに使うものと同じ論理属性型テーブルから導出される。

| 入力形               | TSL論理型 |
| -------------------- | --------- |
| `boolean`            | `bool`    |
| 有限の `number`      | `f32`     |
| 長さ2の有限数値配列  | `vec2`    |
| 長さ3の有限数値配列  | `vec3`    |
| 長さ4の有限数値配列  | `vec4`    |
| 長さ9の有限数値配列  | `mat3`    |
| 長さ16の有限数値配列 | `mat4`    |

たとえば `binding.velocity.add([0, 1, 0])` は、明示的に構築した `vec3` ノードを渡す場合と等価である。文字列、`undefined`、`null`、関数、およびノードは、非リテラルのメタデータを使うTSL演算のためにそのまま渡される。未対応長の有限数値配列、またはノードでも対応リテラルでもないプレーンオブジェクトは、トレースコンパイル中にモジュールpath付きの `NACHI_TSL_BINDING_INPUT_INVALID` で拒否され、GPU送信には到達しない。非有限成分を含む数値配列も拒否される。

既知の制限として、非数値要素を含む混在配列(例: `[1, 'a']`)は、ノード配列メタデータとの
弁別のため素通しされ、GPU送信段で失敗し得る。全数値配列については、形状と有限性の検証を
実施済みである。

### 8.2 シリアライズ可能な登録形式

```ts
const followField = defineTslFunction(
  'game/follow-field',
  ({ position, velocity }) => ({
    velocity: velocity.add(field(position)),
  }),
  1,
);

registry.registerTsl(followField);

const module = tslModule(followField.ref, {
  stage: 'update',
  access: {
    reads: ['Particles.position', 'Particles.velocity'],
    writes: ['Particles.velocity'],
  },
});
```

JSONは `{ kind: 'function-ref', id, version }` のみを格納する。デシリアライゼーションはレジストリを通じてこれを解決する。レジストリIDは安定しており、名前空間化されており、明示的にバージョン管理されていなければならない。欠落しているエントリやバージョン非互換のエントリはコンパイルエラーである。

### 8.3 TSLバージョン境界

パブリックなコールバックは、本ライブラリの `TslExpression<T>` ファサードを受け取る。Three.jsのTSLノードを直接扱うことは、狭く型付けされたアダプターを通じて許容されうるが、生のThree.jsノード型はシリアライズされず、本ライブラリによってsemverの安定性を約束されるものでもない。固定されたThree.jsのTSL APIが変化する際に変更されるべき唯一のパッケージは、このアダプターである。

## 9. エミッターAPI

正規の形は次の通りである。

```ts
interface EmitterConfig<Attributes, Parameters> {
  capacity: number;
  attributes?: Attributes;
  offset?: Vec3;
  parameters?: Parameters;
  spawn: SpawnModule | readonly SpawnModule[];
  init?: readonly InitModule[];
  update?: readonly UpdateModule[];
  events?: Partial<Record<ParticleEventName, EventModule | readonly EventModule[]>>;
  integration?: 'euler' | 'none';
  render: RenderModule | readonly RenderModule[];
  lifecycle?: {
    duration?: number;
    loopCount?: number | 'infinite';
    prewarm?: number;
    startDelay?: number;
  };
}
```

`capacity` は厳格な論理上限であり、正の整数でなければならない。Spawnおよびイベントキューはオーバーフローを明示的にクランプする。GPUバッファを黙って破損させたりリサイズしたりすることは禁止されている。

`offset` は、エフェクトインスタンスフレーム内における当該エミッター要素の原点を平行移動する、有限な
エミッターローカル値である。ランタイムの具現化は、単一の `Emitter.transform` uniformを
`instanceTransform * translate(offset)` として合成する。したがってInit配置、
`space: 'emitter'` を使うすべてのモジュール、エミッターローカルなkill volumeとbounds、sort、
その他のtransform消費点は同じフレームを観測する。timelineの `play()` はエフェクトのpositionと
rotationをそのままcoreへ渡すため、即時または後からplayされるエミッター要素にも同じ合成が成立する。
省略時および厳密なゼロoffsetでは、変更されていないインスタンストランスフォームを維持する。

1つのステージ内の複数のモジュールは、依存関係解析が安全な最適化を証明しない限り、作成者の順序で実行される。1つのイベントに対する複数のハンドラは作成者の順序で実行され、それぞれが正規化された `Emitter.events.<eventName>` キューの依存関係に書き込む。レンダラーモジュールはシミュレーション属性を変更できない。M7は1エミッターあたりちょうど1つのレンダーモジュールをサポートし、追加のレンダーモジュールは `NACHI_RENDER_MODULE_LIMIT` として診断する。複数のレンダーモジュールは描画ごとの間接引数スロットを必要とし、§16に記録されている通りM8以降へ先送りされている。`integration` はデフォルトで `euler` である。M1のコンパイル時正規化は、§4.3の予約済み `$integrate` モジュールを合成し組み込む。`defineEmitter()` における作成時の正規化は、その組み込みを行わない。

M9バッチ1のエミッター継承は `defineEmitter(base, overrides)` を使用する。これはJavaScriptのプロトタイプ
継承を使用せず、返却される定義において `base` のオブジェクトIDを保持することもない。作成時の正規化は、
チェーンを積極的(eager)に通常のエミッターデータへ解決するため、JSON出力は単体の `EmitterDefinition` と
同じ形状を持つ。M12は、この同じ結果へ正規化されるアセット参照形式をエンコードしてもよい。

`capacity`、`integration`、`offset`、`spawn` はchild-wins(子が優先)の値である(`spawn` は1つのポリシーとして
置き換えられ、決して連結されない)。ライフサイクルはフィールドごとのchild-winsである。属性とパラメータは
キーによってマージされるが、継承されたキーは論理型を変更できない。イベントマップはイベント名によって
マージされ、子のハンドラ値は、そのイベントを単位として置き換える。

`init`、`update`、`render` は `EmitterModuleListOverride` を受け付ける。

```ts
const child = defineEmitter(parent, {
  update: {
    mode: 'merge', // default; alternatives are 'append' and 'replace'
    modules: [{ ...gravity(-4), label: 'gravity' }],
    remove: ['drag'],
    order: ['gravity'],
  },
});
```

`merge` は、ラベル付きの子モジュールを同じ親ラベルに一致させ、その場で置き換える。ラベルのない子モジュール
は、正規化された親のインデックスに一致する。一致しない子モジュールは追加される。`remove` はラベルまたは
正規化されたインデックスを受け付ける。`order` は、列挙されたアイデンティティを与えられた順序で先頭へ移動
し、列挙されていないすべてのモジュールは安定した相対順序のままとする。`append` は削除を適用した後、暗黙の
置き換えなしにすべての子モジュールを追加する。`replace` はスタック全体を置き換えるため、`remove` と
`order` を無視する。明示的に `undefined` とされたオーバーライドは、存在しないものとして扱われ、継承された
`capacity`、`spawn`、ライフサイクルフィールド、スキーマエントリ、イベントハンドラを消去しない。未知の
セレクタ、重複/無効な順序セレクタ、ステージの不一致、属性またはパラメータの型不一致、および巡回する生の
作成グラフは、コンパイル診断である。継承の診断は、次の6つの安定した名前を使用する。
`NACHI_EMITTER_INHERITANCE_BASE_TYPE_MISMATCH`、
`NACHI_EMITTER_INHERITANCE_CYCLE`、
`NACHI_EMITTER_INHERITANCE_STAGE_MISMATCH`、
`NACHI_EMITTER_INHERITANCE_TARGET_UNKNOWN`、
`NACHI_EMITTER_INHERITANCE_ATTRIBUTE_TYPE_MISMATCH`、および
`NACHI_EMITTER_INHERITANCE_PARAMETER_TYPE_MISMATCH`。

これはNiagaraのエミッター継承/バージョニングのコードファースト版の等価物である。継承されたスタックの
エントリは、子が識別されたエントリを明示的に上書き/無効化/並べ替えするまで、可視かつ不変のままである。
これは、盲目的な配列の連結(力/レンダラーを二重適用してしまう)と、暗黙の配列全体の置き換え(わずかな
子の編集が親スタック全体をコピーしてしまう)の両方を意図的に回避する。

オーバーライドは、ラベル付きのモジュールをラベルによってアドレス指定する。ラベルのないモジュールは正規化されたステージインデックスによってアドレス指定されるため、編集に対してより不安定である。1つのステージ内での重複するラベルはコンパイルエラーである。

M2のライフサイクル正規化は、最初のアクティベーションの前に `startDelay` を使用し、エミッターを
ループごとに `duration` 秒間実行し、`loopCount` を最初のものを含むアクティベーションの総数として
解釈する(`'infinite'` は発生を決して完了しない)。ライフサイクルオブジェクト自体が省略された場合、
または一部だけが指定された場合のどちらでも、`lifecycle.duration` が未定義なら、`rate` または
`perDistance` を含むエミッターは無限のactive durationを導出する。burst専用エミッターは代わりに、
作成された `cycles * interval` のうち最大の包絡に、静的に証明可能な最大パーティクルlifetimeを
猶予として加えた有限durationを導出する。多サイクルburstがなければ、その導出durationはゼロのままで
ある。明示的な有限の数値durationはすべての作成経路とasset format経路で常に優先され、作成された
`Infinity` は引き続き不正である。`prewarm` は外部から可視になる前に最初のアクティベーションを宣言された
エフェクトローカル秒数だけ進め、`startDelay` は導出されたウィンドウの前に合成される。有限または無限の
`loopCount` の各アクティベーションは、有限の導出包絡全体を繰り返す。duration省略のcontinuous emitter
では最初のアクティベーション自体が無限なので、有限の `loopCount` でも後続activationへ到達しない。
終了は `stop()`、timelineのstop/end、または `release()` が所有する。ループするエミッターには、正の
明示的duration、正の多サイクルburst由来の包絡、またはduration省略のcontinuous spawn policyが必要である。
duration 0の `at(0)` timeline shorthandには後続境界がないため、再生された子が引き続き完了を決める。
正のdurationを持つ最終track boundaryは、その子をtruncateする。

信頼されないアセットは、エミッターの `capacity` が `2^22` を超える場合、PBD距離制約の `iterations` が `64` を超える場合、またはライフサイクルの `prewarm` が `300` 秒を超える場合、割り当てまたは同期的なカーネル展開の前に拒否される。コアのコンパイルとフォーマット検証は同じ制限を強制する。

M2は `burst({ count, cycles?, interval? })`、`rate({ rate })`(エミッターローカル秒あたりのパーティクル数)、`perDistance({ rate })`(`Emitter.transform` によって移動したワールド空間の単位あたりのパーティクル数)を提供する。rateおよびdistanceのモードは端数の余りを保持するため、固定タイムステップは、等価なデルタ分割に対して同じ整数の合計を生成する。Per-distanceは、連続するシミュレーションステップのtransformスナップショットを比較し、その間の直線chordを1回だけ消費する。同じシミュレーションステップより前に `setTransform()` を複数回呼んでも折れ線として蓄積せず、直前にシミュレーションされたtransformから最終current transformまでの純変位だけが距離に寄与する。

H1-7は、通常の各spawnバッチ内で、変換を消費するInit配置を補間する。`i`を物理フリーリスト
スロットおよびalive compaction順から独立した、0始まりのローカルspawn orderとする。`N`粒の
`rate`/`burst`バッチは `phase = (i + 0.5) / N` を使う。直前の端数余りが`r`、スケール済みrateが
`R`、現在の移動線分長が`D`であるper-distanceモジュールでは、粒`i`はアキュムレータの実際の
発火点 `phase = ((1 - r) + i) / (R * D)` を使う。既存のspawn epsilonを覆うため、GPU具現化時
だけ位相を`[0, 1]`へclampする。複数のper-distanceモジュールは、それぞれ独自の余りとバッチを
保持し、時間系モジュールはそのステップで1つのバッチを共有する。

時間系バッチの位相分母 `N` は、要求数、物理capacity、論理可用性に対するCPU側クランプ後の
バッチサイズに固定する。GPU free-list飽和は成功する生成数をさらに減らし得るが、位相は
再分配しない。成功した生成は、そのクランプ済みバッチの最先頭の発火位置を保持する。これは、
最初の利用可能な発火位置を保持し、残りを再分配せず破棄するという、承認済みper-distance飽和
意味論と一致する。

`Emitter.spawnInterpolatedTransform` は、作成者が書き込めるuniformではなく、コンパイラがInitへ
提供する式である。平行移動は `lerp(previous, current, phase)`、回転は最短経路の四元数slerpで
ある。公開transform APIにscaleがないため、補間変換は回転と平行移動のみを対象とする。
`Emitter.previousTransform` は、エミッターローカル`offset`適用後の完全な合成済み変換で
ある。CPUがpreviousとcurrentの行列の厳密な一致を観測した場合、`Emitter.interpolationActive`は
0となり、カーネルは既存の `Emitter.transform` 直読み経路を通る。lerp/slerp式は実行されない。
したがって静止時のbirthは、既存の出力ビットと乱数ストリームを維持する。イベント消費によるInitと
全スロット互換Initは時間/距離位相を持たず、引き続き現在変換を直接読む。

H2-6 module-v2実装はhost uniformではなく別のcompiler提供式である
`Emitter.updateInterpolatedTransform`を追加する。Update段の全emitter-space force、解析collider、
kill volumeは正確なphase `0.5`の1サンプルを共有する。
平行移動はprevious/currentのlerp、回転は最短経路quaternion slerpである。厳密に静止したtransformは
current直読み分岐と従来の出力bitを維持する。NeighborGridのbucket挿入と全visitor lookupは、1回のrebuildで
一貫したgrid frameを持つよう意図的にcurrent終点を使い続ける。このmidpointは時間積分の1サンプル近似であり、
CCD、swept-volume交差、multi-point積分、合成substepではない。薄いvolumeはsample点間で見逃し得る。

H1-7は、ステップ内で `Particles.age` を遡及補正しない。spawn時刻に応じたage補正は将来提案して
よいが、静止エミッターのgolden挙動を暗黙に変えないよう、明示的なオプションでなければならない。

Prewarmの等価性は、同一のシミュレーションステップ列と比較した場合の、決定的なエミッターローカルのSpawn/Init/Update状態をカバーする。これは、スケールされていない `System.*` のクロック値、レンダリング/プレゼンテーション、読み戻しのタイミング、コールバック、外部統合を意図的に除外する。`Particles.lifetime` への書き込みを宣言するモジュールを持たないエミッターは、正確な生存数完了が有効化されていない限り、無制限の保守的なライフタイム(`Infinity`)を持つ。静的に理解される `core/lifetime` 以外のモジュールがlifetimeに書き込む場合も、同じ保守的な `Infinity` ルールが適用される。

宣言されないlifetimeは、`killVolume()` のような明示的なUpdateモジュールが `Particles.alive = false` を書き込むまで生存するパーティクルのためにサポートされる形式でもある。`capacity` への到達が、生存しているスロットを上書きすることは決してない。Spawnは要求をアトミックな空き数と比較し、利用できない呼び出しを破棄し、既存のオーバーフロー診断を通じてそれらを公開する。スロットは、コンパイラが所有する消滅/回収エピローグを通じてのみ利用可能になる。無制限のrateエミッターは、明示的な消滅パスを提供することが望ましい。そうでなければ、capacityは満杯のままとなり、それ以降のスポーンは抑制されたままとなる。

### 9.1 M5/M6イベントとサブエミッター

`onDeath: emitTo(target, { inherit })` はM5の実行可能なイベント契約である。エフェクトのコンパイルは `target` をエミッター要素のキーと照合して解決し、具現化の前に `NACHI_EVENT_TARGET_UNKNOWN` を報告する。各プロデューサのイベント種別は、そのすべてのハンドラによって共有される1つの追加キューを所有する。継承されたフィールドの和集合は、1つのペイロードレイアウトを決定するため、`position` を継承する2つのハンドラが、そのレーンを重複させることはない。

イベントキューは3つのWebGPUリソースを使用する。4ワードのアトミック状態バッファ、二重バンクのvec4ペイロードバッファ、1つの3ワードの間接ディスパッチバッファである。状態ワードは、2つのバンクカウント、累積オーバーフロー数、累積発生イベント数である。ペイロードの割り当ては `2 * producerCapacity * max(1, payloadGroupCount) * 16` バイトを消費する。状態は16バイトを消費し、間接引数はプロデューサのイベント種別ごとに12バイトを消費する。これらのリソースは `CompiledEmitterMeta.storageBuffers` に列挙され、デバイスのストレージバッファ予算に算入される。コンシューマはM2のフリーリストと `dispatchIndirect` パスを使用する。イベント数は別個のスポーンバッチであり、Initは通常通り実行され、その後、継承された属性が宣言された出力を上書きする。

イベントによって消費されるInitは、前のスケジューラフレームから保持された通常のuniform値を使用する。現在フレームのタイミングuniformは、そのエミッターがイベント消費後に進行する際にインストールされる。イベントバンクのuniformは、スケジューラが所有する例外である。エフェクトの完了は、最も深いイベントチェーンのエッジごとに1つのスケジューラフレームを保守的に排出するため、書き込みバンクへの追加は、イベント数の読み戻しなしに消費可能なままである。正確な生存数の読み戻しが有効化されている場合、発生の完了とすべてのイベント消費フレームは、`aliveCountReadbackInterval` にかかわらず最終的なコンパクション/読み戻しを強制する。

アトミックな追加は、プロデューサの容量を超えてバンクカウントをインクリメントすることがあるが、それらのスロットに対してペイロードアドレスが形成されることはない。そのようなレコードは破棄され、累積オーバーフローワードがインクリメントされ、オプトインの読み戻しは `NACHI_EVENT_QUEUE_OVERFLOW` を報告する。コンシューマの容量圧迫は既存の `NACHI_SPAWN_CAPACITY_EXCEEDED` カウンタを使用し、生存しているターゲットのスロットを上書きすることは決してない。キューカウントは、それらの書き込みフレーム内で再帰的に消費されることはない。

複数のプロデューサキューが1つのターゲットへルーティングされる場合、coreは論理tuple
`(target emitter key, source emitter key, event name, handler target, sorted inherited field names)`
の順でlinkを消費する。エフェクト要素の挿入順、object spreadの順序、handler配列の順序は、ターゲットの
容量圧迫時に先に消費されるキューを選ばない。tupleが完全に同一な2つのhandlerは、同じキューレコードを
読み、同じフィールドを継承し、同じemitterをターゲットとするため意味的に区別できず、それらの相対的な
作成順を保持しても結果は変わらない。

このキュー間routing保証は、1つのproducer/eventキュー内のレコードを順序付けない。GPU producerの各
invocationはatomic操作でappend位置を予約するため、勝つappend順は並列hardware raceであり、adapterや
runによって異なり得る。consumer Initと子の`spawnOrder`はそのキューレコード順に従い、queueまたはtarget
の飽和時には同じキュー内の別レコードが勝者になり得る。queue分離は独立producer class間のraceを隔離し、
event target/queueに十分な容量を確保すればoverflow winnerは避けられるが、どちらも1キュー内の並列record
順を決定化しない。emitter seedの分離は乱数streamだけを隔離し、順序の修正ではない。

`onCollision: emitTo(target, { inherit })` はM6において実行可能である。組み込みの解析的コライダーおよび
シーン深度コライダーは、貫通テストが応答を生成するたびに追加を行う。位置補正は追加の前に書き込まれる
ため、`inherit: ['position']` は、`onDeath` で使用されるものと同じvec4ペイロードパスを通じて、解決済み
の表面位置を運ぶ。消費は次のスケジューラフレームのままであり、同じ二重バンクとオーバーフローのルール
によって制限される。ペイロードの `position` と `velocity` は、位置補正と速度応答の後にサンプリングされる。
`mode: 'kill'` の衝突は `onCollision` を追加した後、`alive = false` を書き込むため、両方のハンドラが
存在する場合、コンパイラの消滅エピローグは `onDeath` も追加する。kill以外の応答は `Particles.alive` に
書き込まないため、以前の作成者モジュールによって消滅させられたパーティクルを蘇生させることはできない。
静止接触は厳密な貫通を使用する。重力/積分は、補正されたパーティクルを後のフレームで内部へ戻すことが
あり、貫通の応答が新しく発生するたびに別の `onCollision` を発する。連続的な接触は、M6 v1では、
begin-contactのみのイベントではなく、意図的にイベントストリームである。`onSpawn` は将来のコンパイラが
所有するライフサイクルプロデューサのために予約されており、`NACHI_EVENT_KIND_UNIMPLEMENTED` を生成する。
計画されているカスタムプロデューサの表面は、`tslModule` のコンテキストメソッド
`context.emitEvent(condition)` であり、Updateからのみ有効であり、`onCustom` への条件付き追加へと下位
変換される。条件はスカラーの真偽値TSL式でなければならず、継承されるフィールドは、それでもハンドラの
`emitTo` 宣言から得られる。そのコンテキストが具現化されるまで、`onCustom` を作成すると
`NACHI_EVENT_ON_CUSTOM_UNIMPLEMENTED` を生成する。

### 9.2 M3スプライトおよびメッシュレンダラー

`billboard()` は、シミュレーション属性を変更することなく、パーティクルの色、位置、サイズ、任意の向き
入力を消費する。`flipbook(texture, { cols, rows })` はフレーム0を左上のアトラスセルへ割り当て、左から
右、続いて上から下へ進む。したがって、`flipY` を使用するThree.jsのテクスチャは、アトラス行を
`rows - 1 - floor(frame / cols)` として解決する。`flipY: false` のテクスチャは、`floor(frame / cols)`
を直接使用する。フレーム補間はデフォルトで有効である。エミッターにlifetimeの書き込み元がない場合、
`normalizedAge` はデフォルトのゼロのままとなり、flipbookはフレーム0のままとなる。

モーションベクトル補間には、補間とモーションベクトルの `TextureRef` の両方が必要である。リソースなしでモーションベクトルを要求すると、警告付きで単純な補間へフォールバックする。`interpolate: false` の状態でそれらを要求すると、警告付きで無視される。UVワープはソースのアトラスセルにクランプされるため、フィルタリングが隣接フレームへ漏れ出すことはない。

`cutout.vertices` は4から8の頂点を受け付ける。4頂点は完全な四角形を保持する。5から8の頂点は、四角形に含まれる凸多角形を形成し、テクスチャのコーナーを切り取ることでオーバードローを削減する。三角形分割は `(vertices - 2) * 3` 個のインデックスを使用し、UVは元の四角形の `[0, 1]` 範囲を保持する。

4つのブレンディングモードは、固定されたM3セマンティクスを持つ。`alpha` はソースアルファブレンディングを使用する。`premultiplied` は事前乗算アルファ係数を用いた通常のブレンディングを使用する。`additive` はソースカラーを加算する。`multiply` はデスティネーションにソースカラーを乗算する。ビルボードは、深度テストを有効にし深度書き込みを無効にした透明レンダーパスを常に使用する。メッシュは深度書き込みを有効に保つ。`alpha` と `premultiplied` は透明レンダーパスを選択し、一方 `additive` と `multiply` はオペークなレンダーリストのパスのままである。

ビルボードのアライメントは、`camera-facing`(デフォルト)、`custom-axis`、`velocity-aligned`、`velocity-stretch` のいずれかである。stretch形式は、任意で `factor` によってその速度の寄与をスケールする。メッシュのアライメントは、`none`(デフォルト)、`custom-axis`、`velocity`、`quaternion` のいずれかである。velocityおよびcustom-axisのメッシュモードは、ジオメトリのローカル+Y軸を選択された方向へ回転させ、quaternionモードは `Particles.rotation` を直接消費する。

`orientToVelocity()` は、ビルボード互換の `Particles.spriteRotation` と、ローカル+Yを
`Particles.velocity` へ向けて回転させるクォータニオンの両方を書き込むUpdateモジュールである。その
クォータニオン経路は、Three r185の転置されたTSL `rotate()` のオイラー規約ではなく、直接的な
クォータニオン-ベクトル演算を使用する。`spriteRotation` は、ワールドXY平面に投影された速度角であるが、
レンダラー側のvelocity-alignedビルボードは、その角度をビュー空間で導出する。速度がゼロの場合、
以前の向きが保持される。これはcamera-facingビルボード、または
`meshRenderer({ alignment: { mode: 'quaternion' } })` とともに使用する。レンダラー側の速度アライメント
と組み合わせると、同じ意図が二重に適用されてしまうため、避けることが望ましい。`faceCamera(options)` は
`billboard({ ...options, alignment: { mode: 'camera-facing' } })` の明示的なレンダーステージでの表記で
あり、決定的なUpdateシミュレーションへカメラの状態を持ち込むことはない。

`rotationOverLife()` は `f32` のカーブを受け付け、等方的なスカラー `Particles.spriteRotation` をラジアンで書き込む。軸ごとのオイラー角およびクォータニオンのカーブには、将来の型付き向きモジュールが必要である。`velocityOverLife()` も同様に `f32` のカーブのみを受け付け、すべての速度成分に1つの絶対的な等方スケールを適用する。各Updateは `lut(t) / lut(tPrevious)` を適用する。ここで `tPrevious` は `Emitter.deltaTime` とパーティクルのlifetimeから再構築される。したがって、等価な経過時間の分割は、絶対的なLUT値を繰り返し複合するのではなく、同じスケールに到達する。

`billboard({ soft: true })` は較正されたデフォルトのフェード距離 `0.035` を使用する。`billboard({ soft: { fadeDistance } })` はそれを上書きする。この値はThree.jsのリニアライズされた正規化カメラ深度単位で測定され、正かつ有限でなければならない。このデフォルト値は、受理された `spike-depth` の部分交差較正に由来し、ワールド空間の距離ではない。

`meshRenderer({ geometry, alignment })` は、シリアライズ可能な `GeometryRef` を、具現化時に任意のThree.jsの `BufferGeometry` へ解決する。これは、パーティクルごとの位置、スケール、色を、任意のvelocity、custom-axis、またはquaternionの向きとともに消費する。ビルボードと同様に、メッシュインスタンスはコンパクトな生存インデックス間接指定と生存数間接描画引数を使用する。

### 9.3 M7リボン/トレイルレンダラー

`@nachi-vfx/trails` は、`ribbon()`、`ribbonId()`、そのレンダーコンパイラ登録、Three.jsのWebGPU具現化器を
所有する。アプリケーションは、これらの実装を `registerTrails(createCoreKernelModuleRegistry())` によって
既存のコアレジストリへ追加し、そのレジストリを `VfxSystemOptions.registry` または
`compileEmitter({ registry })` を通じて供給する。コアは、汎用的なレンダー登録コンテキストとライフサイクル
順序付けリソースのみを提供し、trailsの作成用ヘルパーを再エクスポートすることはない。これは、§13で述べた
モジュール/レンダーレジストリ境界の最初の外部パッケージによる証明である。

`ribbon({ width, maxRibbons, uv, taper, blending, map })` は `Particles.position`、`color`、
`ribbonId`、およびコンパイラが所有する `spawnOrder` を読み取る。`ribbonId()` はカスタムの `u32` 属性
`ribbonId` を、定数または交互するmodulo列で初期化する。`maxRibbons` は明示的なスキャン上限(v1では1から64)であり、動的にリサイズされるGPU割り当てではない。`uv.mode: 'stretched'` は各ストリップの順序付け
られた点の序数を `[0, 1]` にマップする。`uv.mode: 'tiled'` はワールド空間のセグメント長を累積し、正の
`tileLength` で割る。テーパー分率は、同じリボンごとのパラメトリック座標に対して評価される。端点の幅は、
対応するテーパー分率が非ゼロの場合にちょうどゼロとなり、一方テーパーのないボディ部分は作成された
`width` を使用する。フラグメントの色は依然として `Particles.color` から読み取られるため、
`colorOverLife()` は直交したままであり、加算、アルファ、乗算、事前乗算アルファのブレンディングは
§9.2の意味を保持する。

リボンの順序は、コンパクション済みの `aliveIndices` を消費してはならない。各スポーンディスパッチの前に、
1回の呼び出しからなるGPU準備カーネルが、`min(requested, freeCount)` の連続した `spawnOrder` 範囲を
アトミックに予約する。スケジューラは準備、スポーン、確定を厳密に順序通りに送信するため、予約と割り当て
の間で `freeCount` が変化することはない。したがって、失敗した飽和スポーン試行は、順序の値もbirth-ringの
スロットも消費しない。成功した呼び出し `i` は、フリーリストの物理スロット選択とは独立に、`base + i` を
受け取る。スポーンは、`spawnOrder % capacity` において、容量サイズのbirth ringへ物理インデックスを
書き込む。レンダリングの前に、`@nachi-vfx/trails` は、各 `ribbonId` についてリングを時系列順にスキャンし、
パーティクルに格納された `spawnOrder` を比較することで古いエントリを拒否し、消滅したパーティクルを
スキップし、隣接する生存ペアをコンパクトなセグメント/UV/幅バッファとレンダラーが所有する間接描画レコード
へ発行する、1回のGPU呼び出しを実行する。したがって、散在する消滅と非決定的な生存コンパクションが、
ストリップを並べ替えたり交差接続したりすることはなく、内部の制御点を消滅させると、その生存している
前者と後者が再接続される。通常の動作はCPUの読み戻しを一切行わない。公開されている読み戻しヘルパーは、
診断およびスモークテスト専用である。

M7バッチ1のリボンには、WebGPUのストレージバッファと間接描画が必要である。WebGL2経路での具現化は、
`NACHI_RIBBON_WEBGL2_UNSUPPORTED` によって明示的に失敗する。黙ったスプライト/ラインへのフォールバック
は存在しない。v1の準備コストは、意図的にシリアルなGPUパスにおいて `O(maxRibbons * capacity)` であり、
順序付けはラップする `u32` の値を使用する。`NACHI_SPAWN_ORDER_WRAP_RISK` は、`2^31` の安全閾値において
一度だけ警告する。カウンタの読み戻しがない場合、通常スポーンの要求累積は保守的な上限である。有効化された
ライフサイクルの読み戻しは、イベントによる誕生を含む実際のGPUカウンタも検査する。32ビットの順序が
ラップする前に、エミッターは再起動されなければならず、M11のシミュレーションキャッシュ作業は、この
カウンタを保持または再基準化しなければならない。

birth ringは、最新の `capacity` 個の成功した誕生のみを保持する。もし1つの異常に長寿命な制御点が生存
している間に、少なくとも `capacity` 個の以降の成功した誕生が、より短命なスロットを次々に消費すると、
そのリングエントリは上書きされ、その点は生存しているにもかかわらずストリップから脱落する。M7バッチ1
には、この状況に対する読み戻しなしのランタイム診断が存在しない。作成者は、必要なすべての制御点が保持
期間内に留まるように、capacity/lifetime/rateのサイズを決めなければならない。

リボンのスムージング、カーブのテッセレーション、width-over-life/パラメトリックな幅カーブ(固定ボディ幅
と端点のテーパーのみ)、制御点ごとのねじれ(twist)、camera-facing幅を超える向きモード、Niagaraのリンク
順序ポリシーのバリアントは、まだ実装されていない。各セグメントもまた、そのサイド・ベクトルを独立に
計算する。幅の広いリボンで鋭いカーブがある場合、これは亀裂を露呈させる可能性があるが、一方Niagaraは点
の接線を隣接セグメント間で平均化する。M7は現在の1エミッターあたり1レンダーモジュールという制限を維持
する。描画ごとの間接スロットと複数のレンダーモジュールは、§16に記録されている通りM8以降へ先送りされて
いる。

縮小されたWebGL2のburstパスは、`spawnOrder` を、そのburst内での呼び出し番号としてのみ割り当てる。これは永続的なグローバルの誕生順序ではない。順序に依存するレンダラーはWebGL2を拒否するため、この制限された意味が黙ってリボンを駆動することはない。

### 9.4 M7ライトおよびデカールレンダラー

`lightRenderer({ maxLights = 8, priority = 'intensity', radiusScale = 1 })` は、上限のある
`THREE.PointLight` プールを使用する。単一呼び出しのWebGPUカーネルは、物理パーティクルストレージを
スキャンし、消滅したものまたは非正の候補を無視し、`Particles.intensity`(または半径を乗じた強度)に
よって上位 `maxLights` 個のエントリのみを保持する。これは、`N × { position, color, intensity, radius,
physicalIndex, spawnOrder }` とカウントを含む1つの固定された小さなバッファを書き込む。ランタイムはこの選択出力を
ダブルバッファリングし、前フレームのバッファに対してちょうど1回の非同期読み戻しを行う。すべての
パーティクル属性を読み取ることは決してない。したがってプールの更新には1フレームのレイテンシがある。
`maxLights` を超える候補数は、一度だけ `NACHI_LIGHT_LIMIT_EXCEEDED` を発し、破棄されたライトは非
アクティブのままとなる。優先度が等しい候補は、unsigned `Particles.spawnOrder` の昇順、すなわち早い
論理birthを勝者として解決する。GPU replacementとCPU readback orderingは同じ
`(priority descending, spawnOrder ascending)` tupleを使い、physical indexは診断metadataとしてのみ残り、
`spawnOrder`より前に勝者を選ばない。light moduleを使う場合にだけ`spawnOrder`を解決済みparticle schemaへ
追加し、非light経路のlayout/budgetを保つ。u32 birth keyがwrapした後、同時に生存する2粒子が同じkeyを持つ
可能性があり、その完全tieのidentityは保証されない。既存のhalf-range警告へ到達し得るapplicationはwrap前に
emitterを再起動しなければならない。`colorOverLife()` は `Particles.color` に書き込み、`lightIntensity()` と
`intensityOverLife()` は `Particles.intensity` に書き込む。レンダラーは両方を読み取るのみであり、
モジュールの直交性を保つ。

プールされたすべてのPointLightは `visible = true` のままである。非アクティブなスロットは
`intensity = 0` を使用する。Three r185は、可視な解析的ライトIDのリストを `LightsNode.customCacheKey`
へハッシュ化するため、フラッシュ中の可視性の切り替えは、シェーダーのバリアントとパイプラインコンパイル
のヒッチを生成してしまう。`dispose()` は冪等であり、すべての強度をゼロにし、プールグループをその
シーンから削除する。レンダラー統合は、エフェクトが完了、エラー、停止、または解放される際にこれを呼び
出さなければならない。`update(renderer, effectState)` は、非アクティブな状態が供給された際にその遷移
を実行する。

Three r185のWebGPUは、固定された可搬な `maxLights` 定数を公開するのではなく、シーンライトごとに1つの
解析的ライトノードを構築する。したがって `/golden-slash/` はオフスクリーンのレンダリング/読み戻しに
よって8、16、32個のPointLightを調査し、各カウントについてウォームアップされたGPUレンダー時間を
`pointLightProbe.gpuRenderMs` として公開する。ランタイムのデフォルトは、シェーダーの成長とライティング
コストを制限するために8のままである。アダプター固有のタイミング値は、可搬なAPI定数ではなく、ベースライン
データである。選択された設計はPointLightプールである。なぜなら、これはポストプロセッシングを必要とせず
に、M7において通常のライティングされたシーンマテリアルへ影響するためである。エミッシブスプライトと
ブルームは、パーティクルライトのセマンティックなフォールバックではなく、M10へ先送りされた補完的な見た目
の作業である。

`decalRenderer({ sizeScale = 1, fadeOverLife = true, map, blending })` は、コンパクション済みの生存
パーティクルごとに、インスタンス化された投影ボックスを描画する。位置、クォータニオンの向き、スカラーの
サイズ、リニアな色、正規化ageは、パーティクル属性から得られる。フラグメントステージは、M6の前フレームの
リニアfloat深度コピーをサンプリングし、`screenUV` をクリップx/yへ変換し、サンプリングされたWebGPUの
NDC zを `[0, 1]` のまま直接使用し、逆投影の前に、左上原点の `screenUV.v` を
`ndc.y = 1 - 2 * screenUV.y` で変換する。その後、カメラのワールド行列を使用してワールド位置を再構築
する。この点は、パーティクルの向き付けられたボックスへ変換される。ボックスの外にあるフラグメントは
透明であり、内側のx/yはデカールのUVを提供する。デフォルトのlifetimeフェードは、アルファに
`1 - normalizedAge` を乗算し、`colorOverLife()` は独立した入力のままである。

M7のデカールはWebGPU専用であり、シーン深度衝突と同じ非MSAA、非sRGB、リニアfloatの深度コピーと、一致
するカメラ/ビューポートを必要とする。`NACHI_DECAL_WEBGL2_UNSUPPORTED`、
`NACHI_DECAL_DEPTH_MSAA_UNSUPPORTED`、`NACHI_DECAL_SCENE_DEPTH_UNAVAILABLE` は明示的な診断である。
キャプチャは単一カメラであり1フレーム遅延するため、動く受け手は1フレームだけ遅れることがある。M7は、
受け手の法線や背面によって投影を拒否することはなく、リバースzはデカールの具現化器においてまだ保護
されていない。統合は、文書化されたforward-zキャプチャを使用しなければならない。CPUの `DecalGeometry`
は使用されない。なぜなら、パーティクル駆動のインスタンスのためにクリップされたCPUジオメトリを再構築
することは、GPUシミュレーションと有界なスケジューリングを損なってしまうためである。

### 9.5 M10透明度の順序付け

alpha/premultiplied描画には独立した2段階の順序がある。粒子単位では、module-v2の`billboard()`、
`meshRenderer()`、`decalRenderer()`は`sorted`を既定で`true`にする。明示`false`はM2のcompact
alive-index配列を選び、death/reuseをまたぐsemantic orderを意図的に持たない。additive/multiplyの
billboard/meshは既定`false`。module-v1 dataは旧意味を保存し、billboard/mesh省略はfalse、decalは常に
particle sortしない。`sorted:true`はalphaまたはpremultipliedだけで有効である。

sorted drawはsimulation/event storageを変更せず、別個のfloat-depth/uint-index bufferを割り当てる。
compactionごとにalive粒子を2冪の範囲へ展開し、最遠camera-z sentinelをsorted prefixに置く。vertex fetchは
`paddedCapacity - aliveCount`から始まるためsentinelは描画されない。`System.viewMatrix`からcamera-space zを
書き、各bitonic `(k,j)` stageを別々のordered compute submissionとして送る。camera-z昇順はThreeの
forward-camera規約でback-to-frontとなる。同depthはphysical indexで比較し、固定inputにはbyte deterministic
だが永続particle identityではない。sortはcompaction後の`VFXSystem.update()`で再構築し、`setCamera()`は
camera stateだけを変更する。

ソートされた間接指定は、レンダラーが所有する描画データであり、新しいシミュレーション順序ではない。
イベント、コンパクション、将来のシミュレーションキャッシュは、引き続き物理/生存ストレージを観測する。
容量は65,536に制限され、次の `2^n` へパディングされる。ディスパッチ数は `log2(P) * (log2(P) + 1) / 2`
であり、各ディスパッチは `P / 2` 組を比較する。エミッターのコンパクション後、bitonicステージごとに
`O(P log^2 P)` 回の比較と1回の順序立てられたバックエンド送信が発生する。WebGL2は
`NACHI_PARTICLE_SORT_WEBGL2_UNSUPPORTED` を報告する。隠されたCPUパーティクルソートのフォールバック
は存在しない。

draw単位では、`VFXSystem`が参加する各alpha/premultiplied compiled drawを、transform済みemitter-local
`sortCenter`（既定はemitter origin）でrankする。rankはfarからnearへ増加し、同depthはsystem内numeric
instance creation sequence、emitter key、draw path/indexの順で決める。v2 billboard/mesh/decalとlegacy
billboard/meshが参加し、v1 decal、ribbon、additive、multiplyは参加しない。runtimeは`(drawIndex, rank)`
assignmentを送り、late materializationや複数registrationをkernel-wide scalarへ潰さない。

Threeはhost orderを厳密に`bucket = base + renderOrderOffset`、automatic drawは
`renderOrder = bucket + (rank + 1) / 2^20`として合成する。sprite/mesh materializerの`base`既定は1000、
decalは10を保持する。`base`、v2 moduleの`renderOrderOffset`、その和はsigned 32-bit integerであり、1 systemの
automatic entry上限は`2^20 - 1`である。従って`bucket`の外部objectは同bucket内の全automatic drawより前、
`bucket + 1`は後に描画される。materialized sprite/mesh/decalはpersistentなhost変更用の
`setRenderOrderBase()`を公開する。`Object3D.renderOrder`への直接代入は次のrank、activation、pool transitionで
上書きされ得る。pool返却は前generation assignmentをclearする。v2 transparent mesh materialはdepth testを
維持しつつ`depthWrite=false`、v1 meshは`true`を保持する。

activeなparticle-sorted draw、または複数automatic transparent drawがcamera未設定なら、影響instanceごとに
`NACHI_ALPHA_SORT_CAMERA_UNSET`を1回記録する。renderer version、exact-number、asset migration、decal spawn
orientation、poolingの契約は[RFC 006](./006-transparent-draw-order.ja.md)を参照。

Weighted OIT(重み付きOIT)は第三の近似的なパスであり、通常、それを経由するオブジェクトについて、
パーティクルソートとエミッターソートの両方を置き換える。`sorted: true` は作成用定義上に残ることが
許されるが、WBOIT内部では品質上の利点を何ら提供しない。統合はコンピュートコストを避けるためにこれを
無効化することが望ましい。通常のソートされたアルファとWBOITのオブジェクトは、明示的な合成境界を持つ
別個のレンダーグループとしてのみ共存できる。

### 9.6 M10ライティングされたビルボードパーティクル

`billboard({ lit: true })` は、スプライトをThree r185の標準的な物理ライティングパスへ組み込む。`lit`
は代わりに `{ normalMap, roughness, metalness }` であってもよい。roughnessはデフォルトで `0.8`、
metalnessは `0` であり、両方とも有限の単位値である。通常の `map` またはflipbookは、ベースカラーと
アルファのままである。接空間の `normalMap` は独立した `TextureRef` であり、`NoColorSpace` のテクスチャ
へ解決されなければならない。これを省略すると、フラットなカメラ向き法線が使用される。

Threeの具現化器は `MeshStandardNodeMaterial` を構築し、その `setupPositionView()` メソッドのみを、
固定されたr185の `SpriteNodeMaterial` の実装で置き換える。これにより `MeshStandardNodeMaterial.setupVariants()`、
`setupLightingModel()`、環境処理、レンダラーが提供する `builder.lightsNode` はそのまま保たれる。
したがって、M7のPointLightプールを含む通常のThreeの解析的ライトは、専用のライトリストや読み戻しなしに、
パーティクルへ影響する。カスタムの `lightingContext` は、Threeの物理マテリアルのセットアップを重複させ、
固定された実装が変化するにつれて黙って乖離してしまうため、却下された。unlitなビルボードは引き続き
`SpriteNodeMaterial` をインスタンス化する。M10を有効化しても、それらのシェーダーグラフやシーンライト
への応答は変化しない。

スプライトジオメトリは、カメラ向きのビューXY平面に直接作成される。`r` を、アライメントとパーティクル
ごとの回転の後にスプライトの頂点パスで使用される正確な回転ノードとする。ビュー空間における法線マップ
の基底は次の通りである。

```text
T = ( cos(r), sin(r), 0 )
B = (-sin(r), cos(r), 0 )
N = (      0,      0, 1 )
n_view = normalize(T*n_tangent.x + B*n_tangent.y + N*n_tangent.z)
```

フラグメントの結果は `normalNode` へ直接割り当てられる。これは意図的なものである。Three r185の
`normalNode` の契約はすでにビュー空間であるため、モデル法線変換を再度適用すると二重変換になってしまう。
`lit` を `velocity-stretch` と組み合わせても、M10では法線基底に対する逆スケール補正は適用されないため、
非一様なストレッチは小さなライティング近似の残差を残すことがある。`/m10-lit/` は、M7のPointLightを
非対称な左右の位置の間で切り替え、フラットにライティングされたペアの測定された明るい側が逆転すること
を要求し、非対称なノーマルマップ付きスプライトをフラットライティングの出力と比較し、不変制御として、
同じunlitな定義を両方のライト位置の下でレンダリングする。

## 10. エフェクト、タイムライン、ランタイムAPI

### 10.1 エフェクトの合成

```ts
const effect = defineEffect({
  elements: {
    arc,
    sparks,
    flash,
    shockwave,
  },
  timeline: [
    at(0.0, play('flash')),
    at(0.05, play('arc'), cameraShake({ strength: 0.3 }), hitStop(40)),
    at(0.08, play('sparks')),
    at(0.1, play('shockwave')),
  ],
});
```

型付きのパブリックパラメータは同じ定義内で宣言される。

```ts
const chargedEffect = defineEffect({
  elements: { flash },
  parameters: {
    'User.intensity': defineParameter('User.intensity', {
      type: 'f32',
      default: 1,
      mutable: true,
    }),
  },
});
```

`defineEffect()` は要素キーを推論する。`play()` と `stop()` はリテラルなターゲット名を保持するため、TypeScriptは未知のターゲットを拒否できる。コアの `defineEffect()` は合成および互換データのファクトリである。タイムラインのターゲットと値のランタイム検証は、定義がコアのタイムラインファクトリで構築された場合を含め、`@nachi-vfx/timeline` に属する。タイムラインエントリは、タイムラインパッケージの検証の過程で時間によってソートされる。作成者の順序が同点を決する。

M9バッチ2は、統合の表面を `@nachi-vfx/timeline` へ移動する。そのパッケージは、タイムラインのデータファクトリを再エクスポートし、独自の `defineEffect()` アダプターと `VFXSystem` の合成ランタイムを供給する。ノーススターの配列形式は引き続き有効である。トラック全体の制御は、同等にシリアライズ可能なラッパーを使用する。

```ts
timeline: timeline([at(0, play('flash')), at(0.05, play('arc'))], {
  duration: 0.5,
  loop: 3,
  speed: 1.25,
});
```

`loop: true` は無限を意味する。正の整数は正確なサイクル数である。ループには正の `duration` が必要であり、durationは最後のエントリより前になることはできない。配列の省略形は、速度1での1サイクルである。各ループ境界において、再生中のすべての要素は即座に停止し、その後、次のサイクルの時刻ゼロのアクションが作成者の順序で実行される。トラックより長生きするライフサイクルを持つ子エミッターは意図的に切り詰められ、一方より短い子は、別の `play()` があるまで停止したままとなる。各 `play()` は新しい子ライフサイクルを作成し、コアのプールされた具現化を再利用することがある。子の `loopCount` がシーケンサのサイクルをまたいで漏れることは決してない。

タイムラインランタイムは、スケジューリングの前に、コアが作成した定義を防御的に再検証する。エントリの時刻は非負かつ有限でなければならず、速度は正かつ有限でなければならない。違反はランタイム診断を生成し、スポーンされたタイムラインインスタンスを `error` として返す。それらはクランプされない。これにより、作成経路をまたいで単一の決定的な時間の意味が保持され、破損した定義がその振り付けを黙って変化させるのではなく、露呈される。

### 10.2 タイムラインアクション

```ts
function at(timeSeconds: number, ...actions: TimelineAction[]): TimelineEntry;
function play<Target extends string>(target: Target): PlayAction<Target>;
function stop<Target extends string>(target: Target): StopAction<Target>;
function cameraShake(options: {
  strength: number;
  duration?: number;
  frequency?: number;
}): CameraShakeAction;
function hitStop(durationMs: number, timeScale?: number): HitStopAction;
function marker(name: string, payload?: JsonValue): MarkerAction;
```

タイムラインの時間はエフェクトローカルな時間である。`hitStop()` は、エフェクトローカルなクロックと、そのエフェクトに明示的にサブスクライブしている統合に影響する。無関係なブラウザやゲームのクロックを凍結させてはならない。`cameraShake()` は、想定されたグローバルカメラを変更するのではなく、登録されたカメラアダプターによって消費される統合コマンドを生成する。

`applyHitStop()` は置換セマンティクスを使用する。再度これを適用すると、積み重ねたり延長したり最大値を
取ったりするのではなく、残りのdurationとtime scaleの両方が新しい呼び出しで置き換えられる。現在の置換は、
すべてのアクティブな子へ転送される。それ以降に再生される子は、その時点で残っている置換を受け取る。

`TimelineEffectInstance` は、次の追加的なcompanion契約も公開する。

```ts
bindCompanion(instance: VfxEffectInstance): void;
unbindCompanion(instance: VfxEffectInstance): void;
```

bind時には、タイムラインの実効time scale(instance scaleとtimeline speedの積)と、残っている
hit-stop置換を直ちにコピーする。以降の `setTimeScale()` 呼び出しとhit-stop適用は、同一フレームの
同期処理として転送される。別々に更新されるcompanion systemは、同じworld step列を消費しなければ
ならない。停止から再開までローカル時刻が厳密に一致するのは、hit-stop action境界がcompanionのupdate
境界でもある場合に限られる。companion systemを先に進めることで停止が丸1フレーム遅れることは防げるが、
非整列のcompanion updateをsub-segment化するものではない。その場合はtimeline側だけがaction位置で分割し、
clock差はcompanion update interval未満になり得る。socket追従統合は
[RFC 005 §3.1](./005-effect-local-clock.ja.md#31-bindcompanionの位相限界とsocket駆動)の消費済みpose
latchパターンを使わなければならない。

bindingは排他的なclock所有権ではなく命令的な転送である。bind時の初期同期はcompanionの現在のtime scaleと
hit stop残余を上書きする。以後のcompanion直接操作とtimeline転送は、各controlについてlast-writer-winsと
なり、unbindは最後の値を残す。bindingは弱参照であり、release済みcompanionは自動的に除去され、timeline
から強参照され続けてはならない。初期同期、残余同期、hit-stop、time-scaleの全転送経路は、状態が `error`
または `released` のcompanionをスキップしなければならない。利用不能なinstanceをbindした場合、または
bind済みcompanionが `error` へ入った場合は、操作を転送せずwarning診断
`NACHI_TIMELINE_COMPANION_UNAVAILABLE` を追加する。

M9は、カメラアダプターを、線形減衰を伴う絶対的な加算的平行移動/回転サンプル、エフェクトID、サイクル、
エフェクトローカルなサンプル時刻を受け取るコールバックとして定義する。これにはThreeのカメラ型は存在
しない。統合はこれを任意のカメラリグへ追加してよく、エフェクトIDによってサンプルを合成してよい。終了は
常にゼロサンプルを発する。ノイズは、エフェクトシード、シェイクシーケンス、軸、周波数格子インデックスに
加え、滑らかな補間を伴う、コアのPCGハッシュを使用する。したがって、等しいシードと等しいワールドステップ
列は、バイト単位で等しいサンプルを生成する。シェイクのdurationはエフェクトローカルな時間で進行するため、
ヒットストップは、その減衰を消費することなく現在のシェイクを保持する。

`marker()` は純粋なデータである。`TimelineEffectInstance.onAction()` は、クロック後・次の送信前の同期
境界において、すべてのアクションを観測する。`onMarker(name, callback)` はそのフィルタリングされた
ゲームプレイ用の形式である。エントリは時間によって安定ソートされ、同点は作成者の順序を保持し、1つの
エントリ内のアクションは引数の順序を保持する。エミッターの `play()` アクションについて、そのイベントの
任意の `emitter` フィールドは、そのアクションによって作成された `VfxEmitterRuntimeView` である。
レンダラーオブジェクトを具現化する統合は、タイムライン定義を単一の巨大なコアエフェクトであると仮定する
のではなく、そのフィールドをキャプチャしなければならない。再生された各エミッターは、独立してスケジュール
される単一要素のサブ定義であり、後で `stop()`、リプレイ、または完了によって置き換えられたり解放されたり
し得る。mesh-fxおよびplay以外のアクションは、このフィールドを未定義のままとする。`on('death', ...)` は、
以下の集約GPUイベントパスのままであり、現在再生中の各子エミッターへ転送される。`onAction`/`onMarker`
コールバックの例外は `NACHI_TIMELINE_ACTION_CALLBACK_FAILED` へ変換され、そのタイムラインインスタンス
のみを `error` へ遷移させ、吸収される。これにより、システムは呼び出し元が供給した完全なワールドデルタ
を消費し、無関係なインスタンスの更新を継続する。

タイムライン/イベントモジュールからのコールバックは、登録されたコールバック参照を使用する。それらは文書化された同期ポイントへキューイングされる。任意のJavaScriptがGPUディスパッチの内部から呼び出されることは決してない。`EffectInstance.on('death', callback)` は意図的に集約専用である。コールバックは、直前のイベントカウンタ読み戻し以降に累積された `{ event: 'death', count }` を受け取り、パーティクルのペイロードやパーティクルごとに1回のコールバックを受け取ることは決してない。イベントの合計とオーバーフローの合計は、通常、権威ある生存数と同じ `aliveCountReadbackInterval` の周期を使用する。発生の完了とイベント消費フレームは、最後の集約が完了前にフラッシュされるように最終的な読み戻しを強制する。そのオプションまたはレンダラーの読み戻しサポートが存在しない場合、GPUイベントは引き続きサブエミッターを駆動するが、JavaScriptのイベント通知やオーバーフロー診断は約束されない。これにより、無条件のGPU/CPU同期ポイントを回避する。

### 10.3 ランタイムインスタンス

`VFXSystem.spawn()` は不変のエフェクトデータを受け取り、`EffectInstance` を返す。位置、回転、シード、ローカルタイムスケール、`User.*` の上書きは、インスタンスに属する。3成分の回転はラジアンでXYZ順のオイラー角である。4成分の回転はクォータニオンである。スポーン時の `parameters` および `instance.setParameter()` は、定義から導出されたマップされた型であるため、未知のパスや誤った値の型はTypeScriptエラーとなり、型のない入力に対してはランタイム検証エラーのままである。`setTransform()` は `Emitter.transform` を更新し、per-distanceスポーンで使用されるデルタを供給する。`attachTo(source)` は、初期化前とスケジュールされた各システムステップにおいて、外部のトランスフォームプロバイダからワールド位置/クォータニオンを読み取る。`detach()` は最後のトランスフォームを保持し、ポーリングを停止する。`createThreeTransformSource(Object3D | Bone)` のようなバックエンド糖衣構文は、エンジンオブジェクトをコアへインポートすることなく適合させる。`release()` は、安全な場合、プールされたリソースを返却する。M2では、`stop()` は即座のスケジューリング停止を意味する。すなわち、パーティクルを進めたり排出したりせず、`release()` までリソースを保持する。将来の明示的なポリシーは、このデフォルトを黙って変更することなく、優雅な排出(graceful drain)を追加する可能性がある。

各正のシミュレーションステップの前に、ランタイムは最後に実際のシミュレーションが消費した変換を
`Emitter.previousTransform`としてスナップショットし、その後に現在ステップのattach/socket/手動変換を
解決する。新規spawn直後とM9プールからの再取得時は、合成済みエミッターoffsetを含めてpreviousをcurrentへ
初期化する。したがって、原点やプールされた前世代の姿勢は補間端点にならない。`setTransform()`は直前に
シミュレーションされた変換からの連続運動として解釈され、補間される。瞬間移動には、将来の明示的な
previous-transformリセット/teleport APIが必要である。H1-7は距離や速度の閾値からteleportを推測しない。

transform履歴は、scalabilityによってinstanceがカリングされている間や、有効なローカル時間がゼロの間
(`timeScale: 0`およびscale 0のhit stopを含む)も、正の各system stepに追従する。その間は
per-distance birthを発生させず、移動距離は再開時の一括発火用に蓄積せず意図的に破棄する。したがって、
再開後の最初のstepは停止中/カリング中の最新transformからだけ補間し、staleな補間ビードを防ぐ。

M9バッチ1は、定義ごとに上限のあるエフェクトリソースプールを追加する。`VfxSystemOptions.maxPoolSize` は、
生存している定義ごとに保持されるエフェクトバンドルの数として、デフォルトで16である。ゼロは保持を無効化
する。`getPooledInstanceCount(definition)` はその定義の現在のカウントを公開する。保持されたリソースと
そのカウントの両方は、定義をキーとする `WeakMap` の値として存在するため、定義を回収しても、容量を消費
し続ける切り離されたシステム全体のカウントが残ることはない。`release()` は、ハンドルを即座に生存中の
スケジューリングから取り除く。システムの更新が進行中の場合、保持/破棄は、そのキューイングされた更新が
確定するまで延期されるため、新しいスポーンが保留中のGPU送信と競合することはない。プール可能性(エラー
状態の除外を含む)は、`release()` が呼び出された際に同期的にキャプチャされる。定義ごとの上限に達すると、
リソースは破棄され、解放されたハンドルは `NACHI_EFFECT_POOL_LIMIT_EXCEEDED` を記録する。

既存のエフェクト定義の `WeakMap` は、引き続き、レンダラーに依存しないコンパイル済みプログラムのキャッシュ
のままである。プールは、具現化されたカーネルとそのGPUストレージ/間接/イベントバッファの、定義ごとの
別個のキャッシュである。バンドルがプールに入る前に、コアはすべてのエミッターについて、レンダラーの
`prepareKernelsForPooling()` フックを呼び出す。アダプターは、バンドルを同期的に描画不可能にしなければ
ならない。間接レンダラーは、描画の `instanceCount` をゼロに書き込み、古くなった外部の描画オブジェクトが
それをレンダリングしてしまう前に、そのワードをアップロード予定に組み込む。再利用は、そのバンドルの周囲
に、新しいパブリックインスタンスと、新しいCPUのライフサイクル、クロック、リスナー、診断、トランス
フォーム、シード、パラメータ状態を構築する。いかなるアクティベーションの前にも、初期化カーネルは、
すべての解決済み属性レーン、フリーリスト、生存/成功/オーバーフローのカウンタ、イベントカウンタ、
`spawnGeneration`、`nextSpawnOrder`、`currentSpawnBase`、容量サイズのbirth ringをクリアする。birth
ringは、そのカウンタがゼロである間、無効値である `0xffffffff` の番兵を使用する。したがって、再利用後の
最初の成功した割り当ては、再びパーティクル世代1、スポーン順序0を持ち、等しい定義/シード/パラメータ/
ステップ列は、§2.6の物理割り当て境界に従い、コールドスポーンとビット単位で一致する。間接描画の
`instanceCount` は、プーリングフックによってゼロにされ、その後、`finalizeIndirect` によってコンパクション
済みの生存数から書き直される。initializeは、描画ワード0(`indexCount`)を上書きしない。

`VFXSystem.update(deltaSeconds?)` は、システムクロックを進め、レンダラーに依存しないカーネルアダプター
を通じて、生存している各インスタンスをスケジュールする。コアは、M1の `KernelTslAdapter`、コンピュート/
間接送信、任意の読み戻し/uniform/リソースフック、デバイスロス通知を含む構造的な `VfxRuntimeRenderer`
ブリッジを受け入れる。バックエンドパッケージは、それをコアへインポートすることなく、そのネイティブな
レンダラーをラップする。`VFXSystem.setCamera({ viewMatrix, projectionMatrix, viewportSize })` は、
列優先(column-major)のカメラ行列と正の深度テクスチャ寸法を、現在のすべてのエミッターへコピーし、以降
のスポーンを同じ状態で初期化する。固定ステップは、まずスケールされていないワールドデルタを分割する。
各インスタンスは、その後、そのサブステップに `timeScale`(およびアクティブなヒットストップのスケール)
を1回乗算して `Emitter.deltaTime` を生成する。`System.deltaTime` と `System.time` はスケールされない
ままである。余分な固定ステップのバックログは、無制限のフレームを生成するのではなく、明示的に破棄される。

M2は散在するWebGPUフリーリストを使用する。Updateは、消滅した物理インデックスをアトミックに追加する。
Spawnは現在の末尾から割り当てる。別個のアトミックなコンパクションパスが、生存パーティクルインデックス
と権威ある生存数を書き込む。4つのカウンタ、フリーリスト領域、コンパクトな生存インデックス領域は、1つの
書き込み可能なライフサイクル状態バッファを共有する。スポーンディスパッチと間接描画の引数は、第二の
間接引数バッファを共有するため、間接ソースが同一のコンピュートパスの同期スコープ内で書き込み可能な
ストレージでもあることは決してない。描画引数は、`BuiltEmitterKernels.drawIndirectOffsetBytes` によって
公開されるバイトオフセットから始まる。コアは意図的にその領域のワード0(`indexCount`)には手を加えない
ため、M3レンダラーは、最初の描画の前にCPU上で一度それをプライムし、そのオフセットで同じリソースを
バインドしなければならない。レンダラーモジュールは、パーティクル属性をバインドする際に
`aliveIndicesOffset` を使用してコンパクトなインデックス間接指定を消費する。必要なアトミクス/間接呼び
出しを欠くWebGL2は、明示的に宣言された縮小 `burst` パスのみを持つ。その初期化カーネルはすべての物理
属性ストレージを書き込むため、`initializeVaryingCount` と `initializeVaryings` は、ノードの具現化の前に、
その完全なtransform-feedback書き込みセットを検証する。バックエンド上限を超えるスキーマと、すべての
`rate`/`perDistance` エミッターは、黙って劣化するのではなく `NACHI_BACKEND_SPAWN_UNSUPPORTED` によって
拒否される。`CompiledEmitterMeta.capabilities` と `backendBudgets` は、これらの区別を検査可能にする。

2つのスポーン順序カウンタと容量ワード分のbirth ringは、解決済みのスキーマが `Particles.spawnOrder` を含む場合にのみ、ライフサイクル状態へ追加される。通常のスプライト、メッシュ、デカールのエミッターは、これらのワードもbirth-ringへの書き込みも負担しない。リボンと、論理birth順でtop-Nを選ぶライトは明示的にopt inする。レンダーコンパイラの予算記述はステージ精度である。リボンの頂点シェーディングは、位置/色とその3つのセグメントバッファをバインドし、別個の準備コンピュートパスによってのみ消費されるライフサイクル状態はカウントしない。

Spawnはスケジューラステップ内でUpdateより前に実行されるため、そのステップで誕生したパーティクルは、即座に完全な1つの `Emitter.deltaTime` 分のエイジングを受け取る。あるステップ内で `$age` がlifetimeに達した場合、コンパイラが所有する消滅処理は、そのパーティクルについて残りの作成者のUpdateモジュールをスキップし、そのステップの生存コンパクションの前にそれを回収する。

正確な完了は `aliveCountReadbackInterval` によるオプトインである。発生が完了した後、権威ある生存数がゼロになるとインスタンスは完了する。それがない場合、コアは、静的に証明可能な最大lifetimeまで保守的に排出する。証明不可能または未宣言のlifetimeは `Infinity` である。

M9の `@nachi-vfx/timeline` システムは、すべてのワールド時間セグメントをコアへ委譲する前に、アクション、ループ、mesh-life、シェイク終了、ヒットストップの境界で更新を分割する。したがって、0.05秒で作成されたヒットストップは、`System.time` を逆行させたり無関係なゲーム時間を凍結させたりすることなく、大きな呼び出し元の更新の残りへ影響し得る。固定ステッピングでは、タイムラインがM2の `FixedStepAccumulator` を所有し、第二のアキュムレータなしに、その固定ワールドステップをコアシステムへ委譲する。タイムラインと子エミッターのクロックは、まったく同じステップ列を消費する。

### 10.4 M11の品質、カリング、重要度

Nachiは4つの順序付けられた `QualityTier` の値を持つ: `low`、`medium`、`high`、`epic`。エミッターは、
`spawnRateScale`、`capacityScale`、および `soft`/`lit`/`sorted` の機能ゲートについて、`quality[tier]`
の下にシリアライズ可能なオーバーライドを宣言してよい。システムのプリセットは、スポーンおよび論理容量に
ついて、それぞれ0.25、0.5、0.75、1.0である。lowはゲートされた3つの機能すべてを無効化する。mediumは
softを有効化する。highはsoft、lit、sortedを有効化し、epicも3つすべてを有効化する。オーバーライドは、その
名前が付けられた値のみを置き換える。すべてのスケールは有限であり `[0,1]` の範囲である。作成された
capacityは物理的な割り当て上限のままであるため、稼働中のティア変更は、既存のGPUストレージを再割り当て
したり切り詰めたりすることは決してない。

`VFXSystemOptions.qualityTier` は、ティアまたは `auto` を受け付ける。省略した場合、後方互換性のため
epicが選択される。`detectDeviceQualityTier()` は、利用可能な場合 `navigator.gpu` のアダプター上限/
機能を取得し、選択されたティア、正規化されたデバイスプロファイル、人間が読める理由を含む
`QualityTierSelection` を返す。WebGL2は常にlowを選択する。WebGPUはデフォルトでmediumを選択し、128 MiB
のストレージバインディング/256呼び出し/8ストレージバッファに加えタイムスタンプクエリの閾値でhighを
選択し、256 MiB/512/10に加えタイムスタンプクエリとshader-f16でepicを選択する。明示的なオーバーライド
が優先され、その旨が報告される。これらのヒューリスティックはポリシーであり、特定の市販GPUモデルについて
の主張ではない。同期的な `VFXSystem` の自動選択は、レンダラーアダプター境界から任意の機能名を発見する
ことはできない。したがって `deviceProfile` が省略された場合、上限はmediumとなる。high/epicの自動選択を
望む呼び出し元は、デバイス検出によって返却される機能を伴うプロファイルを渡さなければならない。ルート
パッケージは `detectDeviceQualityTier()`、`selectDeviceQualityTier()`、およびそれらのデバイスプロファイル/
選択型をエクスポートする。ティアの適用、構造キーの生成、バウンディング/視錐台変換、重要度スコアの数式
は、パブリックAPIではなく、スケジューラ/コンパイラの実装詳細である。

`setQualityTier()` は、CPUのスポーンポリシーのスケーリングと論理容量上限を即座に変更する。これらは
uniform/スケジューラの値であり、同じコンパイル済みプログラムと具現化されたプールを再利用する。作成された
`soft`、`lit`、`sorted` の挙動をオン/オフに切り替えると、レンダラーグラフ、ストレージの間接指定、または
シェーダーインターフェースが変化する。それが再コンパイルの境界である。コンパイル済みプログラムとGPU
プールは、単にティア名によってではなく、結果として得られる構造的なレンダラーバリアントによってキー付け
される。稼働中のインスタンスは、シミュレーション状態を保持するために、既存の構造的バリアントを維持し、
`NACHI_QUALITY_RESTART_REQUIRED` を記録する。次のスポーンは、要求されたコンパイル済み/プールのバリアント
を使用する。これは明示的な再起動境界であり、状態を持つカーネルを黙ってホットスワップすることは決してない。
再起動診断は `(code, destinationTier)` によって重複排除されるため、ティアの往復は各構造的な移行先を一度
だけ報告する。sortedゲートはbillboard、mesh、decal rendererへ対称的に適用される。low/mediumはsorted
variantを除き、high/epicはcorrectness既定を保持する。WebGL2のauto選択はlowを選び、materialization前に
unsupported particle-sort pathを除く。明示high/epicのWebGL2設定はsortedをfalseへoverrideしなければ、
通常のunsupported diagnosticを受け取る。

各エミッターは、保守的なローカル球 `bounds: { center?, radius }` を宣言してよい。省略した場合、半径1000
の球が使用され、視錐台カリングはレンダリングする方向へ誤る。エフェクトの球は、変換されたエミッターの球
の決定的な和集合である。`effect.scalability.culling.distance` は `fadeStart` から `fadeEnd` へ線形に
フェードし、一方 `frustum` は `setCamera()` へ供給されたカメラ行列を使用する。
`VfxCameraState.coordinateSystem` は `webgpu`(`0 <= clipZ <= clipW`、デフォルト)または `webgl`
(`-clipW <= clipZ <= clipW`)を宣言する。透視投影行列だけではそれを識別できないため、near平面の抽出は
その明示的な規約を使用しなければならない。フェードは、ビルボード、メッシュ、デカール、リボン、その他の
マテリアルの不透明度に乗算される `System.visibility` のuniformである。外部のライトプールは同じカーネル
の可視性チャンネルで登録され、フェードがゼロの際に無効化される。フェードがゼロの場合、コアはイベント、
スポーン、更新、コンパクション、ソート、レンダリングの作業をスキップする。その時間契約はNiagara風の
ポーズである。エフェクトローカル時間、エミッターのライフサイクル、パーティクルの状態はすべて凍結される。
システム/ワールド時間は継続する。再進入は、キャッチアップシミュレーションや遅延バーストなしに、正確な
凍結状態から再開する。将来のhidden中進行モードには、明示的なリセット/キャッチアップの契約が必要であり、
推論されることはない。

重要度は、同じカメラ/状態のスナップショットから、スケジューリング境界ごとに一度評価される。ワールド球とユーザー/作成された優先度 `p` について、スコアは次の通りである。

```text
distanceScore = 1 / (1 + viewDistance)
screenOccupancy = clamp((abs(projection[5]) * radius / max(abs(viewZ), radius, 1e-6))^2, 0, 1)
score = 4 * p + 2 * screenOccupancy + distanceScore
```

候補は、スコアの降順、続いて§9.5と同じシステムごとの数値のインスタンス生成シーケンス昇順でソートされる。公開IDの10進文字列は比較しない。`maxActiveInstances` は残りの末尾をカリングする。`maxParticles` は、各受け入れられたインスタンスに、そのティアスケールされた宣言容量を課金する。その予算を超えてしまうであろう、それ以外の点では受け入れられるインスタンスは、既存のパーティクルの更新/描画は続けるが、新しいスポーンを抑制する。この保守的な会計処理は、フレームごとの強制的なGPU読み戻しを回避する。`EffectInstance.scalability` は、アクション(`full`、`spawn-suppressed`、`culled`)、フェード、理由、合計スコア、すべてのスコアの構成要素を公開する。したがって、等しいカメラ/状態/生成シーケンスは、同一のカリングと予算の決定を生成する。

パーティクル予算によるスポーン抑制は、`emitTo` のイベント消費によって要求される誕生ではなく、通常のスケジューラの誕生(`burst`、`rate`、`perDistance`)に適用される。GPUの論理容量クランプは、両方の経路について権威ある上限のままである。

予算の順序付けは、固定された `0.05` のスコアヒステリシスを持つ。前回のスケジューリング境界で受け入れられたインスタンスは、次の予算順序比較についてのみ、そのボーナスを受け取る。公開される重要度スコアは変化しない。したがって、挑戦者は、現在受け入れられているインスタンスをそのバンドより多く上回らなければならず、これにより、閾値付近でのvisible/culledのばたつきと、それが引き起こすであろうポーズ/再開の時間の途切れを防止する。

ティアスケールされた論理容量は、生存数の読み戻しが無効化されている場合であっても強制される。コアは、
ディスパッチされた誕生の保守的な台帳を保持し、静的に証明された最大lifetimeの後にのみ各バッチを期限切れ
にする。未知のlifetimeは `Infinity` であり、意図的に、このモードを再起動まで保守的にする。WebGPUの
アロケータは、その正確なフリーリストの占有状況を用いて、通常およびイベントによる誕生を論理容量の
uniformに対して独立にクランプするため、CPUの推定値が上限超過の誕生を許容することはできない。権威ある
読み戻しは推定値をリセットし、そのサンプル以降の誕生は即座に加算される。論理容量が作成された物理容量と
等しい場合、CPUは古くなった生存サンプルに対してクランプせず、物理的な要求上限まで送信し、正確な空き
スロットのクランプをGPUアロケータに委ねる。これはM11以前の挙動を保持する。WebGL2のフル数量の読み戻しは、
コンパクションのたびに推定値を更新する。

`@nachi-vfx/timeline` は、そのエミッターの子を、親の振り付けの実装詳細として扱う。子のサブエフェクトはエフェクトのカリングを継承せず、タイムラインコアはグローバルな重要度予算を省略する。したがって、子が独立して凍結し、永続的に親の完了をブロックすることはあり得ない。タイムライン/mesh-fxのクロックと完了は親が所有したままである。タイムラインをカリングするアプリケーションは、個々のplayアクションの子に予算を割り当てるのではなく、全体としてそれを行わなければならない。

### 10.5 M11シミュレーションキャッシュ

シミュレーションキャッシュは、シリアライズされたシミュレーションカーネルではなく、レンダラーが利用可能な状態のスナップショットである。ベイクは、ライブ実行と同じシード、モジュールPCG、一定のフレームステップを使用する。

```ts
const cache = await bakeSimulation(system, effect, {
  frames: 121,
  frameRate: 60,
  sampleStartFrame: 120, // warm up for two seconds before recording
  compression: 'float32', // or 'quantized-u16'
  interpolation: 'linear',
  loop: true,
  spawn: { seed: 42 },
});

const player = await replaySimulation(system, effect, cache, {
  loop: true,
  timeScale: 1,
});
player.play();
await player.update(worldDelta);
player.stop();
```

`bakeSimulation()` は供給されたシステムを進行させるため、ライブエフェクトと共有されるシステムではなく、
専用のベイク用システムを受け取ることが望ましい。`fixedTimeStep` が設定されたシステムは
`NACHI_SIM_CACHE_FIXED_TIMESTEP_UNSUPPORTED` として拒否される。そのアキュムレータは、正確な
`1 / frameRate` のキャッシュステップを分割、延期、または破棄してしまう可能性があるためである。
各bake stepはライブ実行と同じsimulation Update dispatchを実行するため、bake runtimeの0を起点として
`Emitter.updateRandomStep` を進める。replayは記録済みframeをuploadし、Update乱数streamを再評価しない。
シミュレーションキャッシュv2は、エミッターのレンダー読み取り属性とライフサイクル間接指定のみを記録する。
Grid2DまたはGrid3Dの状態は記録しない。グリッドを含むエフェクトのベイクは、そのライブステージを実行し、
`SimulationCache.diagnostics` において `NACHI_SIM_CACHE_GRID2D_NOT_RECORDED` または
`NACHI_SIM_CACHE_GRID3D_NOT_RECORDED` を返す。リプレイはエフェクト定義を具現化するが、そのベイクされた
グリッド状態を復元したり進行させたりすることはない。ベイクのインスタンスは、スケーラビリティのアクション
`full` のままでなければならない。`culled` と `spawn-suppressed` のベイク対象は、凍結されたアセットや
空のアセットを生成するのではなく、`NACHI_SIM_CACHE_SCALABILITY_SUPPRESSED` として拒否される。初期化
カーネルが実行されていないエミッターの読み戻しは `NACHI_SIM_CACHE_EMITTER_UNINITIALIZED` として拒否
される。

最初のサンプルは、`sampleStartFrame / frameRate` における初期化済みフレームである。`sampleStartFrame`
はデフォルトでゼロであり、読み戻しの前にウォームアップとしてその回数分の一定のシミュレーションステップ
を進める。それ以降のサンプルは、ちょうど `1 / frameRate` ずつ分離される。キャッシュは
`{ kind, metadata, diagnostics, data: ArrayBuffer }` であるため、JSONメタデータは独立にシリアライズ
可能なままであり、一方バイナリペイロードの転送は通常のアセットローダーを使用できる。メタデータバージョン
2は、エミッターのキー/容量、論理属性の型とコンポーネント数、フレームごとの生存数/物理インデックス、
ロスレスなlineage stream、固定バイトオフセット/ストライド、圧縮、フレームタイミング、補間、ソースの
バックエンド、(lineage集合と整数属性の別個の一致を含む)端点の連続性、ベイク時の品質ティア、フレーム
ごとのアップロードバイト数を記録する。すべてのエミッターは、すべてのフレームについて、各物理スロット
あたり1つのロスレスなu32 `Particles.spawnOrder` lineage値を記録する。v1キャッシュまたはバージョンが欠落
したメタデータは、明示的な再ベイク指示を伴う `NACHI_SIM_CACHE_VERSION_UNSUPPORTED` として拒否される。
v2は古いpayloadを推測したり黙って移行したりしない。ベイクされたソースとは異なるバックエンドでの
リプレイは `NACHI_SIM_CACHE_BACKEND_MISMATCH` として拒否される。v2は、バックエンドをまたぐライフサイクル
または物理スロットの等価性を主張しない。

バイナリペイロード内のマルチバイトのスカラーフィールドは、すべてリトルエンディアンのバイト順序を使用する。ベイクとロードは、ホストのエンディアンを一度チェックし、リトルエンディアンでないホストを `NACHI_SIM_CACHE_ENDIAN_UNSUPPORTED` として拒否する。通常のエミッタースキーマが `Particles.spawnOrder` を含まない場合、ベイクはcache-privateなbake-only compile variantでlineageを具現化する。元のeffect definitionと通常のlive/replay emitter schemaは変更されない。rendererが `Particles.spawnOrder` を消費しないエミッターは、永続的なbirth ringまたはnext-order counterを追加で負担しない。レンダラーが `Particles.spawnOrder`(特にリボン)を消費する場合に限り、キャッシュは既存のライフサイクルbirth-index ringと `nextSpawnOrder` も格納する。リプレイは、レンダラーの準備の前にそれらを復元する。bake-only variantが必要な場合は、それを `releaseUnpooled()` で完全退役させた後、元定義を1回spawn/releaseして通常poolを逐次warmする。variantと通常版のGPU resource bundleを同時保持しないため、peakが二重にならない。観測可能な代償は、bake中の追加compile/materialize経路とsystemのinstance sequence進行である。呼び出し側は前述の専用bake systemを引き続き使用することが望ましい。

属性の選択は、コンパイルされた描画頂点属性と照合された、すべてのレンダーモジュールが宣言する `Particles.*` の読み取りに従う。renderの `optionalReads` は通常のreplay schemaに存在する場合に限り選択する。bake-only lineage variantだけが具現化した属性はoptional readを満たしたことにせず、rendererのfallback意味論を変えない。シミュレーションのみの状態は省略される。値は論理容量順のストリームへデインターリーブされるため、選択されたレーンが、同じパックされたGPUバッファから無関係な、あるいはtransientなレーンを誤ってシリアライズすることはない。レンダーパスによって必要とされるtransient属性は `NACHI_SIM_CACHE_TRANSIENT_RENDER_ATTRIBUTE` である。黙ってそれを保持することは禁止されている。生存メンバーシップ、インデックス、lineageは常にロスレスなu32である。`bool`/`u32` と `i32` の属性はロスレスなu32/i32のままである。浮動小数点属性は、Float32、またはコンポーネントごとのグローバルmin/max u16量子化のいずれかを使用する。非有限の値は、暗黙の置き換えを受けるのではなく、量子化を拒否する。

ループキャッシュは重複した端点を含む: `duration = (frameCount - 1) / frameRate`。端点比較は各端点のlineageから物理スロットへのmapを構築し、同じ生存lineage集合であることを要求し、各lineageに対応付けられた物理スロットの整数属性と浮動小数点属性を比較する。したがって、alive indexの順序と物理スロットの割り当ては連続性の判定基準ではない。metadataは集合の判定結果を `loop.lineageMatch` として公開する。`loop: true` でのベイクには、`lineageMatch`、等しい整数属性、および `loopTolerance` 以下のfloat端点誤差が必要である。そうでなければ `NACHI_SIM_CACHE_LOOP_DISCONTINUITY` を発生させる。再生は、重複した端点をフレームゼロへラップする。これにより、この継ぎ目は、プレイヤー内で不連続性を隠すのではなく、作成/ベイクされた契約となる。ループベイクは、少なくとも1つの完全な作成済み周期分ウォームアップし、パーティクルのlifetime、スポーンの周期、決定的な初期化状態が継ぎ目において周期的であるウォームアップ後のウィンドウを記録することが望ましい。`loop: false` は、ループしないことのオプトアウトである。それでも、ツールのために端点メトリクスを記録するが、不連続な端点を拒否することは決してないため、第二の検証スキップオプションは不要である。

生存lineage値は、エミッターの各フレーム内で一意でなければならない。ベイクは、欠落または重複した生存lineageを `NACHI_SIM_CACHE_LINEAGE_DUPLICATE` として拒否する。unsigned lineage比較を曖昧にしないために、ベイクはウォームアップ中も含め、u32の半範囲(`0x80000000`)以上の `nextSpawnOrder` を `NACHI_SIM_CACHE_LINEAGE_WRAP_RISK` として拒否する。u32がwrapする前にエミッターを再起動し、より短いwindowを再ベイクしなければならない。warmup中のCPU側の保守的なbirth要求上限は、権威あるGPU lifecycle readbackを発生させるtriggerにすぎず、それ自体では拒否しない。allocator saturationや失敗したbirthによって要求上限が実進行を過大評価し得るためである。wrap-risk diagnosticを発生させられるのは、実GPUの `nextSpawnOrder` 値だけである。

M11 v2は、すべてのフレームをGPU常駐にするのではなく、フレームごとのCPUデコードとストレージアップロード
(ネイティブWebGPUアダプターでの `writeBuffer`)を選択する。Webのエフェクトは、一般的に再生前にキャッシュ
をロード/事前計算するため、この経路は、バイナリのダウンロード圧縮を許容しながら、GPUの常駐を既存の現在
フレームの属性/ライフサイクルバッファに限定する。これは、ビルボード、メッシュ、リボン、ライト、デカール
のレンダーリソースバインディングも再利用する。リプレイは、コンパイル済みの物理パッキングと生存間接指定
を再構築する。パーティクルソートやリボンのセグメント構築のようなレンダラー由来の準備は引き続き実行され
得るが、スポーン/初期化/更新/イベント/コンパクションのシミュレーションカーネルは実行されない。すべての
フレームを常駐させるバッファ/テクスチャのモードは、アップロード帯域幅が常駐より高価であるワークロード
のための、将来のオプトインのままである。`writeStorage()` によるリプレイのアップロードは、`seek()`、
`update()`、または初期化が解決される前にフラッシュされるため、即座のデバッグ/ライフサイクルの読み戻しは、
レンダー呼び出しの前であっても、選択されたキャッシュフレームを観測する。

`nearest` は1つの完全なフレームを選択する。`linear` は、隣接する両方のフレームで同じ物理
パーティクルスロットが生存し、両フレームのlineage値も一致する場合に限り、浮動小数点属性を補間する。
異なるlineageに再利用されたスロットと、異なる物理スロットに現れる同じlineageは、いずれも最も近い完全な
フレームを使用する。補間は物理ストレージをまたいで値を移動しない。fractionが `0.5` 未満ならleft frame、
`0.5` 以上ならright frameを選択する。整数属性と生存メンバーシップも同じnearest-frame規則を使用する。
したがって、新規/消滅/再利用されたスロットが無関係な、または古くなった
ストレージに対して補間されることは決してない。同じキャッシュ、再生オプション、時間列が与えられれば、フレーム選択、デコード、アップロード
バイト、描画入力は決定的である。Float32のライブ対リプレイの属性の受容は、バックエンドが読み戻しバイトを
保持する場合、ビット単位の等価性である。M11のスモークテストは、デコードされたfloatの比較には絶対値
`1e-6` を、レンダーターゲットの変換に対応するためにリニアなピクセル閾値 `2 / 255` を使用する。量子化
されたキャッシュは、Float32の基準ではなく、記録されたコンポーネントごとの量子化誤差範囲を使用する。

エミッター `e`、フレーム数 `F`、容量 `C_e`、選択された属性のコンポーネント数 `k_ea`、エンコードされたコンポーネントあたりのバイト数 `b_ea`、選択された物理GPUストレージバッファ `S_e`、ライフサイクル状態ワード数 `L_e` について、予算は次の通りである。

```text
binaryBytes = sum_e F * (4 * C_e + 4 * C_e + birthOrder_e * 4 * C_e + sum_a(C_e * k_ea * b_ea)) + alignmentPadding
metadataBytes = UTF8(JSON.stringify(metadata)).byteLength
totalAssetBytes = binaryBytes + metadataBytes
uploadBytesPerFrame = sum_e (sum_s_in_S_e physicalBytes(s) + 4 * L_e + 4)
```

最初の `4 * C_e` というバイナリ項は固定された生存インデックスstream、2番目は必須のlineage streamである。したがってlineageは、alignment前でエミッターごとに正確に `4 * C_e * F` byteを追加する。`birthOrder_e` は、renderer preparationにbirth ringが必要な場合は1、そうでない場合は0である(そのnext-order scalarはmetadataに留まる)。lineage streamはCPU側のidentity照合であり、通常のフレームごとのreplay uploadへ追加されない。upload式は変更されない。最後のアップロードの `4` は間接インスタンスカウントワードである。共有されたパックストレージとmat3のパディングは、`physicalBytes(s)` において一度だけカウントされる。ランタイムのGPU常駐は、`F` に依存せず、コンパイル済みの現在フレームのストレージにライフサイクル/レンダラーの補助を加えたもののままである。パブリックな `estimateSimulationCacheMemory()` は、上記4つのアセット/アップロードの項を報告する。

WebGL2のパックされたシミュレーションストレージは、グループ0に制限される。振る舞いを持つspawn/init/update
のカーネルがグループ1以上の属性を読み書きする場合、コンパイルは `NACHI_BACKEND_PACKED_STORAGE_UNSUPPORTED`
によって失敗する。そうでなければ、Three r185は、そのグループをtransform-feedbackのデスティネーション
アドレスから省略し、そのアクセスをグループ0へエイリアスさせ、状態を黙って破損させてしまう。コンパイラが
作成するデフォルトパスのみでは、このゲートを発生させない。なぜなら、それは割り当てられたストレージを
確立するだけであり、より高いグループの属性を、後の振る舞いによるアクセスに対して安全にするわけでは
ないからである。振る舞いによるアクセスが完全にグループ0内に留まる縮小WebGL2エミッターは、引き続き有効な
ランタイムのサブセットである。このルールは、それらのM2ランタイム構成を無効化しない。

シミュレーションキャッシュのベイクには、この厳格なゲートの下でサポートされるレンダリング可能なWebGL2
フィクスチャが存在しない。すべてのfloat属性は1つの `packed_float` ストレージへパックされ、ライフサイクル
は `age`、`lifetime`、`normalizedAge` を暗黙的に寄与する。`position` を必要とするレンダリング可能な
ライフサイクルエミッターは、したがって少なくとも6つのfloatコンポーネントを必要とし、それ以外の点では
最小限のスキーマであってもグループ1に達する。したがって、WebGL2のベイクは実質的にサポートされないもの
として扱われ、構造化された診断とともに失敗しなければならない。実装と `/m11-cache/?backend=webgl` の
プローブは、パックされたストレージをエイリアスさせるスキーマを通じて数値的等価性を求めてはならない。

リプレイは、v2において独立に `NACHI_SIM_CACHE_REPLAY_WEBGL2_UNSUPPORTED` として診断される。現在のThree
レンダラー統合には、復元された生存間接指定をバインドできるWebGL2の間接描画具現化パスが存在しない。
プレフィックスを黙って描画しながらリプレイを主張してはならない。パックされた読み戻しはバックエンド固有
である。WebGPUは `particle * groupCount + group` のvec4アドレッシングを保持するが、Three r185のWebGL2
transform feedbackはストレージ/パーティクルごとに1つのvec4を発し、物理的なデスティネーションアドレス
からパックされたグループを省略する。この物理的な制約は、厳格な振る舞いアクセス診断を通じて露呈され、
見かけ上有効なWebGL2のキャッシュバイトへ正規化されることは決してない。

### 10.6 M11デバッグスナップショットとプロファイリング

デバッグはコードファーストのランタイム表面である。エフェクトインスタンスは非同期の論理属性キャプチャを公開し、一方、所有システムは、最新の完了した更新のプロファイラカウンタを公開する。

```ts
const attributes = await instance.debug.captureAttributes('sparks', {
  attributes: ['position', 'velocity', 'age', 'spawnOrder'],
  offset: 0,
  limit: 256,
});

const profile = await system.debug.captureProfile({ gpuTiming: perfBaseline.gpu });
```

`captureAttributes()` は、シミュレーションキャッシュのベイクと同じ具現化されたストレージリソースと物理
から論理へのパッキングメタデータを読み取る。これは、選択された生存エントリごとに1行を返す。コンパクトな
`aliveIndex`、`physicalSlot`、論理型付き値、そして、それらの属性が宣言されている場合の
`spawnGeneration`/`spawnOrder` の系譜を含む。真偽値は真偽値のまま、整数とfloatのスカラーは数値、
ベクトル/行列は論理コンポーネントの配列である(mat3のパディングは決して公開されない)。`attributes` が
省略された場合、宣言されコンパイルされたすべての論理属性を意味する。`limit` が省略された場合、残りの
すべての生存パーティクルを意味する。明示的な非負の `offset`/`limit` は、コンパクトな生存順序を選択する。
すべての結果は `{ totalAlive, returned, offset, limit, truncated }` を含むため、選択やバックエンドの
制限が行を黙って破棄することはない。未知の列や利用不可能な読み戻しは、構造化されたランタイム診断である。

両方のキャプチャAPIは、所有システムの更新キューにおいてシリアライズされる、非同期かつ非変更の操作である。
キャプチャは、それが呼び出された時点ですでに送信されているすべての更新の後に実行され、それが開始した後
に送信されるいかなる更新よりも前に完了する。したがって、呼び出し元が `update()` を並行して発行する場合
であっても、ライフサイクルのメンバーシップ、すべてのストレージの読み戻し、プロファイラカウンタは、単一
のスケジューラ境界を記述する。バックエンドのコマンド完了は、属性のスナップショットを、プレゼンテーション
に対して実質的に1フレーム遅延させることがあり、これは `latencyFrames: 1` として報告される。WebGPUの
メンバーシップは、コンパクトな生存インデックスバッファに従う。WebGL2は、コンパクトな間接ライフサイクル
パスを持たないため、縮小パスの論理的な `Particles.alive` ストレージからメンバーシップを導出する。
エミッターの初期化カーネルが実行される前に属性キャプチャを呼び出すと `NACHI_DEBUG_EMITTER_UNINITIALIZED`
となる。ルートパッケージはインスタンス/システムのキャプチャ表面とスナップショット/プロファイル型を公開
する。フォーマッタ、レンダラー読み戻し、フレーム集約のヘルパーは内部的なものである。

§10.5のバックエンド固有のパックアドレスルールは、WebGL2上では非単射(non-injective)である。Three r185
のtransform feedbackはパックされたグループを省略するため、論理的な `(group >= 1, offset)` は、対応する
`(group 0, offset)` の物理コンポーネントにエイリアスする。厳格なビルドゲートは、黙ったエイリアシングを
許容するのではなく、そのような列にアクセスする振る舞いカーネルを拒否する。グループ0のみの縮小ランタイム
構成は引き続きサポートされる。デバッグキャプチャは、そのゲートを弱めることはない。それでも、コンパイラが
作成するデフォルトパスによってのみミラーされた、宣言された高位グループの列を公開することがある。その
検査のケースでは、キャプチャはデータを捏造したり削除したりするのではなく、物理的なエイリアス値を保持し、
その列を `aliased: true` としてマークし、`snapshot.diagnostics` にコード
`NACHI_DEBUG_WEBGL2_ATTRIBUTE_ALIASED` の警告を追加する。デバッガのDOMは、その値を通常の利用可能な論理
属性として提示するのではなく、エイリアスの注釈を表示しなければならない。エイリアスの注釈は診断上の証拠
であり、その列への振る舞いによるアクセスがサポートされているという約束ではない。

プロファイラカウンタは、トップレベルの `VFXSystem.update()` ごとにリセットされ、その内部で実行される
すべての固定サブステップとプリウォームの作業を集約する。それらの定義は規範的である。

- `alive` は、最新の権威あるGPU/読み戻しの値である。読み戻しが設定されていない場合、それは
  `unavailable` であり、事実として提示される保守的な推定値であることは決してない。
- `capacity` は、エミッターに課される、アクティブなティアスケールされた論理容量である。
- `spawnCount` は、ティア、予算、CPU容量のクランプの後に送信された、通常のスケジューラのスポーン呼び
  出しの数である。並行的な飽和状態におけるGPUフリーリストの拒否は、成功した誕生として再ラベル付けされる
  ことはない。
- `computeDispatches` は、初期化、スポーンの準備/確定、更新、コンパクション、イベント、ソートの各ステージ
  を含む、そのエミッターについて行われた直接または間接のコンピュート/transform-feedbackカーネル送信の
  すべてをカウントする。
- `indirectDraws` は、最も最近完了したレンダーフレームにおいて、そのエミッターについてレンダリング可能に
  なった具現化済みの間接描画レコードの数である。これは意図的に1フレーム遅延している。`update()` の後、
  現在のレンダーの前にFIFOでシリアライズされた `captureProfile()` は、直前の完了したフレームを観測する。
  完全にカリングされた、ポーズされた、または一度も具現化されていないフレームは、完了したフレームになった
  時点でゼロを報告する。WebGL2は、ゼロの代わりに `NACHI_PROFILE_INDIRECT_DRAW_WEBGL2_UNAVAILABLE` を報告
  する。
- `cpuUpdateMs` は、アダプターのプロミスのオーバーヘッドを含む、そのエミッターのスケジューラ作業を待機し
  調整するために費やされたホストのウォールタイムである。これはGPUの実行時間ではなく、その代替として使用
  されることもない。

GPUのタイミングはM11ではパス粒度である。`captureProfile()` は、それ自身でタイムスタンプクエリを割り当て
たり解決したりすることは決してない。アプリケーションは、既存の `nachi.perf-baseline` v1モニタが所有する
キャッシュされた `gpu` レコードを供給し、二重の計装とタイムスタンププールの成長を防止する。利用可能な
`computeMs`/`renderMs` はパスの合計のままであり、エミッター間で分割されることはない。欠落、保留、エラー、
またはサポートされていない値は、理由とnull値を伴うunavailableとして型付けされたままとなる。WebGL2の
コンピュートタイミングは、具体的には `NACHI_PROFILE_GPU_TIMESTAMP_WEBGL2_COMPUTE_UNAVAILABLE` を報告する。
タイムスタンプクエリの欠如がCPUの推定値によって置き換えられることは決してない。正確性/長時間実行の
ループは、タイムスタンプ追跡を無効にしたレンダラーを使用し、その後、確立されたタイムスタンププールの
防御を保持しながら、短いパフォーマンスキャプチャを別個に実行することが望ましい。共有されるパフォーマンス
スナップショットは、Threeがそれを送信済みのコンピュート作業の後にリセットした可能性があるため、ゼロで
あるThreeの現在フレームの `computeCalls` カウンタを、明示的な理由とともに信頼できないものとしてマークする。

### 10.7 M12 Grid2D/Grid3Dデータインターフェースとシミュレーションステージ

シミュレーションステージは、エミッターのSpawn/Init/Update/Event/Renderのモジュールスタックとは別個の、エフェクト要素である。ターゲットは、Grid2DまたはGrid3Dのデータインターフェース要素のキーである。ライブオブジェクト参照の代わりに文字列を使用することで、グラフ全体が宣言的であり続け、壊れた参照を診断可能にする。

```ts
const fluid = defineGrid2D({
  resolution: [64, 64],
  channels: {
    density: { type: 'f32', default: 0 },
    temperature: { type: 'f32', default: 0 },
    velocity: { type: 'vec2', default: [0, 0] },
    pressure: { type: 'f32', default: 0 },
  },
});

const pressure = defineSimStage({
  target: 'fluid',
  phase: 'after-particles',
  iterations: 8,
  update: gridPressureJacobi(),
});

const effect = defineEffect({ elements: { fluid, pressure } });
```

`phase` は `before-particles` または `after-particles` であり、デフォルトは後者である。1つのエフェクトスケジューラ境界において、あるグリッドターゲットのすべてのステージは、それぞれのphase内で作成者の要素順序を保持する。境界は次の通りである。

```text
before-particles grid stages -> particle event/spawn/init/update -> after-particles grid stages
```

独立したグリッドターゲットには、ターゲットをまたぐ可視性の契約はない。グリッド間の依存関係には、将来の明示的なマルチターゲットのデータインターフェース宣言が必要である。グリッドステージは、エフェクトローカルなスケール済みデルタ時間を使用し、エフェクトとともにポーズする。エミッターのプリウォームは現在、内部のプリウォームサブステップごとに1つのGrid2Dステップを織り込むのではなく、通常のパーティクル境界の内部で進行する。したがって、結合されたグリッド/パーティクルのプリウォームには、v1では明示的な外部固定ステップが必要である。

正の `iterations` カウントは、すべて文字通りに扱われる。1回のイテレーションは更新カーネルを送信し、その後、scratchからstateへのコミットカーネルを送信する。これらは別個のバックエンド送信である。WebGPUは、1つのパス内の連続するディスパッチ間でストレージの可視性を定義するが、M10のソート実験は、Three r185を通じて記録された依存ディスパッチが、要求される挙動を提供しないことを観測した。したがってGrid2Dは、ステージとコミットについて保守的に別個のThree送信を使用する。グリッドのクリアは、別個の1回限りの送信である。初期化は、意図的に両方のステージフェーズを `deltaTime = 0` で一度実行する。クリアとあわせて、4x3スケジューリングのスモークテストの2回の通常の更新は、したがって `submissionCount = 13` を主張する。`Grid2DRuntimeView.submissionCount` は、デバッグとスモークテストのアサーションのために、これらの送信を公開する。

Grid2D v1は、ストレージテクスチャではなくストレージバッファを使用する。論理的な `f32` と `vec2` の
チャンネルは、セルごとに1つ以上のvec4レコードへ決定的にパックされるが、すべてのレコードは同一の物理
バッファ内に留まる。したがって、ランタイムの状態は、密度、温度、速度、圧力についてさえ、1つの現在の
バッファと1つのscratchバッファをバインドする。これにより、一般的に小さいデバイスのステージごとの
ストレージバッファ予算をSoAのグリッドチャンネルに費やすことを避け、Three r185の確立された
`instancedArray` ストレージパスと連携する。アドレッシングはx-fastの行優先(`index = y * width + x`)
であり、座標はセル中心であり、境界はクランプされ、`sample()` はTSLで実行される4タップのバイリニア
フィルタリングである。`gridCellIndex()`、`sampleGrid2DBilinear()`、`grid2DSnapshotChannel()` は、テスト
とツールのためのCPUミラーである。

`gridTslModule(factory)` は、`index`、整数のセル座標、エフェクトローカルな `deltaTime`、
`read(channel)`、`sample(channel, cell)` を受け取り、チャンネルへの書き込みを返す。これは、コード
ファーストのスクラッチパッド用エスケープハッチである。インラインファクトリはシリアライズ不可能である。
`defineGrid2DStageFunction(id, factory, version)` の登録は、その正確な `grid2d-function-ref` のみを
シリアライズする。`Grid2DStageRegistry` が実行を供給する。組み込みモジュールは、その安定した
`core/grid2d-*` のソースとJSON設定を直接シリアライズする。

パーティクル転送は、正規化されたパーティクル位置に対するデータインターフェース操作である。WebGPUの
`rasterizeParticles(points, scalarChannel, value)` は、u32の累積バッファをクリアし、1呼び出しあたり
1パーティクルを、スケール4096の固定小数点アトミック加算でディスパッチし、その後、別個の送信でfloatの
グリッド状態へ解決する。重複した堆積は加算的である。値は非負であり `1/4096` に量子化される。負または
非有限の作成された値は、送信前に拒否される。ラスタライゼーションまたはサンプリングに供給されるすべての
点座標は有限でなければならない。`[0, 1]` の外側にある有限な座標は、文書化されたエッジクランプの挙動を
保持する。1回の堆積について、呼び出し元は `value * 4096 <= 0xffffffff` を満たさなければならず、これは
ランタイムが検証する。呼び出し元はまた、1つのセルを対象とするすべての固定小数点堆積の合計が、多くとも
`0xffffffff` であることを保証しなければならない。ランタイムは、その合計を検証するためにCPUのセル
ヒストグラムを実行しない。それを行うことは、GPUラスタライゼーションを重複させることになるためである。
オーバーフローするアトミックな合計は、通常のu32のmodulo-`2^32`のラップセマンティクスを持ち、解決された
floatは、したがって飽和ではなくラップされた値を含む。`sampleParticles(points, scalarChannel)` は1呼び
出しあたり1パーティクルをディスパッチし、グリッドステージで使用されるものと同じセル中心のバイリニア
結果を返す。V1の転送は、1呼び出しあたり多くともグリッドセル数分の点と、スカラーのターゲット/ソース
チャンネルを受け付ける。エミッターのパックされたパーティクルストレージの直接バインディングと、vec2の
転送チャンネルは、後の拡張のままである。これらの明示的なメソッドは、それでもCPU補間結果を捏造すること
を避ける。堆積とサンプリングの両方が、ライブなGPUのGrid2Dストレージに対して実行される。ラスタライゼー
ションのリセット/解決とサンプリングは、それぞれグリッドの全セル容量にわたってディスパッチされるため、
供給される点が少ない場合であっても、それらのディスパッチ作業は常に `C = cellCount` に対して `O(C)`
である。

`getGrid2D()`、`getGrid3D()`、`getNeighborGrid()` によって返される、パブリックな `capture()`、`rasterizeParticles()`、`sampleParticles()` の呼び出しはすべて、`VFXSystem.update()` および属性/プロファイルキャプチャと同じFIFOへ入る。進行中の更新中に行われた呼び出しは、その更新が完了した後にのみ実行され、その送信/読み戻しが終わるまで後続の更新をブロックする。特に、NeighborGridのキャプチャは、その3つの読み取りの間に更新が織り込まれることを許すのではなく、1回の完了した再構築からカウント、スロット、統計を読み取る。

`/m12-grid/` の参照ループは、ソースの注入、一次のsemi-Lagrangian移流、チャンネルごとの指数的散逸、温度
マイナス密度の浮力、繰り返されるJacobi圧力解法、圧力勾配による速度投影を使用する。Jacobiは意図的に近似
的である。8回のイテレーションは局所的な発散を減少させるが、大きなグリッドにおいて低周波の圧力を収束
させることはない。渦度拘束(vorticity confinement)、障害物マスク、MacCormack/BFECC補正、適応的タイム
ステップ、質量保存の移流は存在しない。Semi-Lagrangian移流は安定しているが数値的に拡散的である。クランプ
境界は、物理的に導出された壁条件ではなく、閉じたエッジの複製として振る舞う。速度チャンネルと移流座標は
グリッドセル単位を使用し(速度は秒あたりのセル数)、一方 `gridInject()` の中心と半径は正規化されたグリッド
座標を使用する。注入されるチャンネルの値は秒あたりの加算量であり、注入される速度の値も同様に秒あたりの
セル数で表現される。

シミュレーションキャッシュv2はGrid2Dバッファを記録しない。§10.5で規定されている通り、ベイクはライブのGrid2Dステージを実行するが `NACHI_SIM_CACHE_GRID2D_NOT_RECORDED` を報告し、一方リプレイはGrid2Dシミュレーションをスケジュールしない。ベイクされたGrid2Dの進化が再構築されることはない。

WebGL2は、Three r185のtransform-feedbackフォールバックを通じて、任意のインデックス付きストレージ書き込み、アトミクス、間接ディスパッチを提供することはできない。Grid2Dの具現化は `NACHI_GRID2D_WEBGL2_UNSUPPORTED` を報告する。テクスチャのピンポン、CPU流体、またはprefixのtransform-feedbackシミュレーションが黙って選択されることはない。WebGL2の `/m12-grid/` スモークテストは、この診断のみを主張する。

Grid2Dとシミュレーションステージの定義は、`nachi-effect` envelopeがversion 1だった時点で宣言的サブセットへ
入った。現行serializerは同じclosed payload fieldをcanonical envelope v2で往復変換し、default v1→v2
migrationはgrid/stage定義を書き換えない。インラインのGrid2D TSLは `NACHI_ASSET_INLINE_FUNCTION` を報告する。
登録された参照と組み込みの設定はJSONのままである。

M12バッチ3は、同じ契約を `defineGrid3D()` へ拡張する。Grid3Dのチャンネルは `f32` または `vec3` であり、
アドレッシングはx-fastである(`index = (z * height + y) * width + x`)。座標はセル中心でありクランプ境界
を持つ。論理チャンネルは、1つの現在のバッファと1つのscratchバッファ内のvec4レコードへパックされる。
`sample()` は8タップのトリリニア補間を使用する。すべてのclear、stage、commit、パーティクルラスタ、
resolve、パーティクルサンプルのカーネルは `instanceIndex < cellCount` を保護する。ディスパッチの丸めが、
範囲外のストレージの挙動に依存することは決してない。Grid3Dターゲットは `grid3d-stage-module` のみを
受け付け、Grid2Dターゲットは `grid2d-stage-module` のみを受け付ける。不一致は
`NACHI_SIM_STAGE_TARGET_KIND_MISMATCH` を報告する。両者は同じbefore/after-particleの境界、文字通りの
イテレーションカウント、独立したsubmit契約、ポーズの挙動、ピンポンセマンティクスを保持する。

解像度 `W × H × D` について、`C = W * H * D` とし、`G` をセルごとのパックされたvec4レコード数とする。
現在とscratchをあわせて `2 * C * G * 16` バイトを消費する。v1のグリッドと粒子間の経路は、16バイトの
WGSL配列ストライドを持つvec3の位置ストア、floatのサンプルストア、u32のアトミックストアを確保するため、
その追加の予算は `C * (16 + 4 + 4)` バイトである。したがって、パブリックな `estimateGrid3DMemory()` の
結果は、明示的な割り当て内訳とともに `totalBytes = 32 * C * G + 24 * C` である。一般的なスモークテスト
のスキーマは `G = 2` を使用する。32³は2.75 MiBを消費し、48³はおよそ9.28 MiBであり、すべての次元を倍に
すると、メモリは8倍になる。割り当ての前に、ランタイムは最大の個々のバインディングを、より小さい利用
可能な `maxStorageBufferBindingSize`/`maxBufferSize` の上限と比較し、`NACHI_GRID3D_STORAGE_LIMIT_EXCEEDED`
を報告する。隠された、より低い解像度で再試行することは決してない。Grid2Dは、対称的な安全積とアクティブ
なデバイスバッファのチェックを適用し、割り当ての前に `NACHI_GRID2D_RESOLUTION_INVALID` または
`NACHI_GRID2D_STORAGE_LIMIT_EXCEEDED` を報告する。

Grid3Dの組み込みは、注入、一次のsemi-Lagrangian移流/指数的散逸、y軸の浮力、6近傍のJacobi圧力解法、3軸の
圧力勾配投影の、直接的な3D形式である。速度はグリッドセル毎秒のままであり、一方注入の中心/半径は正規化
されたボリューム座標を使用する。上記のGrid2Dの制限は、Grid3Dにも適用される。一次のバックトレースは数値
的に拡散的であり、イテレーション数が少ない場合に繰り返しのJacobiは低周波の発散を残す(ボリュームが大き
くなるほど、より顕著になる)。エッジクランプは物理的な壁の解法ではなく、v1には障害物マスク、渦度拘束、
MacCormack/BFECC補正、適応的タイムステップ、質量保存の輸送は存在しない。これらの制限は手法の一部であり、
レンダラーによって黙って修正されるものではない。

ボリュームレンダリングは意図的に段階化されている。V1は、Grid2Dで使用されるものと同じストレージバッファ
のデータインターフェース経路で、正規化された3Dパーティクル位置において密度をサンプリングし、それらの
サンプルを深度でレイヤー化されたパーティクルまたは単純なスライス/ビルボードとしてレンダリングする。これ
は、ライブなグリッドとパーティクルの結合を証明し、第二のボリュームリソースシステムを追加することなく、
有界なSwiftShader経路を提供する。将来のステージは、パックされた密度場をフィルタリング可能な3Dテクスチャ
としてバインドし、レイマーチング、空間スキップ、シャドウ統合、時間的リプロジェクション、物理ベースの
吸収/散乱を追加してもよい。完全なレイマーチングはv1のGrid3Dの要件ではない。

`defineGrid3DStageFunction()`/`Grid3DStageRegistry` は、シリアライズ可能なカスタムステージ境界である。インラインの
`grid3DTslModule()` はコード専用のままである。Grid3Dの宣言と登録済み/組み込みのステージは、当時の
`nachi-effect` envelope v1でserializationへ導入された。現行ではpayloadとmodule versionを同じままcanonical
envelope v2で往復変換し、default migrationはenvelope versionだけを変更する。WebGL2の具現化は
`NACHI_GRID3D_WEBGL2_UNSUPPORTED` のみを報告する。CPUのボリュームシミュレーションやテクスチャのピンポンへ
フォールバックすることはない。シミュレーションキャッシュv2は、同様にボリューム状態を記録しないため
`NACHI_SIM_CACHE_GRID3D_NOT_RECORDED` で警告する。

M12バッチ4は、同じbefore-particlesのスケジューラ境界において、パーティクル指向のデータインターフェース
として `defineNeighborGrid()` を追加する。宣言は、`resolution`、立方体の `cellSize`、emitter-localの
最小コーナー `origin`、`cellCapacity`(デフォルト32)を固定する。現在のemitter transformは平行移動と
回転を合成するがscaleを含まないため、`cellSize`、探索距離、PBD距離はworld長のままである。ちょうど1つのエミッター
が、`boids()`、`pbdDistanceConstraint()`、または `neighborGridTslModule()` を通じて、v1のneighbor grid
を参照してよい。消費者がゼロの場合 `NACHI_NEIGHBOR_GRID_UNBOUND` を報告し、複数の消費者がいる場合
`NACHI_NEIGHBOR_GRID_MULTIPLE_EMITTERS` を報告し、キーが欠落している場合
`NACHI_NEIGHBOR_GRID_TARGET_UNKNOWN` を報告する。この単一ソースの制限は、2つの独立してパックされた
エミッタースキーマが1つの一貫したインデックス空間を形成するかのように示唆することを避ける。

すべての通常のパーティクル更新の直前に、ランタイムはクリアカーネルを送信し、続いてバケットカーネルを
送信する。バケットカーネルはエミッターの生存フラグとワールド空間の位置を読み取り、その位置を
`inverse(Emitter.transform)` で写してから、`floor((emitterLocalPosition - origin) / cellSize)` を計算し、
`atomicAdd` によってセルスロットを予約する。同じセル座標helperをbucket挿入と、boids、custom neighbor
visitor、PBDが共有する現在セルlookupに使用する。`Emitter.transform` はinstanceの平行移動/回転と
`EmitterConfig.offset` をすでに合成しているため、spawn transform、`setTransform()`、attachment更新、
offsetにgridが自動追従する。カウント
はセルごとに1つのu32アトミックであり、インデックスは固定されたcell-majorの `cellCount * cellCapacity`
のu32配列である。`cellCapacity` を下回る予約は物理パーティクルインデックスを書き込む。それ以降の予約は
破棄され、`dropped` カウンタをインクリメントし、シミュレーションを停止させることはない。宣言された
ボリュームの外側の位置は別個に破棄され、`outOfBounds` をインクリメントする。`getNeighborGrid(key).capture()`
はカウント、固定スロット、両方のカウンタを読み取る。カウントは試行された挿入を保持するため、容量を
超えることがある。非ゼロの破棄はまた、キャプチャの診断において `NACHI_NEIGHBOR_GRID_CELL_OVERFLOW` を
返す。クリアは、空スロット番兵である `0xffffffff` を書き込む。すべてのclear、bucket、constraintのカーネル
は、その丸められた呼び出しを、その論理的なスロット/セル/パーティクル数に対して保護する。

`setTransform()` は次のrebuildで使用するgrid frameを変更するが、既存のworld-space particle位置を遡及的に
移動しない。したがってm12のlive-transform fixtureは最初のspawnより前にtransformを更新する。particleが
存在した後にemitterを移動するapplicationは、そのworld-space particleを新しく移動したlocal grid volumeの
近くへ残すかどうかを選ばなければならない。

captureは、試行されたcell countの合計から `inBounds` も導出する。`total = inBounds + outOfBounds` が
非ゼロで、`outOfBounds / total` が厳密に50%を超える場合、
`NACHI_NEIGHBOR_GRID_OUT_OF_BOUNDS_DOMINANT` を返す。診断contextは両count、total、ratio、gridの
key/definition、emitter path、`neighbor-grid` kernel種別を含む。所有instanceへ蓄積し、その
NeighborGrid runtime lifetime中は最大1回だけ配送する。複数captureと後続のdominant rebuildは再配送
しない。pool checkoutは新しいruntimeを構築して配送を再armする。snapshotの診断は各capture時点の
rebuildを引き続き表し、cell overflowとdominant out-of-boundsは共存できる。厳密に50%とtotal 0は
発火しない。

バケットの送信は、各パーティクルのworld-space位置と速度もスナップショットする。近傍更新コードはその後、格納された
パーティクルインデックスによってスナップショットを読み取りながら、立方体のセル近傍と各保持された固定
スロットをスキャンする。現在のパーティクルの位置と速度は、代わりにライブな属性から読み取られ、呼び出し
内で一貫している。したがって、以前の更新モジュールからの書き込みは基底状態のままである。近傍の状態のみ
がバケット時点のスナップショットから得られる。このライブな自分自身/スナップショットの近傍というルール
は、組み込みのboidsと `neighborGridTslModule()` のコンテキスト状態に適用される。
`neighborGridTslModule({ grid, radius, access }, factory)` は、コード専用のスクラッチパッド抽象化である。
そのコンテキストは、現在のインデックス/ライブな位置/ライブな速度と、近傍のインデックス/スナップショット
の位置/スナップショットの速度を伴う `forEachNeighbor(visitor)` を公開する。V1は、意図的にこれら2つの
古典的なフロッキング/制約の近傍属性をスナップショットする。任意のカスタム属性のスナップショットは、後の
スキーマ拡張のままである。visitor内部での累積は、セルオフセットと固定スロットがランタイムGPUループで
あるため、TSLローカル(`value.toVar()` に続く `addAssign()`/`assign()`)を使用する。インラインヘルパーは
シリアライズ不可能である。組み込みの `boids()` と `pbdDistanceConstraint()` は通常の宣言的モジュールで
あり、シリアライズされる。

`radius` はセル単位で測定される非負の整数である。パーティクルあたりのスキャンコストは、距離による棄却の前に `O((2 * radius + 1)^3 * cellCapacity)` の分岐であるため、radius 1とデフォルトのcapacity 32が、意図されたリアルタイムのベースラインである(パーティクルあたり最大864回の保持スロットテスト)。固定容量は、メモリとシェーダーの作業を予測可能にするが、過剰に満たされたセルを、勝ったアトミック予約の方へ偏らせる。アトミックな挿入順序はデバイス間または実行間で安定していない。容量未満でも、保持slot順はアトミック予約順であるため、浮動小数点の近傍累積は下位bitが異なり得る。`cellCapacity` を超えると、保持される近傍集合とoverflow勝者が実質的に異なり得る。十分な `cellCapacity` の余裕は損失のある勝者選択を避けるが、正規のbucket順は保証しない。bitwise replayを必要とする順序依存シミュレーションには、将来の決定的neighbor modeが必要である。したがってオーバーフローは、隠されたリサイズやCPUフォールバックではなく、明示的な損失のある近似である。割り当ての前に、最大のスロット/スナップショットのバインディングがアクティブなバッファ上限と照合される。過大な宣言は、黙って解像度や容量を削減するのではなく、`NACHI_NEIGHBOR_GRID_STORAGE_LIMIT_EXCEEDED` を報告する。

`boids()` は、古典的な分離(separation)、整列(alignment)、結合(cohesion)を実装する。整列は近傍の平均速度へ向けて操舵し、結合は近傍の平均位置へ向けて操舵し、分離は `separationRadius` 内の逆二乗変位を累積する。合成された加速度は `maxAcceleration` によって制限され、エミッターローカルなデルタ時間で速度へ積分される。`/m12-neighbors/` の制御群は近傍モジュールを省略し、ライブな群れにおいて実質的により小さい分散半径にゲートするため、ゼロ化された/no-opの近傍読み取りは合格し得ない。

`pbdDistanceConstraint({ distance, stiffness, iterations })` は、最小限のペア距離のpush-outである。各
イテレーションは、clear、bucket/snapshot、そして別個のconstraintの送信を実行する。各パーティクルは、
イテレーションのスナップショットから各貫通補正の半分を独立に計算し、着信するすべての補正を平均し、
`stiffness`(デフォルト1)を適用する。これはインプレースのGauss-Seidel解法ではなく、Jacobi方式である。
`iterations` はデフォルトで1である。Three r185は、1つのコンピュートパス内の依存ディスパッチ間でグリッド
全体のバリアを供給できないため、イテレーション間での再構築と送信は必須である。1回から数回のイテレーション
はソフトな重なりを除去するが、有限のイテレーション数、平均化、オーバーフローの破棄、`stiffness < 1` に
より、剛性は近似的なものとなる。V1には、逆質量重み付け、ピン留めされたパーティクル、compliance/XPBD、
グラフ彩色、摩擦、形状マッチング、密な接触グラフに対する収束の保証はない。厳密に一致するペアもまた、v1
が決定的なジッター方向を注入しないため、一致したままとなる。

PBDの送信によって分離されたイテレーションは、エミッターの通常のUpdateモジュールカーネルより前に実行される。したがって、見た目には `integrate -> pbdDistanceConstraint` と読める作成されたスタックであっても、それでも制約解法が最初に実行され、通常の積分/更新モジュールが2番目に実行される。v1は、この分離されたカーネルの境界をまたいだ作成者スタックの織り込み(interleaving)を約束しない。

Grid2D、Grid3D、NeighborGridのストレージは、プールされないすべてのエフェクトスポーンごとに新しく割り当てられる。現在のエフェクトプールはエミッターのカーネルのみを再利用する。これらのデータインターフェースバッファのプーリングは将来の最適化であり、同じクリア/初期化とFIFOの可視性の契約を保持しなければならない。

Neighbor gridには、ストレージのアトミクスと任意のインデックス付き読み取りが必要である。WebGL2の具現化は
`NACHI_NEIGHBOR_GRID_WEBGL2_UNSUPPORTED` を報告する。CPUのboidsソルバーやtransform-feedbackの近似を選択する
ことは決してない。Neighbor gridの宣言は、当時のenvelope v1でclosed field (`kind`、`version`、`resolution`、
`cellCapacity`、`cellSize`、`origin`)とともに導入された。現行serializationはそれらの宣言fieldを変更せず
canonical envelope v2へ格納し、default v1→v2 migrationはpayload fieldを追加・移行しない。H2-5は既存
`origin`値のruntime解釈をworld-spaceからemitter-localへ変更する。絶対originと変換済みemitterを意図的に
組み合わせたcode/assetは、originをemitter frameへ移行しなければならない。

## 11. シリアライゼーションとレジストリ

### 11.1 正規のドキュメント

アセットのエンベロープは、モジュールとは独立にバージョン管理される。

```ts
interface EffectAssetDocumentV2 {
  format: 'nachi-effect';
  version: 2;
  effect: JsonValue;
}
```

`@nachi-vfx/format` は `effectAssetSchemaV2`、`EffectAssetDocumentV2`型（現行
`EffectAssetDocument` aliasでもある）、`serializeEffect(definition)`、`loadEffect(json)`をexportする。
serializationは正規化された`defineEffect()`出力に対して動作し、上記canonical envelopeだけを出力する。loadは
native v2またはhistorical v1 object（および各JSON文字列表現）を受け付け、設定されたmigration境界を適用して
通常のeagerな`EffectDefinition`を返す。部分的に検証されたdocumentを実行可能な定義として返すことは決してない。
`effectAssetSchemaV1`と`EffectAssetDocumentV1`はinspectionと明示migration tooling向けのcompatibility exportで
あり、現行serializer契約ではない。

グリッドステージのソース文字列は、組み込みの `core/grid2d-*` または `core/grid3d-*` の語彙に対してのみ解決される。シリアライズされたカスタムの `grid2d-function-ref` または `grid3d-function-ref` は、`LoadEffectOptions.grid2DStageRegistry` または `grid3DStageRegistry` を通じて厳密に解決されなければならない。未知の組み込みは `NACHI_ASSET_GRID_STAGE_SOURCE_UNKNOWN` を報告し、登録の欠落は、コアの具現化に到達するのではなく、検証/ロードの過程で `NACHI_ASSET_GRID_STAGE_FUNCTION_UNRESOLVED` を報告する。

現行v2 validationはclosed envelope、effect/emitter/module/access/parameter/attribute/timeline/scalability構造、
既知のtagged value generatorとresource referenceを検証する。format-owned構造の未知fieldはerrorである。module
configのvalidationはversion-awareであり、renderer@2等のowned built-inはclosed nested shapeを使う。それ以外の
registry所有configは、そのvalidator向けの拡張可能なJSON objectのままである。構造validation後、loaderは既存の
`defineEmitter()`と`defineEffect()` normalizationを実行するため、attribute、lifecycle、parameter composition、
M9 merge invariantは単一の契約のままとなる。timeline target/valueのruntime validationは引き続き
`@nachi-vfx/timeline`が所有し、format layerはその責任をcoreへ移さない。

シリアライゼーションは、正規化されたヘルパーの出力、モジュールのバージョン、`User.*` のデフォルト値、不変のカーブ/グラデーションのタプル、(`#RGBA` と `#RRGGBBAA` を含む)カラー文字列、セマンティックなモジュール/タイムラインの配列順序、すべてのレンダー/品質/スケーラビリティのデータを保持する。オブジェクトのキー順序は意味を持たない。これは、`undefined`、非有限の数値、bigint、シンボル、関数、シンボルキー、循環グラフ、クラスインスタンス、Three.js/DOM/GPUオブジェクト、その他の非JSON値を拒否する。診断は、フェーズ `serialize` または `deserialize` と正確なデータパスを伴う、安定した `NACHI_ASSET_*` プレフィックスを使用する。ロードは、未知のフィールド、バージョン、アクション、参照、型不一致を黙って破棄することはない。JSONを保持する設定は、256のネストされたコンテナに制限され、パースされたオブジェクトのスパース配列は拒否されるため、ホストの再帰エラーと配列の穴も `NACHI_ASSET_*` の診断を通じて報告される。

asset envelope versionはmodule versionから独立している。default registryは1-stepのv1→v2 envelope-only migrationを
行う。top-level `version`だけを変更し、inputをmutationせず、effect payloadと全module versionを変更しない。
従ってrenderer module-v1 recordはmigration後もmodule v1である。
`EffectAssetMigrationRegistry.register(fromVersion, toVersion, migrate)`は明示custom-migration境界であり、各stepは
`version`が登録先と厳密に一致する`nachi-effect` envelopeを返さなければならない。その後のvalidationはmigrated
document内のactual module versionを使うため、custom migrationがrenderer@2を生成すればstrict config契約を満たす
必要がある。v1 envelopeはactual render slotへrenderer@2を予約できず、v1-only old readerはcanonical envelope v2を
compile前に拒否する。欠落path、競合登録、cycle、不正resultはdiagnosticである。semver推論とbest-effort downgradeは
存在しない。renderer固有guardとsafe rejectionの詳細はRFC 006 §9が所有する。

### 11.2 リソース

テクスチャ、メッシュ、SDF、ベクトル場、VATデータ、その他の外部アセットは、次のようなシリアライズ可能な参照を使用する。

```ts
interface AssetRef<AssetType extends string> {
  kind: 'asset-ref';
  assetType: AssetType;
  uri: string;
}

type TextureRef = AssetRef<'texture'>;
type FieldRef = AssetRef<'vector-field'>;
type MeshRef = AssetRef<'mesh'>;
type SdfRef = AssetRef<'sdf'>;
```

ランタイムは、アセットリゾルバを通じて参照を解決する。生の `THREE.Texture` や `Object3D` の値は、一時的なコード専用のエフェクトのためにラップされてもよいが、シリアライゼーションは、それらが参照に置き換えられるまで拒否しなければならない。

M9の `timeline/mesh-fx` プレースホルダも参照の境界である。JSONは、そのリソースIDとdurationのみを格納する。`loadEffect()` の後、`bindMeshFxResources()` はそれらのIDをライブなThree.jsのメッシュへ解決し、タイムラインランタイムの `WeakMap` を通じてそれらを帯域外でアタッチする。これは再生の前に必要であるが、JSONドキュメントを変更したり充実させたりすることはない。欠落したリソースは、明示的なタイムライン診断である。Golden #5はこの経路を使用し、JSONからロードされた定義が通常のタイムラインとGPUレンダラーを駆動する一方、同一シードのコード定義が等価性の対照を供給する。

`$` で始まる外部のアセット/ドキュメントIDは、ランタイムが所有するアイデンティティのために予約されている。特に、`$root` はローダーのルートドキュメントを示し、呼び出し元が供給するアセットIDとして使用してはならない。

ASCIIのFGAベクトル場は、レンダラーに依存しない `parseFga()` ヘルパーによってパースされる。ヘッダーは、
3つの正の整数の解像度値の後に、最小および最大のvec3境界が続く。ちょうど
`resolution.x * resolution.y * resolution.z` 個のvec3サンプルが、X優先の順序で続く。
`vectorField({ field, strength, tiling })` は、そのシリアライズ可能な `FieldRef` のみを格納する。
カーネルの具現化の過程で、バックエンドのリゾルバはその参照を境界を意識した3Dテクスチャへマップする。
パーティクルのワールド位置はその境界によって正規化される。FGAのサンプルはコーナー整列である。正規化座標
ゼロは最初のテクセルの中心をアドレス指定し、1は最後をアドレス指定する。軸ごとの変換
`u * (resolution - 1) / resolution + 0.5 / resolution` を使用する。デフォルトは、この変換の前に正規化
された位置をクランプする。`tiling: true` は座標を無制限のままとし、最後/最初のテクセルの継ぎ目をまたぐ
補間を含む、繰り返しサンプラーのラッピングを使用する。これは、`fract()` の後にクランプされたサンプルを
続けるという形での繰り返しを実装するものではない。Float32のフィルタリング可能なバックエンドはトリリニア
サンプリングを使用する。他のバックエンドは最近傍サンプリングへフォールバックする。サンプルは加速度として
扱われ、`Emitter.deltaTime` で積分される。

`MeshRef` のサーフェスリソースは、変形された三角形位置テクスチャと正規化された面積CDFテクスチャを含む。`SdfRef` のリソースは、float距離サンプルと境界/解像度のメタデータを含む。両方ともバックエンド統合によって解決され、エフェクト定義内ではシリアライズ可能な参照のままである。

### 11.3 アセット参照によるエミッター継承

積極的(eager)なcode authoringは、引き続き`defineEmitter(base, overrides)`を使う。canonical v2 documentは
次のelement nodeを受け付け、同じnodeを持つhistorical v1 documentはdefault envelope-only migrationを通して
変更されずここへ到達する。

```ts
interface EmitterAssetExtension {
  kind: 'emitter-extends';
  extends: string; // "#base" or "library://effects#base"
  overrides: EmitterOverrideConfig;
}
```

この参照は、ちょうど1つの `#` を含む。`#element` は現在のドキュメント内のエミッターをアドレス指定し、
一方 `asset-id#element` は同期的な `LoadEffectOptions.resolveAsset(assetId)` のドキュメントリゾルバを
呼び出す。エフェクトの要素キー(エミッター名を含む)は `#` を含むことができない。これにより、`#element`
のようなローカルな参照が曖昧でなくなり、外部リゾルバへ誤って経路指定されることを防ぐ。ローダーは完全な
チェーンを解決し、その後 `defineEmitter(base, overrides)` を呼び出すため、capacity/spawnのchild-wins
挙動、ライフサイクル/スキーマのマージ、ラベル/インデックスのスタックの置き換え/削除/順序、すべてのM9
診断は、その意味を保持する。ロードされた `EffectDefinition` は、結果として得られる積極的なエミッターの
みを含む。プロトタイプ、参照ノード、親のアイデンティティがコンパイルに入ることはない。欠落したリゾルバ/
アセット/要素、エミッターでないベース、無効な参照構文、ローカルまたはドキュメントをまたぐ循環は、
`NACHI_ASSET_*` のデシリアライズエラーである。

### 11.4 実行可能な登録

実行可能なレジストリのカテゴリは3つある。

1. モジュールコンパイラの実装(`type` + `version`)
2. TSLファクトリ(`id` + `version`)
3. JavaScriptのコールバック/統合コマンド(`id` + `version`)

この3つのカテゴリは、それぞれ別個の参照型を使用する。モジュールの検索は `(type, version)` を使用し、TSLは `TslFunctionRef` を使用し、JavaScript/統合コールバックは `CallbackRef` を使用する。`VfxRegistry` は、それに応じて `resolveModule`、`resolveTsl`、`resolveCallback` を公開する。あるカテゴリの参照が、別のカテゴリを通じて解決されることはない。

JSONは参照のみを含む。レジストリはアプリケーション/パッケージのセットアップによって供給され、アセットに埋め込まれることはない。デシリアライゼーションは、すべての登録が存在する前にパースされたドキュメントを生成することがあるが、コンパイルは未解決の必須登録があると成功できない。

`tslModule(functionRef)` と `tslModule(defineTslFunction(id, factory, version))` はどちらも、モジュールソースを同じ `{ kind: 'function-ref', id, version }` というJSON値としてシリアライズする。登録形式は、そのファクトリをモジュール上の列挙不可能な作成メタデータとして保持することがあるが、そのメタデータはシリアライザによって巡回されたり出力されたりすることはない。

同一ID/versionの登録競合は、implementation identityが厳密に同じregistration objectである場合を除きerrorで
ある。version lookupはexactでsemver-compatibleではない。version 2だけが登録されている場合、version 1への
requestは非互換である。envelope-only migrationはこのreference versionを変更しない。referenceを意図的に
upgradeする明示custom migrationは`@nachi-vfx/format`に属し、そのoutputはexactなregistered versionを参照する
必要がある。

M12バッチ1は、既存の `KernelModuleRegistry` コンパイル境界を維持しながら、シリアライゼーションと実行可能な参照の保持を導入する。そのレジストリは、すでに同じモジュールカテゴリの競合ルール(同一の登録オブジェクトは冪等であるが、同一のtype/versionでの置き換えはエラーである)と厳密なバージョン検索を強制している。より広範なマルチカテゴリの `VfxRegistry` ファサードは、JSONのパースのみによって暗示されるものではなく、後のM12のエコシステム作業のままである。

### 11.5 インライン関数のポリシー

`tslModule(callback)` と、将来のコールバックを受け取る作成用ヘルパーは、登録されるまでシリアライズ不可能としてマークされる。コールバックは、列挙不可能な作成メタデータとして保持されることがあるが、それはJSONではない。`serializeEffect()` はその正確なパスを報告し、登録APIを提案する。

effect document内でのsimulation-cache metadata/binaryの埋め込みまたは参照は、将来のasset-format versionで
ある。historical envelope v1にもcanonical envelope v2にも含まれず、どちらもeffect-definition JSONだけを含む。

## 12. エラーモデル

### 12.1 構造化された診断

コンパイラおよびローダーの操作は、次を伴う診断を返すか投げる。

```ts
interface VfxDiagnostic {
  code: string;
  severity: 'error' | 'warning';
  phase: 'compile' | 'deserialize' | 'runtime' | 'serialize';
  message: string;
  path?: string;
  hint?: string;
  context?: {
    emitterPath: string;
    kernel:
      | 'compaction'
      | 'event'
      | 'init'
      | 'neighbor-grid'
      | 'sim-stage'
      | 'sort'
      | 'spawn'
      | 'unknown'
      | 'update';
    neighborGrid?: {
      key: string;
      inBounds: number;
      outOfBounds: number;
      total: number;
      outOfBoundsRatio: number;
      cellCapacity: number;
      cellSize: number;
      origin: [number, number, number];
      resolution: [number, number, number];
    };
  };
}
```

パブリックな診断コードは、テストとツールにとって十分に安定している。人間向けのメッセージは、破壊的リリースなしに改善されてよい。

### 12.2 コンパイル時の検証

以下は、GPU作業が送信される前のコンパイルエラーである。

- 未知のモジュール、関数、コールバック、アセット、属性、パラメータ、イベントターゲット、タイムラインターゲット
- 登録された移行のないバージョンの不一致
- 無効なcapacity、ライフサイクル、カーブ、グラデーション、レンダラーの設定
- 属性の型/デフォルト値の不一致、または互換性のない組み込みの再宣言
- 宣言されていない必須の読み取り、不正なステージの書き込み、読み書きのトレース不一致、依存関係の循環
- フォールバックのない、必要とされるバックエンドのケイパビリティ
- 非決定的またはサポートされていないTSLグラフ
- シリアライゼーション時の未解決のインライン関数
- 既知のデバイス上限を超えるリソースレイアウト

検証は、最初のモジュールで失敗するのではなく、独立した診断を蓄積することが望ましい。部分的にコンパイルされたエフェクトがアクティブなスケジューラに入ることはない。

作成用ファクトリは、エンジンに依存しない静的な設定エラーを早期に検出する。同期的に `VfxDiagnosticError` をthrowし、コンパイル時のレジストリ検証と同じバリデータを使用する。JSONからロードされたモジュールや手動構築されたモジュールデータはファクトリを通らないため、コンパイル時の検査は必ず維持される。エミッタースキーマ、別のエフェクト要素、バックエンド機能、デバイス上限、またはTSLのトレース実行を必要とする制約は、compile専用のままである。

`Particles.lifetime` が割り当てられ、`Particles.age` が存在せず、かつlifecycleが宣言されていない場合、コンパイラは `NACHI_LIFETIME_WITHOUT_AGE` warningを発する。文言はlifecycle宣言または明示的なage書き込みの追加を案内する。宣言されたバッファレイアウトを黙って変更しないため、コンパイラは `age` を暗黙に注入しない。

spawn時のコンパイル診断とカーネルビルド診断は、`VfxSystemOptions.onBuildDiagnostic` にも報告される。プロパティを省略した場合、各errorは `console.error` の1回の呼び出し、各warningは `console.warn` の1回の呼び出しとして出力される。`null` はこの報告を無効化し、関数は既定処理を差し替える。ハンドラがthrowした場合、その例外は封じ込められ、`NACHI_BUILD_DIAGNOSTIC_HANDLER_FAILED` warningとして一度だけ記録される。このwarningを失敗したハンドラへ再帰的に渡すことはない。このフックは通常のランタイム診断を報告しない。

`VfxSystemOptions.onRuntimeDiagnostic` は、将来互換の対応するruntime配送seamである。省略時の既定console
formatter、`null` opt-out、置換handlerの意味は同じである。handlerがthrowしてもupdate/captureをrejectせず、
所有instanceへ `NACHI_RUNTIME_DIAGNOSTIC_HANDLER_FAILED` を1回だけ記録し、それを再帰配送しない。H2-5で
このseamへ接続するのは `NACHI_NEIGHBOR_GRID_OUT_OF_BOUNDS_DOMINANT` だけである。device loss、GPU submission、
attachment、light limit、spawn/event overflow、および他の既存runtime sourceは、H2-12の配送auditが接続するまで
instanceまたはsource localのままである。public seamは、それらがすでに配送されることを意味しない。

### 12.3 ランタイムの検証

ランタイムエラーは、コンパイル時に知ることができない状態のために予約されている。

- デバイスロスまたはレンダラー/バックエンドの除去
- 参照が有効であった後のアセットロードの失敗
- アプリケーションコードからの、型が無効または不変な `User.*` パラメータへの書き込み
- システムの破棄後のスポーン、または解放されたインスタンスの使用
- ランタイムのデバイス上限の変更と外部統合の失敗

デバイスロスおよびその他のランタイムの失敗は、影響を受けるすべての `EffectInstance.state` を `'error'` へ遷移させ、フェーズ `'runtime'` を伴う `VfxDiagnostic` を発する。それらは、もっともらしいFPSを報告し続けたり、セマンティックな挙動を黙って切り替えたりしてはならない。開発ビルドは診断パスを含むことが望ましく、本番ビルドはコンパクトなコードを使用してもよい。

`NACHI_GPU_SUBMISSION_FAILED` は常に `context.emitterPath` と `context.kernel` を伴う。`emitterPath` には送信元のエミッターまたはグリッド要素のpathを格納し、`path` にも同じ値を格納する。messageには両方のコンテキスト値を含める。カーネル種別はspawn、init、update、event、compaction、sort、neighbor-grid、sim-stage、およびunknownフォールバックを区別し、レンダラー例外を原因となった要素の送信まで追跡可能にする。

解放されたインスタンスのメソッド(`setParameter`、`setTimeScale`、`setTransform`、`applyHitStop`、`stop`、エミッターの検査)は `NACHI_INSTANCE_RELEASED` を投げる。`release()` 自体は冪等のままである。デバイスロスは `VFXSystem` によってラッチされる。既存および以降にスポーンされるインスタンスは、二次的な具現化/送信エラーの代わりに、直接 `NACHI_DEVICE_LOST` を受け取る。

の一部ではない。

Threeの描画具現化は、リソースを所有する境界である。具現化されたすべての描画は、そのエミッターカーネルとともに登録されなければならず、これにより、ランタイムの可視性、安定したレンダー順序、間接描画のプロファイリングが、単一の権威あるセットを使用する。描画を再具現化するアプリケーションは、既存のThreeオブジェクトを再利用するか、以前のオブジェクトを明示的に破棄/非具現化しなければならない。カーネルがエフェクトプールに入る前、およびカーネルが永続的に解放される際、Threeランタイムはそれらのオブジェクトの登録を解除し、その所有するジオメトリ、マテリアル、インスタンス属性、補助ストレージ、間接属性を破棄しなければならない。プールされたカーネルは、そのクリーンアップの後にのみ再び具現化されてよい。アプリケーションは、解放されたインスタンスから保持された描画のレンダリングを続けてはならない。

各draw登録は、既定値 `true` の `userVisible` を保持する。Threeオブジェクトの最終可視性は
`runtimeVisible && userVisible` である。runtime可視性はsignificance culling、complete、stop、pooling
遷移の所有権を維持する一方、`setUserVisible(false)` はそれらの遷移をまたぐ永続的なユーザー非表示であり、
`setUserVisible(true)` は現在のruntime値へ制御を戻す。sprite、mesh、decal、ribbon、light-poolの各
materialize結果は `setUserVisible(visible)` を公開する。light drawではpool groupへ適用し、個々の
PointLightは引き続きvisibleのまま、非アクティブslotを強度0で表す。次のruntime遷移で上書きされ得るため、
アプリケーションは永続オーバーライドとして `Object3D.visible` を直接操作してはならない。

### 13.1 `@nachi-vfx/tsl-kit` の単体境界

`@nachi-vfx/tsl-kit` はフレームワーク非依存の導入エントリポイントである。これは `@nachi-vfx/core` や他のいかなるnachiパッケージもインポートしてはならない。すべてのシェーダーヘルパーは、宣言的なJavaScriptの設定オブジェクトを受け取り、通常の `NodeMaterial` へ直接割り当て可能なThree.jsのTSLノードを返す。これらのシェーダー作成用オブジェクトはライブな `Texture` や `Node` のインスタンスを含むことがあるため、§2.2で規定されるシリアライズされたモジュール設定ではない。`mesh-fx` またはコアの登録済みファクトリが、アセット参照からこれらのライブな入力への、後のいかなる変換も所有する。

M8バッチ1は、直接的な `three` のピア依存を選択する。M12のリリース準備は、サポートされるピアを、以前の `>=0.185.0 <0.186.0` の範囲から厳密に `0.185.1` へ絞り込む。実装は、1つのパッケージプライベートなアダプターモジュールを通じて `three/tsl` をインポートするが、第二のTSL抽象やラッパーノード型を公開することはない。これにより、プレーンなThree.jsユーザーのための直接的な相互運用性を保持しながら、TSL APIの変動のための1つの移行ポイントを保持する。アプリケーションはこのピアを供給しなければならない。パッケージは第二のThree.jsランタイムをバンドルしてはならない。

標準の `polarUV(config)` はこのパッケージ内の低レベルのシェーダー関数であり、即座にTSLの `vec2` ノードを返す。コアの作成用型におけるノーススターの `PolarUvFactory` は、別個の、将来の `mesh-fx`/`fxMaterial` 層である。それはシリアライズ可能なUV作成用データを返し、`.flow()` のようなフルーエントな合成を公開する。M8のマテリアルコンパイラは、そのデータを `polarUV()` と `uvFlow()` の呼び出しへと下位変換する。単体の関数がフルーエントなビルダーの戻り値の型を獲得することはなく、この2つの層は構造的に互換にされることはない。

すべてのクロックは明示的である。`uvFlow`、`distortionUV`、`flowMap` は、`time` として数値またはTSLノードを要求する。これらは、Threeのグローバルな `time` ノードやウォールクロック状態を暗黙的に読み取ってはならない。nachiの呼び出し元はエフェクトローカルな時間をバインドし、一方単体の呼び出し元は一般的に `uniform(0)` を供給する。数値定数の検証失敗は、安定したコードとパラメータパスを伴う `TslKitDiagnosticError` を投げる。動的なノードの範囲は、呼び出し元のランタイムの責任のままである。

### 13.2 `@nachi-vfx/tsl-kit` の数学と規約

すべてのUVはThree.jsのジオメトリUV規約を使用する。角度と回転の入力はラジアンである。色は作業空間のTSLカラーである。テクスチャのラップ/フィルタモードは、供給されたThree.jsのテクスチャによって所有されたままである。UVヘルパーは、アルゴリズムが本質的に周期的な位相を要求しない限り `fract` を適用しない。

入力UV `u`、時間 `t`、速度 `s`、中心 `c`、反時計回り(CCW)の回転 `r` について、UVヘルパーは次の通りである。

- `uvFlow`: `u' = u + s t`。結果は意図的にラップされない。
- `polarUV`: `p = u - c` とし、`q = (p.x cos(r) - p.y sin(r), p.x sin(r) + p.y cos(r))` とする。返却
  される `vec2` は `(fract(atan2(q.y, q.x) / 2π + 1/2), length(q))` である。したがって、正のx軸は
  `0.5`、正のy軸は `0.75` であり、正の回転は反時計回りである。実装は、そのパブリックな規約を可視の
  ままにするために、この式を展開する。Three r185のvec2の `rotate()` の経路も標準的なCCWである。知られ
  ている転置/向きの罠は、vec3/オイラーの経路と、表示される行のレイアウトがコンストラクタの列のレイアウト
  と誤認される、手書きの行列リテラルに限られる。それらの経路でのいかなる置き換えも、依然としてGPUの方向
  テストを必要とする。
- `distortionUV`: `n = noise(u + speed t).rg` をサンプリングし、`d = 2n - 1` をデコードし、
  `u' = u + strength d` を返す。

単位法線 `N`、表面からカメラへの単位方向 `V`、指数 `p > 0`、色 `C` について、`fresnel` は `C (1 - clamp(dot(N,V), 0, 1))^p` を返す。これは単体の色付きマスクである。`rimLight` は、意図的に同義語ではなく合成プリセットである。ベースカラー `B`、ライトカラー `L`、非負の強度 `I` について、`B + I L (1 - clamp(dot(N,V), 0, 1))^p` を返す。

`dissolve` はスカラーノイズ `n` をサンプリングし、純粋な閾値入力 `h`(数値またはノード)を受け取り、`vec4(E, A)` を返す。ここで `A = step(h,n)` であり `E = edgeColor * (step(h,n) - step(h + edgeWidth,n))` である。`A` は二値のカバレッジである。これを `opacityNode` へ割り当て、正のマテリアル `alphaTest` を設定することで、フラグメントのカットアウトが実行される。`E` は加算的/エミッシブなエッジの寄与である。lifetimeの評価は、意図的にこのパッケージの外にある。

`flowMap` は標準的な二相サンプリングを実行する。`f = strength (2 flowTexture(u).rg - 1)` をデコードし、`p0 = fract(speed t)`、`p1 = fract(p0 + 1/2)`、`u0 = u - f p0`、`u1 = u - f p1`、`w0 = abs(2p0 - 1)` を定義する。返却されるRGBAノードは `w0 map(u0) + (1 - w0) map(u1)` である。半周期のオフセットと相補的な三角の重みは、正確に2回のベースマップサンプルと1回のフローマップサンプルを要求しながら、リセットの継ぎ目を隠す。

`@nachi-vfx/tsl-kit/math` は、これらの座標/位相ルールのためのCPUミラーを公開する。単体テストはそれらを独立した数値の参照として使用する。`/m8-tslkit/` は、加えて、オフスクリーンのレンダーターゲットを通じて、コンパイルされたGPU結果を測定する。

### 13.3 `@nachi-vfx/mesh-fx` の単体境界とジオメトリの規約

M8バッチ2は、`@nachi-vfx/mesh-fx` をコアに依存しないThree.jsパッケージにする。これは、そのシェーダー部分の
合成対象として `@nachi-vfx/tsl-kit` に依存し、まったく同じThree `0.185.1` のピアを持つ。これは
`@nachi-vfx/core` をインポートせず、シミュレーション状態を割り当てず、GPU作業を送信せず、グローバルな
クロックを読み取らない。パブリックなメッシュファクトリは、通常の `THREE.Mesh` インスタンスを返す。
それぞれが安定した `userData.nachiMeshFx = { kind, version: 1 }` のメタデータを持つため、M9の
`defineEffect()` ランタイムは、このパッケージをコアのライフサイクル型に依存させることなく、ノースター
の `slashArc({...})` の呼び出しによって返されるオブジェクトを適合させることができる。メッシュの構築を
所有するアプリケーションは、代わりに対応する `create*Geometry()` のエクスポートを使用してもよい。

Mesh-fxの診断は、単一の `NACHI_MESH_FX_*` 名前空間を使用する。単体の検証は `NACHI_MESH_FX_INVALID_PARAMETER` と `NACHI_MESH_FX_TEXTURE_REQUIRED` を発する。タイムラインのリソース/ライフサイクル統合は `NACHI_MESH_FX_REFERENCE_INVALID`、`NACHI_MESH_FX_RESOURCE_UNRESOLVED`、`NACHI_MESH_FX_MATERIAL_CLONE_UNSUPPORTED`、`NACHI_MESH_FX_TIME_BINDING_UNSUPPORTED`、`NACHI_MESH_FX_OPACITY_BINDING_CONFLICT` を発する。`NACHI_VAT_FRAME_RANGE` と残りのVAT固有の診断は、その `NACHI_VAT_*` 名前空間を保持する。

すべてのジオメトリは、インデックス付きの三角形リストの `BufferGeometry` である。位置、法線、UVの属性数は等しい。継ぎ目と同心境界は、溶接されるのではなく意図的に複製され、前面の巻き方向は、公開された外向きの法線と一致する。セグメントの値は作成者が制御する整数であり、割り当ての前に静的に診断される。

| ファクトリ    | 座標とトポロジー                                                                                   | UVと法線の契約                                                                                                                                    | 頂点数/インデックス数 |
| ------------- | -------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| `slashArc`    | XY平面のファンストリップ、+X中心の度数スイープ、`radius > innerRadius`、任意の対称的な端点テーパー | `uv=(スイープ分率, 半径方向分率)`、法線+Z                                                                                                         | `2(s+1)` / `6s`       |
| `ring`        | 角度方向の継ぎ目が複製されたXY平面の円環                                                           | `uv=(角度分率, 半径方向分率)`、法線+Z                                                                                                             | `2(s+1)` / `6s`       |
| `cylinder`    | Y軸方向の側面;両端は開いている                                                                     | `uv=(円周分率, 高さ分率)`、解析的な単位半径方向法線                                                                                               | `(r+1)(h+1)` / `6rh`  |
| `cone`        | Y軸方向の側面;底面は開いている;頂点は角度方向の列ごとに一度複製される                              | cylinderと同じUV、解析的な単位傾斜法線 `(cosθ, radius/height, sinθ)`                                                                              | `(r+1)(h+1)` / `6rh`  |
| `magicCircle` | 複製された同心の半径方向バンドに分割されたXY平面のディスク                                         | tsl-kitの `polarUV` 向けの主要な `uv=(x/(2R)+1/2,y/(2R)+1/2)`;`uv1=(角度分率, バンドローカルな半径)` はバンドごとに1つのアイランドを与える;法線+Z | `2k(s+1)` / `6ks`     |

ここで `s` は平面の角度方向セグメント数、`r`/`h` は側面の半径方向/高さ方向セグメント数、`k` はmagic
circleのリング数である。slashの `angle` と `rotation` の入力は、PLANのノーススターである `angle: 140`
の表面を保存するために度数(degrees)を使用する。ring/cylinder/cone/magic-circleの `thetaStart` は
Three.jsに従いラジアンを使用する。Slashのテーパーは、その中心半径を保存しながら、半幅に
`1 - taper * abs(2u - 1)` を乗算する。テーパーが1の場合、各先端で幅がゼロに達する。

magic-circleの主要なUVは、境界の頂点が複製されているにもかかわらず、半径方向のバンドをまたいで連続している。これは、低レベルの直交座標から極座標への変換によって消費されるチャンネルである。`uv1` はすべての同心バンドを分離し、角度方向の継ぎ目を複製し、極座標の主要チャンネルを変更することなく、バンドローカルなテクスチャや将来のマテリアルグループの割り当てを許容する。中心バンドは一致する内側の頂点を持つ。この意図的な退化したエッジは、同じ決定的なカウントを保持し、三角形のラスタライゼーションによって破棄される。

### 13.4 `fxMaterial` の作成、時間、診断

`@nachi-vfx/mesh-fx` は、フルーエントなノーススターの作成形式を所有する。その `polarUV(config)` は不変の作成用データを返し、`.flow({ speed })` は不変のフローステージを追加する。`fxMaterial()` は、そのデータを、作成者の順序で、単体の `@nachi-vfx/tsl-kit` の `polarUV()` と `uvFlow()` のノード関数へ下位変換する。これは、§13.1で確立された層の分離を実装する。低レベルのtsl-kitの戻り値の型はTSLノードのままであり、フルーエントにはならない。

その主要なUVが直交座標のままでなければならないメッシュ(特に開いた円筒ビーム)については、`mesh-fx` は不変の `uvFlow({ speed })` 作成用データも公開する。`fxMaterial()` は、これをジオメトリの変更されていない `uv()` ノードから下位変換する。これは `polarUV().flow()` とは異なる。前者は `(u,v)` を直接移動させ、後者は極座標変換の後に `(angle,radius)` を移動させる。

M8バッチ2は `THREE.MeshBasicNodeMaterial` を返す。任意のベースマップ/色、ディゾルブのエッジとカバレッジ、色付きフレネル、不透明度、深度書き込みの選択、4つのブレンディングモード(`additive`、`alpha`、`multiply`、`premultiplied`)は、1つのノードグラフへ合成される。ディゾルブは、小さな正のalpha-testを伴う不透明度としてその二値カバレッジを使用し、一方そのエッジのRGBとフレネルのRGBはベースの作業空間の色に加算される。テクスチャは、アプリケーションが所有する色空間、ラッピング、フィルタリングを保持する。

`opacity` はスカラー数値またはTSLノードを受け付ける。数値は、省略時の既定値を含め、
`material.fx.opacity` として公開されるパッケージ所有uniformを初期化する。`setOpacity()` は
`[0,1]` の値を検証して書き込み、alpha/additiveを含むブレンディングモードへ即時反映する。
TSLノードは直接合成され、`fx.opacity` はnullになり、`setOpacity()` は代入を拒否する。
この2つの所有モードはAPI契約上排他的である。

`dissolve.uv` は `'static'`、独立した不変の `polarUV()` / `uvFlow()` 作成用データ、または省略を
受け付ける。省略はマテリアルmapのUVを共有して従来のグラフを保ち、`'static'` はジオメトリの未変更な
`uv()` をサンプルする。`dissolve.edgeIntensity` は既定値1の非負な数値発光倍率である。
`dissolve.edgeModulate` の既定は `'none'` であり、`'map'` はベースmapを必須として、エッジRGBへ
作業空間のmap輝度を乗算する。したがって既定値は従来の非変調エッジ出力を保つ。

`dissolve.overLife` は、スカラー/TSLノード、または `[time, value]` のペアの区分線形配列のいずれかを
受け付ける。配列のカーブには、厳密に増加する `[0,1]` の時刻、ゼロと1における端点、`[0,1]` 内の値が
必要である。それらは、フレームごとのJavaScript評価ではなく、ネストされたTSLの選択へ下位変換される。
`time` または `normalizedLife` が省略された場合、`material.fx` は単体での使用のために、パッケージが
所有するuniformと、検証された `setTime()`/`setNormalizedLife()` メソッドを公開する。どちらかの入力が
TSLノードとして供給された場合、それは外部が所有するバインディングとして扱われ、そのセッターは拒否される。
M9のタイムライン統合は、それらのノードバインディングを置き換えない。それはマテリアルを複製し、
ヒットストップとタイムスケールのセマンティクスを含む、パッケージが所有する `material.fx.setTime()` と
`setNormalizedLife()` の制御を通じて、省略された入力を進行させる。`time` を数値として供給することも
同様に固定定数を下位変換し、`setTime()` に代入を拒否させる。したがってタイムラインの作成は、明示的な
`time` を持つ適合されたマテリアルを診断する。必要な書き込み可能なエフェクトローカルなクロックを提供
するには、それを省略する。コアの `curve()` 作成用オブジェクトからmesh-fxのタプル配列への下位変換は、
M9の `@nachi-vfx/timeline` 統合の表面に属する。その `fxMaterial()` ラッパーは、線形補間、厳密に増加する
`[0,1]` のキーと端点をチェックし、その後、不変の `[time,value]` タプルを、コアに依存しない
`@nachi-vfx/mesh-fx` へ渡す。このラッパーはさらに、同じコア `curve()`、タプル配列、数値、TSLノード形式の
`opacityOverLife` を受け付ける。数値とカーブは正規化されたメッシュlifeから区分線形評価され、play時と
タイムライン更新ごとに `fx.setOpacity()` へ書き込まれる。TSLノードはコンパイル時opacityバインディングの
ままである。`opacity` と `opacityOverLife` は同じチャンネルを同時に所有できない。M8はコアの状態を
推論せず、Threeのウォールクロックの `time` ノードを決して読み取らない。

同じパッケージが、エフェクト要素のアダプターを所有する。`slashArc()`、`ring()`、または他のmesh-fx
ファクトリによって返される生の自己記述的なメッシュは、`defineEffect().elements` に直接配置されてよい。
`meshFxElement(mesh, { duration })` は自動的な1秒のlifeを上書きする。生のThreeオブジェクトは、エフェクト
ドキュメントの外に保持される明示的に一時的なリソースであり、そのドキュメントはシリアライズ可能な
`timeline/mesh-fx` のプレースホルダを含む。各エフェクトインスタンスは、メッシュとマテリアルの制御を
複製し、構造的なシーンターゲットを通じてそれを追加し、エフェクトローカルな時間を `fx.time` へ、最新の
`play()` 以降の時間を `fx.normalizedLife` へ書き込む。`stop()`、自然な期限切れ、ループリセット、解放は、
すべてそれをレンダリング不可能にする。タイムラインの所有権は、単体でのmesh-fx利用を保持し、Threeの
ライフサイクルオブジェクトをコアの外に保つ。シリアライズされたプレースホルダから一時的なメッシュ
リソースを復元することは、`@nachi-vfx/format` に属する。

`applyVat()` のmetadataも、mesh-fxをcoreへ依存させずにこのadapterへ参加する。timelineはmaterial clone後に、
graphから到達可能なVAT layerのordered unique unionを再構築しなければならない。position到達性は最後の
absolute-position layerから始まり、すべてoffsetなら最初のlayerから始まる。normal到達性はnormal textureを
指定した最新layerである。どちらかのchannelが選ぶlayerは1つのactive controlを所有し、どちらも選ばないlayerを
clone、返却、駆動してはならない。したがってoffset+normal Aの後にabsolute+normal Bを適用するとBだけを公開し、
Bがnormalを省略すればnormal用Aとposition用Bを元の順序で公開する。timeline `fxMaterial()`では再生成済みの
package graphへ適用する。通常NodeMaterialではThree r185のgeneric cloneが適用済みposition/normal graphを
aliasし得るため、VAT chain直前に取得したnode graphへ戻してから適用する。明示的に作者が指定した
VAT前のposition/normal nodeは外部bindingであり共有を維持する。再適用は単体の合成規約を保つ。すなわちoffset
positionは既存position nodeへ合成し、absolute positionは置換し、VAT normal指定は現在のnormal nodeを置換する。
切断済みbinding metadataはcompactしてよいが、position/normal textureは借用された外部assetであり、timelineの
clone、prepare、error、retained preparation、releaseのどの経路もdisposeしてはならない。

clone可能なVAT metadataは、meshが記録済みmaterialと、最後に該当する`applyVat()`が設定した各channelのrootを
参照し続ける間だけ、positionとnormalについて独立にactiveである。最終positionまたはnormal rootの差し替えは
そのchannelを外部authoringへ戻す。timelineは現在rootを共有clone bindingとして保持し、切り離されたVAT graphを
復活させてはならず、そのgraphだけが所有したclockを返したり駆動したりしてはならない。もう一方のchannelは
独立にactiveなままである。したがって最終position差し替え後も最新VAT normal clockはactiveになり得て、最終
normal差し替えはVAT position clockを無効化しない。両rootまたはmaterialの差し替え後はtimeline所有VAT clockを
残さない。適用間では、normalだけの差し替えは以前のVAT normalを退役させつつ累積position layerを保持し、
position差し替えはその作者rootから新しいVAT chainを開始する。この契約が観測するのはroot identityであり、
同じroot内の変更ではない。所有権を移すcallerはroot参照自体を差し替えなければならない。

`VatConfig.time`を省略すると、sourceと各timeline cloneは別々のVAT uniformを所有する。clone作成時はsource
controlの現在値をsnapshotするが、要素の各`play()`はcloneを0へresetし、以後のtimeline segmentは最新play
からの秒数である`MeshRuntime.elapsed`を書き込む。これはtrack全体の`fx.time`ではない。time scale、speed、
scale 0のpause、hit stop、自然終了、loop reset/replay、新規instanceは要素local lifecycleの意味に従い、
明示stopとすべてのterminal stateは最終値を保持する。内部timeline writerはfiniteかつ非負の時刻だけを
受け付け、非loop要素がVAT clipより長い場合はshader clampで最終frameを保持できるようclip終端以後も許す。
単体`VatControls.setTime()`の厳格な作成range診断は維持する。数値またはTSL nodeの`VatConfig.time`は
non-writableな同じ外部bindingとして各cloneに保持され、timelineは書き込みも診断もしてはならない。

マテリアルcloneは元の作成時configだけを再生するのではなく、spawn時点のスナップショットである。
timelineはパッケージ所有node graphと書き込み可能controlを再生成してから、Three r185の
`MeshBasicMaterial.copy()`状態契約を適用し、両側に対応するowned uniformがある`fx.opacity`、`fx.time`、
`fx.normalizedLife`の現在値を検証済みsetter経由でcopyする。したがって`spawn()`前の`setOpacity()`と、
`side`、depth/stencil/blend/color-write状態、clipping plane、`name`、通常のdeep-JSON copyである
`userData`が保持される。パッケージ所有uniform、clipping plane、user dataはsource/clone間で独立し、
textureなど外部所有リソースはThreeの通常の共有参照規約に従う。source node graph自体はcopyしないため、
clone controlはaliasしない。`opacityOverLife`の排他的なtimeline所有と既存conflict診断は不変である。

`TimelineEffectInstance.setUserVisible(meshKey, visible)`は公開されたmeshごとのuser成分である。cloneが
公開する値は厳密に`mesh.visible = runtimeVisible && userVisible`であり、runtime成分はplay、stop、
自然終了、loop reset/replay、complete、構築失敗cleanup、releaseを所有する。default-trueのuser成分は
それらの遷移とtransform更新を跨いで保持される。user成分をtrueへ戻すことは強制表示ではなく、現在の
runtime値へ制御を戻す。不明/非mesh keyと非boolean値は拒否し、release後は共通のreleased guardを使い、
新しいspawn instanceはdefault-trueの新しいuser成分から始まる。
cloneの`Object3D.visible`への直接代入は永続overrideとしてsupportしない。`setUserVisible()`、play、stop、
自然終了、loop replayによる次のvisibility publishで上書きされ得る。hostはuser所有visibilityに必ず
`setUserVisible()`を使わなければならない。

Three r185の`Mesh.clone()`は`BufferGeometry`参照を共有し、timelineもこの挙動を意図的に保持する。
geometryはinstanceごとのcopyでなく不変借用である。sourceまたはいずれかのcloneからattribute/index/group、
bounding volume、`drawRange`を変更すると全instanceへ波及する。application/resource ownerはsource定義、
active timeline clone、retained preparation objectのすべてが完了するまでgeometryを保持し、その後に限り
`geometry.dispose()`を呼ぶ。timelineのrelease/error cleanupはclone materialをdisposeするが、借用geometryと
source mesh/materialはdisposeしない。
prepareも同じ境界に従う。non-retained成功、preparer失敗、abortでは一時clone materialをdisposeする。
成功した`{ retained: true }`はdetached cloneとmaterialをpreparerへ移譲し、そのownerはretained resourceの
release時にclone materialをdisposeしなければならない。preparerもtimelineも借用geometryはdisposeしない。

クローン時に、timelineはソースメッシュのposition、quaternion、scaleを不変の `authoredLocal` 変換として
取得する。positionと姿勢はエフェクトローカルなオフセットであり、クローン変換は
`effectTransform * authoredLocal` となる。すなわち、エフェクトのquaternionで作者positionを回転してから
エフェクトpositionを加算し、クローンのquaternionはエフェクトquaternionと作者quaternionの積とする。
作者scaleはクローン初期化によって一度だけコピーされる。timelineは `play()`、更新、attachment同期、
`setTransform()` のいずれでもクローンのscaleを上書きせず、アプリケーションコードによる独立したscale
アニメーションを許容する。

静的に知ることができるすべての作成上のエラーは、バッチ1の `TslKitDiagnosticError` の形状(`code`、`path`、`message`)に従って、同期的に `MeshFxDiagnosticError` を投げる。安定したコードは、無効なパラメータと、欠落した/Three製でないテクスチャとを区別する。動的なTSLノードの範囲は、呼び出し元または将来のコンパイラのランタイムの責任のままである。

### 13.5 Blender VATのレイアウトとランタイム

M8バッチ3は、公式のBlender Extensionsカタログを通じて配布され、`flement/VAT-blender-addon` で保守されて
いる `VAT` アドオンを対象とする。これは互換性のソースを正確に特定する。カタログでの配布は、Blender
自身がその拡張機能を作成しているという主張ではない。そのデフォルトのラップされていない画像は、幅が頂点数
に等しく、高さがフレーム数に等しい。水平の1行は、1フレーム分のすべての頂点を含む。フレームは上から下へと
垂直方向に進行する。エクスポータのY反転オプションはその順序を逆転させる。生成されたメッシュは、その頂点
ルックアップのUVセットを `vertex_anim` と名付け、Threeの`glTF`経路はそれを `uv1` として公開する。
Nachiは、そのパーティクルごとのU座標を使用してVATの列を選択し、有効なエクスポータが定義する並べ替えを
保持する。正確なテクスチャ幅対 `geometry.position.count` の検証は、それでも継ぎ目の分割が頂点数を変化
させたインポート済みメッシュを拒否する。`vertexLookup: 'vertex-index'` は、VATと同じ順序で生成された
メッシュのための明示的なフォールバックであり、Blender互換のデフォルトではない。

エクスポータは、オフセットまたは絶対的なオブジェクト空間の位置のいずれかを格納できる。オフセットはランタイムのデフォルトであり、デコードされたサンプルを、マテリアルの既存の `positionNode` または `positionLocal` へ加算する。絶対モードはそれを置き換える。正規化されていない位置出力は、OpenEXRのRGB半精度/全精度floatである。正規化が有効な場合、エクスポータは1つのグローバルな `Min Offset` と `Max Offset` を報告し、エンコードされたゼロは最小値へ、エンコードされた1は最大値へマップされる。したがって `positionEncoding: 'remapped'` は、軸変換の前に、そのスカラーのペアを使用して `p = min + sample * (max - min)` をデコードする。デフォルトの `axisMap: 'xzy'` は、BlenderのZ-upからThreeのY-upへの、エクスポータのThree.jsの例に従うが、単純なY/Zの入れ替えは行列式が-1であるため、カイラリティを変化させてしまう(それは回転ではなく反転である)。行列式が+1の右手系の `(x, z, -y)` 変換には `axisMap: 'xz-y'` を使用する。`xyz` は、すでに変換されたコンテンツのために引き続き利用可能である。ミラーされた `xzy` のデフォルトは、優先される座標系の回転として提示されるのではなく、エクスポータが作成したアセットとの互換性のために保持されている。

v1のランタイムは、位置テクスチャが `NoColorSpace` を伴う `FloatType` または `HalfFloatType` であり、正確な寸法 `(vertexCount, frameCount)` を持ち、mip依存のルックアップがなく、1行あたり1フレームであることを要求する。リマップされたPNGの位置エクスポートは、Blenderアドオンがそれを書き出せるとしても、直接サンプリングされることはない。アプリケーションは、報告されたmin/maxを保持しながら、それをリニアなfloatテクスチャへ変換しなければならない。これは意図的な精度とカラーマネジメントのゲートである。任意の法線テクスチャは、unormのテクスチャであってもよく、デフォルトで `n = normalize(2 sample - 1)` をデコードする。符号付きfloatの法線は明示的な代替である。法線VATのルックアップとデコードはTSLの頂点ステージで発生し、デコードされたオブジェクト空間の法線は、正規化されたビュー空間の法線としてvaryingを通じて渡される前に、モデルビュー法線行列によって変換される。補間されたルックアップUVを通じてフラグメントステージでVATをサンプリングすることは非互換である。

`applyVat(mesh, config)` は、`interpolation: 'nearest' | 'linear'`、両端を含む整数の `frameRange`、`fps`、ループ、フレーム順序、パッケージが所有するかまたは外部のTSL時間入力のいずれかを受け付ける。`frameOrder` はデフォルトで `top-to-bottom` であり、テクスチャのロード後のGPUテクセル行の順序を命名する。互換性プロファイルは、VATテクスチャが `texture.flipY = false` でアップロードされることを前提とする。ローダーがアップロード中に行を反転させる場合、呼び出し元は反対の `frameOrder` を選択しなければならない。別の反転なしにロードされたエクスポータのY反転ファイルは `bottom-to-top` を使用する。

時刻 `t` について、連続的なフレームは `start + fps*t` である。ループはこれを `end-start+1` のmoduloで縮約する。非ループはこれをクランプする。Nearestは最も近いフレームへ丸める。Linearは `floor(frame)` とその次のフレーム(ループモードでのみ末尾を先頭へラップする)をサンプリングし、その後、分数フレームによって混合する。`controls.setTime()` と `setFrame()` は、非ループ、または設定された範囲外の明示的なフレーム値を診断する。外部の動的ノードは、外部の `fxMaterial` の時間/lifeノードとまったく同様に、所有者の責任のままである。

安定したVATの診断は、テクスチャの欠落、非floatの位置テクスチャ、頂点数の不一致、レイアウトの不一致、無効なフレーム範囲、非NodeMaterialの使用を含む。可変の頂点トポロジー、ラップ/ラップアンドクロップのパッキング、複数行フレーム、テクスチャ配列、クリップアトラス、接線/向きVAT、軸ごとのリマップ境界は、v1では互換性がない。アニメーションされる境界は不透明なGPUテクスチャから導出できないため、`applyVat` はデフォルトで視錐台カリングを無効化する。呼び出し元は、保守的なジオメトリの境界を設定した後にのみ、それを再有効化してよい。`/m8-vat/` は、解析的で鏡像化されていないモーションを持つ、生成されたfloatの位置テクスチャとunormの法線テクスチャを使用し、既知の時刻におけるラスタライズされたGPUの頂点位置を、CPUのフレーム補間と比較する。

### 13.6 `@nachi-vfx/post` のパイプライン、ディストーションソース、ブルーム

M10バッチ1は、`@nachi-vfx/post` をコアに依存しないThree r185パッケージにし、M0スパイク4に従って `THREE.RenderPipeline` をその統合ポイントとして固定する。`createPostPipeline()` は、`pass(scene, camera)` をルートとする1つのレンダーパイプライングラフを所有する。設定されたパスの集合は、明示的な `order` の順列によって順序付けられる。`order` が存在しない場合、`distortion -> radialBlur -> bloom` を使用する。設定されたすべてのパスはちょうど一度だけ現れるため、未知、重複、欠落、または存在しないパス名は、暗黙のフォールバックではなく同期的な診断となる。UVドメインのパスは、先行するグラフのためのサンプラークロージャを保持する。ブルームの結果は、そのパステクスチャを通じて公開され、これにより、後のディストーションまたはラジアルブラーが、すでにブルームされた合成を再サンプリングでき、カスタムの順序が、単なるメタデータではなく意味的に実在するものになる。

スクリーンディストーションは、NiagaraのScene Colorディストーションマテリアルのモデルに従う。それは、以前のシーンカラーの合成をオフセットされたスクリーンUVでサンプリングする。シェイクウェーブは、中心、到達半径、リング幅、強度、開始時刻、速度、durationを使用して有効化される入力とともに、半径方向のリングオフセットを追加する。熱かげろうは、羽毛状の矩形領域内に、決定的で滑らかに補間される2つのバリューノイズの成分を追加する。この低周波の連続性は、ピクセルスケールのホワイトノイズではなく、Niagara風の揺らぎを生成する。すべての空間的な値は正規化されたスクリーンUVの単位である。サンプリングは、バックエンド依存のエッジラップを避けるために、インセットされた `[0.001, 0.999]` の境界へクランプされる。

M10のディストーションソースの接続は、パーティクルストレージの読み戻しでも専用のディストーションバッファ
でもなく、**`User.*` を裏付けとするuniformノード**である。ゲームプレイはすでに、ヒット/シェイクウェーブ
のスクリーン中心、開始時刻、半径、幅、強度を所有している。エフェクトは、それらの型付き `User.*` の値を、
`screenDistortion()` へ渡されるTSLのuniformノードへバインドする。数値/タプルのフィールドは、代わりに
`PostPipeline.controls` を通じて書き込み可能な、パッケージが所有するuniformを受け取る。これは、ソースの
数に上限があり、隠れたGPU/CPU同期がゼロであり、Niagaraの通常のパラメータ駆動のディストーションマテリアル
へ直接マップされる。パーティクルの位置を読み戻すことは、ゲームプレイのイベントがすでに知っている値を
重複させながら、レイテンシと同期ポイントを追加してしまうであろう。専用の低解像度のベクトル/ディストー
ションバッファは、数百から数千の独立してシミュレートされる熱パーティクルのための将来の経路であり、
バッチ1では疎なシェイクウェーブのために意図的に割り当てられない。

ディストーションの時間は `fxMaterial` の二層モデルを使用する。`time` を省略すると、パッケージが所有するuniformが作成され、`controls.setTime()` が有効化される。供給された数値は固定定数であり、供給されたTSLノードは外部が所有するものであり、両方ともパッケージのセッターを拒否する。エフェクト/タイムライン統合は、エフェクトローカルなノードを供給するため、ヒットストップとタイムスケールがウォールクロック時間へフォールバックすることは決してない。

ブルームは、Three r185の既存のTSLの `BloomNode` をラップする。その実装はすでに、half-floatのターゲット
への滑らかな輝度閾値の抽出、5段階の徐々にダウンサンプリングされる分離可能なガウシアンブラーのレベル、
WebGPUとWebGL2の両方のバックエンドでの重み付き加算合成を実行している。それを再利用することは、1つの薄い
プリセット/ライフサイクルラッパーのコストで済み、Threeの固定されたTSL境界を追跡する。カスタム実装は、
要求される見た目を変えることなく、少なくとも抽出、ターゲットのリサイズ/破棄、10回のブラーパス、mip
合成、2つのバックエンドの保守を重複させてしまうであろう。したがってこのラッパーは、`soft`、`intense`、
`cinematic` のプリセットと、強度/半径/閾値および解像度スケールのオーバーライドを公開する。

`/m10-post/` はWebGPUのキャンバスを一切プレゼントしない。オフスクリーンのレンダーターゲットを選択し、パイプラインを実行し、Three r185のパディングされたWebGPUの読み戻しをコンパクションし、リニア空間のピクセルを評価し、その後にのみ、視覚アーティファクトのためにバイトを2Dキャンバスへコピーする。読み戻しのチェックはまた、外部が所有する中心、強度、時間のuniformノードを変更し、可視の出力の差異を要求し、`User.*` のGPUバインディング経路をカバーする。これらのパスは深度をサンプリングまたは再構築しないため、forward-z対reverse-zは、それらの数学に変化を与えない。§16の未解決のreverse-z/MSAA/共有深度コピーの制約は、M6の深度衝突とM7のデカール経路、および将来のいかなる深度ゲート付きポストパスにも引き続き適用される。ポスト統合は、その能力を色のみの成功から推論してはならない。

### 13.7 `@nachi-vfx/post` の加重ブレンドOIT

`createWboitPipeline(renderer, transparentScene, camera)` は、2つのアタッチメントを持つThree r185の `RenderTarget` を所有する。`nachiWboitAccum` はRGBA16Fであり、`nachiWboitRevealage` はR8である。透明シーンは、その `mrtNode` が `createWboitOutput(color, alpha, depth)` によって生成される `NodeMaterial` のみを含む。この構造を `outputNode` へ割り当てることは、r185では正しくない。レンダラーのMRTがアクティブな間、`NodeMaterial` はレンダラーのMRTを `material.mrtNode` とマージし、マテリアルのアタッチメント名がコンテキストのデフォルトを上書きする。

リニアな色 `C`、クランプされたカバレッジ `a`、WebGPUの正規化された深度 `z` について、McGuire-Bavoilの重みは次の通りである。

```text
w(a,z) = clamp((min(1, 10a) + 0.01)^3 * 10^8 * (1 - 0.9 clamp(z,0,1))^3,
               10^-2, 3*10^3)
accum      += vec4(C * a, a) * w(a,z)
revealage  *= (1 - a)
out.rgb     = accum.rgb / max(accum.a, 10^-5)
out.a       = 1 - revealage
```

Three r185は、ネイティブなWebGPU上で `MRTNode.setBlendMode()` を通じて、アタッチメントごとのブレンディングをサポートする。Accumは `ONE, ONE, ADD` を使用する。Revealageは1に初期化され、`ZERO, ONE_MINUS_SRC_COLOR, ADD` でR8ターゲットへスカラーのアルファを出力する。R8にはアルファ成分がないため、`ONE_MINUS_SRC_ALPHA` は正しくないであろう。フルスクリーンの初期化描画は、透明シーンの前に `(0,1)` を書き込み、フルスクリーンの合成は、解決された色を呼び出し元の現在の出力ターゲットの上へアルファブレンドする。互換モード/WebGL2のMRTはアタッチメントごとのブレンディングを保持できないため、黙った2パスのセマンティックフォールバックへ分割するのではなく `NACHI_WBOIT_WEBGL2_UNSUPPORTED` として拒否される。

加重OIT(Weighted OIT)は、固定されたフラグメントの多重集合について順序独立であるが、ソートされたアルファ
の近似である。所有する累積ターゲットは、独立した深度アタッチメントも所有する。それは、呼び出し元の
オペークなターゲットに書き込まれた深度を消費できないため、オペークなジオメトリは現在WBOITのフラグメント
を遮蔽せず、壁の後ろのパーティクルが漏れ出すことがある。呼び出し元はオペークなコンテンツを最初にレンダ
リングし、その後WBOITを合成するが、その順序は色の背景のみを供給し、オペークな深度は供給しない。外部の
共有深度オプションは先送りされている。なぜなら、Three r185は1つの `DepthTexture` に1つの所有者
RenderTargetを割り当て、いずれかの所有者を破棄すると共有GPUテクスチャが破棄される可能性があるためで
ある。安全なサポートには、明示的な借用アタッチメントのライフサイクルと、サイズ/サンプル/フォーマットの
検証が必要である。`/m10-sort/` は、`compositeWboitLayers()` に対して2つの既知の重なるレイヤーを検証
し、それらを逆順で繰り返し、意図的なR8のrevealageの量子化を含む線形バイト許容誤差を使用する。

### 13.8 `@nachi-vfx/react` React Three Fiberバインディング

`@nachi-vfx/react` は、第二のランタイムではなく、ライフサイクルアダプターである。`VFXSystemProvider` はR3F
の `Canvas` の内部にマウントされ、供給された `VfxRuntimeRenderer` と現在のR3Fシーンから1つのコアの
`VFXSystem` を作成し、デフォルトで、`useFrame` から `system.update(delta)` を呼び出す前に、アクティブな
R3Fカメラの列優先の `projectionMatrix` と `matrixWorldInverse`、そしてピクセルの `size` を
`system.setCamera()` へコピーする。これにより、距離/視錐台カリング、重要度、シーン深度のuniform、粗い
透明ソートが、レンダリングされたビューと整合したままとなる。`syncCamera` のプロバイダのプロパティは
デフォルトで `true` である。`setCamera()` を自分自身で管理する呼び出し元は `syncCamera={false}` で
オプトアウトできる。非同期の更新の拒否はReactの状態として保持され、Error Boundaryがそれを観測できる
ように次のレンダリングで投げられる。このバインディングが、失敗したフレームをもっともらしい成功した
フレームへ変換することは決してない。

`useVFXSystem()` はそのシステムを返す。`useEffectInstance(definition, options)` はマウント時にスポーンし、アンマウント時、または定義/システム/スポーン専用のシード/優先度が変化した際に、正確なハンドルを解放する。呼び出し元は `definition` をモジュールスコープで定義するか、そうでなければその参照を保持することが望ましい。レンダリング中にそれを再作成すると、意図的にそのリスポーンの契約が発動する。位置、回転、タイムスケール、宣言された `User.*` のパラメータのプロパティは、ライブなインスタンスを更新する。このフックは、自身のコミット済み値のレコードを更新する前に、変更されたパラメータをコアの検証へ転送する。これは、M9のstore-before-forward禁止を保持する。無効な/不変な値は、その時点でエミッターがアクティブでなかったという理由だけで、Reactによって記憶され後で再生されることはない。

`attachTo` は、コアの `EffectTransformSource` またはThreeの `Object3D` のいずれかを受け付ける。後者は `updateWorldMatrix()`、`getWorldPosition()`、`getWorldQuaternion()` によって適合され、初期化前とスケジュールされた各システムステップにおいて、コアによってポーリングされる。アタッチされている間、このサンプリングされたトランスフォームは、スポーン時または後のReactのプロパティを通じて渡された位置や回転を含む、インスタンスの完全なトランスフォームを上書きする。クリーンアップは解放の前にデタッチし、`release()` は権威あるリソース/プールの遷移のままである。`<VFXEffect>` は、同じフックのnullレンダリングコンポーネント形式である。

React、`@react-three/fiber`、Threeはピアである。このパッケージは、本リリースについて、サポートされるThreeのピアを厳密に `0.185.1` に固定し、これら3つのいずれもバンドルしない。単体テストは、`react-test-renderer`、モックされたR3Fのフレーム/シーンコンテキスト、コアテストで使用されるものと同じ構造的なフェイクのランタイムレンダラー境界を使用する。それらはWebGL/WebGPUのキャンバスを必要としない。

R3Fは `state.size` をCSSピクセルで報告する。このバインディングは、シーン深度のテクセルオフセットがデバイスピクセルで動作するため、`viewportSize` を転送する前に、両方の次元に `state.viewport.dpr` を乗算する。ライブなReactのパラメータのプロパティを削除すると、インスタンス上でその宣言されたデフォルト値が復元される。スポーン専用のプロパティ(`seed` と `priority`)はリスポーンを必要とし、意図的にライブパッチされない。

### 13.9 Three r185内部依存の監査

Threeアダプターは `three@0.185.1` に固定されている。Threeのすべてのアップグレードは、以下の内部的な、または
バージョンに結合された表面を監査しなければならない: `renderer._attributes`;ローカルに複製された
`AttributeType` の定数;`setupPositionView` の移植;ストレージノードの `.value` の置き換え;
`texture.renderTarget.samples` のプローブ;`AtomicFunctionNode` の直接構築;`isWebGLBackend`/
`compatibilityMode` の分岐。バージョンの引き上げは、各項目が上流のソースに対して確認され、アダプターの
スモークテストでカバーされるまで、不完全である。

この監査には、playgroundのパフォーマンスツーリングが
`renderer.backend.timestampQueryPool[scope].currentQueryIndex` / `maxQueries` を直接参照することと、
プールのresolve時に `currentQueryIndex` が0へリセットされるという前提も含まれる。これらのフィールドが
改名または削除されるか、リセットのセマンティクスが変わると、適応的ドレインはno-opとなり、タイムスタンプ
プール枯渇warningが再発し、`consoleClean` を強制するページが失敗する。

## 15. PLAN.mdとの相違点

ノーススターのスケッチからの表面構文の意図された乖離はない。コンパイルテストは、そのエミッター、メッシュ要素、エフェクト、タイムライン、ランタイムスポーン、インラインTSLの各形式を使用する。

本RFCは、PLAN.mdにおいて暗黙的であった2つの境界を明示的にする。

1. `tslModule(callback)` それ自体はJSONにはなり得ない。これは作成のためには受け入れられるが、シリアライゼーションの前に、安定した登録済み関数参照へ変換されなければならない。これは、インラインのエスケープハッチと、シリアライズされたモジュール設定はデータであるというルールの両方を満たすために必要である。
2. スケッチ内の `explosionTex` や `noiseTex` のような名前は、シリアライズ可能なエフェクトについて `TextureRef` として型付けされる。生のThree.jsテクスチャを渡すことが許されるのは、明示的に一時的でシリアライズ不可能な定義についてのみである。

これらはシリアライゼーションの制約であり、合成モデルやステージモデルへの変更ではない。

## 16. 未決定事項

M4の視覚的回帰スクリーンショットは、PNGのバイト等価性ではなく、`0.5%` 未満の変化ピクセル比の許容誤差を使用する。蛍(firefly)の加算パスは、等しいシミュレーション状態を異なる描画順序へコンパクションすることがあり、その場合、浮動小数点の加算ブレンディングは、振る舞いの回帰を伴わずに、少数の出力ピクセルの集合を変化させる。正確なシミュレーション/読み戻しのチェックは、決定的でバイトベースのままである。

以下の決定は意図的に先送りされている。実装は、名指しされたスパイクがレビューされる前に、文書化されていない答えをパブリックAPIへ焼き込んではならない。

### スパイク1: 10万パーティクルのTSLコンピュート

SoAは、デバイスのステージごとのストレージバッファ上限によって制約される(WebGPUのデフォルトは8であるが、アダプターはより高いオプトインの上限を公開してよい)。解決済みのスキーマがアクティブなデバイスの上限を超える場合、コンパイルは `NACHI_STORAGE_BUFFER_LIMIT` を報告する。

決定(2026-07-11): SoA-per-attribute+予算診断を基本構成として維持。vec4パッキング(スカラー/小型属性の共有vec4配列への束ね)はM3バッチ1でレンダラ頂点ステージのストレージバッファ予算と統合設計して実装する

追記(2026-07-11、M3バッチ1): vec4パッキングは既定で有効。`f32`/`vec2`/`vec3`、`i32`、`u32`/`bool` の3つの成分型ドメインを別々の共有vec4配列(`vec4`/`ivec4`/`uvec4`)へ格納し、型のビット再解釈は行わない。解決済み論理属性の決定的順序で、最初に収まるvec4グループの最小連続レーンへ割り当てる(first-fit、vec4境界を跨がない)。1粒子あたり複数グループが必要な場合も同じ物理バッファ内のストライドとして保持する。`color`/`quat`/`vec4`/`mat3`/`mat4` は専用バッファを維持する。論理属性から物理buffer/group/laneへの対応は `ResolvedAttribute.physical` と `CompiledEmitterProgram.meta.storageBuffers[].attributes` に公開するが、カーネルおよびレンダラは論理属性名だけを読み書きする。

- SoA対インターリーブ/ハイブリッドストレージ
- 診断で公開されるアライメントとパッキングのルール
- ワークグループサイズとディスパッチの分割
- 固定されたThree.jsのバージョンにおいて、すべてのカスタム属性型がストレージバッファで動作するか
- TSLの読み書きに関するコンパイラのトレース機構
- エミッターの属性スキーマからの `TslParticleBindings` カスタムキーと値型の自動導出
- 実用的なバッファ/デバイスの上限と容量診断

### スパイク2: `drawIndirect` / `dispatchIndirect`

M2は中核となる決定を解決する。物理スロットはアトミックなフリーリストを使用する。なぜなら消滅は散在しており、安定した物理インデックスは、SoAストレージと決定的なアイデンティティを安価に保つためである。コンパクトな生存インデックス領域は、密なレンダー/ディスパッチ順序を供給する。インデックス領域とアロケータ/権威ある生存数/成功/オーバーフローのカウンタは、1つの書き込み可能なライフサイクル状態バッファを共有する。スポーンディスパッチと間接描画の引数は、第二の別個の間接引数バッファを共有するため、間接ソースが同一のコンピュートパスの同期スコープ内で書き込み可能なストレージでもあることは決してない。コアは引数領域を所有し、Spawn/Updateのコンパクションの後にそれらを更新する。CPUの読み戻しは、診断、オプトインの低頻度完了、縮小されたWebGL2のカウント経路のために許可される。誕生と消滅は、現在のスケジューラステップのコンパクションの後、レンダラーに対して可視となる。

並列アダプターをまたぐ決定的な割り当て順序は、意図的に未解決である。将来のオプトインは、永続的なパーティクルIDを割り当てるか、決定的な割り当てパスを使用してもよい。そのコスト、シリアライズされた表現、イベント/シミュレーションキャッシュとの相互作用は未解決のままである。

追記(2026-07-11、M7バッチ1): リボンはallocation/compaction順を決定化せず、別の
`Particles.spawnOrder` とbirth-index ringを用いる。各spawn/event prepareの単一invocationが
連続範囲を予約し、spawn invocation indexを加えたキーを保存するため、physical slotのatomic
競合を順序意味論から分離できる。この決定はリボン順序だけを保証し、一般のfree-list割当順や
加算描画順を決定化するものではない。詳細・制約は§9.3。

### スパイク3: WebGL2バックエンドのパリティ

M2の実用的なライフサイクルランタイムは、現在WebGL2ではサポートされていない。このバックエンドにはアトミクスも間接操作もなく、`rate` と `perDistance` は拒否される。`buildKernels()` は、initializeによって書き込まれるすべての物理属性ストレージを、transform-feedbackの上限と静的に比較し、予算超過のburstエミッターを `NACHI_BACKEND_SPAWN_UNSUPPORTED` によって拒否する。そうしなければ黙って消えてしまうであろう書き込みを持つカーネルを送信することは決してない。initializeの完全な物理書き込みセットが測定された上限に収まる場合、小さなプレフィックスのみのburstスキーマは利用可能なままであり得るが、それはライフサイクルのパリティではなく縮小経路である。

- 非ライフサイクルのどのモジュールケイパビリティがWebGL2バックエンドを通じてコンパイルされるか
- パックされたtransform-feedbackのレイアウトが、縮小されたburstティアを安全に拡張できるか
- 後のマイルストーンによって導入されるストレージテクスチャのフォールバックと上限
- WebGL2が、現在の明示的な拒否の代わりに、`rate`/`perDistance` のためのCPU割り当てカーソルを追加すべきか
- 黙ったセマンティックフォールバックを防ぐために使用される、シリアライズされたケイパビリティ/品質宣言

### スパイク4: 深度アクセスとTSLポストパイプライン

M6バッチ1は、明示的な前フレームの正規化深度カラーコピーを通じて、コンピュート深度衝突を解決する。フラグメントのソフトパーティクルは、そのアクティブなレンダーパスにおいて引き続き `linearDepth(viewportDepthTexture(screenUV))` を使用する。コンピュートは、アタッチメントのフィードバックを避けるため、別個に所有されるサンプリング可能なテクスチャとSystemカメラのuniformを使用する。リバース深度、MSAA深度の解決、WebGL2のパリティ、複数エフェクトをまたぐ共有コピーの所有権は、今後のケイパビリティ作業のままであり、推論されるのではなく診断されなければならない。

M10バッチ1は、このスパイクのcolor-postの半分を、`RenderPipeline` をルートとする§13.6の `@nachi-vfx/post` グラフによって解決する。シェイクウェーブ/熱かげろうのディストーション、ラジアルブラー、ThreeのTSLの `BloomNode` は、すべてシーンカラーの操作として合成されるため、深度の規約を必要としない。これは、reverse-z、MSAA深度の解決、WebGL2のコンピュート深度パリティ、共有深度コピーの所有権を解決するものではない。それらの項目は、深度の利用者にとって明示的に未解決のままである。

M10バッチ2は、§9.5の独立したソートされた間接指定と、§13.7のネイティブWebGPU MRT WBOITによって、アルファの順序付けを解決する。これは、M2のフリーリスト/コンパクションの割り当て順序を解決したり、永続的なパーティクルアイデンティティを作成したりするものではない。深度が等しい場合のソートは、決定的なフレームローカルな同点キーとしてのみ物理インデックスを使用する。65,536の容量上限と `O(log^2 P)` のディスパッチ数は、オプトインの品質契約の一部である。各bitonicステージは別個の送信である。なぜなら、1つのコンピュートパスはディスパッチ間のデバイス全体のバリアを提供しないためである。ソート作業は更新時の作業である。続く更新のないカメラの変更は、意図的に以前の間接指定をそのまま残す。

M10バッチ2レビュー決定(2026-07-12): アルファエミッターのランクは固定された基準値1000を維持し、一方加算
描画はその通常の順序0のままである。加算をランク付けされたセットへ引き入れることは、加算/加算の正確性を
改善することなく、ユーザーの透明シーンオブジェクトとの既存の合成を変化させてしまうため、この分離は省略
ではなく明示的なv1のポリシーである。投影デカールは除外される。なぜなら、それらの有用な可視性を制御する
のは、エミッター中心のアルファ順序ではなく、受け手の深度であるためである。WBOITの外部深度注入は今後の
フォローアップ作業のままである。Three r185のRenderTargetのセッターは `DepthTexture.renderTarget` を
再割り当てし、RenderTargetの破棄はそのキャッシュされた深度テクスチャを破壊するため、借用された共有
アタッチメントを低コストのオプションとして安全に公開することはできない。将来のAPIは、オペークな遮蔽を
有効化する前に、所有権を定義し、寸法、サンプル数、フォーマット、reverse-zの規約、レンダーパスのロード
挙動を検証しなければならない。

リボンのアルファランク統合は今後のフォローアップ作業のままである。それは、既存のトレイル準備や順序付けのセマンティクスを変更することなく、リボンのアルファおよび事前乗算アルファの描画を、エミッター中心のランキングポリシーへ持ち込まなければならない。

### 後のマイルストーンの決定

M11バッチ1決定(2026-07-12): 品質のスポーン/容量のスケールと可視性はランタイムの値であり、soft/lit/sortedの
ゲートは、コンパイルキャッシュとGPUプール内で分離された構造的コンパイラバリアントである。稼働中の構造的な
変更は、パーティクル状態を破棄する代わりに再起動を必要とする。完全にカリングされたエフェクトは、ローカル/
ライフサイクル/パーティクルの時間をポーズし、キャッチアップ作業を一切行わない。重要度は、安定したインス
タンスIDの同点処理を伴う、§10.4の規範的な `4*priority + 2*screenOccupancy + 1/(1+distance)` の数式を
使用する。これは、以下のM11ティア選択の項目を解決する。シミュレーションキャッシュは別個のM11バッチのまま
である。

M11バッチ2決定(2026-07-12、lineage追補2026-07-15): §10.5のシミュレーションキャッシュは、レンダー読み取り
の論理属性、ロスレスな生存間接指定、および必須の物理slotごとのlossless `spawnOrder` lineageを、
metadata+ArrayBufferの形式で格納する。Float32とコンポーネントごとのu16量子化がサポートされる。V2は現在
frameのGPU bufferを再利用し、frameごとに1回のdecode済みuploadを実行する。lineage streamは通常のreplay
uploadを追加せず、全frameのGPU常駐は先送りされている。V1 cacheは再bakeしなければならない。Nearestは
frame全体の選択であり、一方linearは両端点で同じ物理slotがaliveかつlineage一致する場合に限りfloat属性を
補間する。loop cacheは端点を複製し、lineage→physical mapを通じてalive lineage集合と属性を比較する。
WebGL2のpacked simulation storageは、黙ったtransform-feedback aliasを許容するのではなく、group 1以上への
behavioral accessを拒否する。すべてのrenderable lifecycle fixtureがgroup 1に達するため、WebGL2 cache bakeは
実質的にunsupportedで、structured diagnosticを通じて検証される。replayもunsupportedとして明示的に診断
されたままである。これは、以下のsnapshot/custom/transient属性の項目を解決する。

M11バッチ3決定(2026-07-12): §10.6の属性スナップショットは、シミュレーションキャッシュのストレージ読み
戻しを再利用し、論理型付きの行と、明示的なコンパクト/物理の系譜、切り詰めを公開する。プロファイラカウンタ
は、フレームローカルなスケジューラの事実である。GPUタイミングは、引き続き `nachi.perf-baseline` v1が
所有するパスレベルのキャッシュされたサンプルのままである。コアは、第二のタイムスタンプクエリを発行したり、
ヒューリスティックにパス時間をエミッターへ帰属させたりすることはない。これは、GUIエディタをスコープ外に
残しながら、M11のコードデバッガの表面を解決する。

M12バッチ2決定(2026-07-12): §10.7は、Grid2Dとシミュレーションステージをエフェクト要素にする。グリッド
チャンネルは、1つのstateストレージバッファと1つのscratchストレージバッファ内のパックされたvec4セル
レコードを使用する。依存するステージのイテレーションは、別個のupdate/commitの送信である。参照実装の
流体は、semi-Lagrangian移流と、意図的に制限されたJacobi投影を使用する。パーティクルの堆積はu32の固定
小数点アトミクスを使用し、パーティクルのサンプリングはGPUのバイリニア読み取りを使用する。WebGL2は明示
的にサポートされていない。Neighbor grid、Grid3D、障害物場、エミッターストレージへの直接転送は、後の
M12バッチのままである。

M7決定: 上限のあるPointLightプーリングが、セマンティックなライトレンダラーである。GPUのtop-N選択と、
固定された1フレーム遅延の読み戻しは、CPUオブジェクトと同期の両方を制限する。エミッシブ+ブルームはM10へ
先送りされている。なぜなら、ブルームはポストプロセッシングの見た目であり、通常のライティングされたシーン
ジオメトリの照明を置き換えることはできないためである。M7のデカール投影は、向き付けられた四角形への
フォールバックではなく、深度の再構築を使用する。§16のダウングレードは不要であった。CPUの `DecalGeometry`
は、パーティクルごとのGPU駆動の更新には引き続き適さない。

M7決定(2026-07-11): エミッターあたり複数のレンダーモジュールは、後のM7バッチではなく、M8以降へ先送り
されている。それらをサポートするには、描画ごとの間接引数スロットと、協調されたレンダラーリソースの所有権
が必要である。M7は、その作業がスケジュールされるまで、明示的な1レンダーモジュールの制限と
`NACHI_RENDER_MODULE_LIMIT` の診断を維持する。

M8バッチ2決定(2026-07-11): mesh-fxはコアに依存しないままである。手続き的ファクトリは自己記述的なThree
メッシュを返し、M9はそのエフェクト要素のライフサイクルアダプターを所有する。これにより、単体でのジオ
メトリ/マテリアルの作成がコンパイラやスケジューラの型に依存することを避ける。`fxMaterial` は、現在、
単体のuniformと外部から供給されるTSLのクロック/lifeノードを公開する。自動的なエフェクトローカルな
バインディングは、M9のタイムライン統合へ先送りされている。VATは、アセットのテクスチャレイアウト、
フレーム補間、境界、Blenderエクスポータの互換性が、上記の手続き的ジオメトリやマテリアルの契約を変化
させないため、別個のM8バッチのままである。既存の1レンダーモジュールのエミッター制限は、メッシュ
エフェクト要素によって変更されない。それらは、追加のエミッターレンダーモジュールではなく、エフェクト
レベルの非パーティクル要素である。

M8バッチ3決定(2026-07-11): VATは、Blender Extensionsの `VAT` アドオンの厳格な1行1フレームのレイアウト
と、グローバルなスカラーmin/maxリマップに従う。位置テクスチャはfloat/half-floatでリニアである。ラップ/
クロップされたレイアウトや直接のPNG位置レイアウトは、推測されるのではなく診断される。単体のuniform/
外部ノードのクロックの分割は `fxMaterial` に一致し、一方M9は引き続き自動的な要素ライフサイクル
バインディングに責任を持つ。Golden #3はM8ローカルな固定合成を使用する。タイムラインの振り付けは引き続き
M9であり、ブルーム/ポストは引き続きM10である。

H2-10決定(2026-07-16): M9の自動要素lifecycle bindingは、package-owned VAT clockとclone graph再構築を
含む。clockはtrack timeや`fx.time`ではなく、最新playからのmesh element elapsed timeである。明示VAT time
入力は外部所有のまま変更しない。clone可能metadataはmesh-fx内部に留め、timelineがreset/advance/stop/loop
schedulingを所有する。これはtimelineへ載せたVAT meshのminorな挙動変更であり、単体`applyVat()`は不変である。

M9バッチ2決定(2026-07-11): シーケンサのオーケストレーションとmesh-fxのエフェクトアダプターは
`@nachi-vfx/timeline` に存在する。カメラシェイクは、Threeのカメラではなく、コールバック/リグの抽象を対象と
する。トラックのループリセットは、アクティブな子のライフサイクルを切り詰め、時刻ゼロのアクションを再生
する。子エミッターのループは要素ローカルのままである。タイムラインは固定ステップの分割を所有し、コアは
その正確なステップを消費し、ワールド時間、トラック速度、ヒットストップをまたぐ二重のスケーリングを回避
する。生のメッシュリソースは一時的なままであり、一方タイムラインのアクションとメッシュのプレースホルダ
はシリアライズ可能なままである。これは、§13.4のコアカーブの下位変換と、自動的なfxMaterialの
local-time/normalized-lifeのバインディングを解決する。

M9バッチ2レビュー決定(2026-07-12): 現在、各タイムラインの `play()` は単一要素のコアサブ定義を具現化し、
独立してスケジュールされる要素のライフサイクルが分離されたままとなる。したがって、このバッチにおいて、
M5の `emitTo()` のリンクはタイムライン要素のキーをまたぐことができない。プロデューサのサブ定義には
コンシューマのキーが存在しないためである。失敗した子のコンパイルは、現在、明示的な
`NACHI_EVENT_TARGET_UNKNOWN` 診断をタイムラインインスタンスへコピーし、それを `error` へ遷移させる。
黙って `active` のままであることは禁止されている。イベントでリンクされた要素を1つのスケジュールされた
子定義へグループ化することは、すべてのメンバーについて協調されたplay/stopの所有権を必要とするため、
先送りされている。

M10バッチ3決定(2026-07-12): ライティングされたビルボードは、第二のライティングモデルを導入するのでは
なく、MeshStandardの物理ライティング経路とSpriteの位置経路を再利用する。接空間の法線は、ビルボードの
ビュー平面基底によって回転され、ビュー空間の `normalNode` の値として供給される。Golden #2は、現在、
ソートされたスモークと `@nachi-vfx/post` のシェイクウェーブディストーションを合成する。Golden #5はM10で
コード作成され、タイムライン/ポストのスタック全体を演習する。同じ定義をJSONからロードすることは、
暗黙のランタイムケイパビリティではなく、M12のアセットフォーマットの受け入れ項目のままである。

- 予約された `context.emitEvent(condition)` の契約を超えるカスタムイベントのTSLコンテキストの具現化(M5以降)
- WBOITの自動ティア選択は、WBOITがエミッターレンダラーの機能ではなく `@nachi-vfx/post` のパイプラインである
  ため、コアの外部のままである。M11バッチ1は、シリアライズされたエミッターのsoft/lit/sortedのティア
  ゲートを解決する。将来のパイプラインポリシーは、ソートされたアルファとWBOITのどちらかを選択してもよい。
- エミッター継承のためのシリアライズされたアセット参照構文(M9の作成用パッチは積極的に正規化する。M12が
  参照エンコーディングを決定する)
- 優雅な停止/排出ポリシーと、システム横断/グローバルなプール予算(即座の停止と、システムごとに上限のある
  M9のプーリングは確定している)
- neighbor-stageの拡張。Grid2D/Grid3Dのエフェクト要素ステージは、パーティクルの `ModuleStage` の共用体を
  拡張することなく、§10.7によって確定している。完全なボリュームレイマーチングは今後の課題のままである。
- ベクトル場の `space` 選択と明示的なフィールド変換。`vectorField()` は現在、どちらのオプションもなく
  ワールド空間に固定されている。

解決されるまで、これらはコンパイラのケイパビリティに関する疑問である。それらが、既存の作成用データの意味を黙って変化させることがあってはならない。
