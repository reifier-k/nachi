# RFC 002: Effekseer `.efkefc` インポート互換性調査

> Language: [English](./002-effekseer-compatibility.md) / 日本語 (このページ)

- ステータス: 調査完了、実装は保留
- 日付: 2026-07-12
- 推奨: nachi 1.0後に再検討

## 1. 問いと結論

本調査は、nachiが編集可能なEffekseerの`.efkefc`エフェクトを直接インポートすべきかどうかを問うものである。両フォーマットは、有用な非可逆コンバーターを実現できる程度に十分な概念を共有しているが、ランタイムやマテリアルモデルを共有してはいない。したがって、直接的な互換性はパーサー機能ではなく、変換プロダクトとして扱うべき事柄である。

**推奨:** M12の調査は完了として記録し、実装は1.0以降に保留する。1.0後の提案は、スプライトと基本モーションを対象とした、フィクスチャベースかつバージョンゲート方式のコンバーターから着手すべきである。リボン/リング/トラックのレンダリング、F-curve、リソース変換、座標/色ルール、そして未サポート機能の診断についてゴールデンパリティフィクスチャが揃うまでは、汎用的な`.efkefc`互換性を謳ってはならない。

## 2. 参照元と安定性の境界

主な参照元は以下の通りである。

- [Effekseer Tool Reference](https://effekseer.github.io/Helps/17x/Tool/en/ToolReference/index.html): `.efkefc`を編集可能かつランタイムで再生可能なエフェクトファイルとして定義し、親子ノードモデルを説明し、スプライト、リボン、リング、モデル、トラック、F-curve、マテリアル、サウンド、LOD、動的パラメーター、プロシージャルモデルの各サーフェスを列挙している。
- [Effekseer export reference](https://effekseer.github.io/Help_Tool/en/ToolReference/fileExport.html): 再生専用の`.efk`バイナリおよび`.efkpkg`リソースパッケージを、編集可能な`.efkefc`と区別している。
- [Effekseer runtime source](https://github.com/effekseer/Effekseer/tree/master/Dev/Cpp/Effekseer/Effekseer): Sprite/Ribbon/Ring/Trackそれぞれ独立したエフェクトノードローダーとF-curveランタイムクラスが存在し、これらが単一の汎用ビルボード形状ではなく、それぞれ別個のバイナリ/ランタイム契約であることを示している。
- [Effekseer editor binary exporter source](https://github.com/effekseer/Effekseer/tree/master/Dev/Editor/EffekseerCore/Binary): 共通、生成、トランスフォーム、レンダー、サウンド、およびレンダラー固有のバイナリ出力を担っている。

Effekseerは`.efkefc`を、サードパーティ向けの安定した交換フォーマット仕様として公開してはいない。そのソースコード自体が事実上の仕様である。したがって、nachi側のリーダーは、明示的なサポート対象Effekseerバージョンマトリクスとキャプチャ済みフィクスチャを必要とする。「現行エディタが書き出すものをそのまま読む」という方針は、安全な互換性契約とはならない。

## 3. フォーマット構造

有用なアーキテクチャレベルで見ると、`.efkefc`は、旧来のテキスト指向の`.efkproj`ドキュメントとは異なり、バージョン管理されたバイナリコンテナである。エディタデータと、ランタイムが消費可能なコンパイル済みエフェクトの両方を保持する。ランタイムペイロードには以下が含まれる。

1. ヘッダー/バージョンおよびコンテナチャンク
2. カラー、法線、ディストーションの各テクスチャ、マテリアル、モデル、カーブ、サウンドのリソーステーブル/パス
3. スケール、カリング、LOD、動的入力などのグローバルエフェクト設定
4. 再帰的な親子ノードツリー
5. 各ノードごとの、共通のライフタイム/生成/バインディングデータ、スポーン形状、位置/回転/スケール、フォース、デプス/レンダー共通ステート、1つのレンダラー固有ブロック、任意のサウンド、およびF-curve
6. Sprite、Ribbon、Ring、Model、Track、またはレンダリングなし、それぞれのレンダラー固有ペイロード

正確なバイトレイアウトおよびチャンクのリビジョンは、上流の実装詳細である。将来のインポーターは、正規化された中間形式を得るために上流がサポートするエクスポーター/ライブラリを再利用するか、`EffekseerVersion -> decoder`ディスパッチの背後で必要なローダー構造体のみを移植するかのいずれかを取るべきである。サンプルファイルからオフセットを推測する手法は却下する。

`.efkpkg`は別個のパッケージ上の関心事であり、1つ以上の`.efkefc`ファイルと参照リソースを含み得る。エフェクト変換の前段として、パッケージ展開、パス正規化、アーカイブ上限、および信頼できない入力に対する予算設定が必要となる。

## 4. モデルマッピング

| Effekseer概念                       | nachiでの対応候補表現                                                                  | 忠実度                                                                                                                                       |
| ------------------------------------ | ---------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| エフェクトノード / 生成パーティクル | `EmitterDefinition`1つ、ノードツリーは要素と`emitTo()`/タイムライン開始へと変換          | 部分的: Effekseerの親バインディングと子生成タイミングは、死亡/衝突イベントより表現力が高い                                                  |
| Sprite                               | `billboard()` + flipbook/cutout/blending/depthオプション                                 | 基本的なアンリットスプライトでは高い; UVモード、ディストーション/マテリアル詳細は差異あり                                                    |
| Ribbon                               | `@nachi/trails`の`ribbon()`                                                               | 中程度: 連結順序はマッピングできるが、フェイシング、スムージング、幅/カラーステーション、バインディングルールが異なる                        |
| Ring                                 | `@nachi/mesh-fx`のring/プロシージャルメッシュとマテリアル                                | 中〜低: Effekseerのringは視野角、形状/色/UVアニメーションを持つパーティクル単位のレンダラーである一方、nachi現行のringはエフェクトメッシュである |
| Track                                | 幅拡張断面プロファイルを持つribbon/trail                                                 | 低〜中: Effekseerのtrackはfront/middle/backの幅と6つのカラーステーションを持つが、現行の固定幅ribbon契約では表現されない                     |
| Model                                | `meshRenderer()` + 該当する場合は`GeometryRef`/VAT                                       | 部分的: モデルアニメーション、プロシージャルモデル、マテリアル変換には別途アセットが必要                                                      |
| 生成数/間隔/寿命                     | `burst()`/`rate()`、ライフサイクル、`lifetime()`                                         | 中程度: フレームベースのランダム範囲と親トリガーのセマンティクスには変換が必要                                                                |
| 位置/回転/スケールとイージング       | Init/Updateモジュールおよび`curve()`/over-lifeモジュール                                 | 線形/イージングのサブセットについては中程度                                                                                                   |
| F-curve                              | LUTベースの`curve()`、または生成された登録モジュール                                     | 低〜中: Effekseerの軸ごとのキー、タンジェント/補間、オフセット、サンプリング、タイムライン単位は、nachi現行の線形ヘルパーの範囲を超える       |
| 親子バインディング                   | エフェクトトランスフォーム、`attachTo()`、継承されたイベントペイロード                    | 低: bind-at-creation/always/follow-parentのセマンティクスには専用の変換/ランタイムポリシーが必要                                              |
| 動的パラメーター/数式                | 宣言された`User.*`と登録済みTSL                                                          | 低: 式はインタープリタ/コンパイラなしには安全に変換できない                                                                                   |

## 5. 変換可能な最初のサブセット

信頼に足る最初のインポーターは、意図的に絞り込んだプロファイルのみを受け付けることが考えられる。

- スプライトレンダラーの子を持つルート1つ
- 固定/バースト生成、有限の寿命、基本的なpoint/sphere/circleスポーン形状
- セマンティクスに直接対応するモジュールが存在する場合の、固定/ランダム/イージングの位置、回転、スケール
- 重力、および文書化されたローカルフォースのサブセット
- アンリットのカラーテクスチャ、基本的なflipbook、alpha/additive/multiplyブレンディング、depth test/write
- Effekseerのフレーム時間を秒に変換した上での、線形なcolor/scale-over-lifeカーブ
- `AssetRef<'texture'>`に書き換えられた外部PNG参照
- 明示的な座標系、単位スケール、色空間、乱数シードの変換メタデータ

未サポートのノードまたはフィールドは、すべてパス単位の診断を発生させなければならない。「ベストエフォート」モードは、呼び出し側が明示的に要求し、機械可読なロスレポートを受け取る場合に限り、ノードを破棄してよい。デフォルトはfail-closedでなければならない。

## 6. 未サポートまたは高リスクの領域

- `.efkmat`マテリアルグラフ、コンパイル済みマテリアルキャッシュ、カスタムデータおよびディストーションのセマンティクス
- サウンド再生および同期
- 動的数式、動的入力、トリガー、外部モデルスロット
- 正確な親バインディング/継承、および任意の親パーティクル経過時間における子生成
- 高度なribbon/trackジオメトリ、スムージング、スプライン分割、視点依存のringジオメトリ
- モデルアニメーション、`.efkmodel`、`.efkcurve`、プロシージャルモデル、GPUパーティクル固有の機能
- F-curveのBezier/タンジェント挙動、ランダムオフセット、edge/extrapolationモード、正確な60フレームサンプリングルール
- Effekseer固有のタービュランス/フォースフィールド、killルール、衝突/トリガー挙動
- ガンマデフォルトのオーサリングとnachiの線形ワーキングカラーの差異、handedness/軸および単位の変換
- レンダラーのソート/デプス優先度、ソフトパーティクルおよびalpha-cutoffの詳細、LOD/カリングの等価性
- 新しいコンテナまたはノードのバイナリバージョンに対する将来互換性

視覚的な不一致はエフェクトインポーターにおける主要な失敗モードであるため、これらを暗黙のうちに近似することはできない。

## 7. コスト見積もり

| 作業                                                                             |           見積もり |
| -------------------------------------------------------------------------------- | ------------------: |
| バージョン/コンテナの調査、法務/ライセンスレビュー、不正入力ハーネス             | 1〜2エンジニア週    |
| バージョン管理されたデコーダーまたは上流ブリッジ、リソース、正規化された中間モデル | 2〜4週             |
| Sprite/基本トランスフォーム/生成コンバーターおよび診断                           | 3〜5週             |
| F-curve/イージング変換および決定論的タイミングフィクスチャ                       | 2〜4週             |
| Ribbon/ring/track変換およびレンダラーのギャップ対応                              | 4〜7週             |
| マテリアル/モデル/動的パラメーター(部分対応のみ)                                | 6〜10週            |
| クロスバージョンコーパス、ゴールデンスクリーンショット、ドキュメント、fuzz/セキュリティ対応 | 3〜5週    |

実用的な厳格スプライトサブセットはおよそ**8〜12エンジニア週**である。広範な実用的互換性は**20〜30エンジニア週以上**となり、それでも公開された未サポート機能マトリクスが必要となる。その後の保守は上流フォーマットの変更に追随する形になる。

## 8. 1.0後の判断ゲート

以下がすべて真である場合に限り、再検討する。

1. ユーザーが代表的かつ再配布可能な`.efkefc`コーパスを提供し、インポート需要の優先順位を示すこと
2. 上流がサポートする正規化経路、または保守可能なバージョン管理されたデコーダーが選定されていること
3. nachiが専用のring/track契約および非線形カーブ契約を持つか、明確に限定されたロスを許容していること
4. ファイルサイズ、再帰、アロケーション、アーカイブ展開、リソースパスに対するセキュリティ予算が存在すること
5. 受け入れ基準が、サポート対象の全バージョン/プロファイルについて、決定論的な値とスクリーンショットをEffekseerと比較すること

それまでの間、チームは見た目のみの再利用のためにスプライトシートを書き出すか、あるいは同等のnachiエフェクトを手動で作成することができる。M12ではこの調査を完了として記録するものであり、インポーターのコードは含まれない。
