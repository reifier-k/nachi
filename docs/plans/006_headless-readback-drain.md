# 006: ヘッドレス readback の信頼性(初回フルサイズ読み出しが空になる)

- 重大度: 低(検証ツーリング。ただし誤診誘発性は高い)
- 対象: `tools/`(spike ハーネス指針)、必要なら `@nachi-vfx/three` のヘルパー、上流は Three r185
- 状態: 実装済み・受入済み(2026-07-14、受入コミット `ada39ec`)

## 症状

headless Chromium + SwiftShader WebGPU で、**readback を挟まないフレームが十数フレーム
続いた後の最初のフルサイズ `readRenderTargetPixelsAsync()` が全ゼロのピクセルを返す**。
同一フレームで 2 回読んでも 2 回ともゼロ(ダブルリードでは直らない)。事前のフルサイズ
ウォームアップ読み出し 1 回でも直らない。毎フレーム 1×1 の読み出しを挟み続けると
全読み出しが正しくなる。

wuwa-slash 実測: 6 コマ契約シートの 1 コマ目(描画内容ありと要素状態が証明している時刻)だけが
恒常的に真っ黒になり、シーン側の不具合(dissolve、可視性、タイムライン)を疑う長い迂回を生んだ。

## 根本原因(仮説)

Three r185 WebGPU バックエンドの readback ステージングバッファ運用と、多数フレームの
コマンド滞留の相互作用と推定(SwiftShader 固有の可能性もある)。ライブラリ(nachi)側の
バグではない可能性が高いが、検証パイプライン全体の信頼性問題として扱う。

## 改善案

1. **最小再現の切り出し**: nachi 非依存の Three 単体ページ(RenderTarget へ N フレーム描画 →
   1 回読む)で再現するか確認し、再現すれば Three へ upstream issue を起票する。
2. **ツーリング既定の drain**: 再現有無に関わらず、スパイク/golden ページ向けに
   「毎フレーム 1×1 readback で読み出し経路をドレインする」パターンを公式化する。
   置き場所の候補:
   - `apps/playground/src/readback.ts` に `createDrainedReadback(renderer, target)` ヘルパーを
     追加し、golden ページ群から共用する(第一候補。ライブラリ API を汚さない)。
   - もしくは `tools/` のドキュメント(CLAUDE.md の Headless WebGPU constraints 節)への追記のみ。
3. **回帰ガード**: 契約シートを持つページは「全パネル真っ黒でない」ことをチェックに含める
   (wuwa-slash は panelStats の foreground 下限で担保済み。テンプレートとして明文化する)。

## 受け入れ基準

1. 最小再現の結論(Three 単体で再現する/しない)が記録される。
   **結論 (2026-07-14、統括実測・クロスレビュー追認):** Three r185 単体の生 `RenderTarget` +
   `MeshBasicNodeMaterial`、present 無しでは、`drain=0` かつ readback を挟まない 18 / 30 / 60 /
   120 フレームのすべてで初回フルサイズ読み出しが正常となり、再現しなかった。現象は nachi の
   compute pipeline 併用時のみ発現した(`wuwa-slash` で実測)。よって Three への upstream issue は
   起票しない。
2. ドレインヘルパー導入後、drain 無しで発生していた黒キャプチャが、対象ページで再現しない。
3. CLAUDE.md のヘッドレス制約節に 1 行の注意書きが追加される。

## 互換性 / リスク

- ツーリングのみの変更で公開 API に影響なし。
- 毎フレーム 1×1 readback は SwiftShader でのフレーム時間を僅かに悪化させる(実測で
  スパイク所要時間への影響は誤差程度)。実 GPU の視覚検証経路(present)には適用しない。

## 追記: showcase スプリントで判明した関連ツーリング項目 (2026-07-13)

1. **タイムスタンプ解決周期はディスパッチ数依存**: 「6 フレーム毎に resolve」で足りるかは
   ページのコンピュートサブミット数に依存する(2 システム × 4 サブステップ構成では
   2〜3 フレーム毎が必要だった)。周期の手動調整ではなく、クエリプール残量を見て
   自動ドレインするヘルパー(perf.ts へ)を用意する。
2. **spike-runner は非ルート cwd で不可解に失敗する**: 相対 `artifacts/` と `--dist`
   ディレクトリの解決が cwd 依存。起動時に `process.cwd()` がリポジトリルートでなければ
   即座に明示エラーで落とすガードを追加する(古い artifacts を黙って読む事故も防ぐ)。
3. **要素スクリーンショットは CSS スケールで保存される**: `selector` 撮影はレイアウト寸法
   (例 1178×442)であり canvas ネイティブ寸法(1920×720)ではない。ピクセル解析をする
   ページ側チェックは readback 統計(panelStats)を正とし、PNG は視覚確認用と位置づける
   旨をツール README に明記する。
