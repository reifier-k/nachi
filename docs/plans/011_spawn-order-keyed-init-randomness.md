# 011: init 乱数の spawnOrder キー化(スロット再利用下の決定性と分布の正しさ)

- 重大度: 高(決定性/視覚品質。H1-1 検収中に実測で確認)
- 対象: `@nachi/core`(init モジュールの乱数導出: positionSphere / velocityCone / lifetime(range) 等)
- 状態: 提案
- 出典: H1-1 検収(2026-07-13〜14)。showcase-beam のスクリーンショット回帰が発端

## 症状(実測)

`rate()` スポーンで「粒子寿命 < スポーン窓」のエミッタ(死亡とスポーンが継続的に重なり、
フリーリストのスロット再利用が起きる)は:

1. **ラン間非決定**: 同一シード・固定ステップでも、ヘッドレス実測でスクリーンショットが
   2.2〜4.6% 変化する(完全一致ゲート 0.5% 超)。再利用順はコンパクション由来で
   非決定(設計どおり)だが、**init 乱数がスロット(物理インデックス)にキーされている**ため、
   「どのスロットに載ったか」が見た目に直結してしまう。
2. **凍結パターン(分布の実効歪み)**: スロットキー乱数は再利用時に**同じ乱数列を再現**する。
   その結果、可視母集団は「固定された少数の軌跡の繰り返し」になり、名目分布より疎で
   構造的な見た目になる。spawnOrder キーに直すと毎スポーン新鮮なサンプルとなり、
   同じ名目パラメータでも大幅に密度・見た目が変わる(showcase-beam で実証。
   承認済みの見た目がこの凍結アーティファクトに依存していた)。

## 回避策(現状)

showcase-beam はページローカルの `stableRateInit` tslModule(hash(spawnOrder+salt+stream) で
球面/球体位置・コーン速度・寿命を導出)へ置換して決定化した(レビューで分布等価性を確認済み)。
ただしこれは全ユーザーが自作すべきものではない。

## 改善案

1. core の init モジュール群の乱数導出を、物理スロットではなく **`Particles.spawnOrder`
   (+エミッタシード)にキー**する。burst 系(再利用なし)では従来と同分布・同決定性、
   rate/perDistance 系では再利用下でも決定的かつ分布が正しくなる。
2. 既存シード再現性への影響: 生成される具体値は変わる(決定性は保たれるが数列が変わる)。
   スクリーンショット基準の再記録が必要になるため、変更は changeset minor+ゲート付きで行い、
   ゴールデン基準の一括再記録を同一バッチに含める。
3. ハッシュ品質: 隣接 spawnOrder 間のストリーム重なり(order+stream の格子)を避けるため
   `hash(order * K + stream + salt)`(K ≥ ストリーム数)形式とする(H1-1 レビュー指摘)。
4. 完了後、showcase-beam の `stableRateInit` を core モジュールへ戻して回避策を撤去する。

## 受け入れ基準

1. 「rate + 寿命 < スポーン窓」の再現エミッタ(m3-sprites の H1-1 リグレッションを拡張可)で、
   同一シードの連続ラン間スクリーンショット差分が完全一致ゲート内(≤0.5%)。
2. burst 専用エミッタの既存挙動・シード決定性が不変(ゴールデン基準の非意図差分ゼロ)。
3. showcase-beam がページローカル回避策なしで決定性ゲートを通過。

## 互換性 / リスク

- 乱数列の変化は事実上の見た目変更(基準再記録を伴う)。minor として告知。
- GPU コスト: hash 数回/粒子の追加は無視できる規模。
