# nachiへのコントリビューション

> Language: [English](./CONTRIBUTING.md) / 日本語 (このページ)

nachiの改善にご協力いただきありがとうございます。このプロジェクトでは、バグ報告、機能要望、
ユースケースの説明など、Issueを通じたフィードバックを歓迎します。外部からのコード貢献やPull
Requestは受け付けません。

## コントリビューションモデル

外部ユーザーによるPull Requestの作成はリポジトリ設定で制限されています。forkからパッチやPull
Requestを作成してもレビュー・マージは行わないため、コードを準備する前にIssueを作成してください。
MIT Licenseの範囲で、個人利用のためにfork・変更することは引き続き可能です。

Issueでは、発生した問題や実現したい結果を説明してください。プロジェクトの方向性に合う場合、
メンテナーが信頼された内部ワークフローで設計、実装、テスト、マージを行います。これによりレビューと
サプライチェーン上の責任をメンテナー側に保ちつつ、明確な問題報告そのものを価値ある貢献として扱います。
背景となる考え方は
[The Community Pull Request Is Dead](https://stack72.dev/the-community-pull-request-is-dead/)を参照してください。

## バグを報告する

可能な限りバグ報告テンプレートを利用し、次の情報を含めてください。

- 対象パッケージとnachiのバージョンまたはコミット
- OS、Node.js、ブラウザ、GPU、Three.jsなどの関連バージョン
- 最小再現例または正確な再現手順
- 期待する挙動と実際の挙動（エラー出力を含む）
- 確認済みの回避策や調査内容

認証情報、非公開データ、信頼できない指示は含めないでください。セキュリティ上の脆弱性が疑われる場合は、
[SECURITY.md](./SECURITY.md)に従って非公開で報告してください。

## 機能を提案する

API案より先に、解決したい問題またはユースケースを説明してください。望ましい結果、制約、検討した代替案、
影響を受けるパッケージを含めてください。Issueを作成しても、採用や実装時期は保証されません。

## クレジットと共同著者表記

各IssueテンプレートでCo-author creditを希望できます。Issueがリリースされる変更に直接つながり、独自の
診断、再現例、設計上の制約、ユースケース分析など実質的な貢献を含む場合、メンテナーは報告者のGitHub
no-reply identityを`Co-authored-by` trailerとして追加します。重複報告、サポート上の質問、付随的な提案は
通常対象外です。マージ前であれば、Issue上で希望を取り消したり変更したりできます。

## 内部collaborator向け

Pull Requestを作成できるのはリポジトリcollaboratorだけです。内部Pull Requestでは対象Issueをリンクし、
必要なテストと、リリース動作が変わる場合のChangesetを追加し、必須チェックとレビューを通してください。

```sh
pnpm typecheck
pnpm lint
pnpm test
pnpm format:check
pnpm build
```
